# Java 面试资料整理工作规则

## 目标

把本地 Java 面试资料中有面试价值的内容，按主题去重、压缩后归并到 `Learning/` 的现有文档；只有在确实没有对应主题时才新建文档。

原始资料处理完并通过校验后删除。处理不完整、无法解密、文本损坏或 OCR 失败的资料不得删除。

## 工作顺序

1. 建立资料、主题与目标文档映射。
2. 只读提取一份资料的完整内容和目录。
3. 将知识点标记为 `SKIP`、`MERGE`、`REVISE` 或 `CREATE`。
4. 以面试重点为标准压缩内容，写入目标 Markdown。
5. 检查重复、格式、覆盖率和内容完整性。
6. 记录处理结果和来源。
7. 最后才删除该原始资料。

## 内容标准

- 保留：定义、核心原理、关键流程、使用场景、取舍、常见追问、易错点。
- 弱化或跳过：源码逐行分析、过于底层的实现细节、大段代码、重复定义、与面试关系弱的扩展。
- 保持现有 Learning 的问答式、简洁、分层 Markdown 风格。
- 不把不同资料的答案机械拼接；出现冲突时保留更可靠、更新、更适合面试的表述，并标注版本边界。
- 不重复已有知识点；新内容必须带来新的结论、追问角度、场景或纠错。

## Git 约束

- 处理开始前记录基线：`ea220ed9de9d36b762aa439b71462b8c46221b85`。
- 处理全部完成前不执行 `git commit`。
- 不执行 reset、checkout 或其他会覆盖用户修改的操作。
- 每次写入后只检查 diff，不提交。
- 全部资料处理和删除结束后，再统一检查 Git 状态，由用户决定是否提交。
