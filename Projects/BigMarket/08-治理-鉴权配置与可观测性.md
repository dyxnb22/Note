# 治理：鉴权、配置与可观测性

## 1. 三类信任边界

| 请求 | 鉴权方式 | 核心实现 |
|---|---|---|
| 普通用户 HTTP | JWT + jti 吊销 | `TokenAuthInterceptor` / Chatbot JWT 校验 |
| 管理/运营 HTTP | Admin JWT 白名单或 `X-Admin-Token` | `AdminAuthInterceptor` / `OperationalAuthInterceptor` |
| 内部 Dubbo RPC | 内部 RPC token filter | `big-market-starter-dubbo` |

不能把“请求经过网关”当作已鉴权。服务端必须在自己的信任边界内校验来源和身份。

## 2. JWT 登录、校验和吊销

### 2.1 签发

`AuthAccessController.login` 校验学习配置中的账号，`AuthService` 签发 JWT：

| Claim | 用途 |
|---|---|
| `sub/openId` | 用户标识 |
| `jti` | Token 唯一标识，用于吊销 |
| `iat` | 签发时间 |
| `exp` | 过期时间 |

### 2.2 业务服务校验

```text
Authorization: Bearer <jwt>
  → 去 Bearer 前缀
  → 验签
  → 检查 exp
  → 取 jti 查吊销服务
  → 取 openId 写入 request attribute
  → Controller 使用鉴权 userId
```

`_by_token` 用户接口不信任请求体中的 userId，防止用户 A 传 userId=B 操作他人账户。

### 2.3 Logout 为什么需要黑名单

JWT 本身是自包含的，服务端不保存 session；因此签发后默认一直有效到 `exp`。Logout 把 `jti` 写入吊销存储，TTL 取 Token 剩余寿命。

| 模式 | 适用 | 限制 |
|---|---|---|
| InMemory | dev/local/test 单进程 | 不跨服务/实例共享 |
| Redis | Docker 服务拓扑 | auth/market/admin 可共享吊销结果 |

Redis 吊销模式的安全决策：

- 开启 Redis 模式却没有 `RedissonClient`：启动 fail-fast，不静默退回内存。
- Logout 写 Redis 失败：返回失败，不报告假成功。
- 查吊销状态时 Redis 失败：fail-closed，将 Token 当作不可用。

## 3. 管理与运营授权

### Admin Service

- 路径：`/api/*/admin/**`。
- 校验 JWT，并要求 openId 在 `app.admin.user-ids`。
- `public/display` 是明确排除的公开只读接口。

### Market 运营接口

- ERP、activity armory、strategy armory 由 `OperationalAuthInterceptor` 保护。
- 支持静态 `X-Admin-Token` 或管理员 JWT + 白名单。
- Dubbo 敏感方法的无 Token 重载应拒绝，不能因为“是内部 RPC”就默认信任。

## 4. Nacos 动态配置

### 4.1 权威源

Nacos 是平台动态配置的权威源；admin 内存和 Redis 通知都不是第二权威源。

两类配置：

| DataId/类型 | 例子 | 消费者 |
|---|---|---|
| Platform config | Chat provider/model/apiKey/display | chatbot/admin/web public display |
| Runtime switches | `degradeSwitch` / `rateLimiterSwitch` | market |

### 4.2 保存流程

```text
管理员提交 candidate
  → 校验 contentHash（乐观并发）
  → 生成完整 DataId 内容
  → publish Nacos
  → 发布成功才是配置提交点
  → 替换 admin 内存快照
  → listener 让 market/chatbot 整体替换快照
```

为什么 listener 要整体替换不可变快照：

- 避免多个字段逐个更新时，请求读到一半新、一半旧的配置。
- 能正确处理删除和缺少键，回到安全默认值。
- 当前请求持有的快照不会在中途变化。

发布 Nacos 失败必须整体报失败；Redis fan-out 通知失败不回滚已提交的 Nacos，可以返回 `notificationPending` 并依赖 listener/启动拉取最终收敛。

## 5. 降级与限流的边界

### Gateway 技术降级

- 下游超时/失败/熔断打开时返统一 fallback。
- 它解决“下游不可用时客户端不要无限等待”。

### Market 业务降级开关

- Nacos `degradeSwitch=open` 时在进入核心抽奖前拒绝。
- 它解决“运营主动关闭高风险业务”。

技术 fallback 与业务开关的语义不同，不要混成一个“降级”。

## 6. 可观测性

### 一次排障的三个关联键

| Key | 作用 |
|---|---|
| `traceId` | 连接一次 HTTP 请求经过的服务日志 |
| `userId` | 找用户分片，定位账户/订单 |
| `orderId/awardOrderId/outBusinessNo/messageId` | 跨 DB、MQ、Job、RPC 追业务终态 |

Trace ID 不能取代 business ID：同一业务可能经历多次 HTTP/Job 重试，它们有不同 traceId，但必须共享同一稳定业务号。

### 重要业务指标

- Outbox create/fail/manual_pending 数量和最旧年龄。
- `credit_award_task` pending/failed。
- 库存 ledger reserved/manual 积压。
- Chat deducting/refund pending/manual。
- DLQ 待审核数量。
- 远程写对账失败数。

业务指标比“服务 UP”更接近真实完成度。

## 7. 工程验证边界

```text
mvn -B verify -DfailIfNoTests=false
./scripts/acceptance.sh --reuse
./scripts/smoke-security.sh
```

需要明确：

- Context 启动比静态扫描更能证明 Bean/Profile/扫描边界。
- health=UP 不能证明抽奖发奖已闭环。
- `--reuse` 只证明旧卷兼容，`--fresh` 才验证全新初始化。
- secure overlay 需要单独验证，配置文件存在不等于安全已验证。
- 当前仓库是学习/作品集基线，不宣称生产 HA、容量、灾备和灰度能力。

## 8. 本篇面试快答

**Q：JWT 无状态，如何实现登出？**

> 签发时给每个 Token 一个 jti。Logout 时将 jti 写入 Redis，TTL 是 Token 剩余寿命；各业务服务验签和验过期后再查吊销集。Redis 吊销检查故障时 fail-closed，避免黑名单绕过。

**Q：Nacos 配置更新怎么保证一致？**

> Nacos 是唯一权威源，admin 通过 contentHash 做乐观并发检查，发布完整 DataId 成功才返回成功。消费端 listener 将完整内容解析成新的不可变快照后原子替换，删除、缺少和非法值都回到安全默认值。

**Q：如何快速定位一笔发奖卡住在哪里？**

> 先用 traceId 看首次请求，再用 awardOrderId/messageId 跨进程查中奖记录、一级 task、credit_award_task 和 account 流水。根据第一个未达终态的表判断是 MQ 发布、Consumer、二级 Outbox 还是 account RPC 问题，同时查对应 backlog/age 指标。
