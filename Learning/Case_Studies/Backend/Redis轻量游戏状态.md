# Redis 轻量游戏状态

钓鱼佬网页游戏是一个轻量案例：Java 和 Python 两个版本复用同一业务模型，用 Redis 保存高频、短生命周期的游戏状态。

## 建模方式

先定义游戏规则、用户状态、道具、积分和过期行为，再分别设计 API 与 Redis 访问层。业务规则不能因为使用 Redis 就散落在字符串 key 和序列化代码中。

## Redis 适用边界

- 适合高频读写、按用户/场景定位、结构简单且允许按 TTL 管理的数据。
- 需要明确 key 命名、序列化版本、TTL、并发更新、重建和 Redis 不可用时的行为。
- 关键账务、不可丢失事实和复杂查询不应只保存在缓存中。
- 双语言实现的比较重点是业务边界、数据访问隔离、接口一致性和可复现性，不是语法差异。

## 最小检查清单

- 状态是否有版本或更新时间，能否防止乱序覆盖？
- 重复请求是否幂等，计数/道具扣减是否原子？
- 过期状态如何清理，过期后是否允许重建？
- 配置、初始化数据、启动和测试步骤是否记录？
- Redis 故障时是降级、只读、重建还是阻断写入？
