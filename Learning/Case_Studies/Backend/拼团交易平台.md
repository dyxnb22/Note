# 拼团交易平台

拼团是一条交易与营销协作链路，不只是一个下单页面：资格判断、优惠试算、锁单、支付、回调、组队进度、结算、退款和补偿必须分别建模。

## 分层与模块

```text
API → App → Domain → Infrastructure
                  ↑
             Trigger/Task/Message
```

仓库按 `api`、`app`、`domain`、`infrastructure`、`trigger`、`types` 分模块。接口负责协议，应用层编排用例，领域层承载活动/人群/交易规则，基础设施接入数据库、缓存、消息和支付，Trigger 接收定时任务、消息和回调。

## 核心事实

- 试算回答“现在应付多少”；锁单回答“优惠和库存是否暂时占住”。
- 订单、组队、支付、库存和权益分别使用状态机，不能用多个布尔字段隐式表达。
- 支付回调必须验签、校验金额和业务号，并允许重复到达。
- 过期订单释放库存和优惠权益，不能只依赖一次定时任务。
- 成团、未成团、退款、结算和发券都需要可重试、可对账、可人工修复。

## 规则与可靠性

营销规则适合用策略、责任链或规则树组合；资格、价格计算、库存预占和权益发放要拆成不同步骤。跨服务调用使用幂等键、状态条件更新、超时、重试上限和补偿任务。

## AI 排障延伸

项目中的日志标准化、ELK 查询和 MCP 工具可以形成：

```text
日志/指标 → 工具化检索 → Agent 分析 → 结构化诊断报告
```

工具只提供受控查询能力，不能把日志系统和服务器执行权限无条件暴露给模型。
