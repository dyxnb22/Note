# Agent 部署与生产化

本页只讨论 Agent/LLM 应用相对普通 Web 服务新增的生产问题。Docker、环境变量、反向代理、健康检查、CI/CD 和通用回滚流程见 [Backend Delivery](../Backend/Delivery/README.md)，不在这里重复维护。

## 生产对象不只是代码

Agent 系统的一次发布可能同时改变：

- 应用代码和依赖。
- 模型及模型快照。
- System Prompt、Few-shot 和模板。
- Tool Schema、权限和执行器。
- RAG 索引、Embedding、Reranker 和知识版本。
- Workflow 图、状态 Schema 和 Checkpoint 兼容性。
- Eval 数据集、阈值和安全策略。

每个对象都需要独立版本，运行记录必须能还原“哪次请求使用了哪组版本”。只记录 Git SHA 无法复现实际行为。

## 三档运行 Profile

| Profile | 适用场景 | 核心约束 |
| --- | --- | --- |
| Local | 本地实验 | 本地状态、低成本模型、模拟工具、禁止真实副作用 |
| Team | 团队共享服务 | 用户身份、持久化状态、并发隔离、配额和审计 |
| Production | 对外或关键业务 | 高可用、灰度、发布门禁、SLO、灾备和人工兜底 |

Team Server 不等于“更高权限”，而是同一工具合同在多用户、持久化和并发环境中的运行方式。权限仍按用户、租户和任务上下文计算。

## 配置与密钥

Agent 特有配置包括模型、温度、最大输出、上下文预算、工具白名单、循环步数、审批规则和检索参数。

配置必须：

- 绑定明确版本并记录到 Trace。
- 在启动或发布阶段完成类型与范围校验。
- 区分普通配置、敏感密钥和动态策略。
- 支持灰度和快速回滚。
- 防止工具结果、异常和日志泄露密钥。

密钥只通过密钥管理系统或受控环境注入，不能写入 Prompt、知识库、代码仓库或普通配置中心。

## Prompt 与模型版本

Prompt 应像代码一样评审和测试，但不能只按文本文件管理。建议生成不可变的 `prompt_version`，记录模板、变量 Schema、Few-shot、模型和解码参数。

模型升级要做：

1. 离线回归：质量、安全、工具选择和结构化输出。
2. 影子流量：不影响真实用户地比较结果。
3. 小流量灰度：按稳定键分组，避免用户反复切换版本。
4. 监控错误、延迟、成本和人工接管率。
5. 达标后逐步扩大，否则回滚整组版本。

不能因为平均质量上升就忽略长尾安全退化。

## 发布门禁

一次 Agent 发布至少经过：

```text
静态检查与单元测试
→ Tool 合同与权限测试
→ 确定性流程回归
→ 离线 Eval
→ Prompt Injection / 越权用例
→ 成本与延迟预算
→ Checkpoint/Schema 兼容检查
→ 灰度
→ 在线指标与人工抽检
```

安全指标、越权执行和数据泄露属于硬门禁；不能用其他指标的平均提升抵消。

## 状态与持久化

服务重启不能丢失长任务或让副作用重复发生。生产运行时要明确：

- 会话状态、Workflow 状态和长期 Memory 分别存在哪里。
- Checkpoint 的事务边界和 Schema 版本。
- Worker 如何通过 Queue、Lease 和 Heartbeat 认领任务。
- 节点如何使用幂等键。
- 已提交数据库但未发布事件时如何恢复。
- 超过重试上限后如何进入 DLQ 和人工处理。

详细原理见 [Durable Execution 与分布式可靠性](Durable%20Execution与分布式可靠性.md)。

## 限流、预算与背压

至少同时控制：

- 用户级、租户级和全局请求速率。
- 单任务步数、Token、费用和墙钟时间。
- 模型、Embedding、向量库和工具的并发。
- Queue 长度、等待时间和丢弃/降级策略。

限流不能只放在入口。一个请求可能触发多次模型和工具调用，内部 fan-out 会放大真实负载。达到预算时应进入明确的停止、降级或人工接管状态，而不是无限重试。

## 灰度与回滚

灰度键优先使用用户或租户的稳定 Hash，确保同一主体持续落到同一版本。对有状态 Workflow，还要保证恢复时使用兼容的图和状态 Schema。

回滚对象可能包括代码、Prompt、模型、工具、索引和配置。数据库或 Checkpoint Schema 已前向迁移时，简单回滚代码可能无法运行，因此发布前要验证前后兼容窗口。

## 优雅关闭

普通 HTTP 服务只需停止接收请求并等待处理中请求；Agent Worker 还应：

1. 停止认领新任务。
2. 通知正在运行的节点取消或在安全点停下。
3. 在 Lease 到期前提交 Checkpoint 或释放任务。
4. 关闭流式响应并写明终止原因。
5. 确认外部副作用的状态，避免新 Worker 重复执行。

## 降级与 Kill Switch

降级必须预先定义业务语义，例如：

- 高成本模型切换为低成本模型。
- Agent 模式退化为只读检索或固定 Workflow。
- 禁用高风险写工具，仅保留回答能力。
- 检索不可用时明确告知无法验证，而不是无依据生成。
- 全局 Kill Switch 停止新任务，同时保留查询和人工恢复入口。

Kill Switch 的操作权限、传播时延和恢复流程需要演练。

## 上线检查清单

- 所有发布对象是否有版本并写入 Trace？
- Tool 权限是否按身份和资源动态计算？
- 是否有真实副作用的幂等与对账机制？
- Eval 是否覆盖失败、越权、注入和边界输入？
- 是否设置任务、Token、费用、并发和时间预算？
- Checkpoint 能否跨进程和跨版本恢复？
- 灰度指标、停止条件和回滚负责人是否明确？
- 模型或工具全部不可用时系统怎样降级？
- 是否演练过 Kill Switch、DLQ 和人工修复？
