# 部署与生产化

这篇文档解决一个问题：**AI 应用从本地跑通到稳定上线，需要处理哪些工程问题**。

---

## 1. 部署模式选择

| 模式 | 适合场景 | 优点 | 缺点 |
|------|---------|------|------|
| **Serverless（Cloud Functions）** | 低频调用、短任务 | 按用量付费，无需管服务器 | 冷启动延迟、执行时间有上限 |
| **容器（Docker + K8s/ECS）** | 稳定流量、需要更多控制 | 灵活，可扩展，无执行时间限制 | 需要运维知识 |
| **PaaS（Railway, Render, Fly.io）** | 中小规模应用 | 部署简单，有一定弹性 | 成本比纯云稍高 |
| **GPU 实例** | 需要本地推理（非 API 调用）| 成本低于 API（高吞吐场景）| 需要管理 GPU 实例 |

**大多数 AI 应用（调用 LLM API 的）**：容器化部署到 ECS/Cloud Run，配合 Auto Scaling。

### 健康检查端点

Liveness 和 Readiness 要分开，**不要把外部 LLM API 加入 readiness probe**——provider 短暂波动会让 K8s 把你自己的容器误杀，还会产生额外的 API 请求成本。

```text
from fastapi import FastAPI

app = FastAPI()

@app.get("/health/live")
async def liveness():
"""Liveness probe：进程存活即返回 200。
K8s 靠这个判断是否重启容器——只要进程没崩就应该返回成功。"""
return {"status": "alive"}

@app.get("/health/ready")
async def readiness():
"""Readiness probe：验证本地依赖是否就绪。
K8s 靠这个决定是否向容器分配流量——失败时摘流量，不重启。
只检查本服务控制范围内的依赖（数据库、缓存），不检查外部 LLM API。"""
checks = {}

try:
    await db.execute("SELECT 1")
    checks["database"] = "ok"
except Exception as e:
    checks["database"] = f"error: {e}"

try:
    await redis_client.ping()
    checks["redis"] = "ok"
except Exception as e:
    checks["redis"] = f"error: {e}"

all_ok = all(v == "ok" for v in checks.values())
status_code = 200 if all_ok else 503
return JSONResponse(
    status_code=status_code,
    content={"status": "ready" if all_ok else "not_ready", "checks": checks},
)

@app.get("/health/diagnose")
async def diagnose():
"""诊断端点：检查包括 LLM API 在内的所有外部依赖。
不挂到 K8s probe，仅供运维人工触发排查。"""
from httpx import AsyncClient, TimeoutException
checks = {}
async with AsyncClient(timeout=5.0) as http:
    try:
        resp = await http.get(
            "https://api.openai.com/v1/models",
            headers={"Authorization": f"Bearer {settings.openai_api_key}"},
        )
        checks["llm_api"] = "ok" if resp.status_code == 200 else f"http_{resp.status_code}"
    except TimeoutException:
        checks["llm_api"] = "timeout"
    except Exception as e:
        checks["llm_api"] = f"error: {type(e).__name__}"
return checks
```

## 4. Prompt 版本管理

Prompt 是 AI 应用的"代码"，需要像代码一样版本管理。

### 文件化 Prompt

```
prompts/
  v1/
system_prompt.txt
tool_descriptions/
  search_tool.txt
  order_tool.txt
  v2/
system_prompt.txt
...
  current -> v2/  # 符号链接指向当前版本
```

```text
from pathlib import Path

class PromptLoader:
def __init__(self, prompts_dir: str = "prompts", version: str = "current"):
    self.base_path = Path(prompts_dir) / version

def load(self, name: str, **variables) -> str:
    template = (self.base_path / f"{name}.txt").read_text()
    return template.format(**variables)

loader = PromptLoader()
system_prompt = loader.load("system_prompt", product_name="MyApp")
```

### Prompt 变更流程

```
1. 新建 prompts/v3/ 目录，复制当前版本
2. 修改 prompt 文件
3. 在 staging 环境用新版本 prompt 跑 eval
4. 对比 v2 和 v3 的 eval 结果
5. 通过 → 切换 current 符号链接
6. 保留旧版本（可以快速回滚）
```

## 5. 版本管理与灰度发布

### 灰度发布

```text
import hashlib
import random

class FeatureFlag:
def __init__(self):
    self._flags = {}

def is_enabled(self, flag_name: str, user_id: str, rollout_pct: float = 0.0) -> bool:
    """
    rollout_pct: 0.0 = 关闭，1.0 = 全量开放
    对同一个 user_id 结果稳定（不会本次是A下次是B）
    """
    if rollout_pct == 0.0:
        return False
    if rollout_pct == 1.0:
        return True

## 用 user_id hash 确保同一用户始终在同一组
    hash_val = int(hashlib.md5(f"{flag_name}:{user_id}".encode()).hexdigest(), 16)
    return (hash_val % 100) < (rollout_pct * 100)

feature_flags = FeatureFlag()

async def handle_request(user_input: str, user_id: str):
if feature_flags.is_enabled("new_agent_v2", user_id, rollout_pct=0.1):  # 10% 用户
    return await new_agent_v2.run(user_input)
else:
    return await current_agent.run(user_input)
```

## 6. 生产 Checklist

### 上线前必查

```
基础安全
□ 所有密钥通过环境变量注入，不在代码里硬编码
□ API 端点有鉴权（Bearer token / API Key / OAuth）
□ 有请求频率限制（rate limiting）
□ 用户输入有基本验证和长度限制
□ 工具调用有权限边界

可靠性
□ LLM 调用有重试和超时设置
□ 工具调用有错误处理，失败不会让整个请求崩溃
□ Agent 有 max_steps 和 max_cost 上限
□ 有 fallback 策略（LLM API 挂了有降级）
□ 数据库连接有连接池

可观测性
□ 结构化日志（JSON 格式）
□ 每次 LLM 调用记录 token 用量
□ 有错误告警（出了 5xx 会通知）
□ 有 cost 监控（每日/每月花了多少）
□ 有 latency 监控（P95 延迟是多少）

运维
□ 健康检查端点
□ 优雅关闭（SIGTERM 时完成当前请求再停）
□ 容器化，有 resource limits（CPU/内存）
□ 可以快速回滚（prompt 版本、代码版本）
□ 环境变量齐全（在新环境能启动）

测试
□ 核心功能有 eval 测试集
□ 有回归测试，新 PR 会跑
□ staging 环境与 production 配置一致
□ 做过负载测试（知道系统能承受多少并发）
```

## 7. 优雅关闭

```text
import signal
import asyncio

class GracefulShutdown:
def __init__(self):
    self.shutdown_event = asyncio.Event()
    signal.signal(signal.SIGTERM, self._handle_sigterm)
    signal.signal(signal.SIGINT, self._handle_sigterm)

def _handle_sigterm(self, signum, frame):
    self.shutdown_event.set()

async def wait_for_shutdown(self):
    await self.shutdown_event.wait()

## 在 FastAPI 里
@asynccontextmanager
async def lifespan(app: FastAPI):
## 启动时
await startup_tasks()
yield
## 关闭时：等待当前请求完成
await asyncio.gather(*[t for t in asyncio.all_tasks() if t is not asyncio.current_task()])
await cleanup_tasks()
```

## 8. Agent 系统的三档部署 Profile

来自企业级 Agent 平台的部署模式划分——Profile 只改变身份、持久化和运维层，**绝不削弱策略、审批、审计或脱敏能力**：

| 关注点 | Local（本地） | Team Server（团队服务器） | On-Prem（私有部署） |
|--------|-------------|------------------------|-------------------|
| **入口** | CLI 直接调用 | FastAPI `/v2` + 瘦客户端 CLI | 同 Team Server + 组织网络管控 |
| **身份认证** | 本地 `--actor` flag | OIDC bearer token | 组织 OIDC + 秘密管理 |
| **持久化** | 本地文件系统 `.sac/` | PostgreSQL 16 + pgvector | 运营方托管 PostgreSQL |
| **工作流执行** | 进程内同步编排 | 持久化 worker 队列 + lease + heartbeat | 同 Team Server |
| **审计链** | 本地文件 hash-chain | PostgreSQL hash-chain（同语义） | 同 Team Server |
| **GitHub 集成** | 本地 fixture（离线） | GitHub App + Webhook | 内网 GitHub endpoint |
| **可观测性** | 本地 trace 文件 | Trace API + 可选 OpenTelemetry | 集中式 OTel 收集 |

**关键原则**：从 Local 升到 Team Server，不是"放宽限制"，而是"同一套安全合约在多用户环境下执行"。

```python

    ...
```

### Team Server 特有：工作流队列 + DLQ

```python
# 工作流任务入队
async def enqueue_workflow(run_id: str, task_type: str):
    await db.execute("""
        INSERT INTO workflow_queue (run_id, task_type, status, lease_expires_at)
        VALUES ($1, $2, 'pending', NULL)
    """, run_id, task_type)

# Worker 认领任务（带 lease 防止重复执行）
async def claim_next_task(worker_id: str, lease_seconds: int = 300):
    return await db.fetchrow("""
        UPDATE workflow_queue
        SET status = 'running',
            worker_id = $1,
            lease_expires_at = NOW() + interval '$2 seconds',
            attempt_count = attempt_count + 1
        WHERE id = (
            SELECT id FROM workflow_queue
            WHERE status = 'pending' OR
                  (status = 'running' AND lease_expires_at < NOW())  -- 续租超时重试
            LIMIT 1 FOR UPDATE SKIP LOCKED  -- 防止并发冲突
        )
        RETURNING *
    """, worker_id, lease_seconds)

# DLQ（Dead Letter Queue）：超过重试次数的任务转移到 DLQ
MAX_ATTEMPTS = 3

async def move_to_dlq(task_id: str, reason: str):
    await db.execute("""
        INSERT INTO workflow_dlq (original_task_id, reason, failed_at)
        SELECT id, $2, NOW() FROM workflow_queue WHERE id = $1
    """, task_id, reason)
    await db.execute("DELETE FROM workflow_queue WHERE id = $1", task_id)
```

**Lease + SKIP LOCKED 组合**：
- `lease_expires_at`：Worker 挂了，lease 超时后任务回到待认领状态
- `SKIP LOCKED`：多个 Worker 并发抢任务时，跳过已被锁定的行，避免等待

### 选择指南

```
Local  →  个人开发、离线 demo、CI 确定性测试
Team Server  →  多用户共享工作流、需要持久化状态、实时 GitHub 集成
On-Prem  →  组织网络管控、合规要求、自托管基础设施
```

**永远从 Local 开始**。只有在确实需要共享身份、持久化多用户状态或实时集成时，才升级到 Team Server。

## 9. Rate Limiting 实现

AI 应用的限速比普通 API 更复杂——要同时限制请求数和 token 消耗：

```python
import time
import threading
from collections import deque
from dataclasses import dataclass, field

@dataclass
class RateLimiter:
    """滑动窗口限速器，同时限制请求数和 token 数"""
    max_requests_per_minute: int = 60
    max_tokens_per_minute: int = 100_000
    _lock: threading.Lock = field(default_factory=threading.Lock)
    _request_times: deque = field(default_factory=deque)
    _token_times: deque = field(default_factory=deque)   # (timestamp, tokens)

    def check_and_consume(self, estimated_tokens: int) -> tuple[bool, float]:
        """
        检查是否可以发请求，返回 (允许, 需要等待的秒数)
        """
        now = time.time()
        window = 60.0

        with self._lock:
            # 清理 1 分钟前的记录
            while self._request_times and now - self._request_times[0] > window:
                self._request_times.popleft()
            while self._token_times and now - self._token_times[0][0] > window:
                self._token_times.popleft()

            recent_requests = len(self._request_times)
            recent_tokens = sum(t for _, t in self._token_times)

            # 检查是否超限
            if recent_requests >= self.max_requests_per_minute:
                wait = window - (now - self._request_times[0])
                return False, wait

            if recent_tokens + estimated_tokens > self.max_tokens_per_minute:
                wait = window - (now - self._token_times[0][0])
                return False, wait

            # 记录本次消耗
            self._request_times.append(now)
            self._token_times.append((now, estimated_tokens))
            return True, 0.0


# 在 FastAPI 中集成（按用户分别限速）
from fastapi import Request, HTTPException
from collections import defaultdict

user_limiters: dict[str, RateLimiter] = defaultdict(
    lambda: RateLimiter(max_requests_per_minute=20, max_tokens_per_minute=50_000)
)

async def rate_limit_middleware(request: Request, user_id: str, estimated_tokens: int = 1000):
    limiter = user_limiters[user_id]
    allowed, wait_seconds = limiter.check_and_consume(estimated_tokens)
    if not allowed:
        raise HTTPException(
            status_code=429,
            detail=f"Rate limit exceeded. Try again in {wait_seconds:.1f}s",
            headers={"Retry-After": str(int(wait_seconds) + 1)},
        )
```

**分层限速**（生产常见模式）：

```python
# 三层：用户级 < 组织级 < 全局级
class HierarchicalRateLimiter:
    def __init__(self):
        self.global_limiter = RateLimiter(max_requests_per_minute=1000)
        self.org_limiters: dict[str, RateLimiter] = {}
        self.user_limiters: dict[str, RateLimiter] = {}

    def check(self, user_id: str, org_id: str, tokens: int) -> tuple[bool, str]:
        # 从最细粒度开始检查
        for limiter, name in [
            (self.user_limiters.setdefault(user_id, RateLimiter(20, 30_000)), "user"),
            (self.org_limiters.setdefault(org_id, RateLimiter(200, 500_000)), "org"),
            (self.global_limiter, "global"),
        ]:
            ok, wait = limiter.check_and_consume(tokens)
            if not ok:
                return False, f"{name} rate limit: wait {wait:.1f}s"
        return True, ""
```

## 10. AI 应用的 CI/CD 流水线

AI 应用的 CI/CD 比传统应用多了"质量门控"层：

```
代码/Prompt 变更
      ↓
1. 单元测试（工具函数、工具 schema 验证）
      ↓
2. Eval 测试（在测试集上跑 Agent，检查退步）
      ↓
3. 安全扫描（prompt injection 测试，敏感词检测）
      ↓
4. Staging 部署 + Smoke Test（真实 API 调用，小批量验证）
      ↓
5. 生产发布（灰度 → 全量）
```

```yaml
# .github/workflows/ai-ci.yml
name: AI Application CI/CD

on:
  push:
    branches: [main]
  pull_request:
    paths:
      - 'src/**'
      - 'prompts/**'
      - 'eval/**'

jobs:
  unit-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -e ".[test]"
      - run: pytest tests/unit/ -x --tb=short

  eval:
    runs-on: ubuntu-latest
    needs: unit-test
    env:
      ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
    steps:
      - uses: actions/checkout@v4
      - run: pip install -e ".[eval]"

      # 用 Mock Provider 跑快速 eval（不消耗 API 配额）
      - name: Fast eval (mock)
        run: python eval/run_eval.py --mock --test-set eval/core.jsonl

      # PR 时跑完整 eval（用真实 API，小批量）
      - name: Full eval (real API, 50 cases)
        if: github.event_name == 'pull_request'
        run: |
          python eval/run_eval.py \
            --test-set eval/core.jsonl \
            --max-cases 50 \
            --output eval/results/pr-${{ github.event.number }}.json

      - name: Check regression
        if: github.event_name == 'pull_request'
        run: |
          python eval/compare.py \
            --baseline eval/results/baseline.json \
            --current eval/results/pr-${{ github.event.number }}.json \
            --gates '{"pass_rate": -0.03, "avg_faithfulness": -0.02}'

  deploy-staging:
    runs-on: ubuntu-latest
    needs: eval
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to staging
        run: |
          docker build -t my-ai-app:${{ github.sha }} .
          docker push registry/my-ai-app:${{ github.sha }}
          kubectl set image deployment/ai-app app=registry/my-ai-app:${{ github.sha }} \
            -n staging

      - name: Smoke test on staging
        run: python tests/smoke/test_staging.py --env staging --cases 10
        env:
          STAGING_URL: ${{ secrets.STAGING_URL }}

  deploy-production:
    runs-on: ubuntu-latest
    needs: deploy-staging
    if: github.ref == 'refs/heads/main'
    environment: production   # 需要手动审批
    steps:
      - name: Gradual rollout (10% → 50% → 100%)
        run: |
          # 先推 10%
          kubectl patch deployment ai-app -n production \
            -p '{"spec":{"strategy":{"canary":{"weight":10}}}}'
          sleep 300   # 等 5 分钟观察指标
          # 检查错误率（如果告警会中止）
          python scripts/check_canary_health.py --threshold 0.01
          # 全量
          kubectl set image deployment/ai-app app=registry/my-ai-app:${{ github.sha }} \
            -n production
```

**关键原则**：
- **Prompt 变更和代码变更走同一条 CI 流水线**（prompt 文件进 git）
- **Staging 必须用真实 API 做 smoke test**（Mock 发现不了 API 兼容性问题）
- **生产发布需要手动审批**（AI 行为变化影响用户，不适合完全自动化）
- **灰度发布**：先 10% 流量，观察 5 分钟无异常，再全量

## 附录：面试高频

**Q：AI 应用部署时有哪些特别需要注意的地方？**

> 相比普通服务，几个额外关注点：一是 LLM API 调用超时比普通 HTTP 请求长得多（可能 30-60 秒），连接池和超时配置要调整；二是成本监控，LLM API 按 token 计费，没有监控容易超预算；三是 prompt 版本管理，prompt 变了行为就变了，需要像代码一样管理版本和回滚；四是优雅关闭，LLM 调用可能正在执行，SIGTERM 时要等当前请求完成；五是 Agent 的 max_steps/max_cost 限制，防止 Agent 失控消耗大量资源。

**Q：如何做 prompt 的版本管理和灰度发布？**

> Prompt 文件化（放 prompts/ 目录下，按版本号组织），和代码一起进 git。变更流程：新建版本目录 → staging 环境跑 eval 对比 → 通过后切换 current 指向 → 旧版本保留用于快速回滚。灰度发布用 feature flag，根据用户 ID 的 hash 决定走新版本还是旧版本，同一用户结果稳定。上线后观察 eval 指标、错误率、用户反馈，确认没有退步再全量。
