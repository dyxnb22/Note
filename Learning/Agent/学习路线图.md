# AI Agent 工程学习路线图

本指南基于三个层次的资源：
- **笔记库**（本目录）：系统化知识参考；其中模型名、价格、协议版本等高变化内容必须以对应 Provider 的最新文档为准。
- **learn-claude-code 实践**（[本地代码](./实践/learn-claude-code/README.md)）：教学性质的 Harness 复刻项目，包含 20 个渐进式 session；它用于理解设计机制，不代表 Claude Code 官方内部实现。当前以 Notes 内的代码快照为唯一实践入口。
- **SafeCodeAgent**（`~/SafeCodeAgent-enterprise-agent-platform`）：一个具体的企业 Agent 平台案例；它展示一种安全治理取舍，不是行业标准，应与其他实现进行对照。

默认目标是学习 **Coding Agent / Agent Harness**。如果目标是 RAG、客服或流程自动化，应从下方的分支路径进入，不必完整通读所有章节。

推荐把[本地实践目录](./实践/learn-claude-code/README.md)作为本路线的代码实验配套：笔记负责解释边界和生产取舍，`s01-s20` 负责逐章运行、修改和验收。不要再维护一份重复的课程说明；遇到具体机制时回到对应章节的 `code.py` 对照实现。

---

## 学习总览

```
阶段一：最小循环（1-2 周）
  LLM/API 协议 → Agent Loop → Tool Use → 基础权限
  对应 s01-s05，掌握"模型如何提出动作，程序如何执行动作"

阶段二：工程化（2-3 周）
  代码导航 → Context 管理 → 错误恢复 → Durable Execution → Session/Subagent
  对应 s06-s14，掌握"Agent 怎么稳定运行"

阶段三：生产级（2-3 周）
  安全治理 → 验证/Eval → 可观测性 → 成本/部署
  RAG 作为知识型 Agent 分支；对应 SafeCodeAgent 核心模块

阶段四：按需扩展（1-2 周）
  MCP → Workflow/LangGraph → Memory/RAG → Extended Thinking → Computer Use
  这些是场景扩展，不是全部 Agent 的必修前置

阶段五：综合实战（持续）
  用学到的东西自己构建一个完整 Agent 项目
```

## 阶段一：最小循环（1-2 周）

> 目标：理解 Agent 的本质，能自己写出最小可运行的 Agent。

### 读什么

| 顺序 | 笔记 | 重点章节 |
|------|------|---------|
| 0 | `../Python/Python Agent工程化补充.md` | Pydantic、asyncio、类型边界、timeout/retry 和测试边界 |
| 1 | `LLM调用基础.md` | §1-§7；先选一个 Provider 跑通调用、流式输出和错误处理 |
| 2 | `Agent架构与设计.md` | §1-§6 核心概念和四种范式 |
| 3 | `Tool Calling.md` | §1-§6 基础执行循环 |

产品分支（建议并行阅读）：`AI应用产品设计.md` → `Agent产品与人机协同.md`。先定义任务边界、成功证据和人工介入点，再决定模型、工具和 Workflow。

### 动手做

**本地实践对照阅读**（按序，从[实践目录](./实践/learn-claude-code/README.md)进入，每个 session 读 `code.py`，理解后自己默写）：

```
s01_agent_loop     → 最小 while True 循环，理解 stop_reason 控制
s02_tool_use       → TOOL_HANDLERS dict，safe_path，5 个工具
s03_permission     → 三关卡权限：DENY_LIST → 规则引擎 → 人工确认
s04_hooks          → HOOKS 注册表，trigger_hooks() 拦截机制
s05_todo_write     → todo 工具 + rounds_since_todo nag 系统
```

**关键理解点**（读完能回答这些才算过关）：
- `stop_reason == "tool_use"` 和 `"end_turn"` 分别代表什么，由谁设置？
- Anthropic Messages 为什么把 `tool_result` 放在下一条 `role: "user"` 消息里？它与 OpenAI Chat Completions 的 `role: "tool"` 有什么区别？
- `tool_use_id` 必须匹配的原因是什么？
- 三关卡权限为什么不能只有 DENY_LIST？

### 验收

> 不看任何参考，从零写出一个最小 Coding Agent：支持 bash/read_file/write_file，区分工具 schema 与 handler，处理无效参数和工具异常，并对写入/删除操作要求确认。用测试验证正常调用、权限拒绝、工具失败和 max_steps，而不是以代码行数作为唯一标准。

## 阶段二：Harness 工程化（2-3 周）

> 目标：让 Agent 能在复杂任务中稳定运行，不崩溃、不失控、不超预算。

### 读什么

| 顺序 | 笔记 | 重点章节 |
|------|------|---------|
| 1 | `代码 Agent 基础设施.md` | Repository Navigation；Patch/Edit；验证闭环；Git/Worktree |
| 2 | `Context工程.md` | §4 窗口管理；§8 四层压缩流水线；§14 Few-shot；§15 自我修正 |
| 3 | `Durable Execution与分布式可靠性.md` | 状态机；Checkpoint/Journal；Queue/Lease；Retry/Resume |
| 4 | `Agent架构与设计.md` | §9-§18 Harness/Hook/Subagent/Session Journal/写后验证 |
| 5 | `成本与性能工程.md` | §1-§3 成本模型；§6.5 AgentLoop 预算；§8 组件级预算 |

### 动手做

**本地实践对照阅读**：

```
s06_subagent        → 子 Agent 隔离：fresh messages，SUB_TOOLS 排除 task，30 轮上限
s07_skill_loading   → SKILL_REGISTRY 懒加载，两级加载（摘要常驻/完整按需）
s08_context_compact → 四层压缩：tool_result_budget → snip_compact → micro_compact → compact_history
s09_memory          → write_memory_file，_rebuild_index，MEMORY_TYPES（按需学习）
s10_system_prompt   → build_system() 每轮重建，分段组装，Prompt Cache 标记
s11_error_recovery  → RecoveryState，三条恢复路径，指数退避
s12_task_system     → Task dataclass，can_start() 依赖检查，claim_task/complete_task
s13_background_tasks→ threading.Thread daemon，background_lock，通知注入到 user 消息
s14_cron_scheduler  → CronJob，cron_matches()，Scheduler daemon，四层消费
```

**关键理解点**：
- 四层压缩为什么是这个顺序（L3 → L1 → L2 → L4）？
- Subagent 为什么不能有 `task` 工具？防的是什么？
- `build_system()` 在 `while True` 外面调用的时序含义是什么？
- RecoveryState 三条路径分别对应哪三种 API 错误？

### 验收

> 在阶段一的 Agent 基础上，加入：
> 1. 对话历史超过 8K token 时自动摘要压缩
> 2. Session 状态持久化，能够从中断处恢复
> 3. max_steps、token 和 cost 三种预算保护（具体价格从配置读取）
> 4. 工具错误、超时和取消都能结构化返回或安全结束
> 5. 记录每一步的 trace，并能离线 replay 一次失败案例

### 动手做

先完成三个小实验，再阅读企业平台源码：

1. 用 `Agent Eval实验方法.md` 建立 10 个 task 的 baseline/candidate 对比；
2. 用 `Agent安全与威胁建模.md` 为代码 Agent 画信任边界，并写 5 个安全回归 case；
3. 用 `Durable Execution与分布式可靠性.md` 实现 task 状态、checkpoint 和一次 resume。

**SafeCodeAgent 对照阅读**（按模块阅读源码 + 文档）：

```
enterprise-docs/architecture.md
  → 理解"模型输出永远不是执行权限"这个设计哲学
  → 五层治理模型（Policy → RBAC → Grant → Sandbox → Human）

src/safecode/enterprise/rag/retriever.py
  → 权限感知检索：actor_can_access_chunk() 过滤在检索前
  → HybridRetriever：BM25 + semantic + RRF

src/safecode/agent/loop.py + loop_budget.py
  → 四 Mixin 架构：Planner/Budget/Journal/Dispatcher
  → LoopBudget：max_steps/max_tokens/max_cost_usd

src/safecode/agent/validation.py + repair.py
  → VALIDATION_ORDER：test → lint → typecheck → build
  → RepairStrategy 分类器：按错误类型给不同修复 prompt

src/safecode/enterprise/workflow/nodes/registry.py
  → WORKFLOW_NODE_ORDER + NODE_RUNNERS 注册表模式
  → NodePatch 增量更新

enterprise-docs/observability-and-evaluation.md
  → TraceEventType 枚举（22 种事件）
  → 企业级 eval 指标：policy_recall/citation_grounding/approval_bypass_rate

enterprise-docs/deployment-profiles.md
  → Local/Team Server/On-Prem 三档对比
  → "Profile 改变运维层，不削弱安全合约"原则
```

**关键理解点**：
- 单次 Grant 如何防止双重消费（PostgreSQL SERIALIZABLE 事务）？
- Hash-chain 审计为什么每条记录要包含前一条的 hash？
- 权限感知检索为什么必须在检索前过滤而不是检索后？
- Ratchet 基线"只进不退"的工程价值是什么？

### 验收

> 为阶段一、二的 Agent 加入：
> 1. 确定性 Mock Provider + 至少 5 个正常、失败、越权和边界 case，CI 自动跑
> 2. 安全回归：高风险操作（删除文件、发网络请求）需要人工确认
> 3. 结构化 Trace（记录每次 LLM 调用的 token/latency/cost 和工具结果摘要）
> 4. 一个可离线运行的 replay/eval 命令；RAGAS 仅在知识型 Agent 分支中使用
> 5. Docker 容器化和健康检查作为部署练习，不作为 Agent 原理的前置条件

## 阶段五：综合实战项目

> 选一个实际场景，从头构建，综合运用所有知识。

### 推荐项目方向

**选项 A：个人代码助手 Agent**（难度：中，学习价值：高）

```
功能：
  - 读取本地代码库（glob + read_file）
  - 分析代码问题（LLM 推理）
  - 提出修改建议（write_file/edit_file）
  - Human-in-the-loop：危险改动需确认
  - Memory：记住用户的代码风格偏好

用到的知识：
  Tool Calling + 代码 Agent 基础设施 + Context工程 + 安全 + Eval + Durable Execution
  + 产品协同 + 运维回滚

参照：[s20_comprehensive/code.py](./实践/learn-claude-code/s20_comprehensive/code.py)（综合 session）
```

**选项 B：RAG 知识库问答系统**（难度：中，工程广度：宽）

```
功能：
  - 文档 ingestion pipeline（PDF/Markdown → chunk → embed → index）
  - 混合检索（语义 + BM25 + RRF）
  - Agentic RAG（Agent 自主决定是否检索）
  - 引用来源（Citation 对象）
  - RAGAS eval + CI 集成

用到的知识：
  RAG + Agent架构 + Eval + 部署 + 可观测性
```

**选项 C：多 Agent 任务协作系统**（难度：高，适合有一定基础后做）

```
功能：
  - Orchestrator 拆分任务，分发给 Worker Agent
  - Worker 使用不同工具集（search_agent / code_agent / write_agent）
  - LangGraph Supervisor 模式做编排
  - Checkpoint 支持断点续传
  - 完整的 cost/step/quality 监控

用到的知识：
  LangGraph + Workflow + Multi-Agent + Memory + 可观测性 + 成本
```

## 常见坑和注意事项

**读本地实践时**：
- 按 s01 → s20 顺序，不要跳。每个 session 都在前一个基础上加东西。
- 重点不是"能不能运行"，是"为什么这样设计"。
- s08（四层压缩）和 s11（错误恢复）是最值得反复读的两个。

**读 SafeCodeAgent 时**：
- 先读 `enterprise-docs/` 下的文档，再去看源码——文档讲设计意图，源码讲实现细节。
- 重点模块：`agent/loop.py`、`enterprise/rag/retriever.py`、`enterprise/workflow/nodes/`、`audit/`。
- 这个项目的复杂度远超 learn-claude-code，不要试图一次读懂所有代码——按专题读。

**做全库校准时**：
- 模型 ID、价格、MCP/LangGraph/Computer Use API 字段都属于版本化内容，统一标注 Provider、版本和最后核对日期。
- `Agent Eval实验方法.md` 负责 Agent 轨迹和发布门禁；`Eval与测试体系.md` 负责更广的测试技术，不要重复维护同一套指标。
- `Agent安全与威胁建模.md` 负责攻击面和信任边界；`安全与可控性.md` 负责实现模式和代码示例。

**读笔记时**：
- 笔记是工具书，不是小说。遇到具体问题再来查，不要从头到尾背。
- 每个文件末尾的"面试高频"是精华，实际上也是对全文的最好总结。
- 代码示例先确认它属于哪个 Provider；要能默写的是协议不变的核心循环，而不是某个 SDK 的字段名。
- 标记为“核心”的内容优先做出可测试实现；标记为“参考”的内容只在项目需要时阅读。

## 知识地图（全景）

```
                 LLM / Provider 协议
                         ↓
             Tool Calling → Agent Loop → State
                    ↓              ↓
          代码导航 / Patch    Context / Session
                    ↓              ↓
       权限 / 沙箱 / 验证 ←── Eval / Trace

       知识型分支：RAG → Citation → Memory → 权限感知检索
       编排分支：Workflow/LangGraph → Subagent → Multi-Agent
       能力分支：MCP → Browser/Computer Use → 外部系统
       可靠性分支：State Machine → Checkpoint → Queue/Lease → Resume
                         ↓
                  Cost / Deployment
```

**两个项目在地图中的位置**：

```
本地实践目录          → 教学 Harness：覆盖 Agent Loop、工具、权限、Context、Task、MCP
                       强项：底层机制和设计决策的"为什么"；不是 Claude Code 官方源码

SafeCodeAgent        → 一个企业平台案例：覆盖 RAG、治理、审计、Eval、部署
                       强项：企业安全治理、多用户、审计合规；不是唯一正确架构
```
