# AI Agent 工程学习路线图

> 这是一张执行路线，不是知识正文。主题边界与完整目录见 [README](README.md)，跨领域顺序见 [AI / Python / Agent 学习地图](../00_Navigation/AI-Python-Agent学习地图.md)。

## 使用方法

1. 先完成“核心主线”，不要一开始同时学习 RAG、MCP、多 Agent 和 GUI 自动化。
2. 每阶段以可运行产物和测试作为完成证据，不以“读完文章”作为完成标准。
3. 选一个实践仓库做主线；另一个只用于对照设计，不重复抄写相同示例。
4. 遇到具体问题再进入专题文档，完成后回到主线。

## 阶段 0：建立可靠的 API 程序

### 学习重点

- Python 项目结构、类型、测试、日志和异常处理。
- HTTP 的认证、超时、重试、限流与幂等。
- LLM 请求中的消息、采样、结构化输出、流式响应和用量统计。
- 配置与密钥通过环境或密钥系统注入，不写入代码和笔记。

### 完成标准

- [ ] CLI 能读取配置并调用模型。
- [ ] 超时、限流和无效响应有明确错误路径。
- [ ] 日志不泄露密钥或完整敏感输入。
- [ ] 至少有成功、超时和格式错误三个测试。

## 阶段 1：打通最小 Agent 循环

最小循环只有四步：模型决策、工具调用、结果回填、判断终止。先做窄任务，再增加框架。

### 学习重点

- 工具描述、参数 schema、返回值和错误语义。
- 最大步数、超时、预算与重复调用检测。
- 工具注册表与执行器分离；模型不能直接获得任意系统权限。
- 对有副作用的工具加入确认、作用域和幂等键。

### 建议练习

从 [ai-agent-learning](实践/ai-agent-learning/README.md) 的项目 03–05 任选一条连续完成；再阅读 [learn-claude-code](实践/learn-claude-code/README.md) 的最小循环与工具章节作对照。

### 完成标准

- [ ] 至少三个工具使用统一协议注册。
- [ ] 错误结果能回到循环并被模型处理。
- [ ] 循环达到步数、时间或成本上限时确定终止。
- [ ] 重放同一输入可解释每一步为何发生。

## 阶段 2：构建稳定的 Agent Harness

Harness 是包围模型的确定性系统：它管理上下文、权限、状态、执行环境和生命周期。模型能力提升不能替代这层工程。

### 学习重点

- 上下文选择、压缩、缓存与污染控制。
- 文件、Shell、浏览器等工具的隔离和变更验证。
- checkpoint、幂等、重试、取消与人工接管。
- trace、事件日志和 session journal，使失败可以定位和恢复。

### 建议练习

完成 [ai-agent-learning](实践/ai-agent-learning/README.md) 的服务化与工作流项目，或按 [learn-claude-code](实践/learn-claude-code/README.md) 实现权限、Hook、任务状态和会话恢复。不要同时维护两套同功能代码。

### 完成标准

- [ ] 进程中断后能从最近 checkpoint 恢复。
- [ ] 写操作有变更预览、权限边界和写后验证。
- [ ] 上下文超限时按策略压缩，而不是静默截断。
- [ ] 用户可以取消、暂停或接管任务。

## 阶段 3：生产安全、评测与上线

生产门槛不是“多数演示成功”，而是风险可控、质量可测、故障可恢复。

### 学习顺序

1. 用 [Agent 安全与威胁建模](Agent安全与威胁建模.md) 识别资产、信任边界和攻击面。
2. 用 [安全与可控性](安全与可控性.md) 落实最小权限、审批、沙箱和审计。
3. 用 [Agent Eval 实验方法](Agent%20Eval实验方法.md) 设计可证伪实验，再用 [Eval 与测试体系](Eval与测试体系.md) 建立持续回归。
4. 用 [可观测性与调试](可观测性与调试.md) 连接 trace、指标、成本和用户反馈。
5. 依照 [部署与生产化](部署与生产化.md) 设置灰度、SLO、回滚和事故流程。

### 完成标准

- [ ] 高风险动作默认拒绝或要求明确审批。
- [ ] 检索权限在检索前过滤，间接提示注入有隔离与测试。
- [ ] 离线回归集包含成功、拒绝、边界与历史事故样本。
- [ ] 线上质量、延迟、成本和安全指标均有阈值。
- [ ] 完成一次取消、降级和回滚演练。

## 阶段 4：按问题选择分支

| 需求信号 | 专题 | 不应使用的情形 |
|---|---|---|
| 答案依赖私有或持续更新的资料 | [RAG](RAG.md) → [知识系统](知识系统.md) | 静态知识已能放进短提示；检索质量无法评测 |
| 需要跨会话保留用户或任务状态 | [Memory 与状态管理](Memory与状态管理.md) | 单次任务；没有删除、更正和授权机制 |
| 流程步骤稳定、需要分支或恢复 | [Workflow 与编排](Workflow与编排.md) → [LangGraph](LangGraph.md) | 简单循环足够；流程仍在快速变化 |
| 需要跨客户端复用工具与资源 | [MCP 与工具协议](MCP与工具协议.md) | 单进程内部函数已满足需要 |
| 子任务可清晰切分并行或隔离上下文 | [多 Agent 协作](多Agent协作的边界与模式.md) | 任务高度耦合；协调成本高于收益 |
| 必须操作缺少稳定 API 的界面 | [Computer Use 与 GUI Agent](Computer%20Use与GUI%20Agent.md) | 已有可靠 API 或结构化工具 |
| 需要面向用户设计信任和接管体验 | [Agent 产品与人机协同](Agent产品与人机协同.md) | 仅做离线内部脚本 |

新增分支前先记录基线；只有任务成功率、风险、延迟或成本中至少一项得到可重复改善，才保留新增复杂度。

## 实践仓库的分工

### ai-agent-learning：产品工程主线

适合从 API 客户端逐步走向 Tool Calling、服务化、工作流、记忆、RAG、MCP 和评测。以其 [README](实践/ai-agent-learning/README.md) 为唯一项目顺序来源，本路线不复制项目清单。

### learn-claude-code：Harness 机制解剖

适合通过小步实现理解循环、权限、Hook、子 Agent、任务状态、压缩、隔离与会话机制。尽管目录名源于一个具体产品，学习目标是可迁移的架构机制；模型和 SDK 应由运行环境配置，不把示例中的版本号当作推荐。

### 选择原则

- 想做一个可部署服务：以前者为主。
- 想理解代码 Agent 内部机制：以后者为主。
- 两者都学：只选一个实现主项目，另一个用于代码阅读和设计对照。

## 最终验收清单

- [ ] 能解释 Agent 与普通工作流的边界。
- [ ] 能从一次 trace 定位模型、工具、上下文或基础设施故障。
- [ ] 所有副作用动作都有权限、审计、幂等或补偿设计。
- [ ] 质量结论来自固定数据集和可复现实验，而非少量演示。
- [ ] 成本、延迟和成功率可以按任务、模型与工具拆分。
- [ ] 模型、SDK 或协议升级经过回归测试和灰度，而非直接替换。
- [ ] 用户知道系统能做什么、何时会确认、如何停止及如何纠错。

完成这些条件后，再把学习重点从“增加能力”转向“缩小失败面、提升可解释性和运营效率”。
