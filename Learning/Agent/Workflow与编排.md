# Workflow 与编排

Workflow 把一个目标拆成可观察、可控制、可恢复的步骤。它关心的是任务之间的关系、状态如何流动以及失败后怎样继续，不依赖 LangGraph 或某个具体框架。

具体的 LangGraph API、Checkpoint、`Send`、`Command`、`interrupt()` 和 Time Travel 见 [LangGraph](./LangGraph.md)。开放式 Agent 的自主决策循环见 [Agent 架构与设计](./03_Agent架构与设计.md)，跨进程可靠执行见 [Durable Execution 与分布式可靠性](./06_Durable%20Execution与分布式可靠性.md)。

## Workflow、Agent 与普通代码

| 方式 | 控制者 | 适用场景 | 主要风险 |
| --- | --- | --- | --- |
| 普通顺序代码 | 程序 | 步骤固定、分支少、执行时间短 | 复杂后难观察和恢复 |
| Workflow | 图或状态机 | 流程稳定、需要分支、审批、重试或恢复 | 状态和迁移设计不清 |
| Agent | 模型 | 路径开放、需要动态判断和工具选择 | 不确定性、权限和成本 |
| Workflow + Agent | 流程约束模型 | 主流程稳定、局部需要智能决策 | 两套状态相互污染 |

选择原则：能用普通代码清晰表达时，不急于引入图；步骤、分支、等待和恢复成为一等需求时使用 Workflow；只有路径无法预先穷举时，才把局部决策交给 Agent。

## 建模的四个核心对象

### State

State 是流程事实的最小集合。只保存后续节点真正需要的数据，不把日志、缓存、临时对象和不可序列化连接全部塞进去。

一个状态字段应明确：

- 谁创建、谁能修改。
- 是覆盖、追加还是按版本合并。
- 是否允许为空，以及默认值是什么。
- 是否需要持久化和审计。
- 恢复旧版本时能否兼容。

### Node

Node 完成一个边界清晰的动作，例如校验输入、查询资料、调用模型、等待审批或写入结果。节点应尽量满足：

- 输入和输出合同明确。
- 重复执行不会产生不可控副作用。
- 超时、取消和异常能够被上层识别。
- 外部写操作带幂等键或版本条件。
- 业务逻辑可脱离编排器单独测试。

### Edge

Edge 表示合法迁移。条件边应根据结构化状态路由，不依赖自然语言字符串的模糊匹配。每个循环都必须有退出条件、最大步数或预算。

### Checkpoint

Checkpoint 是可恢复状态快照，不等于完整审计日志。它至少要绑定流程实例、状态版本、当前节点和提交时间；恢复时还要确认代码、Schema、Prompt 和工具版本是否兼容。

## 从业务事实开始画图

不要先画节点，再猜状态。推荐顺序：

1. 定义完成条件和不可接受的结果。
2. 列出必须持久化的业务事实。
3. 标记外部副作用和人工决策点。
4. 为每个副作用定义幂等键和确认方式。
5. 再把事实变化映射为节点和边。

简单示例：

```text
接收申请
  → 校验
  → 风险判断
      ├─ 低风险 → 自动执行
      ├─ 中风险 → 人工审批 → 执行/拒绝
      └─ 高风险 → 拒绝
  → 记录结果
  → 通知
```

这里“申请已批准”“执行已提交”“通知已发送”是业务事实；`approval_node` 只是某个实现名称。

## 状态机与迁移约束

状态迁移应显式表达允许的前置状态：

```text
PENDING → APPROVED → EXECUTING → SUCCEEDED
   │           │           └────→ FAILED
   └→ REJECTED └→ CANCELLED
```

执行迁移时同时校验当前状态和版本号，可以避免并发覆盖：

```sql
UPDATE workflow_instance
SET status = 'APPROVED', version = version + 1
WHERE id = :id
  AND status = 'PENDING'
  AND version = :expected_version;
```

受影响行数为 0 表示前置条件已经变化，不能把它当作成功。

## 分支、循环与并行

### 条件分支

分支条件应可测试、可记录。把模型输出先解析成枚举或结构化对象，再用于路由；不要直接把长文本传给条件函数。

### 循环

搜索、反思和修复流程容易形成无限循环。至少设置：

- 最大步数。
- 时间预算。
- Token 或费用预算。
- 相同动作和相同错误的重复检测。
- 无进展时的人工接管或降级路径。

### 并行与汇聚

并行任务需要定义汇聚语义：等待全部、满足法定数量、首个成功，还是超时后使用已有结果。多个分支写同一字段时，要明确 reducer、冲突解决和结果顺序；不能依赖完成先后恰好稳定。

## Human-in-the-loop

人工审批不是简单的布尔开关。审批对象至少包含：

- 将要执行的动作和目标资源。
- 参数、差异和影响范围。
- 风险等级与触发原因。
- 申请人、执行身份和权限来源。
- 批准、修改、拒绝、超时的后续路径。
- 决策人与决策时间的审计记录。

恢复审批流程时，不应盲目重放已经执行的副作用。先查询外部系统或幂等记录，确认动作是否真实发生。

## 失败处理

| 失败类型 | 典型处理 |
| --- | --- |
| 参数或合同错误 | 直接失败，返回可修复信息 |
| 临时网络错误 | 有上限的指数退避重试 |
| 限流 | 尊重服务端等待时间，进入延迟队列 |
| 权限拒绝 | 不重试，升级审批或终止 |
| 外部结果未知 | 查询状态或对账，不能直接重复写 |
| 业务规则不满足 | 记录明确原因，走拒绝或补偿路径 |

重试只适合短暂故障。补偿是新的业务动作，不是数据库回滚；它也可能失败，因此需要独立状态、重试和人工兜底。

## Durable Execution

长流程要能跨进程、跨部署恢复，至少需要：

- 持久化状态和单调递增版本。
- Queue、Lease、Heartbeat 和超时回收。
- 节点级幂等键。
- Outbox 或等价机制连接状态提交与消息发布。
- 失败重试上限和死信队列。
- 可查询的执行历史和人工修复入口。

框架的内存 Checkpointer 只能用于本地实验，不能证明流程具有生产级持久性。

## Workflow 与 Agent 的组合边界

推荐把确定性控制放在 Workflow，把开放判断限制在节点内部：

```text
Workflow：校验 → 检索 → Agent 决策 → 审批 → 执行 → 验证
                         ↑
                只能选择授权工具和结构化动作
```

Workflow 负责预算、权限、重试、审批和终止；Agent 负责在授权范围内选择动作。模型输出永远不是执行权限。

## 可观测性

每个节点至少记录：

- `workflow_id`、`run_id`、`node_id`、`attempt`。
- 输入输出摘要和 Schema 版本。
- 开始、结束、排队和等待时间。
- 外部调用、Token、费用和错误类型。
- 路由决定及其结构化原因。
- 审批和副作用的审计标识。

指标应能回答：哪里慢、哪里失败、重试是否有效、人工等待多久、恢复后是否重复执行。

## 测试方法

1. 单测每个节点的输入输出合同。
2. 表驱动测试所有条件边和非法状态迁移。
3. 注入超时、重复消息、乱序和进程重启。
4. 验证同一幂等键重复执行不会重复产生副作用。
5. 对审批的同意、修改、拒绝和超时分别测试。
6. 用历史 Checkpoint 验证 Schema 迁移和恢复。
7. 对循环验证步数、时间和预算上限。

## 选型清单

采用编排框架前先确认：

- 流程是否真的需要持久化、中断和恢复。
- 状态能否序列化，Schema 如何演进。
- 是否支持所需的并行、等待和人工审批语义。
- Checkpoint 的一致性和存储后端是什么。
- Worker 崩溃后 Lease 如何回收。
- 是否能够导出 Trace、指标和审计数据。
- 框架升级时历史状态是否兼容。
- 是否可以脱离框架测试业务节点。

## 学习与实践顺序

1. 用普通函数实现一个三节点顺序流程。
2. 加入条件分支和结构化状态。
3. 注入临时失败，增加有上限的重试。
4. 加入一个需要人工审批的副作用。
5. 持久化 Checkpoint，并模拟进程重启。
6. 增加并行分支、汇聚和幂等验证。
7. 最后使用 [LangGraph](./LangGraph.md) 实现同一套语义，对比框架带来的能力和约束。
