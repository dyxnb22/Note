# Agent 产品与人机协同

这篇笔记解决一个问题：**如何把 Agent 做成用户愿意使用、能够控制、失败后可以恢复的产品**。

模型能力只是产品的一部分。用户体验还取决于目标定义、工具权限、进度反馈、审批方式、失败处理和责任边界。

## 0. 先判断是否需要 AI

适合 AI 的任务通常输入开放、很难穷举规则、结果允许概率性，并且能够通过验证或反馈持续改善。若规则明确、错误代价极高或必须严格确定，优先使用普通程序或固定工作流；AI 可以只负责提取、排序、草稿或解释，不直接获得最终执行权。

产品立项时先写清：目标用户、触发场景、输入输出、允许改变的资源、失败后果、人工介入点和成功证据。指标同时覆盖任务质量、效率和风险。“像人”“更智能”不是可验收指标。

## 1. 先定义任务，不要先定义模型

一个可交付的 Agent 任务应写清：

```text
用户是谁
触发场景是什么
输入是什么
允许 Agent 改变什么
成功的可验证证据是什么
失败的代价是什么
什么时候必须询问用户
什么时候必须停止
```

“帮我处理一下”不是任务规格。可以先转成结构化目标：

```json
{
  "goal": "修复这个仓库中的缺失 import",
  "scope": ["src/", "tests/"],
  "allowed_side_effects": ["修改源码", "运行测试"],
  "forbidden_side_effects": ["删除文件", "联网发布"],
  "success_evidence": ["相关测试通过", "git diff 可审查"]
}
```

## 2. 自主性梯度

不要只在“全自动”和“完全人工”之间选择：

| 模式 | Agent 可以做什么 | 适合 |
|---|---|---|
| 建议 | 只读、分析、生成草稿 | 高风险和早期产品 |
| 确认后执行 | 准备动作，用户批准后执行 | 修改代码、发消息 |
| 范围内自动 | 在明确 workspace 和预算内执行 | 测试、重构、批量读取 |
| 后台自治 | 定时或异步完成任务 | 低风险、可回滚流程 |

自主性必须和可撤销性、影响范围、用户预期匹配。高自主性不是目标本身，可靠完成任务才是。

## 3. 什么时候澄清

Agent 不应对每个细节都询问，也不应对关键偏好自行猜测。可以用三个判断：

```text
不澄清会不会改变目标？
不澄清会不会产生不可逆副作用？
是否存在两个以上同样合理但结果不同的方案？
```

如果答案为是，应提出最小问题，并带上已经知道的上下文和可选方案：

```text
我发现有两个可行范围：
A. 只修改 src/payment.py
B. 同时更新相关测试和文档

这会影响 3 个文件。你希望选择哪一种？
```

不要只问“请提供更多信息”。好的澄清问题应减少用户的决策成本。

## 4. 进度与可见性

长任务需要让用户知道：

- 当前目标和范围；
- 已完成的步骤；
- 正在等待什么；
- 下一步是什么；
- 消耗了多少时间和预算；
- 如何暂停、取消或接管。

进度不是模型的内部思维内容，而是可审计的状态事件：

```text
task_started
context_loaded
tool_requested
approval_required
tool_completed
validation_failed
waiting_user
task_succeeded
```

## 5. 审批体验

审批界面应让用户回答“我是否同意这个具体动作”，而不是盲目点击确认。至少展示：

- 动作和目标资源；
- 参数和预计副作用；
- 触发原因和相关证据；
- 影响文件、数据或外部系统；
- 允许一次、允许本任务还是拒绝；
- 操作失败后的恢复方式。

审批必须绑定到操作摘要。模型或工具参数变化后，原审批不能继续沿用。

## 6. 失败和接管

用户需要知道任务失败在哪里，而不是只看到“Agent 出错了”。失败消息应包含：

```text
发生了什么
已经完成了什么
没有完成什么
是否产生了副作用
可以选择的下一步
```

接管路径可以是：

- 用户修改参数后继续；
- 用户提供额外文件或权限；
- 人工接手当前步骤；
- 回滚当前任务；
- 结束任务并导出证据。

## 7. 反馈不是一句“满意吗”

把反馈拆成可操作类别：

| 反馈 | 可能根因 |
|---|---|
| 目标理解错 | 输入澄清、任务建模、Router |
| 找错资料 | Context、检索、权限过滤 |
| 调错工具 | schema、工具集合、模型能力 |
| 结果不正确 | 实现、验证器、模型推理 |
| 太慢或太贵 | Context、模型路由、并发、缓存 |
| 不敢使用 | 透明度、审批、可撤销性、隐私说明 |

用户反馈应关联 task、agent version、model、tool trace 和最终结果，才能进入 Eval 和发布流程。

迭代时先修复高频且高损失的失败类别。离线 Eval 负责防止回归，线上灰度验证真实价值；产品约束应反向决定模型、Context、RAG、工具和权限，而不是在技术方案完成后再补交互。

## 8. 产品指标

不要只看“回答质量”。至少观察：

- 任务完成率；
- 首次成功率和人工接管率；
- 澄清问题的接受率；
- 审批拒绝率和撤销率；
- 失败后的恢复率；
- P95 完成时间和每任务成本；
- 高风险动作的越权率；
- 用户重复使用率和投诉率。

指标要按任务类型分片。平均值可能掩盖某类高风险任务的严重退化。

## 9. 上线前检查

```text
任务边界是否明确？
成功是否可验证？
副作用是否可预览和撤销？
用户能否暂停、取消和接管？
失败后是否知道状态和下一步？
是否记录完整审计和 trace？
是否有安全与质量回归集？
是否能快速关闭高风险能力？
```

## 10. 练习与验收

为一个代码修改 Agent 设计三种产品模式：建议、确认后执行、范围内自动。分别写出：

1. 允许的工具和副作用；
2. 用户看到的进度事件；
3. 审批页面展示的字段；
4. 失败和取消后的状态；
5. 任务成功、体验和安全指标。
