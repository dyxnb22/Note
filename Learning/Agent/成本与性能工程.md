# 成本与性能工程

这篇文档解决一个问题：**如何让 AI 应用跑得起、跑得快、跑得省**。

AI 应用的成本和延迟有独特的结构，不能用传统软件工程的直觉套。

## 1. Token 成本模型

### 成本构成

```
总成本 = 输入 token 数 × 输入单价 + 输出 token 数 × 输出单价
```

价格表只是历史示例（价格持续变化）。生产系统应从 Provider 定价页或版本化配置读取，并记录实际 usage；不要把这里的数字当作当前报价。

| 模型 | 输入（/1M token） | 输出（/1M token） | 适合 |
|------|-----------------|-----------------|------|
| gpt-4o-mini | $0.15 | $0.60 | 简单任务，高吞吐 |
| gpt-4o | $2.50 | $10.00 | 复杂任务 |
| claude-3-haiku | $0.25 | $1.25 | 简单快速 |
| claude-3-5-sonnet | $3.00 | $15.00 | 复杂推理 |
| claude-3-7-sonnet (thinking) | $3.00 | $15.00 | 需要长思考链 |

### 成本追踪

```python
from dataclasses import dataclass, field
import threading

@dataclass
class CostTracker:
    total_input_tokens: int = 0
    total_output_tokens: int = 0
    call_count: int = 0
    _lock: threading.Lock = field(default_factory=threading.Lock)

# 价格配置（每百万 token 的美元价格；示例，生产环境从配置读取）
INPUT_PRICE_PER_M = {"gpt-4o": 2.50, "gpt-4o-mini": 0.15, "claude-3-5-sonnet": 3.00}
OUTPUT_PRICE_PER_M = {"gpt-4o": 10.00, "gpt-4o-mini": 0.60, "claude-3-5-sonnet": 15.00}

def record(self, model: str, input_tokens: int, output_tokens: int):
    with self._lock:
        self.total_input_tokens += input_tokens
        self.total_output_tokens += output_tokens
        self.call_count += 1

def total_cost_usd(self, model: str) -> float:
    input_cost = self.total_input_tokens * INPUT_PRICE_PER_M.get(model, 2.50) / 1_000_000
    output_cost = self.total_output_tokens * OUTPUT_PRICE_PER_M.get(model, 10.00) / 1_000_000
    return input_cost + output_cost

tracker = CostTracker()

def tracked_llm_call(messages: list, model: str = "gpt-4o") -> str:
    response = client.chat.completions.create(model=model, messages=messages)
    tracker.record(model, response.usage.prompt_tokens, response.usage.completion_tokens)
    return response.choices[0].message.content
```

## 2. 延迟分析

AI 应用的延迟来源：

```
总延迟 =
  网络延迟（到 API）
  + TTFT（Time to First Token）  ← 主要由 prompt 长度决定
  + 生成延迟（生成 token 数 × 每 token 时间）
  + 工具执行延迟（如果有）
  + 重试延迟（如果触发限速）
```

### TTFT vs TPOT

| 指标 | 含义 | 影响因素 |
|------|------|---------|
| TTFT（Time to First Token） | 从发请求到第一个 token 出来的时间 | prompt 长度、服务器负载 |
| TPOT（Time Per Output Token） | 每生成一个 token 的时间 | 模型大小、并发情况 |
| E2E Latency | 完整响应的总时间 | TTFT + output_tokens × TPOT |

**工程含义**：
- 短 prompt + streaming = 用户感知延迟低（第一个字出来很快）
- 长 prompt 即使 TPOT 快，TTFT 也慢（用户等很久才看到第一个字）
- 所以要压缩 context 长度，不只是为了省钱，也是为了降低延迟

### Semantic Cache（应用级别）

对于相似（不一定完全相同）的 query，缓存 LLM 的回答：

```text
from sentence_transformers import SentenceTransformer
import numpy as np

class SemanticCache:
def __init__(self, similarity_threshold: float = 0.92):
    self.encoder = SentenceTransformer("all-MiniLM-L6-v2")
    self.cache: list[tuple[list[float], str]] = []  # (embedding, response)
    self.threshold = similarity_threshold

def get(self, query: str) -> Optional[str]:
    if not self.cache:
        return None

    query_emb = self.encoder.encode(query)
    similarities = [
        np.dot(query_emb, cached_emb) / (np.linalg.norm(query_emb) * np.linalg.norm(cached_emb))
        for cached_emb, _ in self.cache
    ]

    best_idx = np.argmax(similarities)
    if similarities[best_idx] >= self.threshold:
        return self.cache[best_idx][1]
    return None

def set(self, query: str, response: str):
    query_emb = self.encoder.encode(query)
    self.cache.append((query_emb, response))

semantic_cache = SemanticCache()

def cached_llm_call(query: str) -> str:
cached = semantic_cache.get(query)
if cached:
    return cached

response = call_llm(query)
semantic_cache.set(query, response)
return response
```

**适合场景**：FAQ、文档问答、重复查询多的场景。
**注意**：相似度阈值太低会返回错误答案；动态/实时性要求高的场景不适合。

### Result Cache（确定性输出）

对于工具调用结果，如果工具是幂等的，可以缓存结果：

```text
import hashlib
import json
from functools import wraps

def cache_tool_result(ttl_seconds: int = 300):
"""缓存工具调用结果，适用于幂等查询"""
cache = {}

def decorator(fn):
    @wraps(fn)
    def wrapper(*args, **kwargs):
        cache_key = hashlib.md5(
            json.dumps({"args": args, "kwargs": kwargs}, sort_keys=True).encode()
        ).hexdigest()

        if cache_key in cache:
            cached_result, cached_time = cache[cache_key]
            if time.time() - cached_time < ttl_seconds:
                return cached_result

        result = fn(*args, **kwargs)
        cache[cache_key] = (result, time.time())
        return result
    return wrapper
return decorator
```

```text
from enum import Enum

async def classify_task_complexity(user_input: str) -> TaskComplexity:

complexity_str = response.strip().lower()
```

### 限速控制

```text
import asyncio

class RateLimiter:
def __init__(self, max_per_minute: int):
    self.max_per_minute = max_per_minute
    self.semaphore = asyncio.Semaphore(max_per_minute // 10)  # 控制并发
    self.calls = []

async def acquire(self):
    await self.semaphore.acquire()
    now = time.time()
## 清理 1 分钟之外的记录
    self.calls = [t for t in self.calls if now - t < 60]
    if len(self.calls) >= self.max_per_minute:
        wait_time = 60 - (now - self.calls[0])
        await asyncio.sleep(wait_time)
    self.calls.append(now)
    self.semaphore.release()
```

## 6. 成本优化 Checklist

```
□ system prompt 是否精简？（每次调用都会计费）
□ context 是否有不必要的内容？（历史消息、无关文档）
□ 是否用了 Prompt Caching？（适合长 system prompt）
□ 是否对 FAQ/重复查询做了 semantic cache？
□ 是否根据任务复杂度路由到合适模型？
□ 工具的幂等查询结果是否缓存？
□ 是否在 development 阶段用便宜模型，只在 production 用贵模型？
□ 是否监控了每个功能的平均 token 消耗？
□ 是否设置了 max_tokens 避免超长输出？
```

## 6.5 AgentLoop 预算控制（stuck-loop + token budget）

Agent 可能在某个工具调用上卡死，或者悄悄消耗超预算。需要从循环层面控制：

```python
from dataclasses import dataclass

@dataclass
class LoopBudget:
    max_steps: int = 50
    max_tokens: int = 500_000    # 整个 session 的 input+output token 上限
    max_cost_usd: float = 2.0
    current_steps: int = 0
    current_tokens: int = 0
    current_cost: float = 0.0

    def record_call(self, input_tokens: int, output_tokens: int,
                    input_price_per_1m: float, output_price_per_1m: float):
        self.current_steps += 1
        self.current_tokens += input_tokens + output_tokens
        self.current_cost += (
            input_tokens / 1_000_000 * input_price_per_1m +
            output_tokens / 1_000_000 * output_price_per_1m
        )

    def should_stop(self) -> tuple[bool, str]:
        if self.current_steps >= self.max_steps:
            return True, f"步数上限: {self.max_steps}"
        if self.current_tokens >= self.max_tokens:
            return True, f"token 上限: {self.max_tokens}"
        if self.current_cost >= self.max_cost_usd:
            return True, f"成本上限: ${self.max_cost_usd}"
        return False, ""

# 在 agent_loop 中
budget = LoopBudget()
while True:
    response = client.messages.create(...)
    budget.record_call(
        response.usage.input_tokens, response.usage.output_tokens,
        input_price_per_1m=3.0, output_price_per_1m=15.0
    )
    should_stop, reason = budget.should_stop()
    if should_stop:
        print(f"[budget] 中止: {reason}")
        return

    # stuck-loop 检测
    if stuck_detector.check(block.name, block.input):
        budget_failure_record(reason="stuck_tool_intent")
        return
```

**生产建议**：budget 参数通过配置文件管理，不要硬编码。不同任务类型（quick_fix vs full_refactor）可以设置不同预算。

## 7. Batch API — 离线批量处理（成本节省 50%）

Anthropic Message Batches API 适合大批量、非实时的任务，成本比实时调用低 50%，且不占用实时 API 的速率限制。

### 适合场景

```
✓ 大规模 Eval（对测试集跑批量评估）
✓ 数据处理/标注（批量分类、抽取、翻译）
✓ 离线报告生成
✓ 定期数据摘要（每天凌晨跑，结果早上看）
✗ 不适合：需要实时响应的对话、工具调用链（批量 API 不支持 tool use）
```

### 提交批次

```python
import anthropic

client = anthropic.Anthropic()

for i, doc in enumerate(documents):
```

```python
# 批次通常在几分钟到几小时内完成（取决于规模）
```

### 生产用法：异步轮询 + 结果持久化

```python
import asyncio
import json
from pathlib import Path

async def run_batch_job(documents: list[str], output_path: str):
    # 1. 提交
    batch = client.messages.batches.create(
        requests=[
            {
                "custom_id": f"item_{i}",

    # 2. 等待（也可以单独跑一个脚本来轮询）
    while True:
        batch = client.messages.batches.retrieve(batch_id)

    results = {}
    for result in client.messages.batches.results(batch_id):
        if result.result.type == "succeeded":
            results[result.custom_id] = result.result.message.content[0].text

    Path(output_path).write_text(json.dumps(results, ensure_ascii=False, indent=2))
    print(f"结果已保存到 {output_path}")
```

### 成本对比

不要把某一时点的价目表固化在概念文档。用同一批样本记录实际输入、缓存输入和输出 token，再分别代入实时与 Batch 的官方当日单价：

```text
实时成本 = Σ(各类 token 数 × 对应实时单价)
批量成本 = Σ(各类 token 数 × 对应批量单价) + 额外存储/等待成本
节省比例 = (实时成本 - 批量总成本) / 实时成本
```

Eval、离线抽取等不要求即时响应的任务通常适合评估 Batch；开发交互和在线请求仍需实时 API。是否采用以当前 Provider 能力、SLA 和实测总成本为准，核对入口见 [版本与来源](版本与来源.md)。

## 8. 组件级 Token 预算分配

复杂 Agent pipeline 里，每个组件都消耗 token。不做预算分配容易"入不敷出"：

```python
from dataclasses import dataclass

@dataclass
class TokenBudget:
    """为一次完整请求分配 token 预算"""
    total: int = 100_000          # 总预算（context window 的安全线）

    # 固定成本（每次调用都有）
    system_prompt: int = 3_000
    tool_schemas: int = 2_000     # 工具定义

    # 可变成本（动态分配）
    memory: int = 2_000           # 用户记忆注入
    retrieval: int = 8_000        # RAG 检索结果
    history: int = 10_000         # 对话历史
    reserved_output: int = 4_000  # 为模型输出预留

    @property
    def used_fixed(self) -> int:
        return self.system_prompt + self.tool_schemas

    @property
    def available_for_variable(self) -> int:
        return self.total - self.used_fixed - self.reserved_output

    def allocate(self) -> dict[str, int]:
        """计算各组件实际可用 token"""
        avail = self.available_for_variable
        # 按优先级分配：memory < history < retrieval
        actual_memory = min(self.memory, avail // 5)
        actual_history = min(self.history, avail // 3)
        actual_retrieval = min(self.retrieval, avail - actual_memory - actual_history)
        return {
            "memory": actual_memory,
            "history": actual_history,
            "retrieval": actual_retrieval,
            "total_variable": actual_memory + actual_history + actual_retrieval,
        }


# 在 pipeline 里使用
def build_context_within_budget(
    system: str,
    memories: list[str],
    history: list[dict],
    docs: list[dict],
    user_input: str,
) -> list[dict]:
    budget = TokenBudget()
    alloc = budget.allocate()

    # 按预算截取各组件
    memory_text = truncate_to_tokens("\n".join(memories), alloc["memory"])
    trimmed_history = trim_history_to_tokens(history, alloc["history"])
    trimmed_docs = pack_docs_to_tokens(docs, alloc["retrieval"])

    messages = [
        {"role": "system", "content": system + f"\n\n## 用户记忆\n{memory_text}"},

    messages.extend(trimmed_history)
    if trimmed_docs:
        messages.append({"role": "user", "content": f"[参考资料]\n{trimmed_docs}"})
        messages.append({"role": "assistant", "content": "已收到参考资料。"})
    messages.append({"role": "user", "content": user_input})
    return messages
```

## 附录：面试高频

**Q：如何降低 AI 应用的 token 成本？**

> 几个层次：一是压缩 context，system prompt 精简，去掉无关历史消息；二是 Prompt Caching，把长 system prompt 标记为可缓存，命中后节省 90% input 成本；三是 Semantic Cache，对相似 query 缓存 LLM 回答；四是 Model Routing，简单任务用便宜小模型，复杂任务才用大模型；五是工具结果缓存，幂等查询的结果缓存一段时间。最重要的是先做好监控，知道成本花在哪里，再有针对性地优化。

**Q：AI 应用的延迟怎么优化？**

> 先区分是 TTFT 高还是 TPOT 高。TTFT 高通常是 prompt 太长（要压缩 context）或服务器负载高；TPOT 高通常是模型太大或并发太高。应用层优化：一是 streaming 输出，让用户更快看到第一个字；二是压缩 context 长度；三是并行工具调用；四是对工具结果做缓存；五是用更快的小模型（如果质量够用）。另外，如果是长任务，转成异步后台任务，立即返回 task_id，用户轮询或 webhook 获取结果。
