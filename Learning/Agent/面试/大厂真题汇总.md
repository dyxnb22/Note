> **说明：** 标注 `[据报]` 的是社区收集的面经题，标注 `[典型]` 的是根据该公司业务方向整理的高频题型。

## 阿里巴巴（通义 / 达摩院 / 蚂蚁）

**业务方向：** 通义千问、通义知识库（企业 RAG）、蚂蚁金融 Agent、电商智能客服

### RAG 专项（必考）

**[据报] Q1: RAG 检索准确率低怎么优化？**
> 思路：混合检索（向量+BM25）→ Re-ranking → 查询改写 → 改善 Chunk 策略 → 元数据过滤

**[据报] Q2: RAG 里的幻觉怎么解决？**
> 思路：Prompt 强约束"只用文档里的内容"→ 引用来源 → 置信度过滤 → 验证节点

**[据报] Q3: 向量数据库怎么选？Milvus、Pinecone、Chroma 有什么区别？**
> 思路：规模（Chroma 适合单机小规模、Milvus 适合大规模生产）→ 云托管 vs 自建 → 查询性能 → 生态集成

**[典型] Q4: 企业知识库 RAG 和普通 RAG 有什么区别？**
> 思路：权限控制（用户只能看自己有权限的文档）→ 多租户隔离 → 文档更新实时性 → 非结构化文档解析（PDF/PPT/表格）

**[典型] Q5: 如何处理 RAG 里的表格数据？**
> 思路：表格转 Markdown/JSON → 数值型数据用 text-to-SQL 而不是向量检索 → 混合方案

### Agent 架构

**[据报] Q6: 如何设计一个可以处理千级并发请求的 Agent 服务？**
> 思路：无状态设计（State 存 PostgreSQL）→ 工具调用异步执行 → LLM 调用池化 → 超时控制 → 降级策略

**[据报] Q7: Agent 怎么做 A/B 测试？**
> 思路：不同 Prompt 版本分流 → 用 thread_id 携带实验组信息 → 记录关键指标（任务完成率、用户满意度、Token 消耗）→ 统计显著性

## 腾讯（混元 / 微信 / 游戏 AI）

**业务方向：** 混元大模型、微信 AI 助手、游戏 NPC Agent、企业微信 Bot

### Agent 工程

**[据报] Q1: 怎么保证 Agent 的回答安全合规？**
> 思路：输入过滤（违规词、Prompt Injection）→ 输出过滤（内容安全模型）→ Prompt 约束（拒绝危险请求）→ 人工审核高风险场景

**[据报] Q2: 微信场景的 Agent 和普通 API Agent 有什么不同？**
> 思路：多模态输入（文字/语音/图片）→ 消息截断（微信有字数限制）→ 异步响应（长任务不能让用户等）→ 会话状态管理（微信 session 机制）

**[据报] Q3: LLM 推理服务怎么优化延迟？**
> 思路：KV Cache → Continuous Batching → 量化（INT8/INT4）→ Speculative Decoding → 更快的推理框架（vLLM/TensorRT-LLM）

**[典型] Q4: 游戏 NPC Agent 怎么设计？**
> 思路：角色扮演 System Prompt → 记忆（玩家历史交互）→ 工具（游戏状态 API）→ 情绪/性格一致性 → 响应延迟优化

### 多 Agent

**[据报] Q5: 多个 Agent 怎么协同完成一个复杂任务？**
> 思路：Supervisor 模式 vs Pipeline → 任务分解 → 子 Agent 专业化 → 结果聚合 → 错误传播处理

## 百度（文心 / 百度搜索 / 文库）

**业务方向：** 文心一言、百度搜索增强 Agent、文库知识 RAG、Apollo 自动驾驶 Agent

### 大模型基础（百度偏重基础）

**[据报] Q1: RLHF 是什么？和 SFT 有什么区别？**
> 思路：SFT（监督微调，直接学习人工标注的正确回答）→ RLHF（先 SFT，再用强化学习让模型对齐人类偏好）→ RLHF 更贵但效果更好（尤其对有害内容的控制）

**[据报] Q2: 大模型推理时的 KV Cache 是什么原理？**
> 思路：Transformer 生成 token 时，历史 token 的 K、V 矩阵不变 → 缓存起来 → 每次只计算新 token 的 K、V → 推理速度大幅提升

**[据报] Q3: 量化是什么？INT8 量化会损失多少精度？**
> 思路：把模型参数从 FP32/BF16 压缩到 INT8/INT4 → 减少显存占用、提升推理速度 → INT8 通常精度损失 <1%，INT4 损失更大

### RAG & 搜索结合

**[典型] Q4: 搜索引擎和 RAG 怎么结合？**
> 思路：搜索做粗排（BM25/倒排索引取候选）→ Embedding 做精排 → 结合实时搜索 + 本地知识库 → 引用来源

**[据报] Q5: 文档解析失败怎么处理？（PDF 里的扫描件、表格等）**
> 思路：OCR 处理扫描件 → 表格专用解析器 → 人工标注兜底 → 解析失败的文档单独标记，不进向量库

## 字节跳动（豆包 / 抖音 / 剪映）

**业务方向：** 豆包大模型、抖音内容 Agent、剪映 AI 视频、推荐系统 + LLM

### 工程实践（字节偏重工程）

**[据报] Q1: LLM 服务的 QPS 怎么扩展？**
> 思路：无状态推理实例水平扩展 → 请求队列削峰 → 模型并行（Tensor Parallel/Pipeline Parallel）→ 多模型副本负载均衡

**[据报] Q2: 怎么评估 Agent 的效果？**
> 思路：任务完成率 → 工具调用准确率 → 用户满意度 → LLM-as-Judge → 人工抽样评估 → 和业务指标的关联（如客服场景的解决率）

**[据报] Q3: Streaming 输出怎么实现？前端怎么接收？**
> 思路：后端 SSE（Server-Sent Events）或 WebSocket → 前端 EventSource 接收 → 逐 token 渲染 → 错误重连机制

**[典型] Q4: 推荐系统和 Agent 怎么结合？**
> 思路：Agent 理解用户意图 → 调用推荐系统工具（传结构化 Query）→ 推荐结果由 LLM 做自然语言呈现 → 用户反馈再优化

**[据报] Q5: 如何防止用户用 Agent 生成有害内容？**
> 思路：输入分类器（识别有害意图）→ 输出检测（有害内容过滤）→ 水印（生成内容可溯源）→ 用户行为限速

## 拼多多 / TEMU

**业务方向：** 电商客服 Agent、供应链 AI、广告智能投放

### 系统设计

**[据报] Q1: 高并发下 Agent 的架构怎么设计？**
> 思路：LLM 调用异步化（不阻塞请求线程）→ 工具调用并发执行 → 结果缓存（同类问题复用）→ 优先级队列（VIP 用户优先）→ 限流熔断

**[据报] Q2: Agent 的响应延迟 SLA 怎么保证？**
> 思路：分级超时（工具 5s、整体 30s）→ 超时降级（返回兜底回答）→ 关键路径 Warm-up → P99 延迟监控 + 告警

**[典型] Q3: 电商客服 Agent 怎么设计？**
> 思路：意图识别（退款/物流/投诉/咨询）→ 不同意图走不同工具链 → 订单查询工具 → 退款流程工具 → 无法处理时人工转接 → 满意度收集

**[据报] Q4: Agent 调用第三方 API 失败了怎么处理？**
> 思路：重试（指数退避）→ 备用 API → 缓存上次成功结果 → 降级回答 → 记录失败 + 告警

## 美团

**业务方向：** 外卖配送 Agent、本地服务推荐、供应链优化

**[典型] Q1: 外卖配送场景的 Agent 怎么设计？**
> 思路：任务：根据用户位置和实时订单推荐最优配送路线 → 工具：地图 API、订单系统、骑手状态 → 实时性要求高 → 决策需要快（不能用慢 LLM）→ Plan-Execute 模式

**[典型] Q2: LLM 生成的结果如何接入实时系统（不是纯对话场景）？**
> 思路：结构化输出（JSON）→ 严格的格式校验 → 回退到规则系统 → 不能完全信任 LLM 的数值计算

**[据报] Q3: 怎么控制 Agent 系统的成本？**
> 思路：Prompt Caching → 路由用小模型 → 相似问题用缓存结果 → 统计每类请求的成本，优化成本高的路径

## 蚂蚁集团（金融 AI）

**业务方向：** 金融合规 Agent、风控 AI、支付宝智能客服

**金融场景特殊要求**

**[据报] Q1: 金融场景的 Agent 和普通 Agent 有什么额外要求？**
> 思路：
> - **合规**：回答需要符合金融监管要求（不能给具体投资建议）
> - **可解释**：决策需要可追溯（监管要求）
> - **安全**：用户数据严格隔离，不能让 LLM 看到其他用户信息
> - **准确性高**：金额、利率等数值绝对不能幻觉，必须用工具查询真实数据

**[据报] Q2: 怎么防止 Agent 给出违规的金融建议？**
> 思路：System Prompt 明确禁止特定回答 → 输出分类器检测合规风险 → 高风险回答拦截 + 转人工 → 定期审计 Agent 历史回答

**[典型] Q3: 风控场景用 LLM 有什么挑战？**
> 思路：可解释性（规则引擎 vs LLM 黑盒）→ 延迟（风控需要实时决策）→ 对抗攻击（用户构造 Prompt 绕过风控）→ 现实做法：规则引擎+LLM 混合，LLM 做辅助分析而非最终决策

## 通用高频追问

不管什么公司，以下追问出现频率很高：

1. **"你在项目里用过 LangGraph 的哪些特性？遇到过什么问题？"**
   → 准备一个完整的 DevPilot 案例：StateGraph + ToolNode + MemorySaver + interrupt + stream

2. **"如果让你从头设计一个 Agent 系统，你会怎么做？"**
   → 先问清楚业务场景 → 定义 State → 设计工具 → 确定流程模式（Pipeline/ReAct/Supervisor）→ 考虑容错/安全/成本

3. **"你怎么评估 Agent 的好坏？"**
   → 任务完成率 + 工具调用准确率 + 延迟 + Token 成本 + 用户满意度

4. **"LangGraph 和其他 Agent 框架（AutoGen/CrewAI/Dify）有什么区别？"**
   → LangGraph 更底层可控（代码级别的状态管理）→ AutoGen 更适合多 Agent 对话 → CrewAI 更高级别抽象（快速上手但不灵活）→ Dify 是低代码平台（非工程向）

5. **"你知道 MCP 吗？和 Function Call 有什么区别？"**
   → MCP 是开放协议，工具可跨 AI 系统复用；Function Call 是绑定在 Agent 代码里的
