# RAG

RAG（Retrieval-Augmented Generation）在生成前检索外部证据，让模型基于可追溯资料回答。它主要解决知识更新、私有知识接入和证据引用，不保证模型一定正确。

知识的权限、版本、删除和生命周期见 [知识系统](知识系统.md)。

## 完整链路

```text
数据源
→ 解析与清洗
→ Chunk
→ Metadata
→ Embedding / 词法索引
→ Query 理解
→ Candidate Retrieval
→ Filter / Rerank
→ Context Assembly
→ Generation
→ Citation
→ Eval 与反馈
```

RAG 的错误可以来自每一层。回答错不应直接归因于“模型幻觉”，要区分资料不存在、解析丢失、切分错误、召回失败、排序失败、上下文装配失败和生成未遵循证据。

## Chunk

Chunk 是检索和引用的最小单元。过小会丢失上下文，过大会降低匹配精度并浪费 Token。

切分优先尊重语义结构：标题、段落、列表、函数、类、表格和页面边界。Overlap 只用于保留跨边界信息，不能代替合理的结构切分。

每个 Chunk 至少携带：

- 稳定 `chunk_id` 和 `document_id`。
- 标题路径和来源位置。
- 内容版本与更新时间。
- 权限、租户和数据分类。
- 原文引用定位。

## Embedding

Embedding 把文本映射到向量空间，用相似度召回语义相关内容。选型要考虑语言、领域、维度、上下文长度、延迟、成本和隐私。

建库与查询必须使用兼容的模型和预处理。更换 Embedding 通常意味着新建索引并双写/回填，不能把不同向量空间直接混合。

相似度分数只在同一模型、索引和查询条件下有比较意义，不应把一个固定阈值当作跨场景真理。

## 词法、向量与混合检索

- BM25 擅长专有名词、错误码、精确短语和关键词。
- 向量检索擅长语义改写和概念相似。
- 混合检索分别召回候选，再用 RRF 或学习排序融合。

RRF 基于名次而非原始分数融合，适合不同检索器分数不可比的情况。融合前必须用稳定 ID 去重并保留每一路证据。

## Query 理解

查询可以经过：

- 拼写和实体标准化。
- 对话指代消解。
- 多查询改写。
- 时间、产品、租户等结构化过滤提取。
- HyDE 生成假设文档辅助语义召回。

改写不能悄悄改变用户意图。高风险或精确查询应保留原 Query，并把改写作为额外召回路径。

## Metadata Filter

Metadata 用于权限、时间、文档类型和业务范围过滤。权限过滤必须在候选返回模型之前执行，不能先把无权内容放入上下文，再要求模型“不要引用”。

过滤字段需要正确建索引。过滤过严会导致零召回，过松会扩大噪声和泄露风险，因此要记录过滤前后候选数。

## Reranker

Reranker 对少量候选做更精细的 Query-Document 相关性判断。常见流程是宽召回 `top_k × n`，再重排到最终上下文数量。

Rerank 增加延迟和成本，不能弥补资料未入库或 Chunk 完全错误。应分别评估 Recall 和最终排序指标。

## Context Assembly

上下文装配要处理：

- 去重和相邻 Chunk 合并。
- 来源多样性，避免同一文档占满窗口。
- Token 预算。
- 时间和权威性排序。
- 冲突证据并列展示。
- 明确的来源边界，防止文档指令成为系统指令。

长上下文不是越多越好。无关内容会稀释关键信号，内容中部也可能更容易被忽略。

## Generation 与 Grounding

生成合同应要求模型：

- 只使用给定证据回答可验证事实。
- 区分证据事实、合理推断和未知。
- 证据不足时明确拒答或请求更多信息。
- 每个关键结论关联实际支持它的引用。
- 不把文档中的命令当作系统指令执行。

引用必须由系统根据 Chunk ID 生成和校验，不能让模型凭空编造 URL 或页码。

## Agentic RAG

Agentic RAG 把检索暴露为工具，让模型决定何时查询、如何改写和是否继续检索。它适合多步研究，但增加循环、成本和不可预测性。

必须限制检索次数、查询范围、预算和终止条件，并记录完整工具轨迹。简单问答优先使用确定性检索链路。

## RAG Eval

至少分开评估：

| 层 | 指标或问题 |
| --- | --- |
| 摄取 | 文档是否完整解析、更新及时 |
| 召回 | Recall@K、关键证据是否进入候选 |
| 排序 | MRR、NDCG、关键证据位置 |
| 生成 | Faithfulness、答案相关性、拒答正确性 |
| 引用 | 引用是否存在、是否真正支持结论 |
| 系统 | 延迟、成本、权限泄露、零结果率 |

测试集要包含无答案、冲突答案、过期资料、权限隔离、精确标识符和多跳问题。只测试“资料里有且表述接近”的简单问题，会高估系统能力。

## 排障顺序

1. 正确文档是否存在且版本正确？
2. 解析后的正文是否完整？
3. 正确 Chunk 是否被建立？
4. 权限和 Metadata Filter 是否误删？
5. 候选召回是否包含答案？
6. Reranker 是否把正确候选排低？
7. Context Assembly 是否截断或去重错误？
8. 模型是否忽略、误读或超出证据生成？

## 验收清单

- 能用一组失败样例分别定位摄取、召回、排序和生成错误。
- 能说明 BM25、向量和混合检索的边界。
- 能设计权限先于模型上下文的检索链路。
- 能验证每条引用确实支持对应结论。
- 能为无答案和冲突证据定义明确行为。
