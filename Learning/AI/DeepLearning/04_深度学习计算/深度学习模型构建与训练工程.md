# 深度学习模型构建与训练工程

本章把模型定义、参数管理、持久化、设备和训练循环放在同一个工程边界中理解。目标是从“会调用现成模型”进入“能维护可复现训练程序”。

## 模型层、块与参数管理
`nn.Module` 是 PyTorch 模型的基本单元：在 `__init__` 注册层，在 `forward` 描述数据流。复杂模型应由可复用 Block 组合，而不是把所有逻辑堆在一个 forward 中。

参数通过 `model.parameters()` 交给优化器；命名模块有助于定位参数。共享参数意味着多个位置引用同一个 layer，而非创建两个数值相同的 layer。修改参数前要理解 autograd：推理或手动初始化使用 `torch.no_grad()`，避免把赋值写进计算图。

最小练习：实现一个包含两个自定义 Block 的网络，打印每个参数名称与 shape，并验证共享层两次调用后权重只有一份。

## 自定义层、模型读写与 GPU
自定义层适用于框架没有的变换；只要继承 `nn.Module` 并实现 `forward`。模型保存优先保存 `state_dict()`，加载时先创建同结构模型再 `load_state_dict()`；同时保存配置、标签映射和预处理方式，权重本身不足以复现服务。

GPU 的规则简单：模型、输入、标签在同一个 device；将数据移动放在训练循环内。显存不足时先减 batch size、图像尺寸或序列长度，再考虑梯度累积与混合精度；不要用不断调用 `empty_cache()` 代替定位泄漏。

最小练习：训练一个小模型，保存/重新加载后验证同一输入输出一致；故意让输入留在 CPU，观察并解释 device 错误。

## 训练循环的工程化封装
将训练拆成 `train_epoch`、`evaluate`、`fit`、`save_checkpoint` 和 `metrics`。封装的目的是让实验可靠地替换模型、数据和指标，而不是过早抽象成难读框架。

训练函数接收 model、loader、loss、optimizer、device；验证函数必须 `eval/no_grad`；best checkpoint 按验证指标保存；日志至少包括 epoch、train/val loss、主指标、学习率、耗时。配置使用 dataclass 或 YAML，禁止在多个文件复制超参数。

最小练习：把此前的 Fashion-MNIST 代码整理为可通过命令行改变 batch size、lr、epoch 的脚本。

## 学习与验证

- 运行对应最小代码，记录数据、随机种子、模型、优化器和指标。
- 每次只改变一个变量，对比训练曲线、验证指标、吞吐和显存。
- 把失败归因到数据、目标函数、优化、容量或系统资源，而不是只换模型。
