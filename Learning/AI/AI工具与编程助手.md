# AI 工具与编程助手

这篇文档解决一个问题：**如何真正有效地和 AI Coding Agent 协作**——不幻想它是全能的，也不因为踩坑就放弃。

这篇文档分两部分：
- **方法论（§1–§3，§5–§7）**：描述和工具无关的原则，长期有效
- **操作手册（§4）**：Claude Code 的具体命令和配置，随产品版本会变化

---

## 1. AI 编程工具的几种形态

| 形态 | 代表工具 | 能力范围 | 主要限制 |
|------|---------|---------|---------|
| IDE Assistant | Copilot, Cursor | 自动补全、局部重构、解释代码 | 上下文窗口小，没有文件系统感知 |
| Terminal / CLI Agent | Claude Code, Aider | 读写文件、运行命令、多步骤任务 | 依赖清晰指令，长任务容易漂移 |
| Web Chat | ChatGPT, Claude.ai | 对话式问答、代码讲解 | 没有项目上下文，不能执行代码 |
| Repo Agent | Devin, GitHub Copilot Workspace | 跨文件理解、PR 生成 | 昂贵，需要授权，长任务可靠性差 |
| Browser / Computer Agent | Claude Computer Use | 操作 GUI、浏览器自动化 | 不稳定，慢，适合简单自动化 |

## 2. 能力边界

### 做得好的事

- **单函数/单文件的代码生成**：给清晰需求，生成质量高
- **已知模式的代码转换**：重命名、格式化、翻译语言、添加类型注解
- **解释代码**：读懂陌生代码库、解释复杂逻辑
- **写测试**：给函数生成 unit test
- **debug 已知错误**：把 traceback 粘进去，通常能给出方向
- **文档和注释**：生成 docstring、写 README
- **搜索和浏览代码库**：在大型仓库里找相关文件

### 做不好的事（容易翻车）

- **跨越多个系统的长期任务**：中间状态复杂，AI 容易迷失目标
- **需要系统性架构判断的决策**：它会给一个"看起来合理"的答案，但不一定是正确权衡
- **涉及不熟悉框架的细节**：会幻觉出不存在的 API
- **完全没有测试的黑盒验证**：它生成了，但"是否正确"你才能判断
- **安全相关逻辑**：永远自己 review，不要信任 AI 生成的加密/鉴权代码
- **理解业务上下文**：它不知道你的业务约束，只能看你给的 context

## 3. 高效协作原则

### 给对 Context

AI 的质量上限 = 你给它的 context 质量。

坏 prompt（给 context 不足）：
```
帮我修 bug
```

好 prompt（给足 context）：
```
我在用 FastAPI + SQLAlchemy。
这个函数 get_user_by_email 在并发请求时偶尔返回 None，
但 DB 里确实有这条数据。报错是 XXX。
下面是相关代码。
请帮我分析可能原因。
```

规律：越具体的问题，越好的回答。越模糊的问题，越多废话。

### 小任务切片

```
坏的：帮我做一个完整的 RAG 系统
好的：帮我写一个 chunking 函数，输入是 markdown 文本，
      按标题切块，每块最大 800 token，
      返回 List[Dict[str, str]]，包含 content 和 heading
```

任务越小越精确，质量越高，越容易验证。

### 让 AI 列选项，你来决策

遇到架构决策时，好用法：

```
给我列出这个问题的两种解法，以及各自的权衡。
不要直接告诉我用哪个。
```

你来做最终判断，AI 来列方案和分析权衡。

## 4. Claude Code 使用手册

### 核心命令

| 命令 | 作用 |
|------|------|
| `/add <文件>` | 把文件加入当前对话 context |
| `/remove <文件>` | 从 context 移除文件（节省 token） |
| `/ls` | 列出当前 context 里的文件 |
| `/clear` | 清除对话历史，重置状态 |
| `/compact` | 压缩对话历史（对话太长开始变笨时用） |
| `/help` | 查看所有命令 |
| `Ctrl+C` | 中断当前操作 |

### CLAUDE.md 配置文件

Claude Code 通过 `CLAUDE.md` 给 AI 持久化项目上下文（放在项目根目录，每次对话自动加载）：

```markdown
# CLAUDE.md 示例

## 项目说明
FastAPI + LangGraph 的客服 Agent。

## 关键约束
- 所有 LLM 调用走 services/llm_client.py，不要直接调用 SDK
- 工具函数放在 tools/ 目录
- 不要修改 tests/fixtures/，这些是固定测试数据

## 常用命令
- `make dev`：启动开发服务器
- `make test`：跑测试
```

写什么：项目架构、不能改的约束、常用命令、已决定的技术选型。
不写什么：所有文档内容（太长会被截断）、业务逻辑细节、泛泛的"请写好代码"。

## 5. 常见坑与反模式

| 坑 | 问题 | 正确做法 |
|----|------|---------|
| 过度信任生成的代码 | 可能包含幻觉 API、安全漏洞 | 所有生成代码都 review，关键路径测试覆盖 |
| 让 AI 生成测试验证 AI 生成的代码 | 错误可能一致，形成闭环 | 自己确认核心断言，AI 只写样板 |
| 脏代码堆积 | 每次修 bug 加 if，根因没修 | 定期让 AI 做代码审查而不只是修 bug |
| 对话太长后遗忘 | 早期 context 关注度下降 | 阶段完成后 /clear，新任务带结论重开 |
| 把架构决策外包给 AI | AI 不知道你的团队约束和成本上限 | 架构你来做，AI 列选项分析 |

---

## 6. 安全与审计原则

- 不要把真实密钥放进对话
- 不要让 AI 直接执行你不理解的 shell 命令（生产环境尤其如此）
- 文件修改要 review diff：大型重构让 AI 做，但最终 diff 你要看
- 不要让 AI 有直接推送代码的权限：它可以生成 PR，但 merge 是人来做

---

## 7. 实际效率提升的真实边界

**真正有效的场景**：
- 不熟悉的语言/框架：学习和探索速度 3-5x
- 样板代码（CRUD / schema / test fixtures）：生成速度 10x+
- 代码理解和重构：节省大量"读代码"时间
- Debug：给 traceback + 上下文，通常能快速定位方向

**容易高估的场景**：
- 完整功能的端到端生成（需要大量迭代验证）
- 需要深度业务理解的复杂逻辑
- 长任务的可靠完成（中途需要人介入校准方向）

**结论**：把 AI 当成一个能力极强但不了解业务上下文的开发者——你负责方向和验证，它负责执行和加速。
