# Learning 维护规范

这份规范定义知识库的完成标准。目标不是让文件数量更多，而是让每个入口都能帮助理解、实践或决策。

## 文档类型

| 类型 | 作用 | 最低要求 |
|---|---|---|
| `map` | 说明学习路径、主题边界和先修关系 | 不复制正文；每个链接说明为什么值得进入 |
| `concept` | 解释一个可复用概念或机制 | 定义、机制、边界、反例或误区、关联主题 |
| `guide` | 帮助完成一个任务 | 前置条件、步骤、验证、失败处理 |
| `reference` | 提供稳定查询入口 | 范围清晰、结构可扫描、标明版本边界 |
| `case` | 用事实验证抽象 | 背景、约束、取舍、证据、结果和不可验证部分 |
| `practice` | 承载可运行练习 | 环境、运行命令、预期结果和延伸任务 |

只有标题、两段摘要或通用练习的内容不是独立文档：应合并到模块主文档，或明确标为待发展的 seed。

## 内容标准

一篇成熟主题文档至少回答以下问题中的四个：

1. 它解决什么问题，和相邻概念有什么边界？
2. 关键机制或因果链是什么？
3. 哪些前提下有效，哪些情况下会失效？
4. 有什么真实例子、反例或可运行示例？
5. 如何验证理解或实际结果？
6. 结论来自哪里，哪些信息可能变化？

不要为了套模板而保留空泛章节。没有独立内容时，宁可把相关结论合并为一个紧凑段落。

## 来源与时效

- 稳定基础知识优先引用教材、标准、论文或官方文档。
- API、模型、框架、监管、税制、交易规则和健康建议必须记录核对日期。
- 个人总结可以保留，但应与事实来源、案例证据和已复现结果区分。
- 外部链接应指向支持当前结论的具体页面，不只链接首页。
- 引用不是装饰：正文中的关键结论应能对应到来源。

建议在需要时使用以下 frontmatter；不要求给所有旧文档机械补齐：

```yaml
---
type: concept
status: mature
last_verified: 2026-07-22
---
```

`status` 使用 `seed`、`developing`、`mature`；`last_verified` 只用于存在明显时效性的文档。

## 代码示例

- 标为具体语言的代码块应语法完整，或明确写成 `text` / `pseudocode`。
- 可运行示例应给出依赖、入口和预期结果；长代码移入实践项目，正文只保留关键片段。
- 示例必须体现超时、错误、权限、资源释放等与主题直接相关的失败路径。
- 伪代码不得伪装成可复制运行的生产实现。

## 导航与链接

- `Learning/README.md` 说明目录边界和维护规则。
- `Learning/INDEX.md` 是唯一总入口，负责按目标导航。
- 主题 README 只说明本主题的边界、主线和少量关键入口。
- 模块 README 列出本目录内容，不复制整篇总结和通用练习。
- 使用从当前文件出发的最短相对链接；不写绕回 `Learning/` 的路径。
- 合并或重命名前先检查所有入链，完成后再删除旧文件。

## 合并判断

满足任意两项时，优先合并：

- 两篇文档回答同一个核心问题；
- 核心机制、例子或练习大面积重复；
- 单篇删除模板后只剩少量 bullet；
- 必须依赖另一篇才能形成完整解释；
- 标题不同，但复习时无法说明为什么要分别打开。

保留拆分的理由应是读者目标不同、维护频率不同、篇幅过长，或一篇是概念而另一篇是案例/实践。

## 完成检查

每轮整理结束后检查：

- 没有失效的本地链接和未闭合代码围栏；
- 没有已经完成却仍标为“预定内容”的目录；
- 没有无说明的占位文档；
- 主题边界在 README 中可解释，正文不存在明显双份主文档；
- 技术示例和关键公式经过验证；
- 易变或高风险内容有来源、日期和边界；
- 实践项目与长期知识文档分开维护。
