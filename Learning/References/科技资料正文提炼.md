## 主题矩阵

| 主题 | 精炼结论 | 归位 |
|---|---|---|
| AI Coding/工具 | AI 适合代码导航、生成、迁移、审查和 UI 初稿；必须经过上下文准备、差异审查、编译、测试和安全检查 | `AI`、`Agent` |
| 源码阅读 | 先画项目地图，再按入口、主链路、数据/消息、异常、测试和部署逐层验证；模型只能做导航，不能替代运行证据 | `Case_Studies` |
| 性能优化 | 先找端到端瓶颈，再用吞吐、P99、CPU、GC、网络、数据库和错误率验证；不要只优化局部代码 | `Backend/Architecture`、`Backend/Delivery` |
| 高可用与稳定性 | 事前降低发生、事中降低影响、事后改进；冗余、隔离、超时、降级、恢复、演练和指标要形成闭环 | `Backend/Architecture` |
| DDD/六边形 | 用端口/适配器保护领域核心，按变化原因和数据责任拆边界；架构模式不是固定目录模板 | `Backend/Architecture` |
| 支付/资金 | 收款、付款、退款、清算、结算分别建模；资金事实、渠道事实和账务事实需要幂等、对账和审计 | `Case_Studies/Backend` |
| 并发/运行时 | 并发度由瓶颈和下游容量决定；关注可见性、原子性、取消、背压、资源回收和尾延迟 | `CS`、`Backend` |
| 数据架构 | 冷热分层、归档、读写模型和批量处理要结合访问频率、成本、查询口径和回滚 | `Backend/Data` |
| 模型与推理 | 本地模型/流式推理要评估上下文、首 token 延迟、吞吐、内存、取消和服务隔离 | `AI`、`Agent` |
| 图表与表达 | Mermaid/AI UI 生成适合做初稿；必须人工核对参与者、状态、异常、权限和真实数据 | `Writing_and_Expression` |
| 技术人成长 | 业务理解、需求判断、验证能力、沟通和复盘与编码同等重要 | `Career` |

## 14 份附件的统一结论

- 三方支付：把交易、账务、渠道、清算和结算拆开，资金动作必须可追踪、可对账、可补偿。
- 高并发编程：先判断 CPU、锁、IO、内存、上下文切换或队列瓶颈，再选并发模型并限制并发度。
- 透明协程/运行时改造：性能数字只对特定场景有效，必须灰度、回滚并看 P99、资源和下游压力。
- 技术人核心竞争力：理解用户和结果，主动体验产品并减少协作损耗。
- Doris 冷热分层：迁移与持续写入、Schema 变更、查询口径和成本一起治理。
- 需求评估：拆目标、影响、依赖、风险、资源和维护成本，不用模糊工时掩盖不确定性。
- 软件复杂度：记录新增复杂度、依赖方向、测试、日志和未来删除路径，持续小步治理。
- 稳定性建设：事前、事中、事后分别降低故障概率、影响和复发率。
- GPU/模型：资源调度、显存、吞吐、延迟和模型质量需要同一套基线评估。
- 高可用架构：组件只是手段，RTO/RPO、故障隔离、恢复、演练和审计才是结果。
- DDD 架构：用业务变化和团队/数据责任验证边界，不复制模式名。
- 服务治理：注册、路由、限流、降级、发布、容量和监控随业务增长演进。
- Azure/OpenAI：模型接入还要处理身份、网络、数据、合规、成本、工具和人工兜底。
- 其他工具资料：产品版本、价格、账号和外部链接属于动态信息，不沉淀为稳定事实。
