# 产品文档与需求分析

PRD 是研发理解业务边界的共同语言。阅读时要把“页面描述”翻译成角色、对象、状态、规则、接口、副作用和异常路径。

## 六个问题

| 维度 | 需要确认 |
|---|---|
| 目标与角色 | 谁在什么场景完成什么目标，成功指标是什么 |
| 业务对象 | 用户、活动、队伍、订单、支付单、权益和库存分别是什么 |
| 状态 | 创建、进行、成功、失败、过期、撤销、退款如何迁移 |
| 规则 | 资格、时间、库存、优惠叠加、用户分层和风控如何计算 |
| 外部依赖 | 支付、库存、消息、风控、地图/门店由谁提供能力 |
| 异常与补偿 | 重复点击、并发、超时、回调丢失、库存不足怎么办 |

## 三类案例抽象

- **拼团**：成团人数、有效期、邀请关系、券包库存、未成团退款和核销。
- **电商营销**：签到/游戏/抽奖 → 积分、红包、优惠券 → 购买、留存和复购；抽奖结果不等于权益已发放。
- **任务探店**：任务发布 → 名额预占 → 组队预约 → 内容提交 → 审核 → 奖励发放；预占过期、驳回重提和退款都要有状态。

## 从 PRD 到系统

```text
用户动作 → 状态变化 → 领域事件 → 外部副作用
                         ↓
                  补偿 / 对账 / 审计
```

每个核心功能至少补一张状态表、一张规则表和一张异常路径表。页面原型中的随机数、弹窗和按钮反馈只能作为 UI 证据，不能直接当作生产业务实现。
