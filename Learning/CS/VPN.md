# VPN、Clash 与 VPS 的工作原理

本文的目标不是背诵软件配置，而是理解一条网络请求究竟经过了哪里：Clash 如何接管本机流量、如何选择代理节点、VPS 如何代替客户端访问目标网站，以及响应如何原路返回。

> [!NOTE]
> 日常语境常把 Clash 一类工具称为“VPN 软件”，但严格来说，Clash 更接近支持规则分流的代理客户端。真正的 VPN 通常在网络层建立虚拟网络，例如 WireGuard、OpenVPN；两者都能改变流量路径，但抽象层级和用途并不完全相同。

## 先建立整体认识

不使用代理时：

```text
浏览器 → 操作系统网络栈 → 家庭路由器/运营商 → 目标网站
```

使用 Clash 和远端 VPS 节点时：

```text
浏览器
  → Clash 接管流量
  → 按规则判断 DIRECT / PROXY / REJECT
  → 将需要代理的流量封装并加密
  → 运营商网络
  → VPS 上的代理服务端解密
  → VPS 访问目标网站
  ← 响应沿相反方向返回
```

从目标网站的角度，请求的来源通常是 VPS 的公网 IP，而不是家庭网络的公网 IP。运营商仍能看到设备正在与某个 VPS 通信，以及连接时间和流量大小，但在加密正确的情况下，通常不能直接读取代理隧道内的内容。

## Clash、Clash Verge 与 VPS 分别是什么

| 组件 | 所在位置 | 主要职责 |
| --- | --- | --- |
| Clash 内核 | 本机 | 接收流量、匹配规则、选择节点、执行代理协议 |
| Clash Verge | 本机 | Clash 内核的图形界面，负责配置、启停、查看日志等 |
| 节点配置 | 本机 | 描述服务器地址、端口、协议、认证和加密参数 |
| 代理服务端 | VPS | 接收客户端连接，认证并解密，然后代替客户端访问目标地址 |
| VPS | 数据中心 | 一台有公网网络的虚拟服务器，本身并不自动具备代理能力 |

Clash Verge 不是远程服务，也不是“提供网络”的一方。它主要管理本机上的代理内核。VPS 也不等于节点：只有在 VPS 上安装并配置了与客户端协议相匹配的服务端程序，它才会成为可用节点。

## Clash 如何接管流量

### 系统代理模式

Clash Verge 可以修改操作系统的 HTTP/HTTPS/SOCKS 代理设置，使遵守系统代理设置的应用把请求发送到 Clash 的本地监听端口，例如：

```text
应用 → 127.0.0.1:代理端口 → Clash
```

优点是原理简单；限制是只有支持系统代理的应用才会主动使用它。有些命令行程序、游戏或自行实现网络栈的软件可能绕过系统代理。

### TUN 模式

TUN 模式会创建一个虚拟网络接口，并通过系统路由规则把 IP 流量导向该接口。Clash 从虚拟接口读取数据包，再决定直连还是代理。

```text
应用 → 操作系统路由表 → 虚拟 TUN 网卡 → Clash → 直连或代理
```

它能接管更多不支持系统代理的应用，行为更接近全局 VPN，但也更容易受到权限、路由冲突、DNS 设置和其他 VPN 软件的影响。

### 两种模式的核心区别

| 维度 | 系统代理 | TUN 模式 |
| --- | --- | --- |
| 接管对象 | 主动使用代理设置的应用连接 | 经系统路由进入虚拟接口的 IP 流量 |
| 覆盖范围 | 较窄 | 较广 |
| 所需权限 | 通常较少 | 通常需要创建网卡和修改路由的权限 |
| 常见问题 | 某些应用不走代理 | 路由、DNS、权限或其他 VPN 冲突 |

## Clash 如何决定流量往哪里走

Clash 收到连接后，会根据配置中的规则依次匹配。典型结果有三类：

- `DIRECT`：直接从本机连接目标网站。
- `PROXY`：通过选定的远端节点连接。
- `REJECT`：阻止连接，常用于广告或恶意域名。

规则可能依据域名、域名后缀、IP 地址、地理 IP 数据、端口或进程等条件。例如：

```yaml
rules:
  - DOMAIN-SUFFIX,example.com,PROXY
  - GEOIP,CN,DIRECT
  - MATCH,PROXY
```

规则通常从上到下匹配，命中后停止，因此顺序会影响结果。`MATCH` 常作为兜底规则。

需要区分“代理组”和“节点”：节点对应一台具体服务器；代理组是一种选择策略，可以手动选节点，也可以根据延迟、可用性或回退规则自动选。

## 一次代理访问具体发生了什么

假设浏览器访问 `https://example.com`，规则要求使用 VPS 节点。

1. 浏览器先获得目标域名对应的信息。DNS 查询可能由本机、Clash 或远端服务器完成，取决于配置。
2. 系统代理或 TUN 模式把浏览器流量交给 Clash。
3. Clash 识别目标域名/IP，并按规则选择 `PROXY` 和一个具体节点。
4. Clash 根据节点协议，把原始连接信息和数据封装起来，与 VPS 建立加密或混淆后的连接。
5. VPS 上对应的服务端程序完成认证和解密，得知真正目标是 `example.com:443`。
6. VPS 作为新的客户端，与目标网站建立 TCP/QUIC、TLS 和 HTTP 连接。
7. 网站把响应发给 VPS；服务端程序再通过已有代理连接把响应送回 Clash。
8. Clash 将数据交还浏览器，浏览器通常感觉自己只是在正常访问网站。

这里存在两段不同的网络连接：

```text
第一段：本机 Clash ⇄ VPS 代理服务端
第二段：VPS ⇄ 目标网站
```

代理服务端负责在两段连接之间转发字节。它不是简单地把原始 IP 包“瞬移”到网站；具体是转发连接、数据流还是 IP 包，取决于所使用的协议和模式。

## VPS 为什么能够访问目标网站

VPS 是云服务商数据中心里的一台虚拟机，通常具有公网 IP、操作系统和公网路由。安装代理服务端以后，它会：

1. 在指定的 TCP 或 UDP 端口监听连接。
2. 验证客户端提供的密码、密钥或用户身份。
3. 按所用协议解密和解析客户端请求。
4. 创建到目标网站的新连接。
5. 在客户端连接与目标连接之间双向转发数据。

VPS 能否正常工作还受到以下条件影响：

- 云平台安全组是否放行相应端口。
- VPS 自身防火墙是否允许流量。
- 服务端是否监听正确的地址和端口。
- 客户端与服务端的协议、密钥和参数是否一致。
- VPS 到目标网站的路由是否可达，IP 是否被目标服务限制。
- 域名和 TLS 证书配置是否正确（若协议需要）。

## 节点配置里各字段的意义

一个节点通常包含：

| 字段 | 含义 |
| --- | --- |
| `server` | VPS 的域名或公网 IP |
| `port` | 服务端监听端口 |
| `type` | 双方使用的代理协议 |
| 密码/UUID/密钥 | 客户端身份认证材料 |
| TLS 相关参数 | 是否使用 TLS、验证哪个域名和证书 |
| 传输参数 | 数据如何承载，例如 TCP、WebSocket、gRPC 等 |

这些参数不是由 Clash 单方面决定的。服务端监听什么协议和参数，客户端就必须使用相匹配的设置，否则会在建连、认证或握手阶段失败。

订阅链接通常只是由服务提供方托管的一份配置。Clash Verge 下载并解析它，得到节点、代理组和规则；订阅本身不负责转发流量。

## DNS 为什么经常导致问题

访问域名通常要先做 DNS 解析。若 DNS 查询没有交给 Clash，而是直接发给本地网络，就可能出现：

- DNS 泄漏：本地 DNS 服务可以看到查询了哪些域名。
- 解析污染或错误：拿到无法连接或不正确的地址。
- 规则失效：Clash 只看到 IP，无法按域名规则准确分流。
- 分流结果异常：域名走代理，但其 DNS 查询却走了另一条路径。

Clash 常通过内置 DNS、加密 DNS、`fake-ip` 等机制统一处理解析和域名映射。`fake-ip` 会先向应用返回一个保留范围内的虚拟 IP；应用连接该地址时，Clash 再把它映射回原域名，从而保留域名信息并执行规则。这个虚拟 IP 通常只在本机的 Clash 体系内有意义。

## 代理、VPN 和加密连接的区别

| 概念 | 核心作用 |
| --- | --- |
| 代理 | 由中间服务器代表客户端建立应用连接，常可按域名或规则分流 |
| VPN | 建立虚拟网络接口或网络隧道，使设备或网段像连接到另一个网络 |
| TLS/HTTPS | 加密应用与目标服务之间的内容，不负责决定流量经哪台中间服务器 |

即使不使用代理，HTTPS 也会加密浏览器和网站之间的内容。使用代理后，还可能在本机与 VPS 之间增加一层代理协议保护。两层保护解决的问题不同，不应把“使用代理”简单等同于“所有环节绝对匿名或安全”。

## 常见误区

- **打开 Clash Verge 就一定生效**：还需要启动内核，并启用系统代理或 TUN 等接管方式。
- **选择全局模式就能接管所有应用**：全局模式决定已进入 Clash 的流量如何处理，不代表所有应用都已经被 Clash 接管。
- **VPS 天生就是 VPN**：VPS 只是服务器，必须安装并配置相应服务端。
- **目标网站能看到本机公网 IP**：正常代理时，网站通常看到 VPS IP；但浏览器 WebRTC、直连规则或其他旁路可能暴露额外信息。
- **代理等于完全匿名**：节点运营者、浏览器账号、Cookie、设备指纹、DNS 和支付信息都可能关联身份。
- **测速延迟低就一定快**：实际体验还取决于带宽、丢包、拥塞、VPS 线路、目标站位置和协议开销。

## 排障思路

遇到“开了但不能用”时，按数据链路逐段检查：

```text
应用是否被接管
→ Clash 规则是否选择了预期策略
→ 本机能否连接 VPS 的地址和端口
→ 客户端与服务端参数是否一致
→ 服务端进程是否运行并监听
→ 防火墙/安全组是否放行
→ VPS 能否访问目标网站
→ DNS 是否按预期解析
→ 返回流量是否正常
```

Clash Verge 的连接列表和日志非常重要：可以确认请求是否进入 Clash、命中了哪条规则、选择了哪个节点，以及失败发生在 DNS、连接还是握手阶段。

## 学完后的验收问题

- 能解释 Clash 内核、Clash Verge、节点配置和 VPS 服务端各自负责什么。
- 能画出“应用 → Clash → VPS → 网站 → VPS → Clash → 应用”的路径。
- 能解释系统代理和 TUN 模式为什么覆盖范围不同。
- 能解释 `DIRECT`、`PROXY`、`REJECT` 以及规则顺序。
- 能说明 VPS 本身为什么不是代理，安装服务端后又如何转发流量。
- 能解释为什么客户端和服务端的协议、端口、认证和 TLS 参数必须一致。
- 能解释 DNS 泄漏、`fake-ip` 和域名规则之间的关系。
- 出现连接失败时，能沿本机接管、节点连接、服务端监听和目标访问逐段定位。

## 后续学习顺序

1. 用 Clash Verge 的连接页面观察一次请求命中的规则和节点。
2. 分别体验系统代理与 TUN 模式，观察哪些应用会被接管。
3. 学习 DNS、TCP/UDP、TLS、路由和 NAT 的基础概念。
4. 在合法合规且获得授权的环境中搭建测试 VPS，理解客户端和服务端参数如何对应。
5. 再深入 WireGuard、OpenVPN 与常见代理协议，比较网络层隧道和应用代理的区别。
