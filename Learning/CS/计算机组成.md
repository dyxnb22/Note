# 计算机组成

## CPU 是如何执行程序的？

CPU 执行指令通常经历取指、译码、执行、访存和写回等阶段。程序计数器保存下一条指令地址，寄存器保存操作数和中间结果，算术逻辑单元负责计算，控制器负责协调各阶段。

面试回答重点：CPU 不是直接执行高级语言，而是执行编译或解释后得到的机器指令；程序运行本质上是 CPU 不断读取指令、处理数据并更新状态。

## 32 位和 64 位有什么区别？

主要区别不是“64 位一定快一倍”，而是：

- 64 位 CPU 一次可以处理更宽的数据；
- 64 位地址空间更大，能够寻址更多内存；
- 指针、寄存器和 ABI 等数据结构可能变大，程序占用的内存也可能增加。

实际性能还取决于指令集、缓存、分支预测、编译器和程序访问模式。

## 存储器为什么要设计成层次结构？

寄存器、CPU Cache、内存、SSD 和磁盘的速度依次变慢，容量和单位成本则大致相反。存储器层次结构利用局部性原理，把最近访问或可能再次访问的数据放在更快的层级中。

- 时间局部性：刚访问的数据可能很快再次访问。
- 空间局部性：访问某个地址后，附近地址可能也会被访问。

因此，缓存命中率和访问模式往往比单纯增加 CPU 频率更影响程序性能。

## CPU Cache 一致性解决什么问题？

多核 CPU 各自拥有缓存时，同一份内存数据可能同时存在于多个 Cache 中。一个核心修改数据后，其他核心不能继续使用旧副本，否则会出现线程读到过期值的问题。

缓存一致性协议会跟踪缓存行的状态，在写入时使其他核心的副本失效或更新。实际编程中还要区分缓存一致性和内存可见性：前者由硬件维护缓存副本，后者还涉及编译器、CPU 重排序和语言内存模型。

## 为什么要关注缓存行和伪共享？

CPU 通常以缓存行而不是单个变量为单位加载数据。如果多个线程频繁修改位于同一缓存行的不同变量，即使它们逻辑上互不相关，也可能反复触发缓存行失效，这种现象叫伪共享。

优化方向是让高频写入、相互独立的数据分散到不同缓存行；但填充和对齐会增加内存占用，应结合实际性能数据使用。

## 为什么 `0.1 + 0.2` 可能不等于 `0.3`？

许多十进制小数无法用有限位的二进制浮点数精确表示，存储时只能使用最接近的近似值，运算后再发生一次舍入，所以结果可能出现微小误差。

- 金额计算使用 `BigDecimal` 或整数最小货币单位。
- 浮点数比较通常使用误差范围，而不是直接使用 `==`。
- 浮点数适合测量、科学计算等允许近似的场景，不适合直接承担精确金额计算。

## 中断、DMA 和轮询有什么区别？

- 轮询：CPU 不断检查设备状态，简单但会浪费 CPU。
- 中断：设备准备好后通知 CPU，CPU 暂停当前执行流处理事件，适合不规律的 I/O。
- DMA：设备和内存之间直接搬运数据，减少 CPU 参与，适合网卡、磁盘等大量数据传输。

实际系统通常组合使用：设备用中断通知传输完成，数据搬运由 DMA 完成，CPU 只处理控制和结果。

## 补码与溢出

计算机通常用补码表示带符号整数，因为减法可以转化为加法，且 `0` 只有一种编码。对 n 位有符号补码，表示范围是 `-2^(n-1)` 到 `2^(n-1)-1`。

判断加法溢出的实用规则：两个同号数相加，结果却变成异号，说明超出表示范围；异号数相加不会发生有符号溢出。溢出不是进位本身，而是数学结果超出当前位宽后被截断。Java 的 `int`/`long` 也遵循固定宽度运算，溢出默认回绕，不会自动抛异常。

## 浮点数与 IEEE 754

浮点数通常由符号位、阶码和尾数（有效数）组成。有限位的二进制无法精确表示许多十进制小数，因此运算结果可能出现舍入误差；上层语言中常见的十进制小数精度问题就源于此。

- 比较浮点数通常使用误差范围，而不是直接使用 `==`。
- 金额应使用整数最小单位或 `BigDecimal`，不能依赖二进制浮点数的精确性。
- 阶码溢出可能产生正/负无穷，特殊编码还包括 NaN 和有符号零；具体行为由语言和 IEEE 754 实现决定。

## 指令格式与 RISC/CISC

一条机器指令通常包含操作码和操作数/地址信息。操作码说明“做什么”，寻址方式说明“操作数在哪里”；寄存器寻址、立即数寻址、直接/间接寻址会影响指令长度和访存次数。

- RISC：指令较少且格式规整，通常采用加载/存储架构，便于流水线和硬件实现。
- CISC：指令集更丰富，单条指令可能完成更复杂的操作，便于提高代码密度，但译码和实现更复杂。

不能简单把 RISC 等同于“小型机”、把 CISC 等同于“大型机”；现代处理器往往融合了两类思想，实际性能还取决于微架构、编译器和工作负载。

## 流水线与数据冒险

CPU 可以把已有的执行阶段重叠，让多条指令同时处于不同阶段，从而提高吞吐量，但不会让单条指令的逻辑执行时间必然缩短。

常见冒险包括：

- 结构冒险：多个阶段争用同一硬件资源。
- 数据冒险：后一条指令依赖前一条指令尚未写回的结果。
- 控制冒险：分支跳转目标尚未确定。

处理方式包括增加硬件资源、数据前递、暂停/插入气泡、乱序执行和分支预测。分支预测失败会清空部分流水线，因此分支密集、访问模式不规则的代码可能更难优化。

## 数据校验码的能力边界

- 奇偶校验码通常只能发现奇数个位错误，不能定位或纠正错误。
- 海明码可以通过增加校验位定位并纠正单比特错误；要同时检测双比特错误需要更大的码距或扩展设计。
- CRC 擅长检测传输中的突发错误，但普通 CRC 主要用于检错，不应表述为可以直接纠正任意多位错误；需要纠错时要使用专门的前向纠错编码。
