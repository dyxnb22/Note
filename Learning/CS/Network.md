# 网络模型

## OSI 七层模型

从上到下分别是：

| 层次       | 作用                                             | 典型内容                 |
| ---------- | ------------------------------------------------ | ------------------------ |
| 应用层     | 为应用程序提供网络服务                           | HTTP、DNS、FTP           |
| 表示层     | 数据格式转换、压缩和加密                         | 编码、压缩、加密         |
| 会话层     | 建立、管理和终止会话                             | 会话管理                 |
| 传输层     | 负责端到端传输                                   | TCP、UDP                 |
| 网络层     | IP 寻址、路由转发和分片                          | IP、ICMP                 |
| 数据链路层 | 封帧、MAC 寻址和差错检测                         | Ethernet、ARP            |
| 物理层     | 在网线、光纤、无线信道中传输比特流               | 网线、光纤、无线信号     |

面试中通常重点关注应用层、传输层、网络层和数据链路层。

## TCP/IP 四层模型

| 层次       | 作用                         | 典型协议                         |
| ---------- | ---------------------------- | -------------------------------- |
| 应用层     | 直接面向应用                 | HTTP、HTTPS、DNS、FTP、SMTP      |
| 传输层     | 负责进程到进程的通信         | TCP、UDP                         |
| 网络层     | 负责寻址和路由               | IP、ICMP、ARP                    |
| 网络接口层 | 在具体链路上传输数据         | 对应 OSI 的数据链路层和物理层    |

TCP/IP 模型更贴近实际互联网协议栈，常用它解释一次网络请求的完整链路。

# 应用层

## 应用层常见协议有哪些？

- HTTP/HTTPS：Web 请求与响应。
- DNS：域名解析。
- FTP/SFTP：文件传输。
- SMTP/POP3/IMAP：邮件发送与接收。
- WebSocket：全双工实时通信。
- RPC/gRPC：服务之间的远程调用。

## HTTP 报文结构是什么？

HTTP 请求报文：

- 请求行：请求方法、URL/URI、HTTP 版本。
- 请求头：Host、User-Agent、Content-Type、Cookie 等。
- 空行：分隔头部和请求体。
- 请求体：可选，常见于 POST、PUT 等请求。

HTTP 响应报文：

- 状态行：HTTP 版本、状态码、状态描述。
- 响应头：Content-Type、Content-Length、Set-Cookie、Cache-Control 等。
- 空行：分隔头部和响应体。
- 响应体：HTML、JSON、图片、文件等实际内容。

## HTTP 常见状态码有哪些？

| 类型 | 含义                         |
| ---- | ---------------------------- |
| 1xx  | 提示信息，请求已收到         |
| 2xx  | 请求成功                     |
| 3xx  | 重定向                       |
| 4xx  | 客户端错误                   |
| 5xx  | 服务端错误                   |

常见状态码：

- 200 OK：请求成功。
- 206 Partial Content：范围请求成功，常用于断点续传。
- 301 Moved Permanently：永久重定向。
- 302 Found：临时重定向。
- 304 Not Modified：资源未修改，可以使用缓存。
- 400 Bad Request：请求格式错误。
- 401 Unauthorized：未认证。
- 403 Forbidden：已认证但无权限。
- 404 Not Found：资源不存在。
- 405 Method Not Allowed：请求方法不支持。
- 500 Internal Server Error：服务端内部错误。
- 502 Bad Gateway：网关或代理从上游收到无效响应。
- 503 Service Unavailable：服务暂不可用。
- 504 Gateway Timeout：网关等待上游超时。

## HTTP 缓存如何工作？

HTTP 缓存的目标是减少重复传输和服务端请求。响应可以通过 `Cache-Control`、`Expires` 等字段控制缓存时间和范围；缓存过期后，客户端可以带条件请求向服务端确认资源是否变化。

- `Cache-Control: max-age`：资源在指定时间内可直接使用。
- `no-cache`：可以缓存，但使用前必须向服务端验证。
- `no-store`：不允许缓存，适合敏感数据。
- `private` / `public`：限制只能由浏览器缓存，或允许共享缓存缓存。
- `ETag`：资源版本标识；客户端通过 `If-None-Match` 验证。
- `Last-Modified`：资源修改时间；客户端通过 `If-Modified-Since` 验证。

资源未变化时返回 `304`，客户端继续使用本地副本；已变化时返回新的资源。设置缓存时要同时考虑内容是否敏感、更新是否及时和 CDN/代理是否会共享缓存。

## HTTP 请求方法和幂等性如何理解？

| 方法    | 语义                         | 安全性/幂等性                         |
| ------- | ---------------------------- | ------------------------------------- |
| GET     | 获取资源                     | 安全、幂等                             |
| POST    | 提交数据或创建资源           | 不安全，通常不幂等                     |
| PUT     | 整体更新资源                 | 不安全、幂等                           |
| PATCH   | 局部更新资源                 | 不安全，是否幂等取决于实现             |
| DELETE  | 删除资源                     | 不安全，通常幂等                       |
| HEAD    | 只获取响应头，不返回响应体   | 通常安全、幂等                         |
| OPTIONS | 查询服务器支持的方法         | 常见于跨域预检请求                     |

安全是指不会修改服务端资源；幂等是指执行一次和执行多次的结果一致。

## 什么是跨域？常见解决方式有哪些？

浏览器同源策略要求协议、域名和端口都相同；前端页面访问不同源接口时会触发跨域限制。常见方案是服务端配置 CORS（包括预检 `OPTIONS`、允许的 Origin、方法和请求头），或让反向代理/网关把请求转发为同源路径。JSONP 只适用于旧式 GET 场景，不适合作为通用方案；CORS 允许来源不能无条件写 `*` 后再配凭证。

## HTTP 长连接和消息边界如何处理？

HTTP 长连接指多个 HTTP 请求/响应复用同一个 TCP 连接，避免频繁建立和释放 TCP 连接的开销。

- HTTP/1.0 默认短连接，需要显式使用 `Connection: Keep-Alive` 开启长连接。
- HTTP/1.1 默认长连接，通常不发送 `Connection: close` 就会保持连接。
- 长连接不会永久存在，服务器一般会配置空闲超时时间和最大请求数，到达后主动关闭。

HTTP/1.1 常见的消息体边界判断方式：

- `Content-Length`：声明 body 字节数，接收方按长度读取。
- `Transfer-Encoding: chunked`：分块传输，每个 chunk 自带长度，最后以长度为 0 的 chunk 结束。
- 无 body：请求或响应根据方法、状态码和协议规则判断是否没有消息体；不能只机械地看某一个请求头。

如果同时出现 `Transfer-Encoding` 和 `Content-Length`，发送方本身就不应这样构造报文；接收和转发时必须严格按协议处理，否则容易产生请求走私等安全问题。

## HTTP 断点续传怎么实现？

1. 服务端返回 `Accept-Ranges: bytes`，表示支持按字节范围请求。
2. 客户端记录已下载位置，例如已下载 512KB。
3. 续传时发送 `Range: bytes=512000-`。
4. 服务端返回 `206 Partial Content`，并携带 `Content-Range` 和 `Content-Length`。
5. 如果请求范围超过资源大小，服务端返回 `416 Requested Range Not Satisfiable`。

核心头部是 `Range`、`Accept-Ranges`、`Content-Range` 和 `Content-Length`。

## HTTP、HTTPS 和 TLS 有什么区别？

HTTP 是明文传输，存在窃听、篡改和冒充风险。HTTPS 在 HTTP 和 TCP 之间加入 TLS，通过加密、完整性校验和证书认证解决这些问题。

| 对比项   | HTTP                        | HTTPS                                                        |
| -------- | --------------------------- | ------------------------------------------------------------ |
| 安全性   | 明文传输                    | TLS 加密传输                                                 |
| 默认端口 | 80                          | 443                                                          |
| 连接建立 | TCP 三次握手后直接传输 HTTP | TCP 三次握手后还要 TLS 握手                                  |
| 身份认证 | 无                          | 通过 CA 证书验证服务端身份                                   |
| 防篡改   | 无                          | TLS 消息认证可检测篡改                                       |
| 性能     | 开销较低                    | 有握手和加解密开销，可通过会话复用、HTTP/2、HTTP/3 优化     |

TLS 握手的核心流程：

1. ClientHello：客户端发送支持的 TLS 版本、随机数和加密套件列表。
2. ServerHello：服务端选择 TLS 版本和加密套件，返回随机数并发送证书。
3. 证书校验：客户端校验证书链、域名、有效期和吊销状态等。
4. 密钥交换：现代 TLS 更常用 ECDHE，通过双方临时密钥协商共享秘密；旧的 RSA 流程由客户端生成 pre-master secret 后用服务端公钥加密发送。
5. 生成会话密钥：双方基于随机数和共享秘密生成对称加密密钥。
6. Finished：双方用协商出的密钥发送握手摘要，验证握手没有被篡改。
7. 后续通信：使用对称加密传输 HTTP 数据。

非对称机制主要用于身份认证和密钥协商，真正传输业务数据时使用对称加密。HTTPS 防范中间人攻击依赖证书认证、密钥协商和完整性校验，但不能阻止用户主动信任恶意证书或终端本身被攻破。

## HTTP/1.1、HTTP/2、HTTP/3 有什么区别？

| 版本     | 传输基础   | 主要特性                                       | 主要限制或改进                                               |
| -------- | ---------- | ---------------------------------------------- | ------------------------------------------------------------ |
| HTTP/1.1 | TCP        | 默认长连接、缓存控制、管道化                   | 仍有应用层队头阻塞，管道化实际使用有限                       |
| HTTP/2   | TCP        | 二进制帧、多路复用、HPACK 头部压缩             | 多路复用仍受 TCP 队头阻塞影响；服务端推送已较少使用           |
| HTTP/3   | UDP + QUIC | 多路复用、TLS 1.3、连接迁移、0-RTT             | 解决 TCP 层队头阻塞，但 0-RTT 需要注意重放风险               |

HTTP/2 在一条 TCP 连接上并发多个 Stream；底层 TCP 丢包时所有 Stream 仍可能被阻塞。HTTP/3 基于 QUIC，不同 Stream 可以独立重传，弱网和网络切换体验更好。

## HTTP、Socket 和 TCP 有什么区别？连接建立后为什么会中断？

- HTTP 是应用层协议，定义请求和响应的语义、报文格式和状态码。
- TCP 是传输层协议，提供面向连接、可靠、有序的字节流传输。
- Socket 是操作系统提供的网络编程接口，不是协议。应用程序通过 Socket 使用 TCP 或 UDP 通信。

可以理解为：HTTP 通常运行在 TCP 之上，应用程序通过 Socket 调用系统能力完成网络收发。

连接可能因以下原因中断：

- 任意一端主动关闭连接，发送 FIN。
- 出现异常，发送 RST 强制断开。
- 数据长期得不到 ACK，达到最大重传次数。
- 长连接空闲超过 keepalive timeout，或复用请求数达到上限。
- 网络故障、进程崩溃、防火墙或负载均衡器清理连接。

# DNS

## DNS 是什么？域名有什么层级？

DNS（Domain Name System）用于把域名解析成 IP 地址，通常使用 53 端口，既可能使用 UDP，也可能使用 TCP。

域名从右到左逐级细化，例如 `www.server.com`：

- 根 DNS：`.`。
- 顶级域 DNS：`.com`。
- 权威 DNS：负责 `server.com` 的权威记录。

## DNS 域名解析流程是什么？

以 `www.server.com` 为例：

1. 浏览器、操作系统、hosts 和本地 DNS 缓存先查找，命中则直接返回。
2. 未命中时，客户端请求本地 DNS 服务器。
3. 本地 DNS 查询根 DNS，获得 `.com` 顶级域服务器地址。
4. 查询 `.com` 顶级域服务器，获得 `server.com` 权威 DNS 地址。
5. 查询权威 DNS，获得 `www.server.com` 的 IP。
6. 本地 DNS 按 TTL 缓存结果并返回客户端。
7. 客户端用解析出的 IP 建立连接。

客户端到本地 DNS 通常是递归查询，本地 DNS 到根、顶级域和权威 DNS 通常是迭代查询。

## A 记录和 CNAME 记录有什么区别？

- A 记录把域名直接映射到 IPv4 地址；AAAA 记录映射到 IPv6 地址。
- CNAME 把一个域名映射到另一个规范域名，再由目标域名继续解析，适合 CDN、业务别名和统一切换入口。
- CNAME 便于变更目标，但会增加一次解析链路；配置 DNS 时还要结合 TTL、缓存生效时间和根域名限制。

## DNS 使用 TCP 还是 UDP？

DNS 查询通常使用 UDP，因为 UDP 无连接、开销小、延迟低，适合短小查询。以下场景可能使用 TCP：

- 响应报文过大，UDP 放不下或被截断。
- 区域传送，即 DNS 服务器之间同步记录。
- 某些安全或可靠性要求更高的场景。

DNS 可靠性主要依赖超时重试、缓存和多 DNS 服务器配置。

## DNS 劫持是什么？如何防范？

DNS 劫持是攻击者篡改 DNS 响应，把域名解析到恶意 IP，导致用户访问假网站或被插入广告。

防范方式：

- 使用可信 DNS 服务，避免使用不可信网络提供的 DNS。
- 使用 DNS over HTTPS 或 DNS over TLS，减少解析过程被篡改。
- 域名侧开启 DNSSEC，验证 DNS 记录完整性。
- 通过 HTTPS 证书校验阻止攻击者伪造合法站点。
- 对核心域名做解析监控，发现异常 IP 及时告警。

# Web 认证与会话

## HTTP 无状态，Cookie 和 Session 如何维持状态？

HTTP 无状态是指协议本身不记住上一次请求，每个请求彼此独立。Cookie 是 HTTP 之上的状态补充机制：服务端通过 `Set-Cookie` 把会话标识或少量状态写入浏览器，浏览器后续请求自动带上 Cookie，服务端就能识别用户。

所以准确说法是：HTTP 协议本身无状态，但可以借助 Cookie、Session、Token 等机制实现业务层状态管理。

## Cookie、Session、Token、JWT 有什么区别？

| 对比项     | Cookie                                   | Session                    | Token                   | JWT                      |
| ---------- | ---------------------------------------- | -------------------------- | ----------------------- | ------------------------ |
| 存储位置   | 浏览器                                   | 服务端                     | 客户端                  | 客户端                   |
| 保存内容   | 少量键值对                               | 用户会话数据               | 认证凭证                | Header.Payload.Signature |
| 服务端状态 | 可有可无                                 | 有状态                     | 通常无状态              | 通常无状态               |
| 请求携带   | 浏览器自动携带                           | 通常靠 Cookie 带 SessionId  | 手动放 Header 或 Cookie | 手动放 Header 或 Cookie  |
| 优点       | 简单、自动携带，可设置安全属性           | 敏感数据在服务端，更易管控 | 适合前后端分离和跨端     | 自包含、可签名、防篡改   |
| 缺点       | 容量小，需要注意 CSRF/XSS                | 分布式场景需共享 Session   | 泄露后有风险            | 签发后失效前难主动撤销   |

Session 适合传统单体或服务端渲染；Token/JWT 更适合前后端分离、移动端、微服务和跨域场景，但 JWT 并不等于加密。

## 客户端禁用 Cookie 后 Session 还能用吗？

默认不能正常使用。大多数 Web 应用依赖 Cookie 保存 SessionId，禁用 Cookie 后服务端无法通过请求识别对应 Session。

替代方案：

- URL 重写：把 SessionId 放入 URL，例如 `;jsessionid=xxx`，但容易泄露，不推荐。
- 隐藏表单字段：只适合表单交互，不适合普通链接和 AJAX。
- 改用 Token：由客户端在请求头中主动携带认证凭证。

## Cookie、LocalStorage 和 SessionStorage 如何选择？

| 对比项   | Cookie                                      | LocalStorage                   | SessionStorage                 |
| -------- | ------------------------------------------- | ------------------------------ | ------------------------------ |
| 容量     | 约 4KB                                      | 通常 5MB 以上                  | 通常 5MB 以上                  |
| 请求携带 | 同域请求自动携带                            | 不会自动随请求发送             | 不会自动随请求发送             |
| 生命周期 | 可设置过期时间                              | 默认长期保存，除非手动清理     | 通常随标签页会话结束           |
| JS 访问  | `HttpOnly` 后 JS 不可读                     | 可被前端 JS 读取               | 可被前端 JS 读取               |
| 主要风险 | CSRF；非 HttpOnly 时也怕 XSS                | XSS 可直接读取                 | XSS 可直接读取                 |
| 适合场景 | 会话标识、需要自动随请求发送的数据          | 非敏感本地缓存和页面状态       | 短期页面会话状态               |

Cookie 适合：

- SessionId、短期认证凭证。
- 服务端需要自动随请求接收的数据。
- 需要设置过期时间、`HttpOnly`、`Secure`、`SameSite` 的数据。

LocalStorage 适合非敏感的用户偏好、前端缓存和页面间共享状态。不要把密码、身份证号、银行卡号等敏感明文放进 Cookie 或 Web Storage。

## JWT 的结构、集群优势和泄露处理是什么？

JWT 由三部分组成，用点号分隔：

- Header：令牌类型和签名算法，例如 `alg`、`typ`。
- Payload：声明信息，例如用户 ID、角色和过期时间 `exp`。
- Signature：对 Header 和 Payload 做签名，防止内容被篡改。

Header 和 Payload 只是 Base64URL 编码，不是加密，不能放敏感明文信息。

JWT 适合集群部署，是因为服务端只要能验证签名，就能从 Token 中解析用户身份和权限，不依赖本机 Session。传统 Session 如果保存在单机内存中，需要 Session 复制、粘性会话或 Redis 共享存储。

JWT 的缺点和泄露处理：

- 签发后在过期前默认一直有效，不容易主动撤销。
- Token 体积比 SessionId 大，每次请求携带会增加带宽。
- 存在 LocalStorage 时容易被 XSS 窃取；存 Cookie 时需要防 CSRF。
- 缩短 Access Token 有效期，使用 Refresh Token 续签，并对 Refresh Token 做轮换。
- 维护黑名单或版本号，让指定 Token 失效。
- 用户改密、退出登录或发现异常时，强制刷新密钥或会话版本。

前端存储没有绝对安全的方案，需要结合 XSS、CSRF 防护。认证 Cookie 通常设置 `HttpOnly`、`Secure`、`SameSite`，重要操作再配合 CSRF Token。

# RPC、WebSocket 与 Nginx

## 为什么有 HTTP 还要用 RPC？

RPC 是远程过程调用思想，目标是让调用远程服务像调用本地方法一样。HTTP 是应用层协议，RPC 可以基于 TCP、HTTP/2、QUIC 等实现；gRPC、Dubbo、Thrift 是常见的 RPC 框架或实现。

- HTTP/REST 更适合对外开放接口，通用性强，浏览器和网关支持好。
- RPC 更适合内部服务调用，接口约束强，通常有更完整的序列化、负载均衡、超时和治理能力。
- gRPC 基于 HTTP/2，支持多路复用、流式传输和 Protobuf，适合微服务。

实际项目中常见组合是：外部 API 使用 HTTP/REST，内部服务之间使用 RPC。

## HTTP 长连接和 WebSocket 有什么区别？

| 对比项   | HTTP 长连接              | WebSocket                             |
| -------- | ------------------------ | ------------------------------------- |
| 通信模式 | 请求-响应                | 全双工                                |
| 发起方   | 客户端发请求，服务端响应 | 连接建立后双方都可主动发送            |
| 实时推送 | 需要轮询、长轮询或 SSE    | 原生支持服务端推送                    |
| 协议建立 | 普通 HTTP                | 先 HTTP 握手，再 Upgrade 到 WebSocket |
| 适合场景 | REST API、网页资源加载   | 聊天、游戏、协作编辑、实时行情        |

## Nginx 有哪些负载均衡算法？位于哪一层？

Nginx 常作为七层负载均衡工作在应用层，理解 HTTP 请求后按域名、路径和 Header 转发；使用 `stream` 模块时，也可以做四层 TCP/UDP 代理，按 IP 和端口转发。

常见算法：

- 轮询：默认方式，请求按顺序分配到后端。
- 加权轮询：按权重分配，性能更好的机器分到更多请求。
- IP Hash：根据客户端 IP 分配，适合会话保持。
- 最少连接：优先分配给当前连接数最少的后端。
- URL Hash：按 URL 哈希分配，适合提高缓存命中率。
- 最短响应时间：优先分配给响应更快的后端，通常需要额外模块或商业能力。

# Netty 与高性能网络编程

## Netty 的线程模型是什么？

Netty 基于 Reactor：Boss EventLoop 接收连接，Worker EventLoop 负责读写和执行 `ChannelPipeline` 中的 Handler。一个 EventLoop 通常绑定一个线程和 Selector，可管理多个连接；耗时业务不能长期阻塞 EventLoop，应投递到业务线程池。

## TCP 粘包/拆包怎么解决？

TCP 是字节流，没有消息边界；应用层应定义固定长度、分隔符或“长度字段 + 消息体”等协议边界。Netty 可用 `LengthFieldBasedFrameDecoder` 等解码器先拆帧，再做反序列化；完整原因和通用方案见下方 TCP 小节。

## Netty 为什么性能较高？

- 非阻塞 I/O 与事件驱动，减少连接对应的线程数量。
- Pipeline/Handler 便于复用编解码、鉴权和业务处理链。
- ByteBuf、池化内存和零拷贝减少分配与数据复制；但要注意引用计数释放和内存泄漏。
- 支持 epoll、kqueue 等原生传输，具体收益需要结合平台和压测确认。

# 传输层

## TCP 头部有哪些关键字段？

- 源端口、目的端口：标识通信双方进程。
- 序列号 Seq：标识本报文段第一个字节的序号，用于按序接收、去重和重传。
- 确认号 Ack：表示期望收到的下一个字节序号，用于确认对方数据。
- 数据偏移：表示 TCP 头部长度。
- 控制位：SYN、ACK、FIN、RST、PSH、URG。
- 窗口大小：用于流量控制，告诉对方还能接收多少数据。
- 校验和：检测 TCP 头部和数据是否出错。

常见控制位：SYN 用于建立连接，ACK 表示确认号有效，FIN 用于正常关闭连接，RST 用于异常重置连接。

## TCP 和 UDP 有什么区别？

| 对比项   | TCP                        | UDP                          |
| -------- | -------------------------- | ---------------------------- |
| 连接     | 面向连接                   | 无连接                       |
| 可靠性   | 可靠、有序、不重复         | 尽最大努力交付，可能丢包乱序 |
| 数据形式 | 字节流，无消息边界         | 数据报，有消息边界           |
| 控制能力 | 有流量控制、拥塞控制       | 无内置流控和拥塞控制         |
| 头部开销 | 至少 20 字节               | 固定 8 字节                  |
| 适合场景 | 文件传输、网页、数据库连接 | DNS、音视频、游戏、QUIC      |

## TCP 为什么可靠？粘包如何解决？

TCP 可靠性来自多个机制共同作用：

- 三次握手建立连接，四次挥手释放连接。
- 序列号用于排序、去重和重传。
- 确认应答告诉发送方哪些数据已收到。
- 超时重传和快重传处理丢包。
- 校验和检测传输过程中的比特错误。
- 接收窗口实现流量控制，拥塞窗口实现拥塞控制。

TCP 是字节流协议，没有消息边界。发送方多次写入的数据，接收方可能一次读到；一次写入的数据也可能被分多次读到，这就是常说的粘包/拆包。

解决方式是定义应用层消息边界：

- 固定长度：每个消息固定 N 字节。
- 分隔符：用特殊字符标识消息结束，例如换行。
- 长度字段：自定义协议头，先读长度，再按长度读取 body。

实际项目里最常用的是“长度字段 + 消息体”。

## 怎么用 UDP 实现 HTTP？

HTTP/3 就是基于 UDP 的 QUIC 实现的。UDP 本身不可靠，QUIC 在用户态实现了类似 TCP 的能力：

- 可靠传输：包编号、确认和重传。
- 拥塞控制：根据网络状态调整发送速度。
- 多路复用：多个 Stream 独立传输，减少队头阻塞。
- TLS 1.3 内置：握手更快，安全性更好。
- 连接迁移：通过 Connection ID 识别连接，IP 或网络切换后仍可保持连接。

## TCP 三次握手、异常场景和 SYN Flood 怎么处理？

三次握手：

1. 客户端发送 SYN，携带初始序列号 `client_isn`，进入 `SYN_SENT`。
2. 服务端回复 SYN + ACK，携带 `server_isn`，确认号为 `client_isn + 1`，进入 `SYN_RCVD`。
3. 客户端回复 ACK，确认号为 `server_isn + 1`。客户端立即进入 `ESTABLISHED`，服务端收到 ACK 后也进入 `ESTABLISHED`。

前两次握手不能携带应用层数据，第三次握手可以携带数据。三次握手的目的包括：

- 防止历史连接初始化：旧 SYN 到达时，客户端可以发现异常并发送 RST。
- 同步双方初始序列号：双方都确认对方的 ISN，才能可靠传输。
- 避免服务端在两次握手后就建立连接，减少无效或伪造 SYN 消耗资源。

异常情况：

- 第三次 ACK 丢失：服务端停留在 `SYN_RCVD` 并重传 SYN+ACK；客户端收到后再次发送 ACK。
- 客户端第一个 SYN 丢失：客户端在 `SYN_SENT` 超时重传，通常采用指数退避，次数由系统参数控制。
- 服务端收到重复 SYN：通常再次发送 SYN+ACK，等待客户端确认。

服务端收到 SYN 后，内核先创建半连接并放入半连接队列；收到第三次 ACK 后移入全连接队列，应用进程调用 `accept()` 时取出连接。两个队列都有长度限制。

大量 SYN 包可能打满半连接队列，形成 SYN Flood。常见防护包括 SYN Cookies、合理调整队列和重传参数，并使用防火墙、负载均衡或流量清洗过滤异常流量。

## TCP 四次挥手、TIME_WAIT 和连接关闭异常怎么处理？

以客户端主动关闭为例：

1. 客户端发送 FIN，进入 `FIN_WAIT_1`。
2. 服务端回复 ACK，进入 `CLOSE_WAIT`；客户端收到 ACK 后进入 `FIN_WAIT_2`。
3. 服务端应用处理完剩余数据后发送 FIN，进入 `LAST_ACK`。
4. 客户端收到 FIN，回复 ACK，进入 `TIME_WAIT`。
5. 服务端收到 ACK 后进入 `CLOSED`，客户端等待 2MSL 后进入 `CLOSED`。

四次挥手通常不能合并，是因为服务端收到 FIN 后可以先由内核回复 ACK，但应用层可能还有数据要发送，不能立刻发送 FIN。如果服务端没有剩余数据，ACK 和 FIN 在特定条件下也可能合并，表现为三次挥手。

异常情况：

- 被动关闭方一直不发 FIN：主动方收到 ACK 后进入 `FIN_WAIT_2`；如果是 `shutdown` 只关闭写方向，仍可接收数据。孤儿连接不会无限等待，Linux 通常由 `tcp_fin_timeout` 控制。
- 主动方发送 FIN 后：表示自己不再发送数据；如果只 `shutdown` 写方向，仍可接收对端剩余数据。
- 第一个 FIN 丢失：对端收不到 FIN，主动方因收不到 ACK 超时重传；多次失败后关闭连接。

TIME_WAIT 等待 2MSL 的目的：

- 保证最后一个 ACK 丢失时还能回复对方重传的 FIN。
- 让旧连接中的迟到报文自然消失，避免影响相同四元组的新连接。

谁主动关闭连接，谁通常进入 TIME_WAIT。服务端大量 TIME_WAIT 常见于短连接、长连接空闲超时、达到最大请求数或代理策略主动关闭。可以合理配置 Keep-Alive、超时和最大请求数，并排查异常客户端行为。具体 TIME_WAIT 时长取决于操作系统实现，不宜写成固定值。

## TCP 流量控制、拥塞控制、重传和 Keepalive 有什么区别？

**流量控制：**防止发送方把接收方缓冲区打满。TCP 通过接收窗口 `rwnd` 告知还能接收多少数据；窗口为 0 时，发送方暂停发送并定期发送窗口探测包。

**拥塞控制：**防止发送方把网络打满。发送方维护拥塞窗口 `cwnd`，实际发送窗口约为：

```text
send_window = min(rwnd, cwnd)
```

常见算法：

- 慢启动：`cwnd` 较小时快速增长，整体呈指数增长。
- 拥塞避免：到达 `ssthresh` 后改为线性增长。
- 快重传：收到 3 个重复 ACK 时，不等超时就重传疑似丢失的数据。
- 快恢复：快重传后降低窗口并继续探测网络容量；具体实现会因 TCP 拥塞控制算法而不同。

**超时重传：**发送方根据 RTT 样本估算 RTO，超过 RTO 未收到 ACK 就重传，并通常采用指数退避。RTO 不是固定常数，而是由系统根据网络延迟和抖动动态计算。

**TCP Keepalive：**由内核在连接空闲时发送探测包，多次失败后认为连接断开，适合兜底，但默认探测时间通常较长。HTTP 长连接是应用层连接复用机制；业务需要快速发现异常时，通常还要增加应用层心跳，例如 ping/pong。

# 网络层

## IP、IPv4 和 IPv6 有什么区别？

IP 负责在网络层进行寻址和路由，本身不保证可靠传输。IPv4 使用 32 位地址，地址数量有限；IPv6 使用 128 位地址，地址空间更大，并简化了首部、支持扩展首部和自动配置。

IPv6 不等于天然安全，也不意味着应用层一定不需要鉴权、加密和防火墙；迁移时还要考虑双栈、DNS、负载均衡、访问控制和第三方依赖兼容性。

## ARP 协议如何工作？ARP 缓存是什么？

同一局域网通信时，主机知道目标 IP 但不知道 MAC，会广播 ARP Request；目标主机返回 ARP Reply，发送方把 IP 到 MAC 的映射暂存到 ARP 缓存。跨网段通信时，主机通常解析网关的 MAC，而不是目标主机的 MAC。

ARP 没有内置的身份认证，攻击者可以伪造响应造成流量劫持或中间人攻击；可通过静态绑定、交换机安全策略、网络隔离和加密协议降低风险。

## ICMP、ping 和 traceroute 的原理是什么？

`ping` 通常发送 ICMP Echo Request，目标返回 Echo Reply，用于判断连通性和测量往返时延，但被禁 ICMP 时不能据此断定 TCP 服务不可用。

`traceroute` 逐步增加 IP TTL；中间路由器把 TTL 减到 0 后返回 ICMP 超时报文，发送方据此推断路径上的路由节点。它反映的是探测流量的路径，不一定等于业务 TCP/HTTP 的完整路径。

## NAT 有什么作用？有哪些类型？

NAT 把私网地址转换成公网地址，缓解 IPv4 地址不足并隐藏内部地址。家庭和云环境最常见的是 NAPT：多个内网主机通过不同端口共享一个公网 IP。

NAT 会改变地址或端口，增加端到端通信复杂度；入站连接通常需要端口映射、打洞或中继。不能把 NAT 当成完整安全边界，仍需鉴权、防火墙和应用层安全控制。

## 子网掩码和 CIDR 如何进行子网划分？

CIDR 用前缀长度表示网络位，例如 `/24` 表示前 24 位是网络部分，其余是主机部分。划分子网就是借用主机位增加网络前缀；路由器可以通过最长前缀匹配选择更具体的路由。

计算子网时要明确网络地址、广播地址（IPv4）、可用主机范围和是否存在云平台保留地址。生产环境还要预留扩容空间，避免地址规划过紧。

# 网络场景

## 从输入 URL 到页面展示发生了什么？

以访问 HTTPS 网站为例：

1. URL 解析：浏览器解析协议、域名、端口、路径和参数。
2. 缓存检查：检查浏览器缓存、系统缓存、hosts 和本地 DNS 缓存。
3. DNS 解析：把域名解析成 IP。
4. 获取下一跳 MAC：目标在同网段时通过 ARP 获取目标 MAC，不同网段时通过 ARP 获取网关 MAC。
5. TCP 三次握手：和目标服务器建立 TCP 连接。
6. TLS 握手：校验证书、协商密钥，建立 HTTPS 加密通道。
7. 发送 HTTP 请求：浏览器发送请求行、请求头和可选请求体。
8. 服务端处理：经过负载均衡、网关、应用服务、数据库或缓存生成响应。
9. 返回 HTTP 响应：浏览器接收 HTML、CSS、JS、图片等资源。
10. 页面渲染：解析 HTML 构建 DOM，解析 CSS 构建 CSSOM，执行 JS，完成布局、绘制和合成。

## 网页一直转圈，如何定位问题？

按客户端到服务端的链路排查：

1. 客户端环境：能否访问其他网站，网络、代理、VPN、DNS 配置是否正常。
2. DNS：域名是否解析成功，解析出的 IP 是否正确。
3. 连通性：能否建立 TCP 连接，端口是否开放，防火墙或安全组是否拦截。
4. TLS：证书是否过期、域名是否匹配、TLS 握手是否失败。
5. HTTP：请求是否发出，状态码是多少，是否卡在 TTFB 或下载阶段。
6. 服务端：查看 Nginx、网关、应用日志，确认是否超时、报错、线程池打满或数据库慢查询。
7. 网络质量：抓包查看丢包、重传、窗口过小和 RTT 过高。

常用工具包括浏览器 Network 面板、`curl -v`、`ping`、`telnet/nc`、`traceroute`、抓包工具和服务端日志。

## 如何判断 Server A 和 Server B 连接正常？出错怎么办？

可以分两层判断：

- TCP 层：开启 TCP Keepalive。空闲一段时间后内核发送探测包，多次失败后认为连接断开，但 Linux 默认探测时间通常较长，适合兜底。
- 应用层：定时发送心跳包，例如 ping/pong，可以按业务需求在秒级发现异常。

| 场景               | 现象                    | 处理                              |
| ------------------ | ----------------------- | --------------------------------- |
| 对端正常关闭       | 收到 FIN，`read` 返回 0 | 关闭 socket，释放资源             |
| 对端进程崩溃或重启 | 可能收到 RST            | 关闭旧连接，重新建立连接          |
| 对端宕机或网络中断 | 无响应，直到超时        | 依赖 Keepalive/心跳发现，触发重连 |
| 网络抖动           | 心跳偶发失败            | 设置合理重试和退避，避免频繁重连  |

## 服务端启动但客户端请求不到，如何排查？

- 地址是否正确：域名、IP、端口、协议和路径是否写错。
- DNS 是否正常：域名是否解析到正确 IP。
- 端口是否监听：服务端用 `ss -lntp` 查看，客户端用 `telnet` 或 `nc` 测试。
- 防火墙和安全组：确认服务器防火墙、云安全组和网络策略放行。
- 服务绑定地址：是否只监听 `127.0.0.1`，导致外部无法访问。
- 代理配置：客户端是否配置了错误代理、VPN 或 hosts。
- 网关/Nginx：检查反向代理、upstream 和健康检查。
- 应用日志：根据 4xx/5xx 状态码继续查参数、权限、路由和服务异常。

常见状态码排查：400 看参数格式，401 看认证，403 看权限，404 看路径，500 看应用日志，502/503/504 看网关和上游服务。

## 服务器 ping 不通但 HTTP 能访问吗？

可能。`ping` 使用 ICMP，HTTP 使用 TCP。服务器、防火墙或云安全组可能禁止 ICMP，但允许 TCP 的 80/443 端口访问。因此排查 HTTP 连通性时不应只依赖 ping，还要使用 `curl`、`telnet`、`nc` 或浏览器请求目标端口。

# 网络与 Web 安全

## 什么是 DDoS？如何防范？

DDoS 是分布式拒绝服务攻击。攻击者控制大量机器同时向目标发起请求，耗尽带宽、连接数、CPU、内存或应用处理能力。

| 类型       | 例子                    | 目标                   |
| ---------- | ----------------------- | ---------------------- |
| 网络层     | UDP Flood、反射放大攻击 | 打满带宽               |
| 传输层     | SYN Flood、连接数攻击   | 打满连接队列或连接资源 |
| 应用层     | HTTP Flood、DNS Flood   | 消耗应用处理能力       |
| TLS/会话层 | SSL 握手攻击            | 消耗加解密和会话资源   |

防范方式：

- 使用 CDN、云防火墙、DDoS 高防和流量清洗，在流量到源站前过滤。
- 扩容带宽和服务集群，提升抗压能力。
- 使用限流、黑白名单、验证码和人机识别。
- 对异常 IP、User-Agent 和路径做访问控制。
- 对 SYN Flood 开启 SYN Cookies、合理调整队列并缩短无效连接占用时间。

## SQL 注入是什么？如何防范？

SQL 注入是攻击者把恶意 SQL 片段拼进用户输入，使服务端执行非预期 SQL，例如绕过登录、拖库或删除数据。

- 使用参数化查询或预编译语句，这是最核心的防护。
- 不把用户输入直接拼接到 SQL 字符串中。
- 对输入做格式校验，例如 ID 必须是数字。
- 最小化数据库账号权限，避免应用账号拥有高危权限。
- 对错误信息做脱敏，避免暴露 SQL 结构。

## CSRF 是什么？如何防范？

CSRF 是跨站请求伪造。用户已登录 A 网站，攻击者诱导用户访问恶意页面，恶意页面向 A 网站发起请求；由于浏览器可能自动携带 A 网站 Cookie，服务端可能误以为是用户本人操作。

- CSRF Token：服务端生成随机 Token，提交时携带并校验。
- SameSite Cookie：设置 `SameSite=Lax` 或 `Strict`，减少跨站请求自动携带 Cookie。
- 校验 Origin/Referer：作为辅助来源校验。
- 重要操作增加二次确认，例如支付、改密。
- 避免用 GET 执行有副作用的操作。

CSP 主要用于缓解 XSS，不是 CSRF 的核心防御手段。

## XSS 是什么？如何防范？

XSS 是跨站脚本攻击。攻击者把恶意脚本注入页面，使脚本在浏览器中执行，从而窃取 Cookie/Token、篡改页面或冒充用户操作。

类型：

- 存储型 XSS：恶意脚本存入数据库，用户访问页面时执行。
- 反射型 XSS：脚本通过 URL 或表单提交，服务端反射回页面执行。
- DOM 型 XSS：前端 JS 不安全地操作 DOM，导致脚本执行。

防范方式：

- 输入校验：限制输入格式和长度。
- 输出编码：HTML、属性、URL、JS 场景分别做正确转义。
- 避免使用 `innerHTML` 等危险 API，必要时使用可信 HTML 清洗库。
- 设置 CSP，限制脚本来源。
- Cookie 设置 `HttpOnly`，降低脚本窃取 Cookie 的风险。
