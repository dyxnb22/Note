# 计算机网络

本页聚焦 TCP/IP 网络的分层、寻址、传输和排障。HTTP、TLS、DNS、Cookie 等 Web 协议集中在 [HTTP 与 Web 协议](04_HTTP与Web协议.md)，Socket 与 Java I/O 见 [Java I/O 与网络编程](../Backend/Java/02_JavaIO与网络编程.md)，Web 攻击与防御见 [应用安全](../Backend/Architecture/应用安全.md)。

## 分层模型

| TCP/IP 层 | 主要职责 | 常见协议或对象 |
| --- | --- | --- |
| 应用层 | 定义应用消息与交互 | HTTP、DNS、SSH、SMTP |
| 传输层 | 端到端进程通信 | TCP、UDP、QUIC |
| 网络层 | 跨网络寻址和路由 | IP、ICMP |
| 链路层 | 同一链路中的帧传输 | Ethernet、Wi-Fi、ARP |

OSI 七层模型更适合解释职责，TCP/IP 模型更接近互联网实际实现。排障时从应用症状出发，逐层检查解析、连接、路由和链路，不必机械从第一层开始。

## 一次数据传输

应用数据向下经过封装：

```text
应用消息
→ TCP Segment / UDP Datagram
→ IP Packet
→ Link Frame
→ 物理信号
```

接收端反向解封装。每层只承诺自己的语义：链路成功不代表 TCP 成功，TCP 成功也不代表业务处理成功。

## IP 与路由

IP 地址标识网络接口在某个网络中的位置。路由器根据目标地址和路由表选择下一跳，通常采用最长前缀匹配。

CIDR 使用前缀长度表示网络，例如 `192.168.1.0/24`。判断两个地址是否在同一子网，需要分别与子网掩码按位与并比较网络号。

IPv6 提供更大的地址空间并简化部分报文头设计，但迁移期间常与 IPv4 双栈共存。应用不应假设地址总能塞入 IPv4 字符串或 32 位整数。

## ARP 与邻居解析

IPv4 在局域网内通过 ARP 把下一跳 IP 映射为 MAC 地址。目标不在本子网时，主机解析的是默认网关的 MAC，而不是远端服务器的 MAC。

ARP 缓存减少广播，但缓存陈旧或冲突会导致局域网通信异常。IPv6 使用 Neighbor Discovery 完成类似职责。

## ICMP、ping 与 traceroute

ICMP 用于错误报告和网络诊断。`ping` 常用 Echo Request/Reply 测试可达性和往返时间；`traceroute` 利用 TTL/Hop Limit 逐跳过期来观察路径。

ping 失败不等于 HTTP 一定失败：网络可能屏蔽 ICMP，但允许 TCP/HTTPS。反过来 ping 成功也只证明部分网络路径可达，不证明端口、TLS 和应用健康。

## NAT

NAT 在地址或端口之间建立映射，使多个内网主机共享公网地址，也常用于边界控制。它会破坏端到端可达性，并给 P2P、回调和入站连接带来额外机制。

排查 NAT 问题时确认源/目标地址、端口映射、连接跟踪表和回程路径是否一致。

## TCP 与 UDP

| 维度 | TCP | UDP |
| --- | --- | --- |
| 连接 | 面向连接 | 无连接 |
| 数据语义 | 有序字节流 | 独立数据报 |
| 可靠性 | 重传、排序、流量与拥塞控制 | 由应用按需实现 |
| 头部与延迟 | 相对更高 | 较低 |
| 典型场景 | HTTP/1.1、HTTP/2、数据库连接 | DNS、实时媒体、QUIC 底层 |

UDP 不是“不可靠所以不能用”。当应用能容忍丢失、自己定义确认，或希望在用户态实现传输策略时，UDP 很合适。HTTP/3 的 QUIC 就构建在 UDP 之上。

## TCP 建立连接

三次握手的核心是双方确认彼此的发送和接收能力，并同步初始序列号：

```text
Client → SYN(seq=x)
Server → SYN(seq=y), ACK(x+1)
Client → ACK(y+1)
```

服务端收到 SYN 后维护半连接状态，第三次 ACK 后进入全连接队列。SYN Flood 会消耗半连接资源，可结合 SYN Cookies、队列调优、限流和流量清洗防护。

## TCP 关闭连接

TCP 是全双工连接，两个方向需要分别关闭，因此常表现为四次挥手。主动关闭方进入 TIME_WAIT，用于吸收旧报文并确保最后 ACK 可以重传。

大量 TIME_WAIT 不一定是故障；要结合连接创建频率、端口范围、长连接复用和服务角色判断。随意缩短 TIME_WAIT 可能引入旧报文污染。

## TCP 可靠性

TCP 通过序列号、确认、校验和、重传、滑动窗口和拥塞控制提供可靠有序字节流。

- 流量控制保护接收方，依据接收窗口。
- 拥塞控制保护网络，依据拥塞窗口和丢包/时延信号。
- 重传处理可能丢失的 Segment。
- Keepalive 探测长时间无数据的连接，不等于业务心跳。

应用仍需请求级 Deadline，因为 TCP 连接正常不代表业务能在可接受时间内返回。

## 粘包与拆包

TCP 不保存应用消息边界。一次 `write` 可能被多次读取，多次写入也可能在一次读取中返回。常见协议 framing：

- 固定长度。
- 分隔符。
- 长度字段 + Payload。
- 自描述协议格式。

所谓“粘包”不是 TCP 错误，而是应用错误地把字节流读取边界当成消息边界。

## 网络排障路径

```text
域名是否解析正确
→ 路由是否可达
→ 目标端口是否监听
→ 防火墙/安全组/NAT 是否允许
→ TCP 是否建连
→ TLS 是否握手
→ HTTP 是否返回
→ 网关和应用是否健康
→ 下游和数据库是否超时
```

常用证据：

- `dig`/`nslookup`：DNS 结果。
- `ip addr`、`ip route`：本机地址和路由。
- `ping`、`traceroute`：基础路径。
- `ss`：监听端口和连接状态。
- `nc`、`curl -v`：端口、TLS 和应用响应。
- `tcpdump`：握手、重传、RST 和报文方向。

## 场景判断

### 服务启动但请求不到

确认进程是否存活、监听的是 `127.0.0.1` 还是 `0.0.0.0`、端口是否正确、防火墙和容器端口是否开放，再检查反向代理路由。

### 连接建立后中断

区分 FIN 正常关闭、RST 异常重置、读写超时、空闲连接回收和进程崩溃。客户端、网关、服务端和下游的超时顺序应逐层递减或按预算统一设计，避免上游先超时而下游继续浪费资源。

### 网页一直转圈

从浏览器 Network 面板判断 DNS、连接、TTFB 还是内容下载慢，再沿 CDN、网关、应用、线程池、连接池和数据库向下定位。

## 验收清单

- 能解释数据经过每一层增加了什么头部和语义。
- 能计算 CIDR 网络范围并判断下一跳。
- 能解释 TCP 三次握手、四次挥手和 TIME_WAIT。
- 能区分流量控制、拥塞控制、重传和 Keepalive。
- 能解释为什么 TCP 没有应用消息边界。
- 能用分层证据定位“能 ping 但不能访问”和“端口通但请求超时”。
