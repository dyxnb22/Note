# HTTP 与 Web 协议

本页聚焦 Web 请求从域名解析、TLS 到 HTTP、缓存和会话的完整链路。TCP/IP 底层见 [计算机网络](Network.md)，认证授权与攻击防护见 [应用安全](../Backend/Architecture/应用安全.md)。

## 从 URL 到响应

```text
解析 URL
→ DNS 查询
→ 建立 TCP 或 QUIC 连接
→ TLS 握手
→ 发送 HTTP 请求
→ CDN/网关/应用处理
→ 返回响应
→ 浏览器缓存、解析和渲染
```

每一步都有独立的缓存、超时和失败方式。性能分析应区分 DNS、连接、TLS、TTFB 和内容下载时间。

## DNS

DNS 把域名映射为地址或其他记录。典型解析会经过浏览器/系统缓存、递归解析器，并按需访问根、顶级域和权威服务器。

常见记录：

| 类型 | 作用 |
| --- | --- |
| A | 域名到 IPv4 地址 |
| AAAA | 域名到 IPv6 地址 |
| CNAME | 一个域名指向另一个域名 |
| MX | 邮件服务器 |
| TXT | 验证和策略文本 |
| NS | 指定权威 DNS |

DNS 通常先使用 UDP；响应过大、截断或区域传送等场景会使用 TCP。现代 DNS 也可以通过 TLS 或 HTTPS 加密传输。

TTL 决定缓存时间。变更域名记录前应提前降低 TTL，并考虑递归解析器和客户端缓存不会瞬间全部刷新。

## HTTP 消息

请求由方法、目标、版本、Header 和可选 Body 组成；响应由状态行、Header 和 Body 组成。Header 定义内容类型、缓存、认证、连接和协商等元数据。

没有 `Content-Length` 或分块/帧边界时，接收方无法可靠判断消息结束。HTTP/2 和 HTTP/3 使用二进制帧，不再依赖 HTTP/1.1 的文本消息边界。

## 方法与幂等性

| 方法 | 常见语义 | 是否应幂等 |
| --- | --- | --- |
| GET | 获取资源 | 是 |
| POST | 创建或执行命令 | 通常否 |
| PUT | 用完整表示替换资源 | 是 |
| PATCH | 部分修改 | 取决于操作定义 |
| DELETE | 删除资源 | 语义上是 |

幂等性描述重复执行的业务效果，不代表响应必须完全相同。支付、下单等 POST 可以通过幂等键实现可安全重试。

## 状态码

- 2xx：请求成功处理。
- 3xx：重定向或缓存协商。
- 4xx：客户端请求、认证或权限问题。
- 5xx：服务端无法完成请求。

不要所有失败都返回 200，也不要用状态码替代稳定的业务错误合同。重试逻辑应结合方法幂等性、错误分类和 `Retry-After`。

## HTTP 缓存

缓存分为强缓存和协商缓存：

- `Cache-Control: max-age`、`s-maxage` 控制新鲜时间。
- `no-cache` 表示使用前必须重新验证，不等于完全不存储。
- `no-store` 表示不应存储。
- `ETag` / `If-None-Match` 通过实体标签验证。
- `Last-Modified` / `If-Modified-Since` 通过时间验证。
- `Vary` 告诉共享缓存哪些请求头影响表示。

带用户隐私或认证信息的响应要谨慎配置共享缓存，避免跨用户泄露。

## HTTPS 与 TLS

HTTPS 是 HTTP 运行在 TLS 保护的连接之上。TLS 提供服务器身份认证、传输机密性和完整性。

握手大致包括：协商版本和算法、服务器发送证书、客户端验证域名和信任链、双方完成密钥交换并派生会话密钥。证书有效不只看是否过期，还要看域名、信任链、用途和撤销状态。

TLS 终止点可能在 CDN、负载均衡或应用。终止后的内网链路是否再次加密，应按信任边界和合规要求设计。

## HTTP 版本

| 版本 | 关键特点 |
| --- | --- |
| HTTP/1.1 | 持久连接、文本协议；并发常依赖多连接 |
| HTTP/2 | 二进制分帧、单连接多路复用、头部压缩 |
| HTTP/3 | 基于 QUIC/UDP，连接和流恢复更灵活 |

HTTP/2 解决应用层队头阻塞，但仍受 TCP 丢包影响；HTTP/3 让不同流在传输层更独立。版本升级并不消除服务端处理和下游瓶颈。

## Cookie、Session 与 Token

| 机制 | 保存位置 | 主要作用 |
| --- | --- | --- |
| Cookie | 浏览器，随匹配请求发送 | 保存会话标识和少量客户端状态 |
| Session | 服务端 | 保存会话状态，Cookie 常只存 Session ID |
| Token | 客户端持有凭证 | API 认证与授权上下文 |
| JWT | 自包含 Token 格式 | 携带可验证 Claims |

JWT 不等于天然安全或无需服务端状态。泄露后在到期前可能一直有效，因此需要短生命周期、密钥轮换、受众校验，必要时维护撤销或会话状态。

Cookie 常用安全属性：`Secure`、`HttpOnly`、`SameSite`、合理的 Domain/Path 和过期时间。敏感 Token 不应暴露给不必要的 JavaScript。

## CORS

CORS 是浏览器对跨源读取的限制和授权机制，不是服务器间访问控制。服务端通过允许的 Origin、Method 和 Header 响应；复杂请求可能先发预检 `OPTIONS`。

带凭证请求不能把允许 Origin 简单设为 `*`。允许列表要精确，缓存预检时也要考虑 `Vary: Origin`。

## 长连接、Streaming 与 WebSocket

HTTP Keep-Alive 复用连接，不代表服务器会主动推送。SSE 提供服务器到浏览器的单向事件流；WebSocket 在握手后提供全双工消息通道。

长连接需要单独处理心跳、空闲超时、断线重连、消息顺序、背压和发布期间的连接迁移。

## 断点续传

客户端通过 `Range` 请求部分字节，服务器以 `206 Partial Content` 和 `Content-Range` 返回。资源变更时应使用 ETag 或 `If-Range` 避免把不同版本拼接在一起。

## 排障清单

- DNS 是否解析到预期地址？
- TCP/QUIC 和 TLS 分别耗时多久？
- 证书域名和信任链是否正确？
- 请求是否被浏览器 CORS、Mixed Content 或 Cookie 策略拦截？
- CDN/代理是否错误缓存、重写 Host 或丢失 Header？
- 状态码、TTFB 和响应体能否定位到具体层？
- 客户端超时后，服务端是否仍继续执行副作用？

## 验收清单

- 能从 URL 画出 DNS、连接、TLS、HTTP 和应用处理链路。
- 能解释 `no-cache` 与 `no-store` 的区别。
- 能区分 Cookie、Session、Token 和 JWT。
- 能说明 HTTP/2 与 HTTP/3 分别解决了什么问题。
- 能设计带幂等键、稳定错误合同和缓存语义的 API。
