# 用户态和内核态
<details>
<summary>用户态和内核态的区别？</summary>
	- 用户态运行普通应用程序，权限受限，不能直接访问硬件和内核数据结构。
	- 内核态运行操作系统内核代码，权限最高，可以管理 CPU、内存、磁盘、网络等硬件资源。
	- 应用需要读写文件、网络通信、创建进程等能力时，需要通过系统调用从用户态切换到内核态。
	- 这种隔离可以保护系统稳定性：用户程序崩溃通常只影响自己，不会直接破坏内核。
	- 代价是用户态和内核态切换有成本，要保存现场、切换权限和执行内核逻辑。
</details>
# 进程管理
<details>
<summary>线程和进程的区别是什么？</summary>
	- 进程是资源分配的基本单位，拥有独立地址空间、文件描述符、内存映射等资源。
	- 线程是 CPU 调度的基本单位，同一进程内多个线程共享进程资源，但各自有栈、寄存器上下文和线程局部数据。
	- 进程之间隔离性强，一个进程崩溃通常不会影响另一个进程；线程共享地址空间，一个线程越界写内存可能拖垮整个进程。
	- 线程创建和切换通常比进程更轻量，但多线程需要处理同步、竞态和死锁问题。
</details>
<details>
<summary>进程、线程、协程的区别是什么？</summary>
	- 进程：操作系统资源分配单位，隔离性最好，创建和切换成本最高。
	- 线程：操作系统调度单位，共享进程资源，适合并发执行 CPU 或 IO 任务。
	- 协程：用户态轻量级执行单元，由程序或运行时调度，切换不需要内核参与，成本更低。
	- 协程适合高并发 IO 场景，但如果执行阻塞系统调用或 CPU 密集任务，也可能阻塞调度线程。
	- 面试可总结：进程重在隔离，线程重在并行，协程重在轻量并发。
</details>
<details>
<summary>为什么进程崩溃不会对其他进程产生很大影响？</summary>
	- 每个进程有独立的虚拟地址空间，不能直接读写其他进程的内存。
	- CPU 的内存保护和页表权限会阻止非法访问，越界访问通常只导致当前进程异常退出。
	- 文件、socket 等资源也由内核统一管理，进程退出后内核会回收资源。
	- 但如果崩溃进程持有共享资源，例如共享内存、锁、数据库事务，仍可能间接影响其他进程，所以工程上还要做超时、回收和故障隔离。
</details>
<details>
<summary>进程是分配资源的基本单位，这里的资源指什么？</summary>
	- 独立虚拟地址空间，包括代码段、数据段、堆、栈、内存映射区。
	- 打开的文件描述符、socket、管道、共享内存映射等内核对象。
	- 进程控制块 PCB 中记录的 pid、状态、优先级、调度信息、信号处理信息。
	- 账号权限、工作目录、环境变量、命名空间等运行上下文。
	- CPU 时间片不是进程独占资源，而是由调度器分配给线程执行。
</details>
<details>
<summary>为什么进程之下还要设计线程？</summary>
	- 进程隔离好但创建和切换成本高；线程共享进程资源，创建和切换更轻量。
	- 一个进程内可以有多个执行流，例如 Web 服务同时处理多个请求。
	- 多线程可以让 IO 等待和 CPU 计算重叠，提高资源利用率。
	- 多核 CPU 下，多线程可以真正并行执行。
	- 代价是共享内存会带来数据竞争，需要锁、条件变量、原子操作等同步机制。
</details>
<details>
<summary>多线程比单线程的优势和劣势？</summary>
	- 优势：提高并发能力；IO 等待时其他线程可以继续工作；多核下可以并行；任务拆分后响应更快。
	- 劣势：线程切换有开销；共享数据需要同步；容易出现竞态、死锁、活锁；调试复杂度更高。
	- CPU 密集任务线程数通常接近 CPU 核数更合适；IO 密集任务可以更多，但也要受连接数、内存和下游能力限制。
</details>
<details>
<summary>多线程是不是越多越好？太多会有什么问题？</summary>
	- 不是。线程太多会导致上下文切换频繁，CPU 时间浪费在调度上。
	- 每个线程都有栈和内核调度结构，会占用内存。
	- 线程过多还会增加锁竞争、缓存失效、调试难度和下游压力。
	- 合理线程数要看任务类型：CPU 密集看核数，IO 密集看等待时间、吞吐目标和下游容量。
</details>
<details>
<summary>进程切换和线程切换的区别？</summary>
	- 进程切换需要切换地址空间、页表、内核栈、寄存器上下文等，成本更高。
	- 同一进程内线程切换通常不需要切换地址空间，只需要切换线程栈、寄存器、程序计数器等执行上下文。
	- 如果是不同进程的线程之间切换，本质上也涉及进程地址空间切换。
	- 线程切换虽然更轻，但频繁切换仍会带来调度开销和 CPU cache/TLB 命中率下降。
</details>
<details>
<summary>线程切换为什么比进程切换快，节省了什么资源？</summary>
	- 同一进程内线程共享虚拟地址空间，不需要切换页表。
	- 页表不变意味着 TLB 不需要大规模失效，地址转换缓存更容易保留。
	- 文件描述符、内存映射等进程资源也不需要重新切换。
	- 主要只保存和恢复寄存器、栈指针、程序计数器等线程执行现场。
</details>
<details>
<summary>线程切换详细过程是怎样的？上下文保存在哪里？</summary>
	- 发生时机通常是时间片耗尽、线程阻塞、主动让出 CPU 或更高优先级线程就绪。
	- CPU 进入内核调度逻辑，保存当前线程寄存器、程序计数器、栈指针等上下文。
	- 上下文一般保存在该线程的内核栈或线程控制块 TCB 中。
	- 调度器选择下一个可运行线程，恢复它的上下文，再从上次暂停的位置继续执行。
	- 如果切换到不同进程的线程，还需要切换地址空间相关信息。
</details>
<details>
<summary>进程的五种状态以及如何切换？</summary>
	- 新建：进程刚创建，内核初始化 PCB 和资源。
	- 就绪：具备运行条件，等待 CPU 调度。
	- 运行：正在 CPU 上执行。
	- 阻塞：等待 IO、锁、事件或信号，暂时不能运行。
	- 终止：执行结束或异常退出，等待资源回收。
	- 常见切换：就绪 -> 运行由调度器分配 CPU；运行 -> 就绪通常是时间片耗尽；运行 -> 阻塞是等待事件；阻塞 -> 就绪是事件完成。
</details>
<details>
<summary>进程上下文有哪些？</summary>
	- CPU 上下文：寄存器、程序计数器、栈指针、状态寄存器。
	- 内存上下文：页表、虚拟地址空间、内存映射信息。
	- 内核上下文：PCB、调度信息、打开文件表、信号处理、权限信息。
	- 切换进程时要保存旧进程上下文并恢复新进程上下文，因此比函数调用或线程切换更重。
</details>
<details>
<summary>进程间通信有哪些方式？</summary>
	- 管道：适合有亲缘关系进程之间的单向字节流通信。
	- 命名管道 FIFO：可用于无亲缘关系进程。
	- 消息队列：内核维护消息链表，按消息传递，解耦但有内核拷贝成本。
	- 共享内存：多个进程映射同一段物理内存，速度最快，但需要信号量或锁同步。
	- 信号：用于异步通知，例如终止、暂停、用户自定义事件。
	- socket：可用于同机或跨机器通信，通用性强。
</details>
<details>
<summary>管道有几种方式？</summary>
	- 匿名管道：通常用于父子进程，半双工，生命周期随进程和文件描述符结束。
	- 命名管道 FIFO：在文件系统中有名字，可以让无亲缘关系进程通信。
	- 管道本质是内核缓冲区，一端写、一端读；如果缓冲区满，写端可能阻塞；如果没有数据，读端可能阻塞。
	- 管道适合简单流式通信，不适合复杂结构化消息和大规模高性能通信。
</details>
<details>
<summary>信号和信号量有什么区别？</summary>
	- 信号 signal 是一种异步通知机制，例如 `SIGKILL`、`SIGTERM`、`SIGINT`，用于通知进程发生了某个事件。
	- 信号量 semaphore 是同步原语，用计数器控制多个进程或线程对共享资源的访问。
	- 信号强调“通知事件”，信号量强调“资源计数和互斥同步”。
	- 举例：Ctrl+C 给进程发的是信号；限制最多 10 个线程访问连接池可以用信号量。
</details>
<details>
<summary>共享内存怎么实现的？</summary>
	- 内核创建一段共享内存对象，并把同一段物理内存映射到多个进程的虚拟地址空间。
	- 多个进程读写这段内存时不需要经过内核拷贝，所以速度很快。
	- 共享内存本身不提供同步，必须配合互斥锁、信号量、条件变量等机制。
	- 适合大数据量、高性能 IPC，但要小心并发写、内存可见性和资源回收。
</details>
<details>
<summary>线程间通信有什么方式？</summary>
	- 共享变量：同一进程内线程共享地址空间，可以直接读写共享数据。
	- 互斥锁：保护临界区，避免多个线程同时修改共享数据。
	- 条件变量：一个线程等待条件成立，另一个线程修改状态后通知。
	- 信号量：控制并发访问数量，适合资源池。
	- 读写锁、屏障、事件、阻塞队列也常用于线程协作。
</details>
<details>
<summary>除了互斥锁，还知道哪些锁？分别应用于什么场景？</summary>
	- 自旋锁：线程不阻塞，循环等待锁释放，适合临界区极短且不希望线程睡眠的场景。
	- 读写锁：读多写少时允许多个读线程并发，写线程独占。
	- 悲观锁：认为冲突常发生，先加锁再操作，例如数据库行锁。
	- 乐观锁：认为冲突少，提交时用版本号或 CAS 校验，例如库存扣减版本号。
	- 公平锁：按等待顺序获取锁，避免饥饿，但吞吐可能较低。
	- 可重入锁：同一线程持有锁后可以再次获得，避免递归调用或嵌套调用死锁。
</details>
<details>
<summary>进程调度算法有哪些？</summary>
	- 先来先服务 FCFS：实现简单，但短任务可能被长任务阻塞。
	- 短作业优先 SJF：平均等待时间短，但可能导致长任务饥饿。
	- 时间片轮转 RR：每个进程轮流运行一个时间片，适合分时系统。
	- 优先级调度：优先级高的进程先运行，但要防止低优先级饥饿。
	- 多级反馈队列：综合时间片和优先级，交互式任务响应好，通用操作系统常见。
</details>
# 锁
<details>
<summary>为什么并发执行线程要加锁？</summary>
	- 多个线程同时读写共享变量时，操作可能交错执行，导致结果不确定。
	- 加锁可以把临界区变成互斥执行，保证同一时刻只有一个线程修改共享状态。
	- 锁还提供一定的内存可见性语义，确保一个线程的修改对另一个线程可见。
	- 但锁会降低并发度，所以要控制锁粒度、持锁时间和锁范围。
</details>
<details>
<summary>自旋锁是什么？应用在哪些场景？</summary>
	- 自旋锁获取失败时不会睡眠，而是不断循环检查锁是否释放。
	- 优点是避免线程阻塞和唤醒的上下文切换成本。
	- 缺点是等待期间持续占用 CPU。
	- 适合锁持有时间非常短、CPU 资源充足、内核或高性能并发场景。
	- 不适合临界区长或单核 CPU 场景，否则会浪费大量 CPU。
</details>
<details>
<summary>死锁发生条件是什么？</summary>
	- 互斥条件：资源一次只能被一个线程或进程占有。
	- 请求并保持：已经持有资源的同时继续请求其他资源。
	- 不可剥夺：资源不能被强制抢占，只能由持有者释放。
	- 循环等待：多个线程或进程形成资源等待环。
	- 四个条件同时满足才可能死锁，破坏任意一个条件都能预防死锁。
</details>
<details>
<summary>如何避免死锁？</summary>
	- 固定加锁顺序，避免循环等待。
	- 尽量一次性申请所需资源，减少请求并保持。
	- 使用 tryLock 和超时机制，获取失败主动释放已持有锁。
	- 缩小锁粒度和持锁时间，避免锁内调用外部不可控方法。
	- 通过资源分级、死锁检测、监控线程栈来发现和恢复问题。
</details>
# 内存管理
<details>
<summary>介绍一下操作系统内存管理</summary>
	- 操作系统负责给进程分配和回收内存，并提供虚拟内存抽象。
	- 虚拟内存让每个进程以为自己独占连续地址空间，实际由页表映射到物理内存。
	- 内存管理还包括权限保护、内存共享、页面置换、缺页异常、缓存和 swap。
	- 目标是隔离进程、提高内存利用率、支持大于物理内存的程序运行。
</details>
<details>
<summary>什么是虚拟内存和物理内存？</summary>
	- 物理内存是真实 RAM 地址空间，由硬件实际提供。
	- 虚拟内存是操作系统给进程看到的逻辑地址空间。
	- CPU 访问虚拟地址时，通过 MMU 和页表转换为物理地址。
	- 虚拟内存带来隔离、权限控制、按需加载、共享内存和换页能力。
</details>
<details>
<summary>讲一下页表</summary>
	- 页表记录虚拟页到物理页框的映射关系。
	- 每个页表项通常包含物理页号、有效位、权限位、脏位、访问位等信息。
	- 多级页表可以节省页表内存，因为没有使用的虚拟地址区间不需要分配完整页表。
	- TLB 是页表缓存，用来加速虚拟地址到物理地址的转换。
</details>
<details>
<summary>虚拟地址是怎么转化到物理地址的？</summary>
	- CPU 生成虚拟地址，虚拟地址拆成虚拟页号和页内偏移。
	- MMU 先查 TLB，如果命中，直接得到物理页框号。
	- TLB 未命中则查页表，找到页表项后得到物理页框号并更新 TLB。
	- 物理地址 = 物理页框号 + 页内偏移。
	- 如果页表项无效，会触发缺页异常，由操作系统把页面从磁盘加载到内存或进行异常处理。
</details>
<details>
<summary>程序的内存布局是怎么样的？</summary>
	- 代码段：存放程序机器指令，通常只读。
	- 数据段：存放已初始化的全局变量和静态变量。
	- BSS 段：存放未初始化的全局变量和静态变量。
	- 堆：动态分配内存，由 `malloc/new` 申请，向高地址增长。
	- 栈：函数调用栈、局部变量、返回地址，通常向低地址增长。
	- 内存映射区：动态库、mmap 文件、共享内存等。
</details>
<details>
<summary>堆和栈的区别？</summary>
	- 栈由编译器和运行时自动管理，函数调用进入时分配，返回时释放。
	- 堆由程序员或垃圾回收器管理，生命周期更灵活。
	- 栈空间通常较小，分配释放快；堆空间较大，但分配释放成本更高，可能产生碎片。
	- 栈适合局部变量和函数调用现场；堆适合跨函数、大小动态、生命周期不确定的数据。
</details>
# 中断
<details>
<summary>什么是中断？</summary>
	- 中断是 CPU 暂停当前执行流，转去处理特定事件的机制。
	- 它让 CPU 不必一直轮询设备状态，而是在设备或异常发生时被通知。
	- 中断处理完成后，CPU 通常会恢复原来的执行现场继续运行。
	- 中断是操作系统响应硬件、异常和系统调用的重要基础。
</details>
<details>
<summary>讲讲中断的流程</summary>
	- 设备、异常或软件指令触发中断。
	- CPU 保存当前执行现场，例如程序计数器、状态寄存器等。
	- CPU 根据中断向量表找到对应的中断处理程序。
	- 切换到内核态执行中断处理逻辑。
	- 处理完成后恢复现场，返回被中断的位置继续执行。
	- 对耗时任务，内核通常会把中断处理拆成上半部和下半部，先快速响应，再延后处理复杂逻辑。
</details>
# 网络 IO
<details>
<summary>你了解哪些 IO 模型？</summary>
	- 阻塞 IO：调用 read 后线程阻塞，直到数据准备好并复制完成。
	- 非阻塞 IO：调用立即返回，没数据就返回错误码，应用需要不断轮询。
	- IO 多路复用：一个线程通过 select/poll/epoll 同时监听多个 fd，哪个就绪处理哪个。
	- 信号驱动 IO：内核通过信号通知应用 fd 就绪。
	- 异步 IO：应用提交 IO 请求后继续执行，内核完成数据准备和拷贝后通知应用。
	- 面试重点：同步/异步看数据拷贝阶段是否由调用方等待；阻塞/非阻塞看调用时线程是否等待。
</details>
<details>
<summary>select、poll、epoll 的区别是什么？</summary>
	- select 使用固定大小 fd_set，能监听的 fd 数量有限；每次调用都要拷贝 fd 集合，并线性扫描。
	- poll 用数组保存 fd，突破了 select 的数量限制，但仍然需要线性扫描。
	- epoll 在内核维护关注的 fd 集合，通过事件回调把就绪 fd 放入就绪队列，应用只取就绪事件。
	- epoll 更适合大量连接、少量活跃的场景，避免每次全量扫描。
	- select/poll 是水平触发语义为主；epoll 支持水平触发和边缘触发。
