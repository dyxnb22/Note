# 用户态和内核态

## 用户态和内核态有什么区别？

- 用户态运行普通应用程序，权限受限，不能直接访问硬件和内核数据结构。
- 内核态运行操作系统内核代码，权限最高，可以管理 CPU、内存、磁盘、网络等硬件资源。
- 应用需要读写文件、网络通信、创建进程等能力时，需要通过系统调用从用户态进入内核态。
- 这种隔离可以保护系统稳定性：用户程序崩溃通常只影响自己，不会直接破坏内核。
- 代价是用户态和内核态切换有成本，需要保存现场、切换权限并执行内核逻辑。

# 进程与线程

## 进程、线程和协程有什么区别？

| 对象 | 定位 | 资源与调度 | 特点 |
| ---- | ---- | ---------- | ---- |
| 进程 | 资源分配和隔离单位 | 拥有独立地址空间、文件描述符、内存映射等资源 | 隔离性最好，创建和切换成本最高 |
| 线程 | 操作系统调度单位 | 共享进程资源，但有自己的栈、寄存器上下文和线程局部数据 | 轻量，适合并发执行 CPU 或 I/O 任务 |
| 协程 | 用户态轻量执行单元 | 由程序或运行时调度，切换通常不需要内核参与 | 切换成本低，适合高并发 I/O；阻塞调用或 CPU 密集任务仍可能阻塞调度线程 |

面试可以概括为：进程重在隔离，线程重在并行，协程重在轻量并发。

## 进程拥有哪些资源？为什么进程崩溃通常不影响其他进程？

进程资源包括：

- 独立虚拟地址空间，包括代码段、数据段、堆、栈和内存映射区。
- 打开的文件描述符、socket、管道、共享内存映射等内核对象。
- PCB 中记录的 PID、状态、优先级、调度信息和信号处理信息。
- 账号权限、工作目录、环境变量、命名空间等运行上下文。

每个进程有独立的虚拟地址空间，CPU 的内存保护和页表权限会阻止非法访问，因此越界访问通常只导致当前进程异常退出。文件、socket 等资源由内核管理，进程退出后内核会回收。

但如果崩溃进程持有共享内存、锁或数据库事务，仍可能间接影响其他进程，所以工程上还要做超时、回收和故障隔离。CPU 时间片不是进程独占资源，而是由调度器分配给线程执行。

## 为什么进程之下还要设计线程？多线程是不是越多越好？

线程相比进程更轻量，同一进程内可以共享资源并创建多个执行流：

- I/O 等待时，其他线程可以继续工作，提高资源利用率。
- 多核 CPU 下，多线程可以真正并行执行。
- Web 服务可以在一个进程内同时处理多个请求。

多线程的代价：

- 线程切换、线程栈和调度结构会占用 CPU 与内存。
- 共享数据需要同步，容易出现竞态、死锁和活锁。
- 线程过多会增加上下文切换、锁竞争、缓存失效、调试难度和下游压力。

CPU 密集任务的线程数通常接近 CPU 核数；I/O 密集任务可以更多，但要受等待时间、连接数、内存和下游容量限制。

## 进程切换和线程切换有什么区别？

进程切换需要切换地址空间、页表、内核栈和寄存器上下文，成本较高。同一进程内的线程切换通常不需要切换地址空间，只需要切换线程栈、寄存器、程序计数器等执行上下文。

线程切换更快主要是因为：

- 共享虚拟地址空间，不需要切换页表。
- 页表不变，TLB 不需要大规模失效，地址转换缓存更容易保留。
- 文件描述符、内存映射等进程资源不需要重新切换。

如果切换到不同进程的线程，本质上仍然涉及进程地址空间切换。

线程切换通常发生在时间片耗尽、线程阻塞、主动让出 CPU 或更高优先级线程就绪时。CPU 进入内核调度逻辑，把寄存器、程序计数器和栈指针等上下文保存到线程的内核栈或 TCB 中；调度器选择下一个可运行线程，恢复其上下文后继续执行。

## 进程有哪些状态？上下文包含什么？

进程常见五种状态：

- 新建：刚创建，内核正在初始化 PCB 和资源。
- 就绪：具备运行条件，等待 CPU 调度。
- 运行：正在 CPU 上执行。
- 阻塞：等待 I/O、锁、事件或信号，暂时不能运行。
- 终止：执行结束或异常退出，等待资源回收。

常见状态切换：就绪 → 运行由调度器分配 CPU；运行 → 就绪通常是时间片耗尽；运行 → 阻塞是等待事件；阻塞 → 就绪是事件完成。

进程上下文包括：

- CPU 上下文：寄存器、程序计数器、栈指针和状态寄存器。
- 内存上下文：页表、虚拟地址空间和内存映射信息。
- 内核上下文：PCB、调度信息、打开文件表、信号处理和权限信息。

## 进程调度算法有哪些？

- 先来先服务（FCFS）：实现简单，但短任务可能被长任务阻塞。
- 短作业优先（SJF）：平均等待时间短，但可能导致长任务饥饿。
- 时间片轮转（RR）：每个进程轮流运行一个时间片，适合分时系统。
- 优先级调度：优先级高的进程先运行，但要防止低优先级饥饿。
- 多级反馈队列：综合时间片和优先级，交互式任务响应好，通用操作系统常见。

# 进程间通信

## 进程间通信有哪些方式？

- 管道：适合有亲缘关系进程之间的单向字节流通信。
- 命名管道 FIFO：可用于无亲缘关系进程。
- 消息队列：内核维护消息链表，按消息传递，解耦但有内核拷贝成本。
- 共享内存：多个进程映射同一段物理内存，速度快，但需要信号量或锁同步。
- 信号：用于异步通知，例如终止、暂停和用户自定义事件。
- Socket：可用于同机或跨机器通信，通用性强。

## 管道、信号、信号量和共享内存有什么区别？

**管道：**本质是内核缓冲区，一端写、一端读；缓冲区满时写端可能阻塞，没有数据时读端可能阻塞。匿名管道通常用于父子进程，半双工；FIFO 在文件系统中有名字，可供无亲缘关系进程使用。管道适合简单流式通信，不适合复杂结构化消息和大规模高性能通信。

**信号：**一种异步通知机制，例如 `SIGKILL`、`SIGTERM`、`SIGINT`，用于通知进程发生了某个事件。

**信号量：**一种同步原语，用计数器控制多个进程或线程对共享资源的访问。Ctrl+C 发送的是信号；限制最多 10 个线程访问连接池可以使用信号量。

**共享内存：**内核创建共享内存对象，把同一段物理内存映射到多个进程的虚拟地址空间。多个进程读写时不需要经过内核拷贝，但共享内存本身不提供同步，必须配合互斥锁、信号量或条件变量。

## 线程间通信有哪些方式？

- 共享变量：同一进程内线程共享地址空间，可以直接读写共享数据。
- 互斥锁：保护临界区，避免多个线程同时修改共享数据。
- 条件变量：一个线程等待条件成立，另一个线程修改状态后通知。
- 信号量：控制并发访问数量，适合资源池。
- 读写锁、屏障、事件和阻塞队列：用于不同形式的线程协作。

# 同步与锁

## 为什么并发执行线程要加锁？

多个线程同时读写共享变量时，操作可能交错执行，导致结果不确定。加锁可以把临界区变成互斥执行，并提供一定的内存可见性语义，确保一个线程的修改对另一个线程可见。

锁会降低并发度，因此要控制锁粒度、持锁时间和锁范围，避免在锁内调用外部不可控方法。

## 自旋锁、读写锁、乐观锁和悲观锁有什么区别？

- 自旋锁：获取失败时不睡眠，而是循环等待锁释放；适合临界区极短、CPU 资源充足的场景，不适合长临界区或单核 CPU。
- 读写锁：读多写少时允许多个读线程并发，写线程独占。
- 悲观锁：认为冲突经常发生，操作前先加锁，例如数据库行锁。
- 乐观锁：认为冲突较少，提交时通过版本号、时间戳或 CAS 校验是否被修改；失败后需要重试或返回冲突。
- 公平锁：按等待顺序获取锁，避免饥饿，但吞吐可能较低。
- 可重入锁：同一持有者已经持有锁时可以再次获得，避免递归或嵌套调用死锁。

悲观锁适合冲突频繁、写多的场景；乐观锁适合读多写少、冲突较少的场景，但重试过多也会影响性能。

## 死锁发生的条件是什么？如何避免？

死锁的四个必要条件：

- 互斥：资源一次只能被一个线程或进程占有。
- 请求并保持：已经持有资源的同时继续请求其他资源。
- 不可剥夺：资源不能被强制抢占，只能由持有者释放。
- 循环等待：多个线程或进程形成资源等待环。

避免方式：

- 固定加锁顺序，破坏循环等待。
- 尽量一次性申请所需资源，减少请求并保持。
- 使用 `tryLock` 和超时机制，获取失败时主动释放已持有锁。
- 缩小锁粒度和持锁时间。
- 通过资源分级、死锁检测和线程栈监控发现并恢复问题。

四个条件同时满足才可能发生死锁，破坏任意一个条件都能预防死锁。

## 银行家算法是什么？

银行家算法用于避免死锁，在分配资源前判断分配后系统是否仍处于安全状态。

每个进程声明最大资源需求，系统维护已分配、可用和仍需资源。如果尝试分配后还能找到一个安全序列，让所有进程最终完成，就允许分配；否则暂缓分配。

它思想清晰，但实际系统很难提前知道最大资源需求，因此更多用于理解死锁避免。

# 内存管理

## 操作系统如何管理内存？虚拟内存和物理内存有什么区别？

操作系统负责给进程分配和回收内存，并提供虚拟内存抽象。物理内存是真实 RAM 地址空间；虚拟内存是操作系统提供给进程看到的逻辑地址空间。

虚拟内存让每个进程以为自己拥有独立的连续地址空间，实际由页表映射到物理内存。它还提供进程隔离、权限控制、按需加载、共享内存、页面置换和 swap 能力。

## 页表、段表和多级页表是什么？

页表记录虚拟页到物理页框的映射关系。页表项通常包含物理页号、有效位、权限位、脏位和访问位等信息。多级页表只为实际使用的虚拟地址区间分配下级页表，可以节省页表内存。

段表把程序按逻辑划分为代码段、数据段、栈段等，每段有基址和长度，并进行越界和权限检查。分段符合程序逻辑结构，但容易产生外部碎片；现代系统更多使用分页或段页式结合。

## 虚拟地址如何转换为物理地址？

1. CPU 生成虚拟地址，将其拆成虚拟页号和页内偏移。
2. MMU 先查 TLB；命中时直接得到物理页框号。
3. TLB 未命中时查页表，找到页表项后得到物理页框号并更新 TLB。
4. 物理地址 = 物理页框号 + 页内偏移。
5. 如果页表项无效，会触发缺页异常，由操作系统加载页面或进行异常处理。

TLB 是页表缓存，用于加速虚拟地址到物理地址的转换。

## 什么是缺页异常？触发后会发生什么？

当进程访问的虚拟页没有有效的物理页映射，或访问权限不符合页表设置时，会触发缺页异常。CPU 进入内核态，操作系统判断这是合法的按需加载、换入 swap，还是非法访问；如果需要页面，则准备物理页、更新页表并重新执行被中断的指令，否则向进程报告异常。

## 程序的内存布局是什么样的？堆和栈有什么区别？

常见内存区域：

- 代码段：存放程序机器指令，通常只读。
- 数据段：存放已初始化的全局变量和静态变量。
- BSS 段：存放未初始化的全局变量和静态变量。
- 堆：动态分配内存，由 `malloc/new` 申请，生命周期灵活。
- 栈：函数调用栈、局部变量和返回地址，通常由运行时自动管理。
- 内存映射区：动态库、mmap 文件和共享内存等。

栈分配释放快，但空间通常较小；堆空间较大、生命周期更灵活，但分配释放成本更高，也可能产生碎片。堆和栈的增长方向、具体布局取决于平台和运行时，不宜写成所有系统都固定一致。

## `fork()` 和 Copy-on-Write 如何工作？

`fork()` 创建子进程，子进程拥有父进程虚拟地址空间、文件描述符和信号处理等状态的逻辑副本。现代操作系统不会立即复制所有物理内存，而是使用写时复制（COW）：

- 父子进程开始时共享相同物理页，并把页面标记为只读。
- 任一方写入共享页时，触发缺页异常，内核复制该页后再允许写入。
- 如果 `fork()` 后子进程马上 `exec()`，很多页面不需要复制，可以节省物理内存和 CPU 拷贝时间。
- 代价是第一次写入共享页时会产生缺页异常和页面复制。

## `malloc`、`brk` 和 `mmap` 有什么关系？

- 小块内存通常从堆中已有空闲区域分配，底层可能通过 `brk/sbrk` 扩展堆。
- 大块内存常通过 `mmap` 单独映射，释放时更容易直接归还给操作系统。
- `mmap` 也可把文件或匿名内存映射到进程地址空间，适合大块内存、文件映射和共享内存。
- 具体阈值取决于内存分配器实现和配置。

小对象分配速度快但可能造成堆碎片，大对象使用 mmap 更利于独立释放和管理。

## 操作系统内存不足时会发生什么？页面置换有哪些算法？

内存不足时，操作系统可能回收 page cache，把不常用页面换出到 swap；如果仍不足，Linux 可能触发 OOM Killer，杀掉部分进程释放内存。应用层可能表现为分配失败、延迟升高、频繁换页和系统抖动。

常见页面置换算法：

- FIFO：最早进入内存的页面先淘汰，实现简单但效果不稳定。
- LRU：淘汰最近最久未使用的页面，符合局部性原理，但精确实现成本高。
- LFU：淘汰访问频率最低的页面，适合频率稳定场景。
- Clock：用访问位和循环指针近似 LRU，降低实现成本。
- OPT：淘汰未来最长时间不会访问的页，只能作为理论比较。

# 中断

## 什么是中断？有什么作用？

中断是 CPU 暂停当前执行流，转去处理特定事件的机制。它让 CPU 不必一直轮询设备状态，而是在设备或异常发生时被通知；处理完成后通常恢复原执行现场。

中断可以：

- 提高 CPU 利用率，避免一直轮询设备。
- 支持系统调用，让用户程序安全进入内核请求服务。
- 支持时钟中断，让操作系统实现时间片调度。
- 支持异常处理，例如缺页异常让虚拟内存按需加载成为可能。
- 支持设备驱动响应硬件事件。

## 中断处理流程是什么？

1. 设备、异常或软件指令触发中断。
2. CPU 保存当前执行现场，例如程序计数器和状态寄存器。
3. CPU 根据中断向量表找到对应的处理程序。
4. 切换到内核态执行中断处理逻辑。
5. 处理完成后恢复现场，返回被中断的位置继续运行。

对耗时任务，内核通常把处理拆成上半部和下半部，先快速响应，再延后处理复杂逻辑。

## 中断有哪些类型？

- 硬件中断：外部设备触发，例如网卡收包、磁盘 I/O 完成和定时器中断。
- 软件中断或系统调用入口：程序通过指令主动请求内核服务；具体实现依赖 CPU 架构和操作系统。
- 异常：CPU 执行指令时出现问题，例如缺页异常、除零错误和非法指令。
- 可屏蔽中断：可以被 CPU 暂时屏蔽；不可屏蔽中断通常用于严重硬件故障。

# 网络 I/O

## 常见 I/O 模型有哪些？

- 阻塞 I/O：调用 `read` 后线程阻塞，直到数据准备好并完成拷贝。
- 非阻塞 I/O：调用立即返回，没数据时返回错误码，应用需要再次检查。
- I/O 多路复用：一个线程通过 `select/poll/epoll` 同时监听多个 fd，哪个就绪就处理哪个。
- 信号驱动 I/O：内核通过信号通知应用 fd 就绪。
- 异步 I/O：应用提交 I/O 请求后继续执行，内核完成数据准备和拷贝后通知应用。

同步/异步主要看 I/O 完成是否需要调用方等待或由内核完成后通知；阻塞/非阻塞主要看调用期间线程是否被挂起。两组概念不是同一维度。

## 服务器处理并发请求有哪些方式？

- 多进程：隔离性好、稳定，但进程创建和切换成本高。
- 多线程：比多进程轻量，共享内存方便，但需要处理同步和线程安全。
- I/O 多路复用 + 事件循环：少量线程处理大量连接，适合高并发网络服务。
- Reactor：监听事件，就绪后分发给 Handler 处理，Redis、Nginx、Netty 都有类似思想。
- Proactor：异步 I/O 完成后通知应用处理结果，适合真正异步 I/O 支持较好的系统。

## 什么是 I/O 多路复用？

I/O 多路复用让一个线程同时监听多个文件描述符。当某些 fd 可读或可写时，内核通知应用，应用再处理对应连接。这样不用为每个连接创建线程，可以降低线程数量和上下文切换成本，适合连接数多但单连接活跃度不高的服务。

## `select`、`poll`、`epoll` 有什么区别？

- `select` 使用固定大小的 `fd_set`，能监听的 fd 数量有限；每次调用都要拷贝集合并线性扫描。
- `poll` 用数组保存 fd，突破了 select 的数量限制，但仍需要线性扫描。
- `epoll` 在内核维护关注的 fd 集合，并将就绪事件放入就绪队列，应用只获取就绪事件。
- `epoll` 更适合大量连接、少量活跃的场景，但最终性能仍取决于事件处理和业务逻辑。

`select/poll` 以水平触发语义为主；`epoll` 支持水平触发和边缘触发。

## `epoll` 的水平触发和边缘触发有什么区别？

- 水平触发（LT）：只要 fd 还有数据可读或可写，`epoll_wait` 就会持续通知。
- 边缘触发（ET）：只有 fd 状态发生变化时通知一次。
- ET 通知次数少，但要求应用循环读写直到 `EAGAIN`，否则可能遗漏剩余数据。
- LT 编程更简单，不容易漏事件；ET 更适合高性能服务，但实现要求更高。

## Redis、Nginx、Netty 为什么高性能？

它们都大量依赖事件驱动和 I/O 多路复用，用少量线程处理大量连接，减少线程创建、上下文切换和锁竞争。

- Redis 主要在内存中处理命令，命令执行模型减少了复杂锁竞争；较新版本也使用多线程处理部分网络 I/O。
- Nginx 采用多进程事件驱动模型，worker 之间相对独立，适合高并发反向代理。
- Netty 基于 Reactor 模型和 epoll/kqueue/NIO，封装了高性能网络编程能力。

## 什么是零拷贝？

传统文件发送可能经历“磁盘 → 内核缓冲区 → 用户缓冲区 → socket 缓冲区 → 网卡”，包含多次拷贝和上下文切换。

零拷贝通过 `sendfile`、`mmap` 等机制减少用户态和内核态之间的数据拷贝。典型场景是文件服务器、静态资源发送和 Kafka 日志传输。

零拷贝不是完全没有拷贝，而是减少 CPU 参与的数据复制，尽量让数据在内核缓冲区和网卡之间传递，从而降低 CPU 消耗、减少上下文切换并提升吞吐。
