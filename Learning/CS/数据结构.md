# 数据结构

数据结构的核心不是记住类名，而是理解三件事：数据如何组织、操作要维护什么不变量、不同操作的时间和空间成本是什么。本篇补充底层结构原理；具体题型见 `算法/`，Java 集合实现见 `../Backend/Java/Java集合.md`。

## 复杂度速查

| 结构 | 访问/查找 | 插入/删除 | 关键取舍 |
| --- | --- | --- | --- |
| 数组 | 下标 `O(1)`；按值 `O(n)` | 中间位置 `O(n)` | 连续内存，缓存友好，扩容需要搬迁 |
| 链表 | 按位置 `O(n)` | 已知节点 `O(1)` | 插入灵活，但不支持随机访问 |
| 栈/队列/双端队列 | 队首或栈顶 `O(1)` | 两端 `O(1)` | 分别表达 LIFO、FIFO 和双端操作 |
| 哈希表 | 平均 `O(1)` | 平均 `O(1)` | 依赖哈希质量、容量和冲突处理 |
| 二叉堆 | 堆顶 `O(1)` | `O(log n)` | 只保证堆顶优先级，不保证整体有序 |
| BST | 平均 `O(log n)` | 平均 `O(log n)` | 退化时查找和更新为 `O(n)` |
| AVL/红黑树 | `O(log n)` | `O(log n)` | 用平衡维护最坏复杂度，更新需要旋转或染色 |
| Trie | 按字符串长度 `O(L)` | 按字符串长度 `O(L)` | 与字符串数量弱相关，但空间开销可能较大 |
| 并查集 | 均摊近似 `O(α(n))` | 均摊近似 `O(α(n))` | 擅长连通性，不擅长遍历和删除 |

复杂度结论必须带前提：哈希表的 `O(1)` 是平均情况，BST 的 `O(log n)` 依赖树高，Trie 的 `O(L)` 依赖字符长度和节点存储方式。

## 线性数据结构

### 数组与链表怎么选？

- 数组把元素放在连续内存中，适合随机访问、遍历和批量计算；中间插入或删除需要移动后续元素。
- 链表通过节点和指针连接元素，适合已经定位节点后的插入和删除；查找第 `i` 个元素必须从头遍历。
- 业务代码中不要因为“插入删除多”就默认选链表。如果操作需要按下标访问、缓存局部性或批量遍历，动态数组通常更快。
- 双向链表增加 `prev` 指针，便于向前移动和删除已定位节点，但会增加内存和维护成本。

链表实现的关键边界：空链表、头节点删除、尾节点删除、单节点链表，以及插入/删除后指针是否仍然连通。涉及题型时可结合 [链表](算法/链表.md) 复习。

### 栈、队列和双端队列

| 结构 | 规则 | 常见场景 |
| --- | --- | --- |
| 栈 Stack | 后进先出 LIFO | 括号匹配、递归模拟、撤销、单调栈 |
| 队列 Queue | 先进先出 FIFO | 任务处理、BFS、生产者消费者 |
| 双端队列 Deque | 两端插入和删除 | 单调队列、滑动窗口、同时充当栈和队列 |

数组实现队列时要用循环数组和 `head/tail` 指针，避免每次出队都搬移元素；需要区分“空”和“满”，通常保留一个空槽或额外维护 `size`。

## 哈希表

哈希表通过 `hash(key)` 把 key 映射到桶下标，再在桶内解决冲突。一个可靠实现至少要处理：哈希冲突、扩容、删除、重复 key 和并发访问边界。

### 冲突解决方式

1. **分离链接法**：每个桶保存链表或树。冲突元素仍可插入，删除简单；桶内结构过长时查询会退化。
2. **开放寻址法**：冲突后按探测序列寻找其他空槽，例如线性探测、二次探测。数据集中在数组中，缓存友好，但删除需要墓碑标记或重排。
3. **再哈希/多哈希**：使用多个哈希函数寻找位置，降低聚集概率，但实现和计算成本更高。

### 负载因子和扩容

```text
负载因子 = 已存元素数量 / 桶数量
```

负载因子过高会增加冲突和查询成本；过低会浪费空间。扩容通常是新建更大的桶数组并重新分布元素，成本为 `O(n)`，但可以通过均摊把单次插入成本控制在平均 `O(1)`。

设计哈希 key 时要满足：

- `equals()` 相等的对象必须有相同的 `hashCode()`。
- 作为 key 放入后不要改变参与 `equals/hashCode` 的字段，否则可能再也找不到该 key。
- 哈希函数应尽量让 key 均匀分布；安全场景还要考虑恶意构造冲突。

Java `HashMap` 的数组、链表、树化和扩容细节见 [Java 集合](../Backend/Java/Java集合.md)。

## 堆 Heap

二叉堆是完全二叉树，通常用数组存储。以 0 为起点时：

```text
parent(i) = (i - 1) / 2
left(i)   = 2 * i + 1
right(i)  = 2 * i + 2
```

- 最小堆：父节点不大于子节点，堆顶是最小值。
- 最大堆：父节点不小于子节点，堆顶是最大值。
- 插入后向上调整；删除堆顶后把末尾元素移到堆顶，再向下调整。
- 逐个插入建堆是 `O(n log n)`；自底向上 heapify 建堆是 `O(n)`。

堆只保证“父子之间的优先级”，不保证数组整体有序。因此优先队列、Top K、合并多个有序流和 Dijkstra 等场景适合堆；需要完整排序时应选择排序算法。相关题型见 [堆](算法/堆.md)。

## Trie 字典树

Trie 把字符串的公共前缀合并：根到节点的一条路径对应一个前缀，节点上的结束标记表示某个完整单词。

- 插入、查找和前缀查询的复杂度约为 `O(L)`，`L` 是字符串长度。
- 适合自动补全、词典匹配、前缀统计和路由匹配。
- 节点数量可能接近所有字符串长度之和，字符集较大时空间成本明显。
- 可以用数组保存子节点，访问快但占空间；也可以用 Map，空间更省但常数成本更高。

Trie 的关键是区分“路径存在”和“完整单词存在”：搜索 `app` 时，即使 `a → p → p` 路径存在，也要检查最后节点的 `isWord`。

## 搜索树与平衡树

### 二叉搜索树 BST

BST 的不变量是：任意节点的左子树 key 小于节点，右子树 key 大于节点（重复值策略需要事先约定）。中序遍历得到有序序列。

查找、插入和删除都沿着树高进行：树形均衡时约为 `O(log n)`，按递增顺序插入时可能退化成链表，变为 `O(n)`。

删除节点要分三种情况：

1. 叶子节点：直接删除。
2. 只有一个子节点：让子节点接替当前节点。
3. 有两个子节点：用右子树最小节点或左子树最大节点替换，再删除那个替代节点。

### AVL 树

AVL 要求每个节点的平衡因子满足：

```text
balanceFactor = height(left) - height(right)
balanceFactor ∈ {-1, 0, 1}
```

插入或删除后，沿路径更新高度；失衡时通过左旋、右旋或双旋恢复平衡。AVL 更严格，查找路径较短，但更新时可能进行更多旋转，适合查询明显多于更新的场景。

### 2-3 树与红黑树

2-3 树是每个节点可以保存 1～2 个 key 的多路平衡树，所有叶子处于同一层。插入时通过分裂和向上合并维持平衡。

红黑树可以看作 2-3/2-3-4 树的二叉表示，通过颜色和旋转维护黑高一致。它不像 AVL 那样严格追求高度最优，而是减少更新时的平衡成本，因此更适合插入、删除也频繁的场景。

| 对比 | AVL | 红黑树 |
| --- | --- | --- |
| 平衡程度 | 更严格 | 更宽松 |
| 查找 | 通常路径更短 | 仍保证 `O(log n)` |
| 更新 | 旋转可能更多 | 通常调整更少 |
| 适用 | 读多写少 | 读写较均衡 |

`TreeMap` 使用红黑树；Java 8 `HashMap` 在桶内冲突严重时也可能树化。不要把“红黑树”简单理解成“完全平衡树”，它只保证近似平衡和对数级高度。

## 并查集 Union-Find

并查集维护若干个互不相交的集合，核心操作是：

- `find(x)`：找到元素所在集合的代表元。
- `union(a, b)`：合并两个集合。
- `connected(a, b)`：判断两个元素是否属于同一集合。

最基础实现用 `parent[x]` 指向父节点。为了避免树变高：

- **路径压缩**：`find` 回溯时把节点直接连到根。
- **按秩/按大小合并**：总是把较矮或较小的树挂到另一棵树上。

两种优化同时使用时，单次操作均摊复杂度接近常数 `O(α(n))`。并查集适合连通分量、网络合并、好友关系和 Kruskal 最小生成树；它不擅长查询完整路径，也不适合频繁删除集合关系。

## 图的表示

图由顶点和边组成，可以有向、无向、带权或不带权。

| 表示方式 | 空间 | 适合场景 | 特点 |
| --- | --- | --- | --- |
| 邻接矩阵 | `O(V²)` | 顶点少、边密、频繁判断两点是否相连 | 查询边 `O(1)`，浪费稀疏图空间 |
| 邻接表 | `O(V + E)` | 大多数稀疏图、遍历邻居 | 遍历邻居高效，判断某条边需扫描邻居 |
| 边列表 | `O(E)` | Kruskal、批量排序边 | 结构简单，但不适合按顶点找邻居 |

建图前先确认：边是否有方向、是否有权重、是否允许重边和自环。很多算法错误不是遍历错，而是图模型定义错。

算法选择可以先按图的性质判断：无权最短路用 BFS，非负权单源最短路用 Dijkstra，最小生成树用 Kruskal/Prim；具体算法见 [经典排序与图算法](算法/经典排序与图算法.md) 和 [图论](算法/图论.md)。

## 布隆过滤器 Bloom Filter

布隆过滤器由位数组和多个哈希函数组成：写入元素时把多个位置置为 `1`，查询时只要有一个位置为 `0`，就能确定元素不存在；全部为 `1` 只能说明“可能存在”。

因此它具有：

- **没有假阴性**：插入过的元素不会被判断为不存在（忽略实现错误）。
- **可能有假阳性**：不同元素的哈希位置可能重叠。
- 空间效率高、查询快，但不能直接取出元素本身。

误判率受位数组大小、哈希函数数量和已插入元素数量影响。数据量增长后误判率会升高，需要按容量预估；不能把布隆过滤器当作精确去重或权限校验依据。

标准布隆过滤器不支持删除，因为清零某一位可能影响其他元素。需要删除时可以使用计数布隆过滤器，或采用版本化重建等方案。

典型用途是缓存穿透防护：先用布隆过滤器快速挡住大多数确定不存在的 key，再查询缓存和数据库；它不能替代缓存、数据库唯一约束或最终的业务校验。

## 结构选择的检查顺序

1. 先明确主要操作：随机访问、按序遍历、按优先级取值、前缀查询、连通性判断，还是范围查询。
2. 写出必须维护的不变量：顺序、堆序、BST 左右关系、平衡因子、集合代表元等。
3. 估算数据规模和读写比例，再比较最坏复杂度、内存开销和缓存局部性。
4. 处理边界：空结构、重复值、删除最后一个元素、扩容、溢出、并发访问和非法输入。
5. 最后再考虑是否需要更复杂的结构；能用数组、哈希表或标准库解决时，不要为了学习结构把生产代码手写一遍。
