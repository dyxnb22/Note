    static class Stream implements Comparable<Stream> {
        int id, u, a, b, t;

        public Stream(int id, int u, int a, int b, int t) {
            this.id = id; this.u = u; this.a = a; this.b = b; this.t = t;
        }

        @Override
        public int compareTo(Stream o) {
            if (this.a != o.a) return Integer.compare(o.a, this.a); // 点赞降序
            if (this.b != o.b) return Integer.compare(o.b, this.b); // 评论降序
            if (this.t != o.t) return Integer.compare(this.t, o.t); // 时间升序
            return Integer.compare(this.id, o.id);                  // 编号升序

在牛客网（Nowcoder）的 ACM 模式下，Java 选手最容易在输入输出处理、类名规范以及运行效率上踩坑。既然不能用本地 IDE 的自动补全和调试功能，建议你重点关注以下几个“考场潜规则”：

---

## 1. 核心规范（不遵守直接编译错误）

- 主类名必须是 `Main`：
  代码中必须包含一个 `public class Main`，并且所有的逻辑（包括 `main` 方法）都写在这里面。不要带 `package` 语句。
- 输入处理：
  ACM 模式不像 LeetCode 只需要实现一个函数，你需要手动处理标准输入。
  - 初级版：`Scanner sc = new Scanner(System.in);`（简单，但处理大数据量会超时）。
  - 进阶版：`BufferedReader br = new BufferedReader(new InputStreamReader(System.in));`（效率高）。
- 多组测试数据：
  题目通常包含多组数据，必须使用循环读取。
  `Scanner sc = new Scanner(System.in);<br>while (sc.hasNext()) { // 关键：判断是否还有输入<br>    int a = sc.nextInt();<br>    int b = sc.nextInt();<br>    System.out.println(a + b);<br>}`

---

## 2. 性能陷阱（导致 TLE - 超时）

- 尽量少用 `Scanner`：
  如果题目输入数据量达到 $10^5$ 级别，`Scanner` 的解析速度可能导致超时。改用 `StringTokenizer` 配合 `BufferedReader`。
- 字符串拼接要用 `StringBuilder`：
  在循环里频繁使用 `String += "abc"` 是 Java 选手的死穴，务必使用 `StringBuilder` 进行 `append`，最后一次性输出。
- 输出优化：
  如果输出量极大，`System.out.println` 也会变慢。可以使用 `PrintWriter`。

---

## 3. 盲改代码的“生存技巧”

没有了 IDE 的实时报错，你需要锻炼以下能力：

- 手感与语法熟练度：
  特别是常用集合框架的拼写，如 `ArrayList`, `HashMap`, `PriorityQueue`, `Deque`。记住 `size()` 是方法，而数组是 `.length` 属性。
- 异常捕获：
  如果遇到“运行错误（RE）”，通常是数组越界、空指针或堆栈溢出。ACM 模式下不会告诉你哪一行报错，建议在关键位置加一些 `System.err.println`（注意是 `err`，它不会被计入答案，但能在控制台看到）来辅助定位
