# 智能指针与内部可变性

## 1. `Box<T>`

`Box<T>` 在堆上拥有一个值，栈上只保存指针。它适用于递归类型、大对象、需要稳定间接层的 trait object：

```rust
enum List<T> {
    Cons(T, Box<List<T>>),
    Nil,
}
```

`Box<T>` 仍然是唯一所有权，不等于共享所有权。

## 2. `Rc<T>` 与 `Arc<T>`

`Rc<T>` 允许单线程共享所有权；`Arc<T>` 使用原子引用计数，可以跨线程共享：

```rust
use std::sync::Arc;

let config = Arc::new(String::from("shared"));
let another = Arc::clone(&config);
```

`Arc::clone` 只增加引用计数，不复制底层数据。引用计数解决“谁负责释放”，不解决数据是否可变、是否会发生竞态。

## 3. `RefCell<T>`

`RefCell<T>` 把借用规则从编译期推迟到运行时：

```rust
use std::cell::RefCell;

let value = RefCell::new(0);
*value.borrow_mut() += 1;
```

违反借用规则会 panic。它适合单线程内部可变性和测试替身，不适合跨线程共享。

## 4. `Mutex<T>` 与 `RwLock<T>`

`Mutex` 保证同一时间只有一个锁守卫访问数据；`RwLock` 允许多个读者或一个写者。

```rust
use std::sync::{Arc, Mutex};

let count = Arc::new(Mutex::new(0));
{
    let mut guard = count.lock().expect("mutex is not poisoned");
    *guard += 1;
}
```

锁守卫离开作用域时自动解锁。不要持锁执行慢 I/O、调用未知代码或等待另一个可能需要同一把锁的任务。

## 5. 常见组合

`Arc<Mutex<T>>` 适合多个线程共享、串行修改少量状态，但不是默认架构。替代方案包括：

- 通过消息 channel 把所有修改交给一个 owner。
- 使用原子类型处理简单计数。
- 用不可变快照替代频繁加锁。
- 按 key 分片减少锁竞争。
- 使用 async runtime 的异步锁，但不要用它保护 CPU 长任务。

## 6. `Cell<T>`、原子类型与 `UnsafeCell`

`Cell<T>` 适合 `Copy` 值的单线程内部修改；`AtomicUsize` 等原子类型适合简单跨线程状态；`UnsafeCell<T>` 是内部可变性的底层原语，通常不应直接使用。

原子操作的重点不只是“线程安全”，还包括内存顺序：`Relaxed`、`Acquire`、`Release`、`SeqCst` 各有语义。不了解内存模型时不要自己设计复杂无锁结构。

## 7. `Weak<T>` 与循环引用

两个 `Rc`/`Arc` 互相拥有会让引用计数永远不归零。`Weak` 是不拥有的观察引用：

```rust
let weak = Arc::downgrade(&config);
if let Some(value) = weak.upgrade() {
    println!("{value}");
}
```

树结构通常让父节点拥有子节点，子节点用 `Weak` 指向父节点。

## 8. 选择原则

1. 先用普通所有权和借用。
2. 需要递归或间接层时用 `Box`。
3. 确有共享所有权时用 `Rc`/`Arc`。
4. 确有内部可变性时用 `Cell`/`RefCell` 或锁。
5. 共享状态复杂时优先重新设计数据流，而不是层层包装智能指针。
