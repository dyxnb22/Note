# 模块、可见性与测试

## 1. crate、module 与 package

- package 由 `Cargo.toml` 描述，可以包含多个 crate。
- crate 是 Rust 编译的基本单元，可以是 library 或 binary。
- module 用于组织命名空间和控制可见性。
- 一个 package 可以同时有 `src/lib.rs` 和 `src/main.rs`；它们是不同 crate。

通常把可复用逻辑放在 library crate，把启动、参数解析和依赖装配放在 binary crate。

## 2. 声明和引用模块

现代 Rust 常用文件布局：

```text
src/
  lib.rs
  parser.rs
  storage/
    mod.rs
    file.rs
```

```rust
// lib.rs
mod parser;
pub mod storage;

pub use parser::parse;
```

`mod parser;` 声明并加载模块；`use` 只是引入名字，不会声明模块。

路径关键字：

- `crate::`：当前 crate 根。
- `self::`：当前模块。
- `super::`：父模块。
- `::`：绝对路径的起点，具体取决于上下文。

## 3. 可见性

Rust 默认私有。可以逐步扩大可见范围：

- 私有：只在当前模块及其子模块可用。
- `pub(super)`：对父模块公开。
- `pub(crate)`：对当前 crate 公开。
- `pub`：对 crate 外公开。

优先使用最小可见性。公开 API 是长期承诺，内部实现不要因为“方便测试”全部标成 `pub`。

## 4. API 设计

公开类型和函数应有明确不变量：

```rust
pub struct Port(u16);

impl Port {
    pub fn new(value: u16) -> Result<Self, PortError> {
        if value == 0 {
            Err(PortError::Invalid)
        } else {
            Ok(Self(value))
        }
    }
}
```

私有字段可以避免调用者直接构造非法状态。公开接口尽量使用标准库抽象，例如 `&Path`、`&str`、`&[u8]`，而不是要求调用者必须拥有某个具体容器。

## 5. 单元测试

单元测试通常放在实现文件末尾，用 `#[cfg(test)]` 隔离：

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn parses_positive_number() {
        assert_eq!(parse_number("42").unwrap(), 42);
    }

    #[test]
    #[should_panic(expected = "invalid")]
    fn invalid_input_panics_when_contract_requires_it() {
        panic!("invalid input");
    }
}
```

测试应验证行为和不变量，而不是绑定无关的内部实现。错误测试要检查错误类型或稳定字段，不要过度依赖完整错误字符串。

## 6. 集成测试与文档测试

`tests/` 目录中的每个 `.rs` 文件都是独立的集成测试 crate，只能使用 library crate 的公开 API：

```text
tests/
  parsing.rs
```

文档中的 Rust 代码块默认可能被当作文档测试：

```rust
/// Adds two numbers.
///
/// ```
/// assert_eq!(add(2, 3), 5);
/// ```
pub fn add(left: i32, right: i32) -> i32 {
    left + right
}
```

集成测试适合验证公开行为、模块协作和真实适配器；单元测试适合快速覆盖局部规则。

## 7. 条件编译

```rust
#[cfg(test)]
mod tests {}

#[cfg(target_os = "macos")]
fn platform_name() -> &'static str { "macos" }
```

feature 是 Cargo 提供的条件编译开关。不要让 feature 组合产生大量无法测试的行为分支。

## 8. 测试层次

- 单元测试：函数、类型和局部状态转换。
- 集成测试：多个模块通过公开接口协作。
- 文档测试：示例与 API 同步。
- Golden/snapshot 测试：稳定文本或结构化输出。
- Property test：对大量生成输入验证不变量。
- 端到端测试：从入口到外部适配器的真实流程。

测试替身应只实现需要的边界，不要为了测试复制一套生产逻辑。

## 练习

创建一个 library crate，公开一个 `Repository` 类型，内部拆成 `model`、`parser`、`storage` 三个模块；为其写单元、集成和文档测试，并用私有字段阻止非法构造。

`#rust #module #visibility #testing`
