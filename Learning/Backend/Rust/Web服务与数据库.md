# Web 服务与数据库

这篇讨论通用后端边界，不绑定某一个 Web 框架或数据库驱动。框架 API 会变化，下面的职责划分比具体函数名更重要。

## 1. 一个请求的生命周期

```text
连接
  → 路由
  → 认证/授权
  → 输入解析
  → 业务用例
  → 数据库/外部服务
  → 领域结果
  → HTTP 响应
```

路由层负责找到处理器；处理器负责解析输入、调用用例和映射响应；核心业务逻辑不应依赖 HTTP 请求对象。

## 2. 状态与依赖注入

服务通常有共享的只读配置、连接池和客户端。通过一个明确的 `AppState` 传递依赖，而不是使用全局可变变量：

```rust
struct AppState {
    config: Config,
    database: DatabasePool,
}
```

共享状态要明确 `Send`/`Sync`、生命周期和关闭顺序。请求处理器不应创建新的数据库连接池或 runtime。

## 3. 输入验证

解析 DTO、验证字段、构造领域类型是三个不同步骤：

1. 请求体是否是合法 JSON。
2. 字段是否满足长度、范围和格式。
3. 业务状态是否允许这个操作。

错误响应应区分客户端输入错误、认证失败、权限不足、资源不存在、冲突和服务器内部错误，但不要泄露数据库或内部堆栈细节。

## 4. 幂等、超时与取消

HTTP 请求可能重复发送。创建、支付、发送消息等操作要设计幂等键或去重策略。超时不能自动撤销已经提交的数据库事务或外部副作用。

为每个外部调用设定：连接超时、响应超时、最大响应体、取消行为和有限重试。重试前确认操作是否幂等。

## 5. 数据库连接池

连接池控制并发连接数量，避免每个请求新建连接。池的大小应基于数据库承载能力、请求类型和测量结果决定；过大的池也会造成排队和数据库压力。

连接获取失败、查询超时、事务回滚和连接断开都要进入明确错误路径。不要把连接池的内部错误直接返回给客户端。

## 6. 查询与事务

优先使用参数化查询，不拼接用户输入：

```sql
SELECT id, name FROM users WHERE email = $1
```

事务应尽量短，并明确隔离级别、锁范围、提交失败和重试语义。一个事务内的数据库状态不代表外部消息已经发送；跨系统一致性需要 outbox、幂等消费者或补偿策略，而不是假设事务天然覆盖一切。

## 7. 迁移与 schema

数据库 schema 迁移要版本化、可审查、可在干净数据库和已有数据上测试。删除或重命名字段通常需要分阶段部署：先增加兼容字段，再迁移读写，最后清理旧字段。

## 8. Web 安全基础

- 对输入长度、分页大小和上传体积设上限。
- 默认设置合理的连接、请求和响应超时。
- 正确处理认证、授权、CSRF、CORS 和安全响应头。
- 不把 token、密码和完整请求体写入日志。
- 不把用户输入直接当作 SQL、shell、模板或路径。
- 错误响应统一脱敏，详细诊断只留在受控日志。

## 9. 测试

- 领域单元测试：不依赖网络和数据库。
- handler 测试：验证输入/输出映射。
- 数据库集成测试：使用隔离数据库和真实迁移。
- 契约测试：验证外部服务协议。
- 端到端测试：覆盖认证、事务、超时和错误响应。
