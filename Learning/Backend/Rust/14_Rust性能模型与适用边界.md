# Rust 性能模型与适用边界

## 1. 不要把语言特性当作性能证明

Rust 的“零成本抽象”表示高质量抽象可以在编译后不增加不必要的运行时成本，不表示任何 Rust 程序自动更快。性能主要取决于算法、I/O、分配、缓存局部性、锁竞争、并发模型和外部系统。

先定义指标：吞吐量、延迟、p95/p99、内存峰值、启动时间、二进制大小和能耗。没有指标的优化通常只是猜测。

## 2. Debug 与 release

debug 构建更适合开发反馈，包含较少优化和更多检查；release 构建使用优化，行为和速度可能不同。性能结论必须在接近生产的 release 配置和代表性数据上测量。

## 3. 分配和所有权

常见成本来源：

- 循环中重复创建 `String`/`Vec`。
- 不必要的 `clone`。
- 小对象大量堆分配。
- 频繁扩容。
- 过度使用 `Arc`、锁和动态分发。

可以使用容量预留、缓冲区复用、借用、批处理和更合适的数据布局，但不要为了省一次分配牺牲清晰的所有权或正确性。

## 4. 数据布局

连续的 `Vec<T>` 往往具有良好的缓存局部性；大量指针跳转可能造成缓存 miss。字段顺序可能影响结构体 padding，但应先测量再调整。对于批量数值处理，比较 AoS（结构体数组）与 SoA（字段数组）的实际访问模式。

## 5. 并发性能

线程数过多、锁竞争、channel 排队和上下文切换都可能降低性能。async 适合等待，不适合自动加速 CPU 计算；CPU 工作应使用合理的线程池或并行算法。

吞吐量和延迟常有取舍：更大的批次提高吞吐，但可能增加单个请求等待。背压是保护系统，而不是性能失败。

## 6. 测量工具

- `cargo bench`/Criterion：微基准和回归。
- `perf`、Instruments、火焰图：CPU 和调用栈。
- heap profiler：分配和内存增长。
- `tracing`：请求级耗时和外部调用等待。
- `cargo bloat`：二进制体积。

微基准必须防止编译器消除无效工作，并使用稳定输入。端到端基准应固定硬件、数据、并发、网络和数据库条件。

## 7. 可靠性优先

不要为追求极限性能而删除超时、输入上限、错误处理或资源回收。安全和可靠性边界通常比少一次分配更重要。

## 8. Rust 的适用边界

适合：高并发网络服务、系统工具、性能敏感组件、本地 CLI、需要单二进制分发的服务。

不一定适合：业务逻辑极其简单且团队没有 Rust 能力、主要依赖尚不成熟、交付速度远比运行时性能重要的原型。Go、Java、Python、TypeScript 都可能是合理选择。

语言选择是整体约束权衡，不是技术信仰。

## 练习

为一个文本处理程序建立基准：比较循环和迭代器、预分配和不预分配、单线程和并发版本；记录吞吐、延迟和内存，不凭体感下结论。
