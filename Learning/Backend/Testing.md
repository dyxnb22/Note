# 后端测试体系

后端测试的目标不是追求覆盖率数字，而是用不同层级的证据证明业务规则、组件协作和生产边界在变化后仍然成立。

语言和框架的具体工具见对应目录；Agent 的非确定性评测见 [Eval 与测试体系](../Agent/Eval与测试体系.md)。本页只维护通用后端测试策略。

## 测试金字塔与测试组合

| 层级 | 验证对象 | 特点 |
| --- | --- | --- |
| 单元测试 | 函数、类、领域规则 | 快、定位准，不依赖外部系统 |
| 组件测试 | 一个模块及其适配器 | 使用真实序列化、数据库或消息组件 |
| 集成测试 | 多组件之间的合同 | 验证配置、协议和基础设施行为 |
| 契约测试 | 服务提供者与消费者接口 | 防止独立发布造成不兼容 |
| 端到端测试 | 关键用户链路 | 证据强但慢、脆弱、定位成本高 |
| 性能/韧性测试 | 容量和故障行为 | 验证 SLO、背压、降级与恢复 |

不是所有系统都严格呈金字塔形。数据库、消息和协议较重的服务会需要更多组件测试，但仍应避免所有规则只能通过端到端环境验证。

## 单元测试

单元测试适合纯业务规则、状态迁移、金额计算、权限判断和数据转换。一个好的单元测试：

- 输入、前置状态和预期结果明确。
- 不依赖真实时间、随机数、网络或共享全局状态。
- 失败时能指出哪条规则被破坏。
- 同时覆盖正常、边界和非法路径。

时间、ID、随机数和外部客户端通过接口注入。不要为了“单元”而 Mock 被测对象内部的每个私有调用；测试稳定合同，而不是复刻实现步骤。

## 集成测试

以下行为很难由 Mock 可靠证明，应使用真实组件或兼容容器：

- SQL 方言、索引、约束、事务和锁。
- JSON/Protobuf 序列化兼容。
- Redis TTL、Lua 和并发语义。
- MQ 的确认、重投、顺序和死信。
- HTTP Header、编码、超时和连接复用。

Testcontainers 可为测试启动与生产接近的数据库、中间件或服务。镜像版本应固定，初始化脚本版本化，测试结束后隔离并清理数据。

## 数据库测试

数据库测试至少覆盖：

- 唯一约束、外键和非空约束。
- 事务提交与回滚。
- 并发条件更新和乐观锁。
- 迁移脚本从旧版本升级。
- NULL、重复值、时区和边界日期。
- 关键 SQL 的执行计划或性能预算。

只使用 H2 等内存替代品可能掩盖 MySQL/PostgreSQL 方言和锁行为差异。关键 Repository 应在真实目标数据库上验证。

测试数据应通过工厂或 Builder 表达业务含义，不要在每个测试里复制几十个无关字段。每个测试独立拥有数据，不能依赖执行顺序。

## API 测试

API 测试验证：

- 请求和响应 Schema。
- 状态码与稳定错误码。
- 认证、授权和租户隔离。
- 参数边界、未知字段和非法编码。
- 幂等键与重复请求。
- 分页稳定性和排序。
- 超时、取消和限流响应。

测试不仅要断言 HTTP 200，还要断言业务状态、数据库副作用和审计记录。

## 契约测试

契约测试证明消费者使用的字段、状态码和交互在提供者新版本中仍然成立。它适合独立发布的服务和 SDK。

兼容原则：

- 新增可选字段通常向后兼容。
- 删除、改名、改变类型或枚举语义通常不兼容。
- 消费者不应因为新增未知字段而失败。
- 提供者应在 CI 中验证仍被支持的消费者契约。

契约测试不能替代少量端到端验证，因为网关、认证和部署配置仍可能出错。

## 消息与异步任务测试

必须覆盖：

- 同一消息重复投递。
- 消息乱序。
- 业务提交成功但 ACK 失败。
- 消费过程中进程退出。
- 重试达到上限进入死信。
- 补偿本身失败。
- 多消费者竞争同一任务。

断言最终业务事实和幂等记录，不只断言某个 Mock Publisher 被调用一次。

## 性能测试

先定义容量模型和 SLO，再压测：

- 目标吞吐、并发和数据规模。
- p50、p95、p99 延迟。
- 错误率和超时率。
- CPU、内存、GC、线程池和连接池。
- 数据库扫描、锁等待和缓存命中。
- 队列积压与恢复速度。

同时做基线测试、阶梯加压、突发流量和长时间稳定性测试。压测环境与生产差距必须写入结论，不能把测试 QPS 直接当生产容量。

## 故障与韧性测试

主动注入：

- 网络延迟、超时和连接重置。
- 下游 429/5xx。
- 数据库主从切换或连接耗尽。
- MQ 重复、积压和短暂不可用。
- 进程重启、节点摘除和磁盘不足。

验证系统是否按设计限流、熔断、降级、重试和恢复，并确认重试没有造成流量放大或重复副作用。

## 测试替身的边界

| 替身 | 用途 |
| --- | --- |
| Fake | 用简化实现提供可运行行为 |
| Stub | 返回预设结果 |
| Mock | 验证特定交互 |
| Spy | 包装真实对象并观察部分行为 |

Mock 外部系统合同，而不是 Mock 每一层内部调用。过度验证调用顺序会让重构破坏大量测试，却没有增加业务正确性的证据。

## CI 门禁

推荐层次：

```text
提交前：格式、静态检查、快速单测
Pull Request：全部单测、组件测试、迁移检查、安全扫描
合并后：契约、集成、关键端到端
发布前：回归、性能基线、迁移演练
生产灰度：SLO、错误率和业务指标
```

不稳定测试要有负责人和修复期限，不能无限重跑到绿色。隔离失败测试只能是临时措施，并保留风险可见性。

## 覆盖率与变异测试

覆盖率只能说明代码被执行，不能证明断言有效。优先关注关键业务分支、失败路径和边界条件。变异测试通过主动修改判断或返回值检查测试是否能发现错误，可用于高价值纯逻辑模块。

## 最小验收清单

- 核心业务规则有快速、确定性的单元测试。
- 数据库约束和事务在真实目标数据库上验证。
- API 有错误、权限、幂等和边界测试。
- 消息链路覆盖重复、乱序、崩溃和死信。
- 服务间有兼容策略和契约证据。
- 发布前有迁移、回滚和性能基线。
- 生产故障模式至少进行过一次可控演练。
