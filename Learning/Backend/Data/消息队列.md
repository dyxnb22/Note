# 基础概念与选型

## 什么是消息队列？

消息队列是应用之间异步传递消息的中间件。生产者发送消息，消费者按规则拉取或接收并处理。

它的核心价值是：

- 异步：非主链路任务不必阻塞主请求。
- 解耦：生产者和消费者不需要直接调用。
- 削峰：消费者按自身处理能力消费，保护下游服务。

代价是需要额外处理可靠性、重复消费、顺序性、消息积压和运维复杂度。

## 消息队列有哪些使用场景？

- 异步处理：下单后的短信、邮件、积分、优惠券等任务。
- 系统解耦：订单服务发布事件，库存、积分、通知服务分别消费。
- 削峰填谷：秒杀、抢券等流量高峰先进入队列。
- 数据分发：业务事件被搜索、数仓、风控、推荐等多个消费者订阅。

从设计模式看，消息队列主要体现生产者-消费者模式和发布-订阅模式，也是事件驱动架构的常用基础设施。

## Kafka、RocketMQ、RabbitMQ 怎么选型？

| MQ | 适合场景 | 主要特点 |
| --- | --- | --- |
| Kafka | 日志、埋点、流计算、数据管道 | 吞吐高、分区日志、生态成熟、可回放 |
| RocketMQ | 订单、支付、库存、营销等业务消息 | 事务、延时、顺序、重试和死信能力较完整 |
| RabbitMQ | 中小规模业务解耦、任务分发 | AMQP、路由灵活、延迟低、使用成熟 |

选型时综合考虑吞吐、延迟、消息模型、事务/延时/顺序能力、可靠性、运维成本和团队熟悉度。

# 通用可靠性与消费治理

## 消息如何避免丢失？

需要分别检查生产者、Broker 和消费者三个环节：

- 生产者：等待 Broker 确认，发送失败重试；必要时使用事务消息或 Outbox。
- Broker：开启持久化和副本，根据业务选择同步刷盘、同步复制或异步策略。
- 消费者：业务处理成功后再确认消息，避免先 ACK 后执行业务。

消费成功但 ACK 前宕机，消息可能再次投递，所以可靠消费必须和幂等设计一起考虑。

## 消息重复消费和幂等写怎么解决？

MQ 通常只能做到至少一次投递，网络超时、消费失败和重试都会造成重复消费。业务侧需要保证幂等：

- 数据库唯一键：以订单号、业务流水号或消息业务 ID 建唯一索引。
- 状态机：只允许合法状态流转，例如“待支付 -> 已支付”。
- 消费记录表：消费前记录消息，插入冲突则说明已经处理。
- Redis `SET NX`：适合短期去重或防抖；资金、库存等关键业务仍应以数据库为准。

处理成功和记录幂等状态最好放在同一个数据库事务中，不能只依赖进程内存。

## 消息顺序如何保证？

MQ 通常只保证同一队列或分区内有序，不保证全局有序。

- 发送端按订单号、用户 ID 等业务 key 路由到同一队列/分区。
- 消费组内同一队列/分区同一时间只交给一个消费者。
- 消费者内部也不能让同一 key 并发处理，否则落库顺序仍可能错乱。

顺序性会限制并发度，因此只对真正需要顺序的业务 key 保证局部顺序。

## 如何保证业务和消息最终一致？事务消息怎么实现？

业务落库和发送消息不是同一个本地事务，可能出现“业务成功但消息失败”或“消息成功但下游失败”。常见方案：

- 事务消息：先发送半消息，执行本地事务，再提交或回滚；Broker 超时后回查本地事务状态。
- Outbox：业务数据和消息记录写入同一个数据库事务，再由后台任务可靠投递。
- 补偿机制：重试、状态查询、定期对账和人工修复。

实际系统通常追求最终一致性，而不是为所有消息强行引入分布式事务。

## 如何处理消息积压？

先定位原因：消费者宕机、消费逻辑变慢、下游数据库慢、队列/分区过少或突发流量过大。

临时处理：

- 扩容消费者，提高批量消费能力。
- 暂停非核心逻辑，给慢下游限流或降级。
- 优化数据库写入、同步 I/O 和外部接口调用。
- 必要时转储到新 Topic，增加分区和消费者并行追赶。

注意：消费者数量超过队列/分区数量后，新增消费者无法继续提高并行度；转储和扩容还必须评估幂等与顺序影响。

## 什么是死信队列？

死信队列（DLQ）用于存放无法正常消费的消息，常见进入原因：

1. 消费失败次数超过最大重试次数。
2. 消息 TTL 到期。
3. 队列达到最大长度，无法容纳新消息。

不同 MQ 的实现：

- RocketMQ：通常为消费者组创建对应的死信队列。
- RabbitMQ：通过 `x-dead-letter-exchange` 配置死信交换机。
- Kafka：没有原生 DLQ，通常将失败消息发送到另一个 Topic，例如 DLT。

生产环境需要为死信队列配置告警、原因记录、人工排查和重新投递机制。

# 消息队列架构设计

## 如何设计一个消息队列？

核心模块包括 producer SDK、Broker、consumer SDK、存储层、元数据管理和监控告警。

Broker 至少需要支持：

- Topic/Queue 管理、消息写入和读取。
- 持久化、索引、副本和故障转移。
- ACK、消费位点、重试和死信队列。
- 消费者负载均衡和消息顺序约束。
- 堆积量、消费延迟、TPS、失败率、磁盘和网络监控。

## 消息中间件如何做到高可用？

- Broker 集群部署，避免单点故障。
- 队列或分区配置副本，Leader 故障后由副本接管。
- 生产端配置超时、重试和幂等发送。
- 消费端支持失败重试、死信和消费位点恢复。
- 持续监控 Broker 存活、磁盘、网络、堆积、延迟和错误率。

# RocketMQ

## RocketMQ 有什么特点？

RocketMQ 更偏业务消息，适合订单、支付、库存和营销等场景，支持事务消息、延时消息、顺序消息、重试和死信。

它以 Topic 下的 Message Queue 为核心，并通过 NameServer 提供路由发现。业务上可以按订单号、用户 ID 等 key 做顺序和分片。

## RocketMQ 和 Kafka 有什么区别？

| 对比项 | RocketMQ | Kafka |
| --- | --- | --- |
| 核心模型 | Topic + Message Queue | Topic + Partition 日志 |
| 主要优势 | 业务消息语义丰富 | 吞吐高、生态强、可回放 |
| 典型能力 | 事务、延时、顺序、重试、死信 | 流式处理、日志管道、批量消费 |
| 路由发现 | NameServer | 集群元数据 |

日志、埋点和流式处理通常优先 Kafka；交易链路和复杂消息语义通常更适合 RocketMQ。

## RocketMQ 延时消息的原理是什么？

延时消息通常按延时等级或定时机制先写入特殊主题，到达投递时间后再转回真实 Topic，消费者才能消费。

它适合订单超时关闭、支付超时检查等场景，但不适合作为高精度定时任务系统。

## RocketMQ 如何实现事务消息？

1. 生产者发送半消息，消费者暂时不可见。
2. 生产者执行本地事务。
3. 本地事务成功则提交消息，失败则回滚消息。
4. Broker 长时间未收到结果时，回查生产者的事务状态。

生产者需要根据事务表或订单状态回答回查，不能只依赖进程内存。

## RocketMQ 如何保证消息顺序？

发送端按业务 key 将消息固定路由到同一个 queue，Broker 保证 queue 内有序，消费端使用顺序消费模式，让同一 queue 同时只由一个消费线程处理。

这样只能保证局部顺序，而且会降低并发度，因此不应追求全局顺序。

# Kafka

## Kafka 的核心模型是什么？

Kafka 是分布式日志型消息系统，核心概念包括 Topic、Partition、Broker、Producer、Consumer Group 和 Offset。

- Topic 可以拆分为多个 Partition，每个 Partition 是有序追加日志。
- Consumer Group 内的消费者分摊 Partition。
- 不同 Consumer Group 可以独立消费同一份消息。
- Kafka 主要由消费者主动拉取消息，并自行维护消费 Offset。

拉模式便于消费者按自身处理能力控制速率，也适合批量拉取和消息回放。

## Kafka 为什么快？

- Append-only 日志以顺序写为主，减少随机 I/O。
- Page Cache 减少用户态和内核态之间的数据拷贝。
- 零拷贝降低网络发送时的 CPU 拷贝开销。
- 批量发送、压缩和分区并行提高磁盘与网络利用率。

## Kafka 如何保证顺序？Partition 和消费者数量是什么关系？

Kafka 只保证单个 Partition 内有序，不保证跨 Partition 全局有序。同一订单或用户的消息应使用相同 key 路由到同一 Partition。

同一个 Consumer Group 内，一个 Partition 同一时刻只分配给一个消费者：

- 消费者少于 Partition 数时，一个消费者可能负责多个 Partition。
- 消费者多于 Partition 数时，多余消费者会空闲。
- 消费能力受 Partition 数限制，不能只靠盲目增加线程扩展。

## Kafka 的 ISR 和消息确认机制是什么？

ISR（In-Sync Replicas）是与 Leader 保持同步的副本集合，只有 ISR 内的副本才有资格被选为新 Leader。Follower 如果长时间落后，会被移出 ISR。

生产端通过 `acks` 控制确认级别：

- `acks=0`：不等待确认，最快但可能丢消息。
- `acks=1`：Leader 写入后确认，Leader 故障时可能丢消息。
- `acks=all`：ISR 副本确认后返回，可靠性更高。

生产环境常将 `acks=all` 与 `min.insync.replicas` 配合使用。消费端主要提交 Offset，提交过早可能导致业务失败后消息不再重放，因此仍需业务幂等。

## Kafka 的 Rebalance 是什么？如何减少？

Rebalance 是消费者组重新分配 Partition 的过程，常见触发原因：

- 消费者加入、退出、崩溃或重启。
- Topic 增加 Partition。
- 消费者心跳超时。
- 单次处理时间超过 `max.poll.interval.ms`。

Rebalance 会影响消费延迟。可以通过静态成员 `group.instance.id`、合理设置 `session.timeout.ms` 和 `max.poll.interval.ms`、减少单次拉取量以及优化消费耗时来降低发生频率。

# RabbitMQ

## RabbitMQ 有什么特点？

RabbitMQ 基于 AMQP，核心概念包括 Producer、Exchange、Queue、Binding 和 Consumer。

- Exchange 支持 direct、fanout、topic、headers 等路由方式。
- 支持确认、持久化、死信、延迟插件和优先级队列。
- 适合业务解耦、任务分发和中小规模异步处理。
- 海量日志吞吐通常不如 Kafka。

## RabbitMQ 的底层架构是什么？

Producer 不直接把消息发送给 Queue，而是先发送到 Exchange。Exchange 根据类型、routing key 和 Binding 规则，把消息路由到一个或多个 Queue。Consumer 从 Queue 消费消息，并通过 ACK 告知 Broker 是否处理成功。

高可用通常依赖集群、Quorum Queue、持久化和确认机制。

# ActiveMQ

ActiveMQ 是较传统的 JMS/AMQP 消息中间件，核心模型包括 Queue（点对点）和 Topic（发布订阅）。面试重点关注可靠性：持久化消息通常需要持久化目的地、持久化消息和可靠确认共同配合；消费失败应通过重投、重试上限和死信队列避免无限循环。新系统选型还要结合吞吐、延迟、事务/延时消息能力和维护现状，不能只按“支持 JMS”决定。

Topic 的非持久订阅者离线时通常收不到离线期间的消息；持久订阅通过客户端标识让 Broker 为离线订阅者保留消息，重新连接后补发，但会增加存储和管理成本。Queue 更接近点对点负载均衡，一条消息通常只由一个消费者成功处理。
