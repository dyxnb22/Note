# 消息队列场景
<details>
<summary>什么是消息队列？</summary>
	- 消息队列是应用之间异步传递消息的中间件。生产者把消息发送到队列或主题，消费者再按规则拉取或接收并处理。
	- 它把“调用方”和“被调用方”解耦，调用方不必等待下游处理完成，只要消息成功进入 MQ，就可以继续后续流程。
	- 面试回答可以抓三个关键词：异步、解耦、削峰；再补一句代价：引入 MQ 后要额外处理可靠性、顺序性、重复消费和运维复杂度。
</details>
<details>
<summary>消息队列怎么选型？</summary>
	- Kafka：吞吐高、生态成熟、适合日志、埋点、流式数据、事件管道；缺点是传统业务消息能力不如 RocketMQ 细腻。
	- RocketMQ：业务消息能力强，支持延时消息、事务消息、顺序消息，适合交易、订单、支付、营销等场景。
	- RabbitMQ：协议成熟、路由灵活、延迟低，适合中小规模业务解耦；但大吞吐和海量堆积能力通常不如 Kafka/RocketMQ。
	- 选型话术：先看业务目标，再看吞吐、延迟、消息模型、事务/延时/顺序支持、运维成本和团队熟悉度。
</details>
<details>
<summary>消息队列使用场景有哪些？</summary>
	- 异步处理：下单后发送短信、邮件、积分、优惠券等非主链路动作，缩短接口响应时间。
	- 系统解耦：订单服务只发布订单事件，库存、积分、通知服务各自消费，减少服务之间的直接依赖。
	- 削峰填谷：秒杀、抢券等流量高峰先进入队列，消费者按自身处理能力慢慢消费，保护数据库和下游服务。
	- 数据分发：一份业务事件可以被多个消费者订阅，用于搜索索引、数据仓库、风控、推荐等。
</details>
<details>
<summary>消息重复消费怎么解决？</summary>
	- 先承认现实：MQ 很难承诺业务绝对只消费一次，所以业务侧必须做幂等。
	- 常见做法：业务唯一键去重、数据库唯一索引、Redis set/nx、状态机判断、消费流水表。
	- 处理顺序：消费者收到消息后先检查是否处理过；未处理则执行业务并记录处理结果；已处理则直接返回成功。
	- 注意不要只依赖内存去重，因为服务重启、扩容、迁移都会丢状态。
</details>
<details>
<summary>消息丢失怎么解决？</summary>
	- 生产端：发送消息后等待 broker 确认，失败要重试；本地事务和消息发送之间可以用事务消息或 outbox 表兜底。
	- Broker 端：开启持久化和副本机制，关键业务不要只写内存；根据业务选择同步刷盘、同步复制或可接受的异步策略。
	- 消费端：业务处理成功后再确认消息；如果先 ack 再执行业务，消费者宕机会导致消息丢失。
	- 面试可分三段答：生产者丢、broker 丢、消费者丢，分别用确认、持久化/副本、消费成功后 ack 解决。
</details>
<details>
<summary>使用消息队列还应该注意哪些问题？</summary>
	- 一致性：主业务成功但消息失败，或消息成功但下游失败，都要有补偿和对账机制。
	- 幂等性：重复投递、消费重试、网络超时都会导致同一消息被处理多次。
	- 顺序性：如果业务依赖顺序，需要让同一业务 key 进入同一队列/分区，并由单消费者顺序处理。
	- 可观测性：要监控堆积量、消费延迟、失败重试、死信队列、broker 状态。
	- 降级预案：下游不可用时要能暂停消费、限速、转死信、人工补偿。
</details>
<details>
<summary>消息队列的可靠性、顺序性怎么保证？</summary>
	- 可靠性靠完整链路：生产确认、broker 持久化和副本、消费者成功处理后 ack。
	- 顺序性靠分片约束：同一订单、同一用户、同一业务聚合根按 key 路由到同一队列或分区。
	- 消费端要避免同一 key 被多个线程并发处理，否则即使 MQ 投递有序，业务落库也可能乱序。
	- 可靠性和性能经常冲突，比如同步刷盘更可靠但更慢，需要按业务等级取舍。
</details>
<details>
<summary>如何保证幂等写？</summary>
	- 数据库唯一键：例如订单号、消息 id、业务流水号建唯一索引，重复写入直接失败或转为查询结果。
	- 状态机：只允许状态从合法前置状态流转到目标状态，例如“待支付 -> 已支付”，重复支付回调不会再次扣减。
	- 幂等表：消费前插入消费记录，插入成功才执行业务，插入冲突说明已处理。
	- Redis NX：适合短期去重或防抖，但关键资金、库存类业务最好仍落数据库。
</details>
