# SQL 基础

## SQL 和 NoSQL 的区别

| 维度       | 关系型数据库（SQL）                         | NoSQL                                |
| ---------- | ------------------------------------------- | ------------------------------------ |
| 数据模型   | 二维表，Schema 明确，关系清晰               | KV、文档、列族、图等，结构更灵活     |
| 查询能力   | SQL 标准化程度高，擅长 JOIN、聚合和复杂查询 | 查询方式依赖具体产品                 |
| 事务一致性 | 通常完整支持 ACID 事务                      | 依赖具体产品，部分更偏最终一致性     |
| 扩展方式   | 常见读写分离、分库分表，水平扩展成本较高    | 通常更容易进行水平扩展               |
| 适用场景   | 订单、支付、账户、复杂业务查询              | 缓存、日志、内容存储、海量高并发读写 |
| 典型产品   | MySQL、PostgreSQL、Oracle                   | Redis、MongoDB、Cassandra            |

**核心结论：**关系型数据库更擅长强事务、复杂查询和数据一致性；NoSQL 更强调数据结构灵活、高并发和水平扩展。两者通常结合使用，而不是相互替代。

**注意：**NoSQL 不等于不支持事务，不同产品的事务、一致性和查询能力差异很大。

## 数据库三大范式

| 范式 | 核心要求                                 | 解决的问题               |
| ---- | ---------------------------------------- | ------------------------ |
| 1NF  | 字段值具有原子性，一个字段只存一个业务值 | 字段中存储重复或集合数据 |
| 2NF  | 满足 1NF，非主属性完全依赖候选键         | 部分依赖                 |
| 3NF  | 满足 2NF，非主属性不能传递依赖候选键     | 传递依赖                 |

-   2NF 主要针对**联合候选键**；如果候选键只有一列，通常不会出现部分依赖。
-   实际业务不一定严格追求最高范式，可为查询性能适当冗余数据。
-   反范式设计的典型场景包括订单快照、历史价格和用户名冗余。

## MySQL JOIN 连表查询

JOIN 用于根据关联字段连接多个表。

| 类型       | 说明                                               |
| ---------- | -------------------------------------------------- |
| INNER JOIN | 返回两个表都匹配的数据                             |
| LEFT JOIN  | 保留左表全部数据，右表无匹配补 NULL                |
| RIGHT JOIN | 保留右表全部数据，左表无匹配补 NULL                |
| FULL JOIN  | MySQL 不支持，可用 LEFT JOIN UNION RIGHT JOIN 模拟 |

JOIN 优化

-   让优化器根据过滤效果、索引和扫描成本选择合适的驱动表，不要机械地认为一定是“小表驱动大表”。
-   JOIN 关联字段建立索引，提高查询效率。
-   避免 JOIN 条件字段使用函数、表达式和隐式类型转换，防止索引失效。
-   复杂多表查询可通过适当冗余减少 JOIN。

## MySQL 如何避免重复插入数据

1.  建立唯一索引是防止重复数据的根本方案。
2.  `INSERT IGNORE`：重复键时忽略插入，不报错。
3.  `INSERT ... ON DUPLICATE KEY UPDATE`：重复键时执行更新操作。
4.  “先查再插”存在并发问题：
    -   两个线程同时查询不存在；
    -   同时执行 INSERT；
    -   最终必须依靠数据库唯一约束拦截。

## CHAR 和 VARCHAR 区别

| 对比     | CHAR                   | VARCHAR                  |
| -------- | ---------------------- | ------------------------ |
| 存储方式 | 定长存储，不足补空格   | 变长存储，按实际长度保存 |
| 空间占用 | 固定长度               | 实际长度 + 长度标识      |
| 适用场景 | 长度固定的数据         | 长度变化的数据           |
| 示例     | 性别、状态码、固定编码 | 用户名、地址、备注       |

补充：

-   `VARCHAR(n)` 中 `n` 表示字符数量，不是字节数量。
-   字符实际占用字节数由字符集决定，例如 `utf8mb4` 一个字符最多占 4 字节。

## INT(1) 和 INT(10) 区别

| 项目      | 说明                             |
| --------- | -------------------------------- |

| n 的含义  | 历史上的显示宽度，不影响存储范围 |
| 取值范围  | 与 INT 类型本身有关，与 n 无关   |
| ZEROFILL  | 用于不足位数时左侧补 0 显示      |

## 外键约束作用

| 方面       | 说明                                 |
| ---------- | ------------------------------------ |
| 作用       | 保证引用完整性，避免孤儿数据         |
| 优点       | 数据一致性交给数据库保证             |
| 缺点       | 增加写入、删除、迁移和分库分表复杂度 |
| 互联网实践 | 常用业务逻辑保证关系，少用物理外键   |

## IN 和 EXISTS 区别

| 对比      | IN                         | EXISTS                 |
| --------- | -------------------------- | ---------------------- |
| 判断方式  | 判断值是否在结果集中       | 判断子查询是否存在记录 |
| 关注内容  | 返回的数据值               | 是否有结果             |
| 常见场景  | 子查询结果较小             | 依赖外层查询逐行判断   |
| NULL 影响 | 容易受到 NULL 三值逻辑影响 | 只判断行是否存在       |

注意：

-   `EXISTS` 子查询返回什么列不重要：

```
EXISTS (SELECT 1 FROM table WHERE ...)
```

-   不要认为 `EXISTS` 一定比 `IN` 快，MySQL 优化器可能进行半连接优化。

| 聚合函数   | `COUNT`、`SUM`、`AVG`、`MAX`、`MIN`                          |

## SQL 查询语句的逻辑执行顺序

```text
FROM / JOIN / ON → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

因此，`WHERE` 中不能使用 `SELECT` 中定义的别名，而 `ORDER BY` 通常可以使用别名。聚合函数在分组后参与计算，不需要单独作为一个执行阶段记忆。

# 执行流程与存储引擎

## 一条 SQL 请求在 MySQL 中如何执行

| 阶段 | 组件     | 作用                                 |
| ---- | -------- | ------------------------------------ |
| 1    | 连接器   | 建立连接、认证用户、校验权限         |
| 2    | 查询缓存 | MySQL 8.0 已删除，历史版本中命中率低 |

## MySQL 常见存储引擎

| 引擎   | 特点                                           | 适用场景             |
| ------ | ---------------------------------------------- | -------------------- |
| InnoDB | 默认引擎，支持事务、行锁、外键、崩溃恢复、MVCC | 绝大多数 OLTP 场景   |
| MyISAM | 不支持事务和行锁，表锁，读性能较好但崩溃恢复弱 | 历史遗留或只读场景   |
| MEMORY | 数据在内存中，重启丢失                         | 临时数据、缓存类场景 |

## InnoDB 和 MyISAM 区别

| 维度       | InnoDB                           | MyISAM                             |
| ---------- | -------------------------------- | ---------------------------------- |
| 事务       | 支持 ACID                        | 不支持                             |
| 锁粒度     | 行锁为主，也有表级锁             | 表锁                               |
| 外键       | 支持                             | 不支持                             |
| 索引结构   | 聚簇索引，主键叶子节点存整行数据 | 非聚簇索引，索引叶子节点存数据地址 |
| 崩溃恢复   | 支持 redo log 恢复               | 恢复能力弱                         |
| `COUNT(*)` | 需要按可见性统计，通常较慢       | 保存表总行数，无 WHERE 时很快      |
| 适用场景   | 高并发读写、事务场景             | 读多写少、历史场景                 |

# 索引

## 索引分类

| 分类方式   | 类型                                   |
| ---------- | -------------------------------------- |
| 按数据结构 | B+Tree 索引、Hash 索引、Full-text 索引 |
| 按物理存储 | 聚簇索引、二级索引                     |
| 按字段特性 | 主键索引、唯一索引、普通索引、前缀索引 |
| 按字段数量 | 单列索引、联合索引                     |

## InnoDB 索引如何实现

| 类型                 | 叶子节点存储      |
| -------------------- | ----------------- |
| 主键索引（聚簇索引） | 完整行数据        |
| 二级索引             | 索引字段 + 主键值 |

关键概念：

-   **回表**：
    -   通过二级索引找到主键；
    -   再根据主键查询聚簇索引获取完整数据。
-   **覆盖索引**：
    -   查询字段全部存在二级索引中；
    -   不需要回表。

## B+Tree 为什么适合作为 MySQL 索引

| 特性                  | 原因                                     |
| --------------------- | ---------------------------------------- |
| 非叶子节点只存索引    | 单个节点存更多 key，降低树高度，减少 I/O |
| 叶子节点存数据/主键值 | 可以直接定位数据                         |
| 叶子节点有序链表      | 支持范围查询和排序                       |
| 多路平衡树结构        | 适合磁盘页读取                           |

核心：

-   树越矮 → 查询需要访问的磁盘页越少。
-   叶子节点有序 → 范围查询效率高。

## 查到 B+Tree 叶子节点后，页内如何查找记录

1.  InnoDB 数据页中的记录按照主键顺序组织。
2.  页内维护页目录（Page Directory）。
3.  页目录通过槽（slot）记录记录分组位置。
4.  查询时：
    -   先在页目录中二分查找；
    -   再在对应分组内遍历查找。

## 主键应该怎么设计？自增 ID 和 UUID 怎么选？

**主键设计原则**

| 原则         | 原因                       |
| ------------ | -------------------------- |
| 唯一且非空   | 满足主键基本要求           |
| 尽量短       | 减少聚簇索引和二级索引空间 |
| 尽量递增     | 减少页分裂和随机 I/O       |
| 避免业务字段 | 防止业务变化导致主键修改   |

**自增 ID 优势**

-   InnoDB 主键索引是聚簇索引。
-   自增 ID 通常顺序插入，数据追加到 B+Tree 尾部。
-   减少页分裂，提高写入性能。

**UUID 问题**

-   无序，容易随机插入。
-   长度较大，占用更多索引空间。
-   增加页分裂和碎片。

## 联合索引和最左匹配原则

对于联合索引：INDEX(A, B, C)

| 查询条件              | 是否使用索引                     |
| --------------------- | -------------------------------- |
| `A=?`                 | ✅ 可以                           |
| `A=? AND B=?`         | ✅ 可以                           |
| `A=? AND B=? AND C=?` | ✅ 可以                           |
| `B=?`                 | ❌ 通常不能有效使用               |
| `A=? AND C=?`         | ✅ A 可用，C 通常不能用于索引定位 |

原因：B+Tree 按：A → B → C 排序

**范围条件**

范围查询会影响后续列匹配：

例如：

```
WHERE A = ?
AND B > ?
AND C = ?
```

通常：

-   A 用于定位；
-   B 用于范围扫描；
-   C 不能继续用于索引排序定位。

**联合索引设计原则**

-   高频查询字段优先考虑。
-   等值查询字段通常放在范围查询字段之前。
-   排序、分组需求可以结合索引设计。
-   区分度低的字段单独作为最左列，可能导致优化器放弃索引。

## 索引下推

1.  ICP 是 MySQL 5.6 引入的索引优化。
2.  没有 ICP：
    -   索引找到记录；
    -   回表获取完整数据；
    -   Server 层过滤剩余条件。
3.  有 ICP：
    -   在索引层先判断部分条件；
    -   不满足的记录直接过滤；
    -   减少回表次数。
4.  ICP 不会让不符合最左匹配的字段参与索引定位。

例子：联合索引 `(A, B, C)`，条件 `A = 1 AND C < 10`。A 能用于索引定位，C 不能跳过 B 继续定位，但 C 在索引中，可能通过索引下推先过滤一部分记录。

## 可能导致索引利用不充分的情况

| 场景           | 示例                    |
| -------------- | ----------------------- |
| 左模糊查询     | `LIKE '%abc'`           |
| 索引列使用函数 | `DATE(create_time)=xxx` |
| 索引列参与计算 | `age + 1 = 18`          |
| 隐式类型转换   | 字符串字段与数字比较    |
| 违反最左匹配   | 联合索引缺少最左字段    |
| OR 条件不合理  | OR 一边无索引，可能无法有效使用索引 |
| 选择性太低     | 优化器认为全表扫描更优              |

**优化原则** 不要改变索引列：

```
WHERE DATE(create_time) = '2026-01-01'
```

改为：

```
WHERE create_time >= '2026-01-01'
AND create_time < '2026-01-02'
```

## 覆盖索引

覆盖索引不是额外的索引类型，而是一种查询效果：查询所需字段全部包含在索引中，因此不需要回表。它主要通过减少磁盘 I/O 和访问聚簇索引的次数来优化查询，常见于联合索引。

## 索引是不是建得越多越好？

**适合建立索引**

| 场景            | 示例                    |
| --------------- | ----------------------- |
| 唯一性字段      | 用户 ID、订单号         |
| 高频 WHERE 条件 | 查询过滤字段            |
| JOIN 关联字段   | 被驱动表关联字段        |
| 高频排序/分组   | ORDER BY、GROUP BY 字段 |
| 多字段查询      | 联合索引                |

**不适合建立索引**

-   数据量很小的表。
-   区分度很低的字段，例如性别、状态。
-   频繁更新但查询很少的字段。
-   已被其他联合索引覆盖的冗余索引。

**索引成本**

-   占用额外磁盘空间。
-   INSERT、UPDATE、DELETE 需要维护索引。
-   索引过多增加优化器选择成本。
-   增加数据库维护复杂度。

## 单列索引和联合索引都存在时，MySQL 会选哪个？

影响索引选择的因素：

| 因素         | 说明             |
| ------------ | ---------------- |
| 过滤效果     | 能过滤多少数据   |
| 扫描行数     | 需要扫描的数据量 |
| 是否覆盖索引 | 是否避免回表     |
| 是否满足排序 | 是否减少额外排序 |
| 回表成本     | 二级索引回表开销 |

验证方式：

-   `EXPLAIN`
-   `EXPLAIN ANALYZE`
-   慢查询日志

# 事务与 MVCC

## 事务的 ACID 特性是什么？InnoDB 怎么实现？

| 特性               | 含义                           | InnoDB 实现                    |
| ------------------ | ------------------------------ | ------------------------------ |
| 原子性 Atomicity   | 要么全部成功，要么全部失败     | undo log 回滚                  |
| 一致性 Consistency | 事务前后数据满足约束和业务规则 | 原子性、隔离性、持久性共同保证 |
| 隔离性 Isolation   | 并发事务互不干扰               | 锁 + MVCC                      |
| 持久性 Durability  | 提交后即使宕机也不丢           | redo log + WAL                 |

## MySQL 并发事务会产生哪些问题

| 问题       | 含义                                                         |
| ---------- | ------------------------------------------------------------ |
| 脏读       | 读到其他事务未提交的数据                                     |
| 不可重复读 | 同一事务内多次读同一行，结果不同，通常由 update/delete 导致  |
| 幻读       | 同一事务内按相同条件多次查询，结果集行数变化，通常由 insert 导致 |
| 丢失更新   | 多个事务同时更新同一数据，后提交覆盖先提交结果               |

## 事务隔离级别

| 隔离级别            | 脏读 | 不可重复读 | 幻读                       | 实现特点                             |
| ------------------- | ---- | ---------- | -------------------------- | ------------------------------------ |
| 读未提交 RU         | 可能 | 可能       | 可能                       | 直接读最新数据                       |
| 读提交 RC           | 不会 | 可能       | 可能                       | 每条语句生成新的 Read View           |
| 可重复读 RR         | 不会 | 不会       | 快照读不会，当前读靠锁解决 | 事务内复用 Read View，MySQL 默认级别 |
| 串行化 Serializable | 不会 | 不会       | 不会                       | 读写都加锁，事务串行执行             |

注意：InnoDB 的 RR 比标准 RR 更强，普通快照读通过 MVCC 避免幻读，当前读通过 next-key lock 避免幻读。

## MVCC 的实现原理

**MVCC 三大组成**

**1. 隐藏字段**

每行记录包含：

| 字段         | 作用                   |
| ------------ | ---------------------- |
| trx_id       | 最近修改该行的事务 ID  |
| roll_pointer | 指向 undo log 历史版本 |

**2. undo log 版本链**

数据更新时：

```
当前版本
 ↓
undo log 旧版本
 ↓
更旧版本
```

如果当前版本不可见，会沿版本链寻找可见版本。

**3. Read View**

Read View 用于判断某个版本是否对当前事务可见。

核心理解：

-   事务创建 Read View 时，会记录当前活跃事务。
-   已提交事务的数据可以看到。
-   未提交事务的数据不可见。
-   找不到可见版本时，通过 undo log 找历史版本。

**4. 可见性判断**

判断一个版本是否可见：

1.  自己事务修改的数据 → 可见。
2.  Read View 创建前已经提交的数据 → 可见。
3.  Read View 创建后才开启的事务修改的数据 → 不可见。
4.  创建 Read View 时正在执行的事务修改的数据 → 不可见。

不可见时：沿 undo log 版本链寻找历史可见版本。

**5. RC 和 RR 的区别**

| 隔离级别 | Read View 创建方式                   |
| -------- | ------------------------------------ |
| RC       | 每次快照读创建新的 Read View         |
| RR       | 第一次快照读创建 Read View，后续复用 |

因此：

-   RC：可能看到其他事务提交的新数据。
-   RR：同一事务内普通查询结果保持一致。

## 一条 UPDATE 是原子性的吗？为什么？

是原子性的。

InnoDB 执行 UPDATE 时会先记录 undo log，保存旧值。如果事务失败或回滚，可以用 undo log 恢复到更新前状态。同时 UPDATE 会对目标记录加锁，避免多个事务同时修改同一行导致冲突。

## 大事务有什么问题

-   锁持有时间长，容易造成阻塞、死锁和锁等待超时。
-   undo log 膨胀，回滚成本高。
-   长事务会阻碍 undo 版本清理，导致历史版本堆积。
-   提交时产生大量 redo log/binlog，影响 I/O。
-   主库大事务提交后，从库回放也需要较长时间，容易造成主从延迟。

优化方向：拆小批次提交，缩短事务时间，避免事务中做远程调用或复杂计算。

# 锁

## MySQL 中有哪些锁？

按粒度：

| 类型   | 说明                                                         |
| ------ | ------------------------------------------------------------ |
| 全局锁 | `FLUSH TABLES WITH READ LOCK`，让整个库只读，常用于全库逻辑备份 |
| 表级锁 | 表锁、元数据锁 MDL、意向锁                                   |
| 行级锁 | InnoDB 的记录锁、间隙锁、临键锁                              |

按模式：

-   共享锁（S 锁）：读锁，多个事务可同时持有。
-   排他锁（X 锁）：写锁，与其他 S/X 锁互斥。

InnoDB 行锁算法：

| 锁            | 含义                                     |
| ------------- | ---------------------------------------- |
| Record Lock   | 锁住某条索引记录                         |
| Gap Lock      | 锁住两条记录之间的间隙，不锁记录本身     |
| Next-Key Lock | Record Lock + Gap Lock，锁住左开右闭范围 |

InnoDB 的行锁是加在索引上的，不是直接加在数据行上的。

## 表锁和行锁有什么区别？

| 维度     | 表锁                  | 行锁                   |
| -------- | --------------------- | ---------------------- |
| 粒度     | 锁整张表              | 锁部分索引记录         |
| 并发度   | 低                    | 高                     |
| 加锁开销 | 小                    | 较大                   |
| 冲突范围 | 大                    | 小                     |
| 死锁概率 | 低                    | 相对更高               |
| 适用场景 | DDL、备份、大批量操作 | 高并发单行或小范围更新 |

## 如果 UPDATE 的范围条件不是主键或索引，还会阻塞吗？

1.  InnoDB 行锁实际上加在索引记录上。
2.  WHERE 条件没有索引：
    -   可能导致全表扫描；
    -   锁范围扩大；
    -   增加锁等待和死锁概率。
3.  RR 隔离级别下范围更新可能涉及：
    -   记录锁（Record Lock）；
    -   间隙锁（Gap Lock）；
    -   Next-Key Lock。
4.  高并发更新必须保证 WHERE 条件命中合适索引。

## 串行化隔离级别通过什么实现

Serializable 通过加锁让事务串行化执行。

在 InnoDB 中，普通 `SELECT` 也会变成加共享锁的当前读，写操作需要排他锁，因此读写、写写之间会互相阻塞。

# 日志

## MySQL 日志

| 日志      | 层级        | 类型       | 作用                         |
| --------- | ----------- | ---------- | ---------------------------- |
| redo log  | InnoDB      | 物理日志   | 崩溃恢复、保证持久性         |
| undo log  | InnoDB      | 旧版本信息 | 回滚、支持 MVCC              |
| binlog    | Server      | 逻辑日志   | 主从复制、备份和增量恢复     |
| relay log | Server 从库 | binlog 副本 | 接收并回放主库 binlog        |

记忆方式：redo/undo 是 InnoDB 层，binlog 是 Server 层。

## binlog

**binlog 特点**

-   Server 层日志，所有存储引擎都可以使用。
-   记录 DDL、DML 等数据变化。
-   不记录普通 `SELECT`。
-   采用追加写方式，生成多个 binlog 文件。

**binlog 格式**

| 格式      | 记录内容   | 特点                       |
| --------- | ---------- | -------------------------- |
| STATEMENT | 原始 SQL   | 日志小，可能存在一致性问题 |
| ROW       | 行数据变化 | 最安全，日志较大           |
| MIXED     | 自动选择   | 两者折中                   |

生产环境通常使用：ROW 格式

## undo log

undo log 记录数据修改前的旧版本，主要作用：

-   事务回滚：执行失败时根据 undo log 恢复旧值。
-   实现 MVCC：快照读需要通过 undo log 版本链读取历史版本。

不同操作的 undo：

-   INSERT：回滚时删除新插入的记录。
-   DELETE：回滚时把旧记录恢复回来。
-   UPDATE：回滚时把更新列恢复为旧值。

## redo log、undo log 和 binlog 如何配合？

日志类型和职责见上表，面试时重点说明：redo log 负责“恢复本机”，binlog 负责“复制和归档”，undo log 负责“回滚和 MVCC”。

**为什么不能只用 binlog？**

-   binlog 不知道 InnoDB 哪些脏页已经刷盘，无法直接恢复数据页状态。
-   redo log 可以根据数据页修改记录重做已提交但未刷盘的数据。
-   因此，redo log 负责“恢复本机”，binlog 负责“复制和归档”，两者职责不同。

**redo log 如何保证持久性？**

采用 WAL（Write-Ahead Logging）：先写 redo log，再异步刷数据页。

```text
修改 Buffer Pool → 写 redo log → 事务提交 → 异步刷数据页
```

redo log 的特点：记录数据页物理修改、顺序写入，并通过 checkpoint 标记已经落盘的数据范围。

刷盘策略由 `innodb_flush_log_at_trx_commit` 控制：

| 值   | 含义                           |
| ---- | ------------------------------ |
| 1    | 每次提交刷 redo log，最安全    |
| 0    | 每秒刷一次，性能高但可能丢数据 |
| 2    | 写系统缓存，不一定立即落盘     |

生产环境通常使用 `1`。

**redo log 和 binlog 的两阶段提交**

1.  InnoDB 写 redo log，并标记为 `prepare`。
2.  Server 层写入并刷盘 binlog。
3.  InnoDB 将 redo log 标记为 `commit`。

崩溃恢复时：

| 状态                               | 处理             |
| ---------------------------------- | ---------------- |
| redo log 没有 prepare              | 事务未提交，回滚 |
| redo log prepare，但 binlog 不完整 | 回滚             |
| redo log prepare，binlog 完整      | 提交             |

核心规则：redo log 处于 `prepare` 且 binlog 完整时，认为事务可以提交。

## Doublewrite Buffer

1.  InnoDB 数据页通常为 **16KB**。
2.  如果写入过程中宕机，可能只写入部分数据页，导致页损坏。
3.  Doublewrite 会先保存完整页副本，再写真实数据文件。
4.  恢复时：
    -   先用 Doublewrite 修复损坏页；
    -   再通过 redo log 重做修改。

# 性能调优

## EXPLAIN

`EXPLAIN` 用于查看 SQL 执行计划，判断查询是否合理。

| 字段  | 含义                                                         |
| ----- | ------------------------------------------------------------ |
| type  | 访问类型，常见从好到差：const、ref、range、index、ALL        |
| key   | 实际使用的索引                                               |
| rows  | 预估扫描行数                                                 |
| Extra | 额外信息，如 Using index、Using where、Using filesort、Using temporary |

常见优化目标：避免全表扫描，减少扫描行数，减少回表，避免临时表和文件排序。

## 发现一张表查询很慢，怎么排查和优化？

**第一步：定位问题**

使用：

-   慢查询日志；
-   监控平台；
-   SQL 执行耗时。

找到具体慢 SQL。

**第二步：分析执行计划**

使用：

```
EXPLAIN SQL
```

关注：

-   是否全表扫描（type=ALL）。
-   是否命中索引。
-   扫描行数（rows）。
-   是否发生回表。
-   是否出现 filesort、temporary。

**第三步：优化 SQL 和索引**

检查：

**索引问题**

-   WHERE 条件是否有索引。
-   JOIN 字段是否有索引。
-   ORDER BY / GROUP BY 是否需要索引。
-   是否存在索引失效：

例如：

```
WHERE DATE(create_time)=?
```

**SQL 问题**

-   避免 `SELECT *`。
-   使用覆盖索引减少回表。
-   深分页的具体优化方案见“深分页（LIMIT 大偏移量）怎么优化”。

**第四步：架构优化**

如果 SQL 和索引已经优化：

-   数据归档；
-   冷热数据分离；
-   分库分表；
-   缓存。

## EXPLAIN 选择的索引不合适，怎么干预？

**1. 更新统计信息**

```
ANALYZE TABLE table_name;
```

让优化器获得更准确的数据分布。

**2. 检查索引设计**

包括：

-   联合索引顺序是否合理。
-   是否存在冗余索引。
-   索引选择性是否合适。

**3. 优化 SQL**

例如：

-   减少无效条件。
-   避免索引失效。
-   让过滤条件更加明确。

**索引提示**

| 方式         | 作用             |
| ------------ | ---------------- |
| FORCE INDEX  | 强制使用指定索引 |
| USE INDEX    | 建议优先考虑索引 |
| IGNORE INDEX | 忽略指定索引     |

## 深分页（LIMIT 大偏移量）怎么优化？

问题根因：LIMIT 100000, 10 会扫描并丢弃前 100000 行，偏移量越大越慢，全表扫描无索引可用。

| 方案       | 原理                                                         | 优点                   | 缺点           | 适用场景           |
| ---------- | ------------------------------------------------------------ | ---------------------- | -------------- | ------------------ |
| 游标分页   | 使用上一页最后一条记录作为游标，例如 `WHERE id > ? LIMIT 10` | 利用索引定位，性能稳定 | 不支持随机跳页 | 列表分页，推荐使用 |
| 延迟关联   | 先查询目标记录 ID，再根据 ID 查询完整数据                    | 减少回表，支持跳页     | SQL 更复杂     | 需要跳页场景       |
| 限制深分页 | 限制最大页数，引导用户增加筛选条件                           | 实现简单，成本低       | 用户体验受限   | 普通业务系统       |

# 高可用与扩展

## MySQL 主从复制流程

主从复制基于 binlog：

1.  主库事务提交时写入 binlog。
2.  从库 I/O 线程连接主库，读取 binlog。
3.  从库把接收到的 binlog 写入本地 relay log。
4.  从库 SQL 线程读取 relay log，并在从库重放，最终实现数据同步。

MySQL 复制默认是异步复制，主库提交成功不等于从库已经同步完成，所以会有主从延迟。

基于主从复制可以做读写分离：写请求走主库，普通读请求走从库，降低主库读压力。但刚写完马上读、资金库存等强一致读通常要走主库，避免读到从库旧数据。

## 主从延迟怎么处理？

常见原因：

-   主库存在大事务或写入压力过大。
-   从库硬件性能不足。
-   从库回放 binlog 的速度较慢。
-   网络延迟或抖动。

处理方案：

-   拆分大事务，减少单次批量写入。
-   提升从库配置，开启并行复制。
-   强一致性读直接走主库。
-   监控复制延迟，延迟过高时摘除从库或降级处理。
-   半同步复制只能降低日志传输延迟，不能解决从库回放延迟。

## 分库和分表是什么？有什么区别？

分库是把数据拆到多个数据库实例中，主要解决单库连接数、CPU、IO、存储容量瓶颈。

分表是把一张大表拆成多张表，主要解决单表数据量过大导致的查询和维护成本问题。

拆分方式：

-   垂直分库/分表：按**业务或字段拆**，比如订单库、用户库；或者把大字段拆到扩展表。
-   水平分库/分表：按用户 ID、订单 ID、时间等规则把**同结构数据拆到多个库或表**。

代价：

-   跨库 JOIN 困难。
-   分布式事务复杂。
-   全局唯一 ID、分页、排序、聚合更复杂。
-   扩容和数据迁移成本高。

所以分库分表通常是数据量和并发真的到瓶颈后再做，优先通过索引、SQL、缓存、读写分离等手段优化。
