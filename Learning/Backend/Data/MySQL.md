# MySQL

MySQL 这篇的重点不是把所有语法点背完，而是建立一条比较稳定的理解主线：数据怎么存、SQL 怎么跑、索引怎么加、事务怎么保证一致、出了性能问题怎么排查。

## 1. 先建立总框架

理解 MySQL，最值得抓住的几件事是：

- 表和索引怎么组织数据
- 一条 SQL 是怎么被执行的
- 事务、锁、MVCC 怎么一起工作
- 日志系统怎么支撑恢复和复制
- 性能瓶颈通常会出现在哪里

一句话概括：

> MySQL 的核心能力，主要由 B+Tree 索引、事务机制、锁与 MVCC、日志系统以及执行计划优化共同支撑。

## 2. SQL 基础

### SQL 和 NoSQL 有什么区别？
- SQL 通常指关系型数据库，数据以表结构组织，强调结构化数据、事务一致性和 SQL 查询能力。
- NoSQL 指非关系型数据库，常见模型包括 KV、文档、列族、图等，更强调灵活结构和水平扩展。
- 但 NoSQL 不是一个统一能力集合，不同产品差异很大，不能把某个产品特性当成全部 NoSQL 的共性。

### 数据库三大范式是什么？
- 第一范式（1NF）：字段具有原子性，不能再拆。
- 第二范式（2NF）：在 1NF 基础上，非主属性必须完全依赖候选键。
- 第三范式（3NF）：在 2NF 基础上，非主属性不能依赖其他非主属性。

### MySQL 怎么做连表查询？
常见 JOIN：

| 类型 | 含义 |
| --- | --- |
| `INNER JOIN` | 只返回两表都匹配的数据 |
| `LEFT JOIN` | 保留左表全部数据，右表不匹配则为 `NULL` |
| `RIGHT JOIN` | 保留右表全部数据，左表不匹配则为 `NULL` |
| `FULL JOIN` | MySQL 不直接支持，可用 `LEFT JOIN UNION RIGHT JOIN` 模拟 |

### MySQL 如何避免重复插入数据？
- 建唯一索引或唯一约束，从数据库层保证唯一性。
- 使用 `INSERT IGNORE`，冲突时忽略该行。
- 使用 `INSERT ... ON DUPLICATE KEY UPDATE`，冲突时转更新。
- 业务侧“先查再插”在高并发下并不可靠，最终仍要靠唯一约束兜底。

### CHAR 和 VARCHAR 有什么区别？
- `CHAR(n)` 是定长字符串，适合长度稳定的数据。
- `VARCHAR(n)` 是变长字符串，更适合长度波动大的字段。
- 一般情况下，长度不固定的业务字段优先用 `VARCHAR`。

### INT(11) 里的 11 表示什么？
- 它不是存储字节数，也不是最多存 11 位数字。
- 它只是历史上的显示宽度概念。
- 真正的 `INT` 始终占 4 字节。

### TEXT 和 VARCHAR 有什么区别？
- `VARCHAR` 适合普通业务字段。
- `TEXT` 适合较长文本，如文章内容、日志详情。
- `TEXT` 在默认值、索引、排序、临时表等方面限制更多。

### IN 和 EXISTS 有什么区别？
- `IN` 常用于子查询结果集较小的情况。
- `EXISTS` 更适合外层表大、内层能快速命中的场景。
- 现代优化器会做一定重写，但面试时这样理解通常足够。

### SQL 的逻辑执行顺序是什么？
- `FROM -> JOIN -> ON -> WHERE -> GROUP BY -> HAVING -> SELECT -> DISTINCT -> ORDER BY -> LIMIT`
- 这也是为什么 `WHERE` 里通常不能直接用 `SELECT` 别名，而 `ORDER BY` 往往可以。

## 3. 执行流程与存储引擎

### 一条 SQL 请求在 MySQL 中如何执行？
- 连接器建立连接、认证和权限校验。
- 解析器做词法和语法分析。
- 预处理器检查表和字段。
- 优化器选择索引和执行计划。
- 执行器按计划调用存储引擎。
- 存储引擎真正读写数据页、索引和日志。

### MySQL 常见存储引擎有哪些？
| 引擎 | 特点 | 适用场景 |
| --- | --- | --- |
| `InnoDB` | 默认引擎，支持事务、行锁、MVCC、崩溃恢复 | 绝大多数 OLTP 场景 |
| `MyISAM` | 不支持事务和行锁，表锁，崩溃恢复弱 | 历史遗留或只读场景 |
| `MEMORY` | 数据在内存中，重启丢失 | 临时数据、缓存类场景 |

### 为什么 InnoDB 会成为默认引擎？
- 支持事务
- 支持行级锁
- 支持崩溃恢复
- 支持 MVCC
- 综合能力更适合现代业务系统

## 4. 索引

### 索引有哪些分类？
- 按数据结构：B+Tree、Hash、Full-text。
- 按物理存储：聚簇索引、二级索引。
- 按字段特性：主键索引、唯一索引、普通索引、前缀索引。
- 按字段数量：单列索引、联合索引。

### 为什么 InnoDB 默认用 B+Tree？
- 适合页式存储和磁盘 IO。
- 树高低，查询稳定。
- 支持范围查询和排序。
- 相比 Hash，B+Tree 不只支持等值查询，也支持最左匹配和范围扫描。

### 什么是聚簇索引和二级索引？
- 在 InnoDB 中，主键索引通常就是聚簇索引，叶子节点直接存整行数据。
- 普通索引、唯一索引等通常是二级索引，叶子节点存主键值，再回表取整行数据。

### 为什么建议主键尽量短、稳定、递增？
- 主键越短，二级索引越省空间。
- 主键稳定可以减少更新主键带来的整行移动和维护成本。
- 递增主键更有利于顺序插入，减少页分裂。

### 联合索引为什么要遵循最左匹配？
- 联合索引底层按最左列优先排序。
- 只有从最左列开始连续命中条件时，才能更充分利用索引的有序性。
- 例如 `(a, b, c)`：
  `where a = ?` 可以用；
  `where a = ? and b = ?` 可以用；
  `where b = ?` 通常不能直接高效利用。

### 什么是覆盖索引？
- 如果查询所需字段都能从索引里直接拿到，就不需要回表，这就是覆盖索引。
- 它通常可以减少 IO，提高查询性能。

### 什么情况会导致索引失效？
- 左模糊或全模糊
- 对索引列做函数或表达式计算
- 隐式类型转换
- 联合索引不满足最左匹配
- `OR` 一侧无索引
- 返回数据量太大时，优化器主动放弃索引

## 5. 事务与 MVCC

### 事务的 ACID 特性是什么？InnoDB 怎么实现？
- 原子性依赖 undo log。
- 一致性由原子性、隔离性、持久性共同保证。
- 隔离性依赖锁和 MVCC。
- 持久性依赖 redo log 和 WAL。

### 事务隔离级别有哪些？
- 读未提交
- 读已提交
- 可重复读
- 串行化
- MySQL InnoDB 默认是可重复读。

### 脏读、不可重复读、幻读分别是什么？
- 脏读：读到别人还没提交的数据。
- 不可重复读：同一事务里多次读同一行，结果不一致。
- 幻读：按条件查询时，多次读取到的记录数量或范围发生变化。

### 什么是 MVCC？
- MVCC 是多版本并发控制。
- 它通过隐藏字段、undo log、Read View 等机制，让读尽量不加锁地看到合适版本的数据。
- 核心目标是提升并发下的读性能。

### 为什么 InnoDB 在可重复读下还能较好解决幻读？
- 快照读主要靠 MVCC。
- 当前读如 `select ... for update`、`update`、`delete`，主要靠 next-key lock 限制范围内插入，从而降低幻读问题。

### 为什么不建议大事务？
- 锁持有时间长
- undo log / redo log 压力大
- 回滚成本高
- 主从延迟更明显
- 崩溃恢复时间更长

## 6. 锁

### InnoDB 常见锁有哪些？
- 共享锁（S）
- 排他锁（X）
- 行锁
- 表锁
- 间隙锁
- Next-Key Lock

### 为什么说 InnoDB 本质上锁的是索引？
- InnoDB 的行锁是基于索引项加锁实现的。
- 如果查询没走索引，可能退化成扫描更多记录，带来更大范围的锁影响。

### 什么情况下容易发生死锁？
- 两个事务以不同顺序加锁同一批资源时，最容易发生死锁。
- 常见优化：
  固定加锁顺序；
  缩短事务；
  尽量走索引；
  大事务拆小。

## 7. 日志系统

### redo log、undo log、binlog 分别是什么？
- `redo log`：InnoDB 引擎层物理日志，用于崩溃恢复。
- `undo log`：记录修改前信息，用于回滚和 MVCC。
- `binlog`：Server 层逻辑日志，用于主从复制和归档恢复。

### 为什么既要 redo log，又要 binlog？
- redo log 负责“恢复本机”。
- binlog 负责“复制和归档”。
- 两者职责不同，所以都需要。

### 什么是两阶段提交？
- 为了避免 redo log 和 binlog 状态不一致。
- 提交时先写 redo log prepare，再写 binlog，最后标记 redo log commit。
- 崩溃恢复时可以据此判断事务是否真正提交成功。

## 8. 性能调优

### EXPLAIN 有什么作用？
- 用来查看 SQL 执行计划。
- 常重点关注 `type`、`key`、`rows`、`Extra`。
- 用来判断是否走索引、扫描是否过大、是否出现 `filesort` 或 `temporary`。

### SQL 调优一般从哪些方面入手？
- 看执行计划，判断是否走索引。
- 调整索引，尤其是联合索引顺序。
- 尽量少用 `select *`。
- 避免在索引列上做函数、计算和隐式转换。
- 拆复杂 SQL，减少无意义连表。
- 控制深分页。

### 为什么深分页慢？怎么优化？
- `limit 100000, 20` 这类深分页需要先扫描并丢弃前面大量记录。
- 常见优化：
  使用游标或基于上次最大 ID 的方式分页；
  先走覆盖索引查主键，再回表取详情；
  尽量减少大范围排序和扫描。

## 9. 主从复制与分库分表

### MySQL 主从复制大致流程是什么？
- 主库写 binlog。
- 从库 IO 线程拉取 binlog 写入 relay log。
- 从库 SQL 线程或并行回放线程重放 relay log。
- 从而把数据同步过去。

### 为什么会出现主从延迟？
- 主库写入压力大
- 从库回放能力不足
- 大事务
- 从库还承担查询压力
- 网络抖动

### 什么时候需要分库分表？
- 当单库单表在容量、并发、IO、维护窗口等方面接近瓶颈时。
- 并且已经尝试过索引优化、读写分离、冷热分离等手段，仍无法满足需求。

## 10. 一页答题模板

如果被问“你怎么理解 MySQL”，可以这样回答：

> 我会把 MySQL 理解成一个以 InnoDB 为核心的关系型数据库系统，它通过 B+Tree 索引组织数据，通过事务、锁和 MVCC 解决并发一致性，通过 redo log、undo log 和 binlog 解决恢复与复制。实际开发里最常见的问题通常集中在索引设计、SQL 执行计划、事务粒度、锁冲突和主从延迟这些地方。

## 11. 一页总结

- MySQL 的核心主线是索引、事务、锁、MVCC、日志和执行计划
- InnoDB 之所以常用，是因为它把事务和高并发能力补全了
- 索引不是越多越好，关键在于是否和查询模式匹配
- 事务问题很多时候最终会表现成锁冲突、延迟或恢复成本
- 调优通常从执行计划、索引设计和事务范围开始
