# Redis

Redis 不只是“缓存工具”，而是一个基于内存的数据结构服务器。真正理解 Redis，重点不在背命令，而在于知道它适合解决什么问题、为什么快、哪里容易出问题。

## 1. 先建立总框架

Redis 最常见的价值有 4 类：

- 做缓存，减轻数据库读压力
- 做高频数据结构，如计数器、排行榜、去重集合
- 做协调工具，如分布式锁、延迟队列
- 做高性能中间层，承接热点流量

但 Redis 也不是银弹，它的风险点通常在：

- 数据最终还是以内存为核心
- 一致性天然弱于关系型数据库
- 热 key、大 key、缓存雪崩都可能把系统打抖
- 高可用、持久化、淘汰策略都需要明确设计

## 2. 数据结构与使用场景

### Redis 常见数据类型和应用场景是什么？
- String：缓存对象、计数器、分布式锁、Session。
- Hash：对象缓存、购物车、用户画像。
- List：简单队列、消息列表、时间线。
- Set：去重、点赞、抽奖、共同关注。
- Zset：排行榜、延迟队列、按分数排序。
- Bitmap：签到、布尔状态统计。
- HyperLogLog：海量 UV 统计。
- GEO：附近的人、门店距离。
- Stream：支持消费组和 ACK 的消息流。

### Redis 底层有哪些核心数据结构？
- SDS
- dict
- listpack
- quicklist
- intset
- skiplist
- Redis 会在省内存和操作效率之间做权衡：小对象优先用紧凑结构，数据变大后再切换到更适合查询或修改的结构。

### String 为什么不用 C 字符串，而用 SDS？
- O(1) 获取长度
- 支持二进制安全
- 减少缓冲区溢出风险
- 通过空间预分配和惰性释放减少频繁内存重分配

### Hash、List、Set、Zset 底层怎么实现？
- Hash：小对象时可能用 listpack，变大后转成哈希表。
- List：主要由 quicklist 实现，本质是双向链表结合紧凑列表。
- Set：元素少且全是整数时可能用 intset，否则用哈希表。
- Zset：通常由跳表和哈希表共同实现。

## 3. Redis 为什么快

### Redis 为什么快？
- 主要基于内存操作
- 数据结构设计高效
- 主命令执行流程单线程，减少锁竞争
- 使用 I/O 多路复用
- RESP 协议简单
- 后台线程处理 fsync、lazyfree 等耗时任务

### Redis 单线程指的是什么？
- 通常指命令解析、命令执行、数据读写主流程由主线程串行完成。
- 但 Redis 并不是整个进程只有一个线程，还会有 BIO、lazyfree、I/O 读写辅助线程等。

### 什么是 I/O 多路复用？
- Redis 用一个线程同时监听多个连接。
- 哪个连接准备好读写，由操作系统通知。
- 这样不必为每个连接创建一个线程，也能高效处理大量并发连接。

## 4. 原子性、事务和 Lua

### Redis 如何保证原子性？
- 单条命令由主线程串行执行，天然具备原子性。
- 多条命令一起保证原子性时，通常用 Lua 脚本或 `MULTI/EXEC`。
- 需要“读-判-写”整体不可打断时，Lua 更合适。

### Redis 事务能像数据库事务一样回滚吗？
- 不能。Redis 事务能保证命令按顺序连续执行，但没有传统数据库那种完整回滚语义。
- 某条命令执行失败时，其他命令仍可能继续执行。

### 为什么 Lua 脚本常用于 Redis 场景？
- Lua 脚本会在 Redis 中一次性串行执行。
- 非常适合“先读取条件，再判断，再写入”这种需要整体原子性的逻辑。

## 5. 持久化

### Redis 持久化方式有哪些？
- RDB
- AOF
- 混合持久化

### RDB 和 AOF 有什么区别？
- RDB：时间点快照，恢复快、文件紧凑，但两次快照之间可能丢数据。
- AOF：按写命令追加日志，数据更完整，但文件更大、恢复更慢。
- 混合持久化：结合两者优点，通常先写 RDB，再接增量 AOF。

### AOF 重写是什么？
- 不是简单压缩旧文件。
- 而是根据当前内存状态生成一份能恢复现状的最简命令集，从而减小 AOF 文件体积。

## 6. 过期删除与内存淘汰

### 过期删除和内存淘汰有什么区别？
- 过期删除回答的是：TTL 到了以后什么时候删。
- 内存淘汰回答的是：内存满了以后删谁。

### Redis 的过期删除策略有哪些？
- 定时删除：到期立刻删，但成本高。
- 惰性删除：访问时发现过期才删。
- 定期删除：周期性抽查一批 key 删除。
- Redis 实际上主要使用“惰性删除 + 定期删除”。

### Redis 的内存淘汰策略有哪些？
- `noeviction`
- `allkeys-lru`
- `volatile-lru`
- `allkeys-lfu`
- `volatile-lfu`
- `allkeys-random`
- `volatile-random`
- `volatile-ttl`
- 生产里较常见的是 `allkeys-lru` 或 `allkeys-lfu`。

## 7. 高可用与集群

### Redis 主从同步里的全量同步和部分同步怎么做？
- 全量同步通常由 `PSYNC` 触发，主库生成 RDB 发给从库，再补发缓冲期间的增量命令。
- 部分同步依赖 replication id、offset 和 `repl_backlog_buffer` 补发缺失数据。

### 哨兵模式解决了什么问题？
- 主要解决主从架构下的高可用问题。
- 包括节点监控、自动故障转移、通知客户端新主节点地址。
- 但哨兵不负责数据分片。

### Redis Cluster 怎么分片？
- Redis Cluster 通过 16384 个 hash slot 分片。
- key 先做 CRC16，再对 16384 取模，落到对应 slot。
- 这不是一致性哈希，而是固定槽位分片。

### Cluster 和哨兵模式有什么区别？
- 哨兵模式：解决高可用，不做数据分片。
- Cluster：既做高可用，也做分片扩展。
- 如果只是避免单点故障，哨兵通常就够；如果单机容量或吞吐不够，才需要 Cluster。

## 8. 缓存设计

### 如何保证 Redis 和 MySQL 缓存一致性？
- 常见模式是 Cache Aside。
- 读：先查缓存，未命中再查数据库并回填。
- 写：先更新数据库，再删除缓存。
- 再配合重试、延迟双删、binlog 订阅等机制，保证最终一致。

### 缓存雪崩、击穿、穿透是什么？怎么解决？
- 雪崩：大量 key 同时失效，或 Redis 本身故障。
- 击穿：单个热点 key 过期，大量请求同时打到底层。
- 穿透：请求的数据根本不存在，每次都打到数据库。
- 常见手段：随机 TTL、互斥锁、空值缓存、布隆过滤器、限流、多级缓存。

### 为什么会有大 key 和热 key 问题？
- 大 key：单个 key 对应 value 很大，删除、迁移、同步都会抖动。
- 热 key：某个 key 访问量极高，可能把单个节点打成瓶颈。
- 大 key 常见处理：拆分、异步删除、结构优化。
- 热 key 常见处理：本地缓存、多副本、预热、限流、读写分离。

## 9. Redis 分布式锁

### Redis 分布式锁怎么实现？
- 最基础的做法是：

```bash
SET lock_key unique_value NX PX 30000
```

- `NX`：只有 key 不存在时才设置成功。
- `PX`：设置过期时间，避免死锁。
- `unique_value`：标识锁持有者，释放时必须校验。

### 为什么释放锁时要用 Lua 脚本？
- 因为“先判断 value，再删除 key”如果分成两条命令，中间可能被别人插入，导致误删别人的锁。
- Lua 脚本可以把校验和删除做成原子操作。

## 10. 一页答题模板

如果被问“你怎么理解 Redis”，可以这样回答：

> Redis 是一个基于内存的高性能键值数据库，支持多种数据结构，常用于缓存、计数、排行、去重、分布式锁和热点数据承载。它快的原因主要是内存操作、高效数据结构和事件驱动模型，但实际使用时也要重点关注持久化、高可用、缓存一致性、热 key 和大 key 等问题。

## 11. 一页总结

- Redis 本质上是内存型数据结构服务器
- 数据类型决定它不仅能做缓存，还能做计数、排行、去重和协调
- 快的核心来自内存、高效结构和事件驱动
- 高并发场景里重点关注缓存一致性、热 key、大 key、雪崩击穿穿透
- 高可用和持久化策略决定它能不能稳定跑在线上
