# Redis

## 数据结构与编码

## Redis 有哪些常见数据类型？分别适合什么场景？

Redis 是 key-value 数据库，但 value 支持多种数据结构。

| 类型 | 常见场景 |
| --- | --- |
| String | 缓存对象、计数器、分布式锁、Session |
| Hash | 对象缓存、用户信息、购物车 |
| List | 简单队列、最新列表、栈/队列 |
| Set | 去重、点赞、抽奖、共同关注、交并差集 |
| Zset | 排行榜、延迟队列、按分数排序 |
| Bitmap | 签到、在线状态、布尔状态统计 |
| HyperLogLog | 海量 UV、近似去重计数 |
| GEO | 附近的人、门店距离查询 |
| Stream | 带消息 ID、消费组和 ACK 的轻量消息队列 |

## Java 客户端与 Spring Data Redis 怎么选？

- Jedis 的连接对象不是线程安全的，并发使用时应通过连接池管理，不能把单个 Jedis 实例作为全局共享对象。
- Spring Data Redis 的常见 Spring Boot 配置使用 Lettuce；具体客户端由依赖和连接工厂决定，选型时要看版本和线程模型。
- `RedisTemplate` 默认的对象序列化通常是 JDK 序列化，可读性差、体积也可能较大。
- 生产环境要显式统一 key 和 value 的序列化策略：可以自定义 `RedisTemplate` 使用 JSON，也可以使用 `StringRedisTemplate`，由业务显式完成 JSON 序列化和反序列化。
- 序列化策略一旦确定，应在不同服务之间统一；修改类型信息或字段结构时要考虑旧缓存的兼容和淘汰。

## Redis 底层有哪些核心数据结构？

Redis 会根据数据规模和编码条件，在节省内存与操作效率之间做权衡。

| 用户类型 | 小数据编码 | 大数据编码 |
| --- | --- | --- |
| String | SDS | SDS |
| Hash | listpack | dict |
| List | quicklist/listpack | quicklist |
| Set | intset | hashtable |
| Zset | listpack | skiplist + dict |

其他核心结构包括：

- SDS：Redis 自己实现的动态字符串，也用于 key 和 AOF 缓冲区。
- dict：哈希表，用于 Hash 和 Redis 的全局 key 空间。
- listpack：紧凑的连续内存结构，Redis 7.0 后逐步替代 ziplist。
- quicklist：由多个 listpack 节点组成的双向链表。
- skiplist：支持有序遍历、范围查询和排名。
- intset：整数 Set 的紧凑编码。

## String 为什么使用 SDS，而不是 C 字符串？

SDS 记录已使用长度和已分配容量，具备以下优点：

- 获取长度是 O(1)，不需要遍历到 `\0`。
- 二进制安全，可以保存包含 `\0` 的数据。
- 拼接前可以检查容量，降低缓冲区溢出风险。
- 支持空间预分配和惰性释放，减少内存重分配。
- 保留末尾 `\0`，仍可兼容部分 C 字符串函数。

## Redis 哈希表如何扩容和 rehash？

dict 扩容时会同时使用 `ht[0]` 和 `ht[1]`，通过渐进式 rehash 避免一次性迁移大量数据阻塞主线程。

1. 为 `ht[1]` 分配新空间，并将 `rehashidx` 设为 0。
2. 每次请求或定时任务迁移一部分 `ht[0]` 的桶。
3. rehash 期间查询、删除、更新需要检查两张表。
4. 新增数据直接写入 `ht[1]`。
5. 迁移完成后释放旧表，让 `ht[1]` 成为新的 `ht[0]`。

## Zset 底层是怎么实现的？为什么使用跳表？

大数据量的 Zset 通常由两种结构组成：

- dict：保存 member 到 score 的映射，按成员查分数是 O(1)。
- skiplist：按 score 和 member 排序，支持范围查询、排名和顺序遍历。

小数据量时通常使用 listpack；超过配置阈值后转为 skiplist + dict。score 相同时，按 member 的字典序排序。

跳表是多层有序链表，底层保存全部节点，上层作为索引，平均查找复杂度为 O(logN)。Redis 使用跳表主要因为：

- 实现简单，插入和删除只需调整局部指针。
- 适合范围查询和顺序遍历。
- 内存场景不需要像 B+ 树那样围绕磁盘页进行优化。

## ziplist、listpack 和 quicklist 有什么区别？

- ziplist：早期的紧凑连续内存结构，省内存，但节点的 `prevlen` 变化可能引发连锁更新。
- listpack：不记录前一个节点长度，减少连锁更新，Redis 7.0 后逐步替代 ziplist。
- quicklist：双向链表加多个 listpack，兼顾两端操作效率和内存利用率，主要用于 List。

## 线程模型与原子性

## Redis 为什么快？单线程和 I/O 多路复用分别做什么？

Redis 的高性能来自多个因素：

- 数据主要在内存中，数据结构操作效率高。
- 命令执行主流程由一个线程串行完成，避免锁竞争和线程切换。
- I/O 多路复用让一个线程可以监听大量连接。
- RESP 协议简单，解析成本低。
- 后台线程处理异步释放内存、文件关闭等耗时任务。

Redis 6.0 后可以使用 I/O 多线程分担 socket 读写，但命令执行仍由主线程完成。`KEYS`、大范围 `ZRANGE`、大 key 的 `HGETALL` 和 `DEL` 仍可能阻塞主线程。

## Redis 的 I/O 模型是怎样的？

Redis 使用 Reactor 模型：

1. 客户端连接生成 socket fd。
2. Redis 将 fd 注册到事件循环。
3. 内核通过 epoll、kqueue 等机制监听读写事件。
4. 事件就绪后，Redis 读取请求、执行命令并返回响应。

Redis 6.0 前，网络事件和命令执行主要在主线程；Redis 6.0 后，网络 I/O 可以由多线程分担，但数据修改和命令执行仍保持串行。

## Redis Pipeline 解决什么问题？

Pipeline 把多条命令一次性发送给 Redis，再批量读取响应，减少网络往返次数，适合批量读写和延迟敏感的场景。它只是通信层面的批处理，不提供事务隔离，也不能保证多条命令失败时自动回滚；需要原子性时仍应考虑事务或 Lua。

## Redis 如何统计 UV？

UV 是按用户或 IP 去重后的访问人数；PV 只统计访问次数。小规模可用 Set 精确去重，但内存随用户数增长。海量 UV 通常用 HyperLogLog：

- `PFADD uv:2026-07-16 user_id...`：写入并自动去重。
- `PFCOUNT uv:2026-07-16`：统计近似基数。
- `PFMERGE uv:week key1 key2...`：合并多个时间片。

HyperLogLog 内存固定且很小，但有约 0.81% 的标准误差，适合趋势分析，不适合精确计费或结算。按天、小时拆 key 便于查询和过期清理。

## Redis 如何实现延时任务？

优先使用专业 MQ 的延时消息；若用 Redis，可将任务放入 **Sorted Set**，`score` 为到期时间，后台 worker 扫描到期任务并投递到消费队列，再执行幂等处理。

- 多实例消费时，要用 Lua 或带锁的方式原子地“取出并标记”，避免重复执行。
- Redis 过期事件通知是在服务删除 key 时触发，受惰性删除、定期删除、通知丢失和 Pub/Sub 不持久化影响，不能作为唯一可靠机制。
- 任务必须保留重试、失败补偿和定时扫描兜底；Redisson 的延时队列本质上也是有序集合加消费队列。

## Redis 如何保证原子性？事务、Lua 和 WATCH 有什么区别？

- 单条命令：命令执行期间不会被其他命令插入，通常具备原子性。
- MULTI/EXEC：命令先排队，EXEC 后按顺序连续执行，但没有传统数据库的回滚能力。
- Lua：把读、判断、写封装为一个整体执行，适合分布式锁解锁、库存扣减等场景。
- WATCH：监听 key 是否被修改，适合实现乐观锁；如果 key 被修改，事务会放弃执行。

事务错误需要区分：入队阶段的语法错误可能导致整个事务不执行；执行阶段的类型错误只会让对应命令失败，其他命令仍会继续执行。

## 持久化

## Redis 有哪些持久化方式？

| 方式 | 原理 | 优点 | 缺点 |
| --- | --- | --- | --- |
| RDB | 某一时刻的内存快照 | 文件小、恢复快、适合备份 | 两次快照之间可能丢数据 |
| AOF | 追加记录写命令 | 数据丢失较少、可读性较好 | 文件较大、恢复时需要重放 |
| 混合持久化 | RDB 文件加增量 AOF | 兼顾恢复速度和数据完整性 | 实现和运维更复杂 |

生产环境常根据数据重要性选择 AOF、RDB 或混合持久化。持久化不能替代数据库，缓存数据通常可以接受一定程度的数据丢失。

## RDB 和 AOF 分别是怎么工作的？

### RDB

1. 通过 `save` 配置、`BGSAVE` 或主从全量同步触发。
2. Redis fork 子进程。
3. 子进程把 fork 时刻的内存数据写入临时 RDB 文件。
4. 写完后替换旧文件。

父子进程通过写时复制 COW 共享内存页。主进程继续处理请求，被修改的内存页才会复制，因此快照保持 fork 时刻的状态；但 fork 仍可能带来内存和 CPU 开销。

### AOF

写命令先追加到 AOF 缓冲区，再按 `appendfsync` 策略刷盘：

- `always`：每条命令刷盘，安全但性能较低。
- `everysec`：通常每秒刷盘一次，是常见折中方案。
- `no`：交给操作系统决定，性能好但数据丢失风险更高。

### AOF Rewrite

AOF rewrite 不是压缩旧日志，而是根据当前数据生成等价的新 AOF。rewrite 期间产生的增量命令会被暂存，子进程完成后由主进程追加到新文件，再替换旧文件。

## 过期删除与内存淘汰

## 过期删除和内存淘汰有什么区别？

| 对比项 | 过期删除 | 内存淘汰 |
| --- | --- | --- |
| 触发条件 | key 设置的 TTL 到期 | 内存达到 `maxmemory` |
| 解决问题 | 删除已经过期的 key | 为新写入腾出空间 |
| 常见策略 | 惰性删除 + 定期删除 | LRU、LFU、random、TTL 等 |

## Redis 的过期 key 会立即删除吗？

不会。Redis 采用两种方式：

- 惰性删除：访问 key 时检查 TTL，发现过期后再删除。
- 定期删除：周期性抽样过期字典，清理过期 key，并限制单次清理时间。

这样可以避免大量 key 同时过期时集中消耗 CPU，但冷数据过期后可能暂时占用内存。

## Redis 有哪些内存淘汰策略？

常见策略如下：

| 策略 | 含义 |
| --- | --- |
| `noeviction` | 不淘汰，写入失败 |
| `volatile-lru` | 在有 TTL 的 key 中淘汰最近最少使用的 |
| `allkeys-lru` | 在所有 key 中淘汰最近最少使用的 |
| `volatile-lfu` | 在有 TTL 的 key 中淘汰访问频率最低的 |
| `allkeys-lfu` | 在所有 key 中淘汰访问频率最低的 |
| `volatile-random` / `allkeys-random` | 随机淘汰 |
| `volatile-ttl` | 在有 TTL 的 key 中优先淘汰剩余 TTL 较短的 |

LRU 更关注最近访问时间，适合热点变化快的场景；LFU 更关注访问频率，适合长期热点明显的场景。纯缓存通常考虑 `allkeys-lru` 或 `allkeys-lfu`。

## 高可用与集群

## Redis 主从同步中的全量同步和部分同步怎么实现？

核心命令是 `PSYNC`。

### 全量同步

1. 从库发送 `PSYNC`。
2. 主库无法满足部分同步时执行 `BGSAVE`。
3. 主库发送 RDB，从库清空旧数据并加载 RDB。
4. RDB 生成和传输期间的新写命令暂存在该从库的 replication buffer 中。
5. 从库加载完成后，主库发送这些增量命令。

### 部分同步

1. 主从双方维护 replication id 和复制偏移量。
2. 主库用 `repl_backlog_buffer` 环形缓冲区保存最近的写命令。
3. 从库重连时携带旧 replication id 和 offset。
4. 如果缺失数据仍在 backlog 中，主库只补发缺失部分；否则退化为全量同步。

replication buffer 用于全量同步期间暂存命令；`repl_backlog_buffer` 用于断线后的部分同步，二者不要混淆。

## Sentinel 是如何实现高可用的？

Sentinel 负责监控、故障判断、选主、通知和故障转移。

- 主观下线：某个 Sentinel 认为 master 超时无响应。
- 客观下线：达到 quorum 的 Sentinel 都认为 master 下线。
- Leader 选举：Sentinel 之间通过多数投票选出执行故障转移的 Leader。
- 选主：过滤不健康、断线过久或优先级为 0 的副本，再优先选择 `slave-priority` 数值更小、复制偏移量更新的副本；仍相同时按 runid 选择。
- 切换完成后，其他副本复制新 master，并通知客户端更新主节点地址。

## Redis Cluster 是怎么分片和路由的？

Redis Cluster 使用哈希槽，不是一致性哈希：

- 集群共有 16384 个 slot。
- key 经 CRC16 计算后对 16384 取模，得到所属 slot。
- master 节点负责一部分 slot，扩缩容本质上是迁移 slot。
- 访问错误节点时返回 `MOVED`；迁移过程中可能返回 `ASK`。

多 key 操作要求 key 位于同一个 slot，可以使用 hash tag，例如 `user:{1001}:profile` 和 `user:{1001}:orders`。跨节点事务、Lua 和批量操作会受到限制。

## Redis 主从和 Cluster 能保证强一致性吗？

不能直接认为它们提供强一致性。主从复制默认异步，主库写成功后可能还未同步到副本，故障切换时可能丢失最近写入。

可以使用 `WAIT`、`min-replicas-to-write` 和 `min-replicas-max-lag` 降低风险，但它们不能把 Redis 变成严格强一致存储。关键数据仍应以数据库或其他强一致存储为准。

## 缓存设计

## 为什么使用 Redis？Redis 和 MySQL 有什么区别？

Redis 适合高频、简单、低延迟的数据访问，常用于缓存、计数、排行榜、分布式锁和轻量队列。MySQL 更适合复杂查询、事务、强一致和长期可靠存储。

Redis 在许多简单读写场景中更快，主要因为：

- 数据主要在内存中。
- 数据结构和命令模型更简单。
- 命令执行和网络协议开销较低。

不能简单理解为 Redis 可以替代 MySQL；实际性能还取决于数据规模、命令、索引、网络和缓存命中率。

## 本地缓存和 Redis 缓存有什么区别？

| 对比项 | 本地缓存 | Redis 缓存 |
| --- | --- | --- |
| 延迟 | 极低，无网络开销 | 有网络开销 |
| 共享性 | 每个实例一份 | 多实例共享 |
| 容量 | 受单机内存限制 | 可通过集群扩展 |
| 一致性 | 多实例间较难保持 | 更容易统一管理 |
| 常见用途 | 一级缓存、热点数据 | 二级缓存、共享缓存 |

常见架构是本地缓存 + Redis 二级缓存，但需要处理失效通知和短暂不一致。

## 如何保证 Redis 和 MySQL 的缓存一致性？

常见方案是 Cache Aside，通常追求最终一致性。

### 读流程

1. 先读缓存。
2. 未命中时查询数据库。
3. 将结果写回缓存并设置 TTL。

### 写流程

1. 先更新数据库。
2. 再删除缓存。

删除缓存通常比更新缓存更稳妥，因为更新缓存可能基于旧值计算，也容易遗漏字段。删除失败时可以使用消息队列重试、订阅 binlog 异步删除或其他补偿机制。

延迟双删只能降低并发读写造成脏数据的概率，不能严格保证一致性；强一致场景需要锁、版本号或其他更严格的方案。

## 缓存雪崩、击穿和穿透有什么区别？

| 问题 | 含义 | 常见解决方案 |
| --- | --- | --- |
| 雪崩 | 大量 key 同时过期，或 Redis 故障 | TTL 加随机值、多级缓存、限流降级、高可用 |
| 击穿 | 单个热点 key 过期，大量请求同时回源 | 互斥锁、逻辑过期、后台刷新 |
| 穿透 | 请求的数据缓存和数据库都不存在 | 参数校验、缓存空值、布隆过滤器 |

### 布隆过滤器原理

布隆过滤器由位图和多个哈希函数组成：

- 写入时，将多个哈希位置设为 1。
- 查询时，任意位置为 0，说明一定不存在。
- 所有位置为 1，只能说明可能存在。

它有误判可能，没有误判不存在；普通布隆过滤器也不能直接删除元素。

## 分布式锁与高并发场景

## Redis 分布式锁怎么实现？

加锁使用带唯一 value 和过期时间的原子命令：

```text
SET lock_key unique_value NX PX 10000
```

- `NX`：只有 key 不存在时才设置，保证互斥。
- `PX`：设置过期时间，避免客户端宕机后永久持锁。
- `unique_value`：标识锁的持有者，避免误删其他客户端的锁。

解锁不能直接 `DEL`，必须用 Lua 原子判断 value 后再删除：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
  return redis.call("DEL", KEYS[1])
else
  return 0
end
```

注意：锁的过期时间要覆盖业务执行时间；长任务需要续期机制。关键业务还要考虑锁过期后旧客户端继续执行的问题，必要时使用 fencing token 或其他更严格的方案。RedLock 有争议，不应把它当作所有场景的默认方案。

## 什么是大 key？有什么危害？

大 key 没有统一阈值，需要结合数据大小、元素数量、访问频率和延迟目标判断。常见经验是 String 超过 1 MB、集合超过 1 万个元素时需要重点关注。

危害包括：

- 单次操作耗时长，阻塞主线程。
- 网络传输、主从同步和 AOF rewrite 成本高。
- 删除时释放大量内存，造成延迟抖动。
- 在 Cluster 中造成数据倾斜。

发现方式：`redis-cli --bigkeys`、`MEMORY USAGE key`、slowlog，以及 CPU、QPS 和带宽监控。

解决方式：拆分 key、避免 `HGETALL`/`SMEMBERS` 全量读取、使用 `SCAN` 或分页，并用 `UNLINK` 异步删除大对象。

## 什么是热 key？怎么解决？

热 key 是访问集中在某一个或少数 key 上。它可能导致单个节点的 CPU、带宽或连接处理能力成为瓶颈。

常见发现方式：业务埋点、`redis-cli --hotkeys`、单节点 QPS/带宽/CPU 监控。

解决方式：

- 只读热点放入本地缓存。
- 将热点复制为多个 key，读请求随机打散。
- 使用副本承担读请求，但接受复制延迟。
- 配合限流、降级和大 key 拆分。

## Redis 如何实现消息队列？

| 方案 | 优点 | 局限 |
| --- | --- | --- |
| List | 简单，支持阻塞消费 | 没有天然消息 ID、ACK 和消费组 |
| Pub/Sub | 实时、使用简单 | 不持久化，消费者离线会丢消息 |
| Stream | 支持消息 ID、持久化、消费组、ACK 和 pending | 仍需自行设计重试、幂等和监控 |

List 常用 `LPUSH` + `BRPOP`；Stream 常用 `XADD`、`XREADGROUP`、`XACK`。核心业务、复杂重试或高可靠场景优先考虑专业 MQ。

## 秒杀场景如何处理高并发和超卖？

常见方案是 Redis 预扣库存 + Lua 原子校验 + MQ 异步下单：

1. 活动开始前把库存加载到 Redis。
2. 请求先做登录、活动时间和限流校验。
3. Lua 原子检查库存和“一人一单”条件。
4. 校验通过后预扣 Redis 库存，并投递 MQ。
5. 立即返回排队结果，由消费者异步创建订单。
6. 数据库使用条件更新作为最终兜底：

```sql
UPDATE goods
SET stock = stock - 1
WHERE goods_id = ? AND stock > 0;
```

还需要考虑 MQ 消费幂等、失败补偿、库存回补、防刷、限流和降级。数据库锁适合做最终一致性兜底，不适合单独承受秒杀流量。
