# 大数据基础

## HDFS 的核心角色和读写流程

HDFS 适合大文件、顺序读写和高吞吐场景：

- NameNode 保存文件名、目录、块到 DataNode 的元数据；DataNode 保存实际数据块。
- 文件被切成多个 Block 并复制到多个 DataNode，副本提供容错和并行读取。
- 写入时客户端向 NameNode 获取目标 DataNode，再按流水线写入副本；读取时先拿元数据，再就近读取数据块。
- NameNode 的 `fsimage` 是元数据快照，`edits` 记录快照后的变更；checkpoint 会合并二者，避免重启时重放过长日志。

## Secondary NameNode 是什么？

Secondary NameNode 主要负责定期把 NameNode 的 `fsimage` 与 `edits` 合并成新的 checkpoint，减少 NameNode 重启时需要回放的日志量。它不是 NameNode 的实时备份，也不能在 NameNode 故障后直接接管服务；高可用场景应使用专门的 standby NameNode 机制。

## MapReduce Shuffle 做了什么？

Map 端先把中间结果按分区写入内存，达到阈值后 spill 到本地磁盘，并按 partition/key 排序；可用 Combiner 先做局部聚合，减少网络数据。Reduce 端执行 copy、sort/merge、reduce，把属于自己的分区拉取后合并处理。

Shuffle 的主要瓶颈是磁盘、网络和数据倾斜。优化方向是合理分区、局部聚合、压缩中间结果、避免小文件和处理热点 key。

## Spark 的窄依赖、宽依赖和 `reduceByKey`

- 窄依赖：父分区只被一个子分区使用，如 `map`、`filter`，通常无需 Shuffle。
- 宽依赖：一个父分区被多个子分区使用，如 `groupByKey`、`reduceByKey`，会触发 Shuffle。
- `reduceByKey` 能在 Map 端先局部聚合，通常比直接 `groupByKey` 少传输数据、占用更少内存；但仍要注意热点 key 和分区倾斜。

面试先区分“存储层 HDFS”和“计算层 MapReduce/Spark”，再从数据分区、Shuffle、容错和数据倾斜回答，不必堆配置参数。

## Hive 解决什么问题？

Hive 是构建在大数据存储之上的 SQL 数仓工具，本身不负责长期保存数据；Hive SQL 会被编译成 MapReduce、Tez 或 Spark 等执行任务，适合离线批处理和多表分析，不适合低延迟 OLTP。

- 分区按时间、业务类型等字段切目录，查询带分区条件可减少扫描。
- 分桶按哈希把数据拆成固定桶，适合抽样、关联和数据组织，但不是索引。
- 重点优化是避免小文件、减少无谓列/分区扫描、处理数据倾斜并选择合适执行引擎。

## HBase 的核心模型是什么？

HBase 是基于 HDFS 的分布式列族数据库，按 `RowKey → Column Family → Qualifier → Timestamp` 组织数据；表按 RowKey 划分为 Region，由 RegionServer 提供读写，底层数据经过 MemStore、WAL 和 HFile 持久化。

- 适合海量数据、按 RowKey 高并发随机读写；不适合复杂 JOIN、临时聚合和随意多条件查询。
- RowKey 设计决定数据分布和查询方式，应避免时间递增等导致热点 Region，必要时加盐或哈希前缀。
- Hive 偏离线分析，HBase 偏在线随机访问，二者可以共享 HDFS 但解决的问题不同。

## HBase 写入和刷盘怎样完成？

一次典型写入链路是：客户端根据元数据定位 Region → RegionServer 先追加 WAL → 更新内存中的 MemStore → 返回写入确认。WAL 先于 MemStore 持久化，RegionServer 故障后可以通过日志恢复尚未刷盘的数据。

当 MemStore 达到阈值或触发定期刷盘时，数据会写成 HFile；多个 HFile 经过 Compaction 合并，减少读放大和文件数量。Region 过大还会拆分，RowKey 设计需要同时考虑查询范围、写入热点和 Region 的均匀分布。

## YARN 的核心组件是什么？

- ResourceManager：全局资源管理和调度。
- NodeManager：管理单个 Node 上的容器、资源和任务状态。
- ApplicationMaster：负责某个应用的任务协调，并向 ResourceManager 申请资源。

任务提交后由 ResourceManager 分配首个容器启动 ApplicationMaster，再由它申请并协调后续任务。容量不足、队列配置或单任务资源申请过大都可能导致任务长时间排队。
