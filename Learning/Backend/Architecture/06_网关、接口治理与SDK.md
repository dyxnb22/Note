# 网关、接口治理与 SDK

网关的价值是统一协议、路由、鉴权和治理；SDK 的价值是把稳定契约和适配细节交给调用方复用。两者都不应成为业务规则的垃圾桶。

## 网关职责

- 协议转换：HTTP、RPC、JSON-RPC、SSE 和流式协议之间建立明确的错误映射。
- 路由与发现：按服务、版本、租户或能力选择目标，并处理健康检查和灰度。
- 鉴权与授权：验证身份、工具/接口权限、租户边界和调用配额。
- 流量治理：限流、超时、熔断、负载均衡、重试和请求大小限制。
- 可观测：记录请求 ID、业务号、调用方、目标、耗时、状态和下游错误。

网关不应替代领域服务校验；权限、业务状态和资源归属仍由领域边界负责。

## Nginx 作为反向代理

Nginx 常位于边缘或应用前方，负责 TLS 终止、静态文件、反向代理、负载均衡和基础限流。请求匹配要区分前缀、精确、正则和静态目录规则，变更后使用配置检查并在灰度环境验证实际 URI 重写结果。

Master 进程负责读取配置和管理 Worker，Worker 使用事件驱动模型处理连接。事件循环不应被同步磁盘、脚本或上游长时间阻塞；上传、下载、SSE 和 WebSocket 需要单独配置缓冲和超时。

常见错误要沿链路定位：

| 状态 | 常见含义 | 优先检查 |
| --- | --- | --- |
| 502 | 无法获得有效上游响应 | 上游进程、端口、协议、连接重置 |
| 503 | 服务暂不可用或主动拒绝 | 限流、无健康实例、维护降级 |
| 504 | 等待上游超时 | 上游耗时、代理超时、连接池和数据库 |

网关重试可能让已经到达上游的写请求重复执行，因此只对明确幂等、可恢复的请求开启，并把重试次数计入整条调用预算。

## 接口契约

一份稳定接口至少明确：请求/响应 schema、错误码、幂等键、超时预算、分页/排序、版本策略和兼容性。DTO 与领域对象分离，第三方接口通过防腐层转换。

## SDK 设计

- 用统一接口隔离供应商差异，核心代码只依赖抽象。
- 统一配置、鉴权、序列化、重试、超时、日志和错误模型。
- 流式接口要定义增量事件、结束事件、异常事件和取消语义。
- 版本升级保持向后兼容；破坏性变更使用新版本或适配层。
- SDK 不替调用方隐藏业务幂等和权限责任。

## AI/MCP 网关的特殊点

AI MCP Gateway 需要额外管理工具元数据、参数 schema、Session 生命周期、SSE/Streamable 传输、JSON-RPC 错误和工具权限。分布式部署时，Session 必须共享、路由到同一节点或由网关转发，否则连接切换会丢上下文。

## 治理检查

- 是否能在不改业务代码的情况下增加路由、限流或观测？
- 是否能区分网络失败、业务拒绝、参数错误和下游异常？
- 重试是否只作用于可恢复错误？
- 接口和工具是否最小授权、可审计、可取消？
- 是否有版本、灰度、回滚和兼容策略？
