# 任务、消息与补偿

任务解决“以后还要做什么”，消息解决“把事实通知给谁”。两者可以共用可靠投递和重试机制，但语义不能混为一谈。

## 本地任务消息模型

```text
事务内产生业务事实
        ↓
写入任务/事件记录
        ↓
HTTP 或 MQ 通知
        ↓
成功确认 / 失败重试 / 超限转人工
```

适合支付回调后的发券、库存释放、积分返利、订单关闭和跨系统同步。核心业务提交成功后，外部通知可以异步完成，但任务记录必须和业务事实处在可恢复的边界内。

## 任务状态

建议至少包含：`WAITING`、`PROCESSING`、`SUCCESS`、`RETRY`、`DEAD`。记录业务类型、业务号、目标、payload 版本、重试次数、下次执行时间、错误原因和锁定时间。

- 抢占任务要有租约或超时释放，避免 worker 崩溃后永久卡住。
- 重试按指数退避和上限执行；不可恢复错误直接进入死信。
- 失败任务可由定时扫描兜底，但扫描必须分页并限制并发。
- 人工重放前先检查最新业务状态，避免补偿已经成功的副作用。

## 消息策略

| 策略 | 用途 | 注意事项 |
|---|---|---|
| Spring Event | 同进程解耦 | 进程崩溃或跨服务时不可靠 |
| HTTP 通知 | 简单跨服务调用 | 超时、签名、重试和幂等必须明确 |
| MQ | 削峰、解耦、异步 | 消费幂等、顺序、堆积和死信 |
| 延迟消息/定时扫描 | 超时关单、过期释放 | 不能只依赖一次触发 |

## 切面与入口

可以用 AOP 统一记录任务、拦截需要通知的业务动作，但切面只负责捕获和持久化，不应隐藏关键领域规则。任务来源、状态迁移和副作用仍要在应用/领域层可见。

## 设计检查

- 任务记录是否与业务事实建立唯一业务号？
- 重复投递和乱序消费是否可预测？
- 消费成功但 ACK 丢失时会发生什么？
- 任务处理超时后谁释放租约？
- 队列堆积、死信和对账差异谁负责？
- 是否支持暂停、重试、重放和人工终止？
