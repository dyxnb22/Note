# 场景方案库

这里把 `xfg-planet` 的 98 个场景压缩成可复用的问题域。完整原文仍在源库；本文件只保留判断框架、方案骨架和高频边界，避免维护 98 份重复问答。

逐条检索入口见[98 个场景逐项索引](./场景逐项索引.md)。

## 统一答题框架

```text
目标与规模 → 业务事实/状态 → 核心链路 → 数据与副作用
                         ↓
                并发、失败、观测、补偿
```

先问清用户、QPS、数据量、峰值、可用性、一致性和成本；再画入口、应用服务、领域规则、数据库、缓存、消息和任务；最后说明替代方案、失败场景和验证方式。

## 问题域索引

| 问题域 | 典型场景 | 通用解法 |
|---|---|---|
| 数据与性能 | 大表、慢查询、重复数据、文件导入/导出 | 分批、索引、异步、流式、限速、校验和回滚 |
| 缓存与高并发 | 热 Key、大 Key、秒杀、排行榜、UV | 预热、TTL、Lua、分片、削峰、重建和降级 |
| 交易与支付 | 下单、支付、退款、超时关单、分账 | 状态机、条件更新、回调验签、幂等、对账 |
| 库存与权益 | 库存扣减、红包、优惠券、积分、抽奖 | 资格/计算/占用/发放拆分，唯一号和补偿 |
| 消息与一致性 | MQ、跨服务调用、第三方回调 | 本地任务、重试、死信、补偿、人工重放 |
| 接口与架构 | 网关、RPC、聚合层、微服务选型 | 契约、超时、限流、依赖方向、数据归属 |
| 安全与治理 | 登录限制、敏感词、评论审核、接口防刷 | 风险分级、限流、审计、人工介入和可解释规则 |
| 读写模型 | 评论列表、Feed、搜索、报表 | 按查询模式建模，区分事实表、读模型和缓存 |

## 高复用模板

### 资源竞争

```text
请求资格校验 → 原子预占 → 异步落库 → 失败释放/对账
```

预占必须有过期时间；最终落库仍需数据库约束；Redis、MQ 或 worker 故障时要能重建和修复。

### 支付与超时

```text
创建待支付单 → 支付回调/超时任务竞争状态迁移
                    ↓
             支付成功 / 关闭 / 退款
```

支付回调和关单都只能把“待支付”迁移到合法下一状态，失败方读取最新状态再决定退款或忽略。

### 批量数据处理

分片、游标、流式读写、幂等分片号、失败重试和结果校验必须同时设计；不要用一次性加载和单事务覆盖千万级数据。

### 第三方同步

把请求、回调、重试和最终状态分开记录；验签、金额/业务号校验、幂等和对账是必需项。不能把一次 HTTP 返回成功当作业务完成。

## 选择与验证

- 小规模系统优先简单架构，复杂度要由业务规模和失败成本证明。
- 分布式事务不是默认答案；先判断能否用状态机、消息和补偿实现最终一致。
- 任何高并发方案都要配压测、指标、故障注入和差异对账。
- 任何“已完成”结论都标注课程阅读、源码核验、复现验证或独立实现。
