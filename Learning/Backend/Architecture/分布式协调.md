# 分布式协调

分布式协调处理“谁负责、谁先执行、资源是否仍归我所有”等问题。典型机制包括租约、选主、分布式锁、注册发现和分布式 ID。

## 租约比永久所有权更现实

节点崩溃或网络分区后，其他节点无法可靠判断它是否永久死亡。因此协调系统通常授予有期限的 Lease：持有者必须续约，过期后所有权才能转移。

Lease 需要定义：

- 服务端时间还是客户端时间作为准绳。
- TTL、续约周期和最大暂停容忍。
- 续约失败后持有者何时停止工作。
- 新旧持有者并存时如何保护资源。

## 分布式锁

锁的价值不是“成功写入一个 key”，而是让受保护资源拒绝旧持有者继续写。可靠方案通常需要 fencing token：每次获取锁得到单调递增令牌，下游资源只接受不小于当前版本的写入。

```text
Client A 获取 token=41 → 长暂停
Client B 在租约过期后获取 token=42 → 写入成功
Client A 恢复并携带 token=41 → 下游拒绝旧写入
```

如果下游不校验 fencing token，仅依赖 Redis `SET NX PX` 和“只删除自己的锁”，仍无法完全防住进程暂停超过租约的情况。

选择锁之前先考虑能否使用：

- 数据库唯一约束。
- 条件更新或乐观锁。
- 单分区串行消费。
- 幂等键。
- 业务状态机。

这些方案常比跨系统锁更容易证明正确。

## 选主

Leader 负责串行化某类决策或任务调度。选主必须和任期/版本绑定，旧 Leader 在网络恢复后不能继续写。

多数派协调服务适合保存少量关键元数据，不适合作为高吞吐业务数据库。业务 Worker 应能处理 Leader 切换、重复调度和未确认任务。

## 注册发现

注册中心维护服务名到实例的映射。一个完整方案包括：

- 实例注册和续约。
- 健康检查与故障怀疑。
- 客户端缓存和更新推送/拉取。
- 实例摘除与优雅下线。
- 注册中心不可用时的降级行为。

注册表内容可能短暂陈旧，因此调用方仍需超时、重试上限、熔断和连接池保护。

## ZooKeeper

ZooKeeper 通过层次化节点、版本号、临时节点和 Watch 提供协调原语。典型用途是选主、配置元数据和成员管理。

注意：

- Watch 是变更通知，不是永久消息队列；客户端要重新注册并在重连后校验状态。
- 临时节点与 Session 绑定，长时间停顿可能造成 Session 过期。
- 顺序节点可用于排队，但仍要处理客户端重试和创建结果未知。
- ZAB/共识保证协调状态顺序，不自动保证业务数据库写入。

## 分布式 ID

ID 方案要按需求选择：全局唯一、趋势递增、是否暴露时间、是否可离线生成、时钟回拨、吞吐和可用性。

| 方案 | 优点 | 风险 |
| --- | --- | --- |
| 数据库号段 | 简单、趋势递增 | 号段服务与容量规划 |
| Snowflake | 本地高吞吐、趋势递增 | 时钟回拨、机器号分配 |
| UUID | 无中心、生成简单 | 长、索引局部性较差 |
| Redis/协调服务自增 | 全局递增 | 中心依赖和持久性 |

业务订单号和数据库主键可以分开：前者满足展示、校验和隐私需求，后者满足存储与关联效率。

## 验收清单

- 能解释 Lease 过期为什么不代表旧持有者已经停止。
- 能说明 fencing token 如何保护下游资源。
- 能判断唯一约束或状态机是否可以替代分布式锁。
- 能设计注册中心不可用时客户端的行为。
- 能根据排序、吞吐、隐私和时钟条件选择 ID 方案。
