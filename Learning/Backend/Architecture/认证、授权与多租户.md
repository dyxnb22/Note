# 认证、授权与多租户

认证回答“你是谁”，授权回答“你能做什么”，审计回答“你做过什么”。Cookie、Session、JWT 和 TLS 的协议层见 [HTTP 与 Web 协议](../../CS/04_HTTP与Web协议.md)；注入、XSS、CSRF、SSRF、上传和安全测试见 [应用安全](应用安全.md)；本页关注服务端如何把身份变成可验证的权限决策。

## 身份与会话

密码只存强哈希；第三方登录使用 OAuth/OIDC 的授权码流程：客户端带 `state` 跳转授权页，回调拿一次性 code，服务端校验 `state` 和 redirect URI 后换取 token，再用第三方唯一用户标识绑定本地账号。access token 应短期有效，refresh token 可撤销且需轮换。验证 token 的签名、issuer、audience、过期时间，不信任客户端提交的 user id 或 role。

多实例部署时，传统 Session 要么共享 Redis/数据库，要么改用可验证的 Token；共享 Session 还要考虑过期、失效、故障丢失和敏感数据最小化。

## Token 续租和登录校验如何分工？

如果采用滑动过期策略，不能只在“需要登录”的拦截器中刷新 Token：用户访问不需要登录的页面时，该拦截器不会执行，Token 也就不会续租。常见分工是：

- 外层拦截器负责读取并按策略刷新 Token，覆盖需要续租的请求范围。
- 内层拦截器负责判断登录状态、解析身份并执行权限校验，只放行明确的白名单路径。

刷新逻辑还要处理并发刷新、旧 Token 是否立即失效、设备维度和撤销列表。对新系统而言，短期 access token + 可撤销且轮换的 refresh token 通常比无限延长单个 Token 更容易控制风险。

## 如何限制登录暴力尝试？

按账号、IP、设备等维度组合限流，避免单一维度误伤或绕过；用 Redis 记录失败次数并设置窗口 TTL，判断、递增和过期时间更新要保证原子性。达到阈值后进入冷却期，并结合验证码、设备风控和审计日志；错误提示不要暴露账号是否存在。

## 授权模型

RBAC 用角色聚合权限，简单易管；ABAC 按用户/资源/环境属性判断，表达力强但难审计；资源级授权必须在服务端对“当前主体 + 目标资源 + 动作”判定。前端隐藏按钮不是权限控制。

## 多租户

租户隔离可在数据库、schema、行级字段或独立实例实现，隔离越强成本越高。无论选择何种模式，每次查询和缓存 key 都要带 tenant scope；后台任务、搜索索引、对象存储和日志也不可遗漏。用测试验证跨租户 ID 不可读取。
