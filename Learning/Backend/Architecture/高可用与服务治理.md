# 高可用与服务治理

这篇的目标不是背术语，而是回答一个很实际的问题：服务一旦拆多、链路一旦变长，系统怎样才能在部分组件出问题时不被一起拖垮。

## 1. 为什么需要服务治理

单体系统里一次请求可能只经过一个进程和一个数据库，问题相对集中。

到了分布式或微服务场景，一次请求可能会经过：

```text
gateway -> service A -> service B -> cache / db / mq
```

链路一长，风险就来了：

- 某个下游慢，整条链路都会慢
- 某个依赖挂了，上游线程可能被全部占满
- 重试处理不好，会把故障放大
- 高峰流量可能把最脆弱的依赖打垮

服务治理的核心目标就是三件事：

- 控制故障传播范围
- 保住核心功能
- 让系统可以恢复

## 2. 高可用到底在讲什么

高可用不是“永远不出故障”，而是系统出故障时仍然尽量可服务、可恢复、可降级。

常见关注点：

- 单点故障有没有消除
- 下游超时会不会把上游拖死
- 容量不足时能不能限流
- 非核心能力挂了，核心链路能不能继续
- 故障发生后是否能快速切流、回滚、恢复

## 3. 超时、重试、幂等

这是最基础的一组三件套。

### 3.1 为什么一定要设超时

远程调用不能无限等，否则线程、连接、协程、连接池都会被慢请求耗尽。

面试表达可以这样说：

> 超时本质上是在限制故障等待成本。如果一个下游已经异常，还无限等待，就会把局部故障拖成全链路雪崩。

### 3.2 重试不是越多越好

重试适合处理短暂抖动，但有几个前提：

- 失败原因确实可能是临时性的
- 接口要么天然幂等，要么业务上能防重
- 重试次数和间隔要可控

常见坑：

- 下游已经过载，上游还同时重试，雪上加霜
- 多层服务各自重试，调用量被指数放大
- 写请求没有幂等，重复扣款、重复下单

### 3.3 幂等为什么关键

在分布式系统里，重复请求是常态，不是例外。来源包括：

- 用户重复点击
- 网关超时后客户端重试
- MQ 重复投递
- 支付平台重复回调

常见幂等手段：

- 幂等号 / request_id
- 唯一索引
- 状态机约束
- 去重表或去重缓存

一句话理解：

- 重试解决“请求可能没成功”
- 幂等解决“请求可能成功了不止一次”

## 4. 限流、熔断、降级

这是防止系统被流量和故障打穿的核心手段。

### 4.1 限流

限流是控制进入系统的请求速率，避免系统超出承载能力。

常见算法：

- 计数器
- 滑动窗口
- 漏桶
- 令牌桶

常见落点：

- 网关层限流
- 接口级限流
- 用户/IP/租户维度限流
- 下游资源维度限流

面试里要讲清楚的一点：

限流不是为了“拒绝用户”，而是为了在容量不足时优先保住系统整体可用。

### 4.2 熔断

熔断是下游故障明显时，主动快速失败，避免继续把请求打到故障服务上。

可以理解为电路保险丝：

- 错误率高
- 超时多
- 响应极慢

达到阈值后，熔断器打开，一段时间内直接失败；之后再半开探测，确认恢复后关闭。

适合场景：

- 第三方支付接口偶发雪崩
- 某个推荐服务高延迟
- 某个非核心依赖持续超时

### 4.3 降级

降级是保核心、砍非核心。

常见例子：

- 推荐服务挂了，首页只展示基础内容
- 评论服务异常，商品详情页仍可看主信息
- 用户画像拿不到时，用默认策略继续返回

降级本质上是业务取舍，不只是技术开关。

## 5. 隔离

没有隔离，故障就会传染。

常见隔离方式：

- 线程池隔离
- 连接池隔离
- 服务集群隔离
- 核心与非核心链路隔离
- 读写流量隔离
- 大客户与普通流量隔离

例如：

- 秒杀服务和普通下单服务拆开
- 后台管理接口和用户主链路隔离
- 推荐、搜索等重资源服务与交易链路隔离

## 6. 雪崩、击穿、穿透

这是缓存和高并发系统里非常高频的一组问题。

### 6.1 缓存雪崩

大量 key 同时失效，流量一起打到数据库。

常见处理：

- TTL 加随机值
- 多级缓存
- 热点数据永不过期 + 异步刷新
- 限流和降级

### 6.2 缓存击穿

某个热点 key 失效瞬间，大量请求同时查库。

常见处理：

- 互斥锁
- singleflight / 请求合并
- 热点 key 提前续期

### 6.3 缓存穿透

请求的数据根本不存在，每次都打到数据库。

常见处理：

- 参数校验
- 布隆过滤器
- 空值缓存

## 7. 容灾与多机房

当系统规模再大一步，就不只关心单实例和单机房了。

常见容灾层级：

- 同机房多实例
- 跨可用区部署
- 异地多机房

要解决的问题：

- 流量怎么切
- 数据如何同步
- 切流后是否会读到旧数据
- 哪些服务要求强一致，哪些接受最终一致

现实里最常见的取舍是：

- 核心写链路优先正确性
- 查询类和推荐类优先可用性

## 8. 监控、告警、压测

没有观测能力，就谈不上治理。

最少应该看这些指标：

- QPS
- RT / P95 / P99
- 错误率
- 超时率
- CPU、内存、磁盘、网络
- 线程池、连接池、MQ 堆积

常见误区：

- 只看平均响应时间，不看长尾
- 只看服务指标，不看业务成功率
- 没压测就直接上线限流值和线程池参数

## 9. 面试里的回答套路

如果被问“怎么保证系统高可用”，可以按这个顺序答：

1. 先去单点，服务多实例部署
2. 调用链路设置超时、限流、熔断、降级
3. 对核心和非核心能力做隔离
4. 关键写操作设计幂等、重试、补偿
5. 用监控告警和压测做容量管理
6. 更高要求场景再补多机房和容灾切流

这样回答会比零散堆概念更像真实做过系统的人。

## 10. 常见追问

### 为什么重试有时会把系统打垮？

如果下游已经慢或过载，上游大量重试只会继续加压；如果多层服务都重试，流量会被层层放大，最终把线程池、连接池和下游实例全部耗尽。

### 限流、熔断、降级的区别是什么？

限流是控制进入量；熔断是发现下游明显异常后主动快速失败；降级是保核心功能、关闭或简化非核心能力。三者目标不同，但都在减少故障扩散。

### 为什么说幂等是分布式系统的基本功？

因为重复请求几乎无法完全避免。只要有超时、重试、消息重复投递、用户重复点击，就必须保证重复执行不会产生错误副作用。

### 高可用和一致性冲突时怎么取舍？

要看业务。交易扣款、库存确认这类核心写链路更偏一致性；推荐、统计、缓存读取这类场景通常更偏可用性，允许短暂不一致后再补偿。

## 11. 一页总结

- 超时控制等待成本
- 重试解决短暂失败，但一定要配幂等
- 限流保护容量边界
- 熔断阻止故障扩散
- 降级优先保核心功能
- 隔离避免局部故障拖垮全站
- 容灾解决更大范围故障
- 监控和压测决定你是否真的可治理
