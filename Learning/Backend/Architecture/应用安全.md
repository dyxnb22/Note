# 应用安全

应用安全关注输入、身份、权限、数据和副作用如何跨越信任边界。网络层协议见 [HTTP 与 Web 协议](../../CS/HTTP与Web协议.md)，身份模型和多租户细节见 [认证、授权与多租户](认证、授权与多租户.md)，配置与密钥生命周期见 [配置、密钥与环境治理](配置、密钥与环境治理.md)。

## 先画信任边界

至少标出：

- 浏览器、移动端和第三方客户端。
- CDN、网关和应用服务。
- 数据库、缓存、消息系统和对象存储。
- 管理后台、运维入口和 CI/CD。
- 第三方 API、Webhook 和文件来源。

每次跨边界都要重新确认身份、输入、权限和审计，不能因为“来自内网”就默认可信。

## 认证与授权

认证回答“你是谁”，授权回答“你能对哪个资源做什么”。授权应在服务端根据当前主体、租户、资源、动作和上下文计算。

常见模型：

- RBAC：角色映射权限，适合职责稳定的组织。
- ABAC：按主体、资源和环境属性判断，表达力强。
- ACL：在具体资源上维护主体权限。
- Capability：持有不可伪造能力即获得受限操作权。

前端隐藏按钮不是授权。对象级操作必须校验资源归属，防止只验证“用户已登录”却允许修改他人资源。

## Session、OAuth 2.0 与 OIDC

- Session 适合服务端维护登录状态，浏览器通常只保存随机 Session ID。
- OAuth 2.0 是授权框架，用于委托访问，不是身份协议。
- OIDC 在 OAuth 2.0 之上增加身份层和 ID Token。

浏览器和移动应用优先使用 Authorization Code + PKCE。必须校验 `state`、`nonce`、redirect URI、issuer、audience、签名和过期时间。

Access Token 生命周期应较短；Refresh Token 要轮换、可撤销并安全保存。不要把 ID Token 当作任意下游 API 的 Access Token。

## 密码与凭证

密码使用专用慢 Hash（如 Argon2id、bcrypt）和独立 Salt，不能使用可逆加密或普通 SHA。登录接口需要速率限制、异常检测和安全的账户恢复流程。

API Key、数据库密码和签名密钥：

- 不进入仓库、日志、URL 和 Prompt。
- 通过密钥管理系统注入。
- 具有最小权限、所有者和到期时间。
- 支持轮换、撤销和泄露响应。

## 输入验证与注入

所有外部输入先做长度、类型、格式、范围和业务约束校验。

- SQL 使用参数绑定；动态列名和排序字段使用白名单。
- Shell 避免字符串拼接，使用参数数组和受限命令。
- 模板输出按 HTML、JavaScript、URL 等上下文分别编码。
- LDAP、XPath、NoSQL 查询同样存在注入风险。
- 日志要防止换行注入和敏感数据泄露。

仅靠黑名单过滤危险字符串通常不可靠，应采用结构化 API、白名单和最小权限。

## XSS

XSS 让攻击者内容在其他用户浏览器中执行。主要防线：

- 按输出上下文编码。
- 避免不安全地插入 HTML。
- 富文本使用成熟 Sanitizer。
- Cookie 设置 `HttpOnly`、`Secure` 和合适 `SameSite`。
- 使用 Content Security Policy 降低成功后的影响。

CSP 是纵深防御，不能替代正确编码。

## CSRF

CSRF 利用浏览器自动携带 Cookie 发起用户未授权的请求。防线包括：

- SameSite Cookie。
- 不可预测且绑定会话的 CSRF Token。
- 校验 Origin/Referer 作为辅助证据。
- 高风险操作再次认证或确认。

使用 Authorization Header 且 Token 不被浏览器自动附带的 API，风险模型与 Cookie 会话不同，但仍要防 XSS 窃取 Token。

## CORS

CORS 只控制浏览器是否允许某个 Origin 读取响应，不是认证或防火墙。允许 Origin、Method、Header 和 Credentials 要最小化。

带凭证时不能使用通配 Origin；动态回显 Origin 前必须匹配严格允许列表，并正确设置缓存 `Vary`。

## SSRF

SSRF 诱使服务端访问攻击者指定的地址，可能探测内网、云元数据或本机管理接口。

防线：

- 只允许必要协议和目标域名。
- DNS 解析后校验所有返回地址，阻止私网、回环和链路本地地址。
- 重定向后重新校验目标。
- 出站网络策略限制可访问范围。
- 云元数据使用新版本防护和最小 IAM 权限。
- 响应大小、时间和重定向次数设上限。

## 文件上传

不能只信任文件扩展名或客户端 `Content-Type`。应检查大小、实际类型、文件名、压缩包展开比例和恶意内容。

上传文件使用服务端生成的存储名，放在不可执行、与应用隔离的位置；下载时设置安全的 Content-Type 和 Content-Disposition。图片、PDF 和 Office 文件也可能包含主动内容或解析器漏洞。

## 反序列化与依赖安全

不对不可信数据使用 Java 原生反序列化、任意类型实例化或危险 YAML 配置。JSON 映射只允许预期类型和字段，并限制深度、集合大小和数字范围。

依赖要锁定版本、生成 SBOM、扫描已知漏洞，并验证构建来源。升级后仍需测试，因为“无已知 CVE”不等于不存在逻辑风险。

## 多租户隔离

租户标识来自验证后的身份上下文，不能直接信任请求参数。隔离必须贯穿：

- 数据库查询和唯一约束。
- 缓存 Key。
- MQ Topic/消息属性。
- 对象存储路径。
- 搜索和向量检索过滤。
- 日志、导出和后台任务。

测试时要主动尝试跨租户读取、更新、缓存命中和批量导出。

## 审计

高风险操作记录主体、租户、动作、资源、结果、时间、来源、审批和关联 Trace。审计日志要防篡改、限制访问并定义保留期。

日志中不记录密码、完整 Token、私钥和不必要的个人数据。脱敏要在进入日志管道前完成。

## 错误处理

对外错误不泄露堆栈、SQL、内部地址和密钥；对内保留可定位的错误类型、TraceId 和上下文。认证失败响应避免暴露“用户存在但密码错误”等可枚举信息。

安全失败应默认拒绝高风险动作，但要避免认证或策略服务故障造成无限重试和雪崩。

## 安全测试

- SAST、依赖和密钥扫描。
- API 权限矩阵和对象级越权测试。
- SQL/命令/模板注入用例。
- XSS、CSRF、CORS、SSRF 和上传测试。
- Token 过期、撤销、密钥轮换和 Session 固定测试。
- 多租户隔离和缓存污染测试。
- 限流、暴力破解和异常流量测试。
- 发布前威胁建模与高风险变更复核。

## 最小验收清单

- 每个入口都能标出认证、授权和输入验证位置。
- 每个写操作都校验资源级权限和租户归属。
- SQL、Shell、模板和 URL 访问使用结构化安全边界。
- 密钥有所有者、权限、轮换和泄露响应流程。
- 上传、导出、Webhook 和管理后台有单独威胁模型。
- 安全事件能通过审计日志还原，且日志不泄露凭证。
