# Architecture

这里回答系统如何拆分、如何承受失败，以及如何把设计取舍落到可验证的链路上。

## 主题

| 文档 | 关注点 |
|---|---|
| [系统设计](./系统设计.md) | 设计题框架和典型场景 |
| [分布式](./分布式.md) | CAP、锁、事务、RPC 和注册发现 |
| [高可用与服务治理](./高可用与服务治理.md) | 超时、限流、熔断、降级和容灾 |
| [DDD 与领域建模](./DDD与领域建模.md) | 事实、聚合、分层和适配 |
| [可靠性与一致性](./可靠性与一致性.md) | 幂等、状态、补偿、对账 |
| [任务、消息与补偿](./任务、消息与补偿.md) | 任务状态、通知、重试和死信 |
| [网关、接口治理与 SDK](./网关、接口治理与SDK.md) | 协议、路由、鉴权、限流和适配 |
| [组件化与中间件设计](./组件化与中间件设计.md) | Starter、策略、插件和配置 |
| [场景方案库](./场景方案库.md) | 98 个场景的问题域模板 |
| [场景逐项索引](./场景逐项索引.md) | 98 个场景的逐条检索入口 |

建议顺序：`系统设计 → DDD → 分布式/可靠性 → 任务与消息 → 网关/组件 → 场景`。

边界：数据库、缓存和 MQ 的实现细节看 `Data`；部署、发布和运维看 `Delivery`；这里保留跨组件的设计取舍。
