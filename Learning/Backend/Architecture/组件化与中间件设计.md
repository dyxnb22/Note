# 组件化与中间件设计

组件不是把代码打包成 Jar，而是把重复问题沉淀为稳定接口、默认实现、扩展点和可观测边界。

## 组件的最小结构

```text
契约/API → 配置与自动装配 → 核心执行器 → 扩展点 → 观测与失败处理
```

- **契约**：明确输入、输出、错误和版本兼容。
- **配置**：提供合理默认值，允许按环境覆盖，避免把密钥写入默认配置。
- **自动装配**：Starter 只负责接入和装配，不隐藏业务行为。
- **核心执行器**：把策略、路由、任务或通知流程做成可测试的服务。
- **扩展点**：策略、责任链、规则树、Plugin 或 SPI 用于稳定变化点。
- **治理**：日志、指标、Trace、超时、重试、开关和降级必须可配置。

## 可迁移组件模式

| 组件方向 | 稳定抽象 | 关键边界 |
|---|---|---|
| 本地任务消息 | 任务记录、通知策略、重试调度 | 事务事实、幂等和补偿 |
| 动态线程池 | 线程池配置、刷新、监控 | 配置变更、运行中任务和容量 |
| API 网关 | 路由、协议、鉴权、限流 | 业务规则不能下沉到网关 |
| 支付 SDK | 统一支付接口、适配器、回调 | 签名、金额校验、幂等和退款 |
| 规则组件 | 责任链、规则树、策略装配 | 规则版本、顺序和可解释性 |
| Spring Boot Starter | 自动装配、配置属性、扩展接口 | 默认值、条件装配和兼容性 |
| 字节码/插件 | Agent、插件点、运行时增强 | 类加载、回滚、性能和安全 |

## 发布与维护

Maven 多模块和 Archetype 可以复制经过验证的工程骨架，但脚手架要保持轻量：只提供通用结构、默认配置和示例，不预先绑定具体业务。发布前验证构建、最小示例、配置覆盖、版本兼容和失败行为。

## 设计检查

- 组件是否解决重复且稳定的问题？
- 用户是否能替换核心实现而不修改业务调用？
- 默认配置是否安全、可解释、可覆盖？
- 失败是否有错误类型、重试边界和降级结果？
- 运行中动态变更是否可观测、可回滚？
- 文档是否包含最小接入示例和不适用场景？
