# DDD 与领域建模

DDD 的价值不是增加目录层级，而是把业务事实、状态和变化责任放到稳定边界内。

## 建模顺序

1. 从用户、目标和业务动作开始，列出对象、角色、状态和外部依赖。
2. 区分资格判断、规则计算、资源占用、权益发放和外部副作用。
3. 按数据归属和业务不变量划分限界上下文，再确定聚合边界。
4. 写出状态迁移和异常路径，最后决定 API、表、缓存、消息和任务。

## 推荐分层

```text
API/Trigger → Application → Domain → Infrastructure
```

- **API/Trigger**：协议、鉴权、定时任务和消息入口；不承载领域规则。
- **Application**：编排一次用例、事务和端口调用；不把所有业务逻辑堆成脚本。
- **Domain**：状态、规则、不变量、领域服务和事件；不依赖 MySQL、Redis、MQ 或具体 SDK。
- **Infrastructure**：仓储、缓存、消息、支付、模型和外部系统适配器。

## 对象边界

| 对象 | 作用 | 常见风险 |
|---|---|---|
| DTO/Req/Res | 接口协议 | 直接泄漏数据库或领域内部字段 |
| PO | 持久化结构 | 被误当成领域对象 |
| Entity | 有身份和生命周期 | 只有字段，没有行为和不变量 |
| Value Object | 表达值和约束 | 被拆成无意义的基础类型 |
| Aggregate | 维护一组不变量 | 聚合过大导致锁和事务范围失控 |
| ACL/Adapter | 隔离外部模型 | 第三方字段污染核心领域 |

## 典型状态

订单、组队、支付、库存和权益都应使用显式状态机。例如：

```text
待支付 → 已支付 → 已完成
   └→ 已关闭 → 退款/补偿
```

状态迁移必须带条件、唯一业务号和操作者；支付回调、超时任务和用户操作同时到达时，只有合法的状态迁移可以成功。

## 实践原则

- 领域规则优先用代码表达，配置只承载可控、可版本化的变化。
- 外部依赖通过端口接入，便于替换、测试和故障隔离。
- 一个业务事实只在一个边界内拥有最终解释权。
- 复杂链路拆成同步决策和异步副作用，给每个副作用补齐重试、对账和人工兜底。
