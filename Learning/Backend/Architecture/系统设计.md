# 系统设计

这篇的重点不是记住多少组件，而是形成一套稳定的答题顺序。系统设计题最怕的不是不会，而是脑子里有很多点，却讲不成一条线。

## 1. 设计题的总框架

任何系统设计题，都尽量按下面这个顺序回答：

1. 先问清需求和约束
2. 再画核心链路
3. 再识别瓶颈
4. 最后讲取舍和兜底

可以把它记成一句话：

> 先边界，后主链路；先核心，后扩展；先瓶颈，后优化。

## 2. 一开始要问什么

### 系统设计题应该先澄清哪些问题？

- 核心功能是什么，哪些是必须做，哪些可以后置。
- 用户规模、QPS、峰值流量有多大。
- 数据量级大概多大，读多写多还是写多读少。
- 对一致性要求高不高，能不能接受最终一致。
- 对延迟、可用性、成本的要求分别是什么。
- 是否有热点、峰值、节假日活动这类特殊场景。

### 为什么设计题不能上来就堆 Redis、MQ、ES？

- 因为组件只是手段，不是答案。
- 如果没有先讲清瓶颈和约束，就很难说明为什么要引入这些组件。
- 面试官更想听到的是：你是否知道系统为什么会出问题，以及这些组件在链路里解决的具体问题。

## 3. 画主链路时怎么组织

最常见的主链路可以按这几层展开：

```text
client
-> gateway / lb
-> application service
-> cache / db / mq / search
-> async worker
-> monitor / alert
```

讲主链路时，尽量回答这几个问题：

- 请求从哪里进来
- 先经过哪些校验
- 核心写路径落在哪里
- 哪些步骤同步做，哪些异步做
- 数据最终怎么落库、怎么查出来
- 如果某一步失败，如何重试、补偿、降级

## 4. 容量和瓶颈怎么想

### 设计题为什么要有容量意识？

- 因为不同量级下，方案差异很大。
- 日活一万和日活一亿，缓存、数据库、分片、异步化的必要性完全不同。
- 即使不精确估算，也要给出大概判断：是单机能扛，还是必须水平扩展。

### 常见瓶颈一般在哪？

- 流量入口：网关、负载均衡、鉴权。
- 热点数据：缓存击穿、热点 key。
- 数据库写压力：库存扣减、订单落库、索引冲突。
- 异步链路：MQ 堆积、消费失败。
- 单点组件：单 Redis、单数据库、单任务调度器。
- 跨服务一致性：支付、库存、订单状态流转。

## 5. 高频题型一：秒杀系统

### 如何设计一个秒杀系统？

- 秒杀的核心矛盾是：瞬时高并发请求与有限库存之间的冲突。
- 设计目标不是让所有请求都成功，而是让系统在洪峰下仍然可控，并且不超卖。
- 前端层：倒计时、按钮置灰、验证码、静态资源 CDN，减少无效流量。
- 网关层：限流、防刷、黑名单、活动鉴权。
- 缓存层：把商品信息和库存预热到 Redis，用 Lua 脚本做原子预扣。
- 异步层：预扣成功后写入 MQ，订单异步创建，达到削峰目的。
- 数据库层：真正扣减库存和创建订单时，仍要有条件更新和唯一约束兜底。
- 兜底：订单失败要补偿库存，库存差异要定时对账。

### 秒杀如何防止超卖？

- Redis 预扣必须原子执行。
- 数据库更新也要带条件，例如 `count > 0`。
- 同一用户维度要有唯一约束，防止重复下单。
- Redis 和数据库不一致时，靠补偿和对账收敛。

### 秒杀如何削峰？

- 入口限流，先把绝大多数无效流量拦住。
- 请求排队，通过 MQ 或内存队列按后端处理能力消费。
- 异步下单，先返回“排队中”，减少同步等待。
- 热点隔离，把秒杀链路和普通交易链路拆开，避免拖垮主站。

## 6. 高频题型二：订单超时取消

### 订单 30 分钟未支付自动取消，怎么设计？

- 下单时写入订单状态和过期时间。
- 推荐做法是延时消息：订单创建后发一条 30 分钟后的检查消息。
- 消费时判断订单是否仍为待支付，若是则取消订单并释放库存。
- 定时扫描可以作为兜底方案，负责修正延时消息遗漏。
- Redis 过期通知实现简单，但可靠性不够，通常不单独作为主方案。

### 支付成功和超时取消并发怎么办？

- 关键是订单状态机。
- 支付只能把 `WAIT_PAY` 改成 `PAID`。
- 取消只能把 `WAIT_PAY` 改成 `CANCELED`。
- 更新语句带状态条件，谁先改成功谁生效。
- 支付回调、延时消息、定时任务都可能重复，所以相关逻辑必须幂等。

## 7. 高频题型三：短链接系统

### 如何设计一个短链接系统？

- 核心功能分两段：长链接转短码；短码解析并重定向。
- 短码生成方案常见有：
  自增 ID 转 62 进制、
  哈希加冲突处理、
  分布式 ID 转 62 进制。
- 存储层通常用 MySQL 保存映射关系：`short_code -> long_url`。
- 热点短链可以放 Redis，加快重定向查询。
- 统计点击量时通常用 `302`，因为每次都能经过服务端。
- 防滥用要做：频率限制、黑名单、URL 校验、有效期控制。

### 短链接系统的难点是什么？

- 短码冲突
- 热点短链读压力
- 恶意刷链和滥用
- 点击统计与主重定向链路的性能平衡

## 8. 高频题型四：缓存与 Redis 扩容

### 单 Redis 扛不住了，怎么扩？

- 先区分问题类型：读瓶颈、写瓶颈、热点瓶颈还是容量瓶颈。
- 读压力大：本地缓存、从库读、多副本、Redis Cluster。
- 写压力大：按 key 分片，或直接上 Redis Cluster。
- 热点 key：拆 key、多级缓存、请求合并、热点续期。
- 大 key：拆结构，避免迁移和网络阻塞。

### Redis Cluster 扩容要注意什么？

- 扩容本质是 slot 迁移。
- 客户端要支持 MOVED / ASK 重定向。
- 大 key 和热点 key 会放大迁移成本。
- 最好低峰期操作，并提前准备监控和回滚方案。

## 9. 高频题型五：负载均衡

### 常见负载均衡方案有哪些？

- DNS 负载均衡：适合入口层和跨地域调度。
- 四层负载均衡：基于 IP 和端口，性能高。
- 七层负载均衡：基于 HTTP 特征做路由，更灵活。
- 客户端负载均衡：服务消费者自己选实例。
- 服务端负载均衡：统一 LB 负责转发。

### 常见负载均衡算法有哪些？

- 轮询
- 加权轮询
- 随机 / 加权随机
- 最少连接
- 一致性哈希

### 负载均衡怎么做高可用？

- LB 自己不能是单点。
- 后端实例要有健康检查和故障摘除。
- 网关或 LB 要配合超时、限流、熔断。
- 多机房场景下还要考虑切流和容量冗余。

## 10. 设计题里的通用优化手段

无论题目是什么，最后通常都能落到这些优化手段：

- 缓存：减轻数据库压力
- 异步化：缩短主链路、削峰
- 分库分表或分片：扩展写能力和容量
- 限流、降级、熔断：控制故障范围
- 幂等、重试、补偿：收敛异常链路
- 对账、监控、压测：让系统可恢复、可观测

## 11. 设计题里的常见扣分点

### 系统设计题最容易答散的地方是什么？

- 只列组件，不讲主链路。
- 只讲正常流程，不讲异常和兜底。
- 不区分核心链路和非核心链路。
- 完全没有容量和瓶颈意识。
- 讲了高并发，却没讲幂等、限流、补偿和监控。

### 面试官为什么喜欢追问“如果挂了怎么办”？

- 因为系统设计不只是把功能做出来，更重要的是系统在压力和故障下如何表现。
- 一个看起来能跑的方案，如果没有超时、限流、降级、回滚、补偿，其实很难算真正可上线。

## 12. 一页答题模板

如果被问“系统设计题你一般怎么答”，可以这样说：

> 我通常先确认需求边界，包括流量、数据量、一致性和可用性要求；然后画核心链路，说明请求从入口到落库怎么走，哪些是同步、哪些是异步；再识别瓶颈，比如热点数据、数据库写压力、消息堆积和单点问题；最后再补优化和兜底手段，例如缓存、MQ、限流、幂等、补偿、监控和压测。这样能避免一上来只堆组件，而是围绕问题本身展开。

## 13. 一页总结

- 设计题先问边界，再讲主链路
- 组件不是答案，瓶颈和取舍才是答案
- 高频题最终都会落到缓存、异步、限流、幂等、补偿
- 正常流程只能说明“能跑”，异常流程才说明“能上线”
- 说清为什么这样设计，比多背几个组件名字更重要
