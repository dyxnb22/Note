# 设计题答题框架

## 系统设计题应该怎么回答？

1. 先问清需求：用户规模、QPS、数据量、核心功能、可用性要求、一致性要求，以及是否有峰值场景。
2. 再画核心链路：入口层、业务服务、缓存、数据库、消息队列、异步任务、监控告警。
3. 然后识别瓶颈：流量入口、热点数据、库存一致性、数据库写压力、消息堆积、单点故障。
4. 最后讲取舍和兜底：限流、降级、幂等、重试、补偿、数据对账、容量预估、压测和监控。

面试不要只堆组件，要把“为什么这么设计”和“出问题怎么兜底”讲清楚。

# 秒杀系统

## 秒杀系统如何设计？如何防止超卖和削峰？

秒杀的核心矛盾是瞬时高并发和有限库存，目标是把大部分请求挡在数据库之前。

**入口层：**按钮置灰、倒计时、验证码或滑块、静态资源 CDN，减少无效请求和恶意刷接口。

**网关层：**限流、黑名单、风控、接口鉴权、防重复提交，把流量控制在服务可承受范围内。可以按 IP、用户、活动等维度限制 QPS。

**缓存层：**商品信息和库存预热到 Redis，用原子扣减或 Lua 脚本完成“判断库存大于 0 + 扣减库存”，避免请求直接打到数据库。

**异步下单：**Redis 预扣成功后写入消息队列，订单服务异步创建订单，削峰填谷。接口可以先返回“排队中/抢购中”，再通过轮询、推送或订单页查询结果。

**数据库层：**最终落库时仍要做库存校验、唯一订单约束和幂等处理，例如：

```sql
update stock
set count = count - 1
where sku_id = ? and count > 0;
```

还可以对“活动 id + 用户 id”建立唯一约束，防止同一用户重复下单。

**热点隔离：**秒杀库存、订单表、Redis Key 和服务集群尽量与普通交易隔离，避免秒杀流量拖垮主站。

**兜底：**Redis 和数据库可能短暂不一致，预扣库存后如果订单创建失败，要通过补偿任务释放库存，并监控队列堆积、失败率和库存差异，定期进行异步对账。

# 订单超时取消

## 订单超时取消与支付并发如何处理？

创建订单时记录过期时间，例如 `expire_time = create_time + 30min`，订单初始状态为待支付。

订单取消方案：

- 延时消息：创建订单后发送一条 30 分钟延时消息，到期后消费者检查订单是否仍未支付，未支付则关闭订单并释放库存。
- 定时扫描：定时任务分页扫描过期未支付订单，适合兜底和补偿，但要控制实时性和扫描压力。
- Redis 过期通知：实现简单但可靠性一般，通知可能丢失，不能作为唯一方案。

支付成功和超时取消可能并发执行，需要用订单状态机控制：

- 支付成功只能把“待支付”改为“已支付”。
- 取消只能把“待支付”改为“已取消”。
- 数据库更新带条件：`where order_id = ? and status = 'WAIT_PAY'`，谁先更新成功谁生效。
- 失败的一方读取最新状态后按业务处理：已支付则不能取消，已取消则支付回调走退款或失败处理。

取消订单、支付回调、延时消息和定时任务都可能重复触发，因此必须保证幂等。释放库存、退优惠券、关闭支付单等动作最好异步化，并配合补偿任务和对账。

# Redis 扩容

## Redis 扩容和 Cluster 迁移要注意什么？

先区分读瓶颈、写瓶颈、热点瓶颈和容量瓶颈，再给对应方案：

- 读压力大：增加本地缓存、合理使用 Redis 从节点读、使用 Redis Cluster 分片；从节点读取要注意主从延迟。
- 写压力大：按业务 Key 分片到多个 Redis 节点，或使用 Redis Cluster 的 hash slot 分片。
- 热点 Key：拆 Key、加随机后缀、多级缓存、请求合并、限流，避免所有请求集中访问同一个 Key。
- 大 Key：拆分数据结构，避免单个 Key 过大导致网络传输、阻塞和迁移问题。
- 高可用：使用哨兵或 Cluster、主从复制和故障转移，同时关注脑裂和数据丢失窗口。

Redis Cluster 通过 hash slot 做分片，扩容本质是把部分 slot 迁移到新节点。迁移期间：

- 客户端要支持 `MOVED` / `ASK` 重定向，否则可能出现请求失败。
- 扩容前评估大 Key 和热点 Key，否则可能迁移慢、阻塞高、负载不均。
- 关键业务尽量在低峰期操作，并提前准备回滚、监控和容量验证方案。

# 可重入分布式锁

## 如何设计一个可重入的分布式锁？

可重入指同一个持锁者已经持有锁时，可以再次加锁而不会被自己阻塞。这里的持锁者不能只依赖线程 ID，通常由服务实例 ID、线程 ID 和请求 ID 等信息组成唯一标识。

Redis 可以用 Hash 结构表示锁：Key 是锁名，Field 是持锁者标识，Value 是重入次数。

- 第一次加锁：如果锁不存在，则创建 Hash，将 Field 设置为当前持锁者标识，Value 设为 1，并设置过期时间。
- 重入加锁：如果锁已存在且 Field 匹配当前持锁者，则计数加 1，并刷新过期时间。
- 解锁：只有 Field 匹配当前持锁者才能解锁；计数减 1，减到 0 才删除锁 Key。
- 原子性：判断锁归属、加减计数、设置过期时间和删除锁都要通过 Lua 脚本保证原子性。
- 防误删：锁过期后可能已被其他客户端获得，原持有者必须先比较标识再删除，不能直接 `DEL`。
- 看门狗：如果业务执行时间不确定，需要续期；续期也只能由当前持锁者执行。

# 负载均衡

## 负载均衡有哪些方案、算法和高可用措施？

常见负载均衡方案：

| 方案             | 特点与适用场景                                                   |
| ---------------- | --------------------------------------------------------------- |
| DNS 负载均衡    | 简单、成本低，适合跨机房或入口级调度；但生效慢，故障摘除受 DNS 缓存影响 |
| 四层负载均衡    | 基于 IP 和端口转发，性能高，常见于 LVS、云负载均衡 TCP 层     |
| 七层负载均衡    | 基于 HTTP URL、Header、Cookie 等路由，常见于 Nginx、Envoy、网关 |
| 客户端负载均衡  | 客户端从注册中心获取服务列表并自行选择实例，常见于微服务 RPC |
| 服务端负载均衡  | 请求先到统一 LB，再转发到后端实例，调用方不感知实例变化      |

常见负载均衡算法：

- 轮询：请求依次分配到每台机器，简单但不考虑机器性能差异。
- 加权轮询：给性能更好的机器更高权重，适合机器规格不同的场景。
- 随机 / 加权随机：实现简单，大量请求下分布较均匀。
- 最少连接：优先发给当前连接数少的机器，适合长连接或请求耗时差异大的场景。
- 一致性哈希：相同 Key 尽量落到同一节点，适合缓存和有状态路由，节点增减时迁移量较小。

高可用措施：

- LB 自身不能是单点，通常要多实例部署，配合 VIP、云 LB、Keepalived 或 DNS 做容灾。
- 后端实例要做健康检查，失败实例及时摘除，恢复后再逐步放量。
- 网关或 LB 要配置限流、熔断、超时和重试策略，避免放大下游故障。
- 多机房场景要考虑就近访问、故障切流、数据一致性和容量冗余。

# 短链接系统

## 如何设计一个短链接系统？

核心需求：长 URL 转短码、短码重定向到长 URL、高并发读取和防滥用。

**短码生成：**

| 方案                   | 特点                                                         |
| ---------------------- | ------------------------------------------------------------ |
| 自增 ID 转 62 进制     | 最简单，数字 + 大小写字母，6 位可表示约 568 亿条            |
| 哈希                   | 例如 MurmurHash 取前 6 位，冲突时加盐重试                   |
| 预生成池               | 提前生成短码存入 Redis，写入时直接取，无冲突风险             |
| 分布式 ID 转 62 进制  | 例如雪花算法，兼顾唯一性和性能，适合生产环境               |

**存储设计：**

- MySQL 保存映射关系：`short_code(PK)`、`long_url`、`user_id`、`expire_at`、`click_count`。
- Redis 缓存热门短链，保存 `short_code → long_url` 映射，TTL 与过期时间对齐。

**重定向方式：**

- 301 永久重定向：浏览器可能缓存，后续请求不一定经过服务端，不适合精确、持续的点击统计。
- 302 临时重定向：通常每次经过服务端，可以统计点击量、来源等，适合需要统计的场景。

**高可用和防滥用：**Redis 集群缓存抗读压力，MySQL 主从，多地部署；同时增加 IP 限流、黑名单、短链有效期和创建频率限制。
