# 设计题答题框架
<details>
<summary>系统设计题应该怎么回答？</summary>
	- 先问清需求：用户规模、QPS、数据量、核心功能、可用性要求、一致性要求、是否有峰值场景。
	- 再画核心链路：入口层、业务服务、缓存、数据库、消息队列、异步任务、监控告警。
	- 然后识别瓶颈：流量入口、热点数据、库存一致性、数据库写压力、消息堆积、单点故障。
	- 最后讲取舍和兜底：限流、降级、幂等、重试、补偿、数据对账、容量预估、压测和监控。
	- 面试不要只堆组件，要把“为什么这么设计”和“出问题怎么兜底”讲清楚。
</details>
# 秒杀系统
<details>
<summary>如何设计一个秒杀场景？</summary>
	- 秒杀核心矛盾是瞬时高并发和有限库存，所以目标是把大部分请求挡在数据库之前。
	- 前端层：按钮置灰、倒计时、验证码或滑块、静态资源 CDN，减少无效请求和恶意刷接口。
	- 网关层：限流、黑名单、风控、接口鉴权、防重复提交，把流量控制在服务可承受范围内。
	- 缓存层：商品信息和库存预热到 Redis，用原子扣减或 Lua 脚本完成库存预扣，避免直接打数据库。
	- MQ 异步化：Redis 预扣成功后写入消息队列，订单服务异步创建订单，削峰填谷。
	- 数据库层：最终落库时要有库存校验、唯一订单约束、幂等处理，防止重复下单和超卖。
	- 兜底：预扣库存后如果订单创建失败，要通过补偿任务释放库存；同时监控队列堆积、失败率和库存差异。
</details>
<details>
<summary>秒杀如何防止超卖？</summary>
	- Redis 预扣库存必须是原子的，常用 Lua 脚本完成“判断库存 \> 0 + 扣减库存”。
	- 数据库更新也要加条件，例如 `update stock set count = count - 1 where sku_id = ? and count > 0`。
	- 对用户维度加唯一约束，例如活动 id + 用户 id，防止同一用户重复下单。
	- Redis 和数据库可能短暂不一致，所以要有异步对账和补偿，不能只相信单边数据。
</details>
<details>
<summary>秒杀如何削峰？</summary>
	- 入口限流：网关按 IP、用户、活动维度限制 QPS。
	- 排队机制：请求先进入 MQ 或内存队列，后端按消费能力处理。
	- 异步下单：先返回“排队中/抢购中”，订单创建结果通过轮询、推送或订单页查询。
	- 热点隔离：秒杀库存、订单表、Redis key、服务集群都尽量和普通交易隔离，避免拖垮主站。
</details>
# 订单超时取消
<details>
<summary>订单到了半个小时未支付就取消，怎么设计？</summary>
	- 创建订单时记录过期时间，例如 `expire_time = create_time + 30min`，订单初始状态为待支付。
	- 推荐方案一：延时消息。创建订单后发送一条 30 分钟延时消息，到期后消费者检查订单是否仍未支付，未支付则关闭订单并释放库存。
	- 推荐方案二：定时扫描。定时任务分页扫描过期未支付订单，适合兜底和补偿，但实时性和扫描压力要控制。
	- 推荐方案三：Redis 过期通知。实现简单但可靠性一般，通知可能丢，不适合作为唯一方案。
	- 关键点：取消订单必须幂等，支付回调和取消任务并发时要用状态机控制，例如只能从“待支付”流转到“已支付”或“已取消”。
	- 释放库存、退优惠券、关闭支付单等动作最好异步化，并配合补偿任务和对账。
</details>
<details>
<summary>支付成功和超时取消并发怎么办？</summary>
	- 用订单状态机解决：支付成功只能把“待支付”改为“已支付”，取消也只能把“待支付”改为“已取消”。
	- 数据库更新带条件，例如 `where order_id = ? and status = 'WAIT_PAY'`，谁先成功谁生效。
	- 失败的一方读取最新状态后按业务返回：已支付则不能取消，已取消则支付回调走退款或失败处理。
	- 所有消息和回调都要幂等，因为支付平台回调、延时消息、定时任务都可能重复触发。
</details>
# Redis 扩容
<details>
<summary>大流量网站里单 Redis 扛不住，怎么解决？</summary>
	- 读压力大：加本地缓存、增加 Redis 从节点读、使用 Redis Cluster 分片、热点 key 做多副本或本地缓存。
	- 写压力大：按业务 key 分片到多个 Redis 节点，或者使用 Redis Cluster 自动分槽。
	- 热点 key：拆 key、加随机后缀、多级缓存、请求合并、限流，避免所有请求打到同一个 key。
	- 大 key：拆分数据结构，避免单个 key 过大导致网络传输、阻塞和迁移问题。
	- 高可用：哨兵或 Cluster，主从复制，故障转移，同时关注脑裂和数据丢失窗口。
	- 面试表达：先区分读瓶颈、写瓶颈、热点瓶颈、容量瓶颈，再给对应方案。
</details>
<details>
<summary>Redis Cluster 扩容时要注意什么？</summary>
	- Redis Cluster 通过 hash slot 做分片，扩容本质是把部分 slot 迁移到新节点。
	- 迁移期间客户端要支持 MOVED/ASK 重定向，否则会出现请求失败。
	- 扩容前要评估大 key 和热点 key，否则迁移慢、阻塞高、负载不均。
	- 关键业务要在低峰期操作，并提前准备回滚和监控方案。
</details>
# 可重入分布式锁
<details>
<summary>如何设计一个可重入的分布式锁？</summary>
	- 可重入的意思是同一个线程或同一个业务执行上下文已经持有锁时，可以再次加锁而不会被自己阻塞。
	- Redis 里可以用 Hash 结构表示锁：key 是锁名，field 是持锁者标识，value 是重入次数。
	- 第一次加锁：如果锁不存在，则创建 hash，设置 field 为当前线程/请求标识，value 为 1，并设置过期时间。
	- 重入加锁：如果锁已经存在且 field 是当前持锁者，则计数加 1，并刷新过期时间。
	- 解锁：只有 field 匹配当前持锁者才能解锁；计数减 1，减到 0 才删除锁 key。
	- 上述判断、加减计数、设置过期时间都要用 Lua 脚本保证原子性。
</details>
<details>
<summary>可重入分布式锁为什么需要唯一持锁者标识？</summary>
	- 因为锁过期后可能被其他客户端获得，原持有者如果直接 delete key，会误删别人的锁。
	- 标识通常由服务实例 id + 线程 id + 请求 id 组成，能区分不同执行上下文。
	- 解锁时必须先比较标识再删除或减计数，这一步要和删除操作原子执行。
	- 如果业务执行时间不确定，还需要看门狗续期，但续期也只能由当前持锁者执行。
</details>
# 负载均衡
<details>
<summary>你看过哪些负载均衡方案？</summary>
	- DNS 负载均衡：简单、成本低，适合跨机房或入口级调度，但生效慢，故障摘除受 DNS 缓存影响。
	- 四层负载均衡：基于 IP 和端口转发，性能高，常见于 LVS、云负载均衡 TCP 层。
	- 七层负载均衡：基于 HTTP URL、Header、Cookie 等做路由，能力更灵活，常见于 Nginx、Envoy、网关。
	- 客户端负载均衡：客户端从注册中心拿服务列表，自己选择实例，常见于微服务 RPC。
	- 服务端负载均衡：请求先到统一 LB，再转发到后端实例，调用方不感知实例变化。
</details>
<details>
<summary>常见负载均衡算法有哪些？</summary>
	- 轮询：请求依次分配到每台机器，简单但不考虑机器性能差异。
	- 加权轮询：给性能更好的机器更高权重，适合机器规格不同的场景。
	- 随机/加权随机：实现简单，大量请求下分布较均匀。
	- 最少连接：优先发给当前连接数少的机器，适合长连接或请求耗时差异大的场景。
	- 一致性哈希：相同 key 尽量落到同一节点，适合缓存和有状态路由，节点增减时迁移量较小。
</details>
<details>
<summary>负载均衡怎么做高可用？</summary>
	- LB 自身不能是单点，通常要多实例部署，配合 VIP、云 LB、Keepalived 或 DNS 做容灾。
	- 后端实例要做健康检查，失败实例及时摘除，恢复后再逐步放量。
	- 网关或 LB 要有限流、熔断、超时和重试策略，避免把故障放大。
	- 多机房场景要考虑就近访问、故障切流、数据一致性和容量冗余。
</details>
# 短链接系统
<details>
<summary>如何设计一个短链接系统？</summary>
	核心需求：长URL → 短码（6位）、短码 → 重定向长URL、高并发读、防滥用。<br><br>短码生成方案：<br>① 自增ID转62进制（最简单，数字+大小写字母，6位可表示约568亿条）<br>② 哈希（MurmurHash取前6位，冲突则加盐重试）<br>③ 预生成池（提前生成短码存Redis，写时直接取，无冲突风险）<br>生产推荐：分布式ID（如雪花算法）转62进制，兼顾唯一性和性能。<br><br>存储设计：<br>• MySQL存映射关系：short_code(PK), long_url, user_id, expire_at, click_count<br>• Redis缓存热门短链（short_code → long_url），TTL与过期时间对齐<br><br>重定向方式：<br>• 301 永久重定向：浏览器缓存，下次直接访问长URL，服务端无法统计点击<br>• 302 临时重定向：每次经过服务端，可统计点击量、来源等<br>• 生产统计场景用 302<br><br>高可用：Redis集群缓存抗读压力，MySQL主从，多地部署。<br>防滥用：IP限流、黑名单、短链设置有效期、创建频率限制。
</details>
