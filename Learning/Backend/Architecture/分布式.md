# 分布式理论
<details>
<summary>介绍一下 CAP 理论</summary>
	- CAP 指一致性 Consistency、可用性 Availability、分区容错性 Partition tolerance。
	- 一致性：用户读到的数据必须是最新的，或者所有节点看到的数据一致。
	- 可用性：每个请求都能在合理时间内得到非错误响应。
	- 分区容错性：网络分区、节点之间通信失败时，系统仍能继续工作。
	- 分布式系统里网络故障无法完全避免，所以 P 通常必须接受；真正的取舍是在 CP 和 AP 之间。
	- CP 代表优先保证一致性，必要时牺牲部分可用性，例如强一致配置中心、分布式锁。
	- AP 代表优先保证可用性，允许短暂不一致，后续通过同步和补偿达到最终一致，例如很多业务缓存和异步系统。
</details>
<details>
<summary>CAP 和 BASE 是什么关系？</summary>
	- BASE 是 Basically Available、Soft state、Eventually consistent，即基本可用、软状态、最终一致。
	- 它不是推翻 CAP，而是在 AP 方向上的工程实践：服务先保证可用，再通过异步复制、重试、补偿、对账让数据最终一致。
	- 面试可以这样讲：强一致成本高，很多互联网业务更关注可用性和吞吐，因此常用 BASE 思路设计最终一致系统。
</details>
# 分布式锁
<details>
<summary>用 Redis 怎么实现分布式锁？</summary>
	- 加锁用 `SET key value NX PX expire_ms`，必须同时满足互斥和过期时间，避免死锁。
	- value 要放唯一标识，例如 UUID，释放锁时只允许持有者删除自己的锁。
	- 解锁要用 Lua 脚本保证“判断 value + 删除 key”是原子操作，避免误删别人刚获得的锁。
	- 业务执行时间可能超过锁过期时间，所以要么设置合理过期时间，要么使用看门狗续期机制。
	- Redis 分布式锁适合短临界区和可容忍少量异常的业务；强一致要求特别高时要谨慎。
</details>
<details>
<summary>Redis 分布式锁有哪些坑？</summary>
	- 过期时间太短：业务没执行完锁就过期，其他线程进入临界区，导致并发问题。
	- 过期时间太长：持锁服务宕机后恢复慢，影响可用性。
	- 非原子解锁：先 get 再 delete 可能误删别人的锁，需要 Lua 保证原子性。
	- 主从切换：主节点写入锁后还没同步到从节点就宕机，新的主节点没有这把锁，可能出现多个客户端同时持锁。
	- 可重入、续期、公平性都不是 Redis 原生命令直接提供的，通常要靠 Redisson 等客户端封装。
</details>
<details>
<summary>还有没有其他方法实现分布式锁？</summary>
	- 数据库锁：用唯一索引插入锁记录，或者用行锁控制互斥；实现简单但性能一般，适合低频场景。
	- ZooKeeper 锁：利用临时顺序节点和 watcher，节点断开后锁自动释放，一致性更强，适合对正确性要求高的场景。
	- etcd 锁：基于 lease 和 revision，也适合服务协调、选主、配置一致性等场景。
	- Redis 锁：性能高、实现方便，适合高频短任务，但要处理过期、续期和主从切换风险。
	- 选型话术：性能优先选 Redis；一致性和协调语义优先选 ZooKeeper/etcd；简单低频任务可用数据库。
</details>
# 分布式事务
<details>
<summary>分布式事务的解决方案你知道哪些？</summary>
	- 2PC：协调者先询问各参与者能否提交，再统一提交或回滚。优点是强一致，缺点是同步阻塞、协调者单点、性能较差。
	- 3PC：在 2PC 基础上增加阶段降低阻塞风险，但实现复杂，实际业务中不常直接使用。
	- TCC：Try 预留资源，Confirm 确认提交，Cancel 释放资源。适合核心业务，但侵入性强，每个服务都要写三套逻辑。
	- Saga：把长事务拆成一系列本地事务，每步有对应补偿动作。适合长流程，但只能保证最终一致。
	- 事务消息：本地事务成功后通过可靠消息驱动下游，常用于订单、支付、库存等最终一致场景。
	- 本地消息表 / Outbox：业务数据和消息表写入同一个本地事务，再由后台任务投递 MQ，失败可重试和补偿。
</details>
<details>
<summary>阿里的 Seata 框架了解过吗？</summary>
	- Seata 是分布式事务框架，常见模式有 AT、TCC、Saga、XA。
	- AT 模式对业务侵入较低，会记录 undo log，一阶段提交本地事务并释放本地锁，二阶段根据全局事务结果提交或回滚。
	- TCC 模式需要业务实现 Try、Confirm、Cancel，适合对一致性要求高且业务可控的场景。
	- Saga 模式适合长事务，把流程拆成多个本地事务，失败时按补偿动作回滚。
	- 面试重点：Seata 能降低分布式事务开发成本，但并不是银弹；要考虑性能、锁冲突、补偿、幂等、空回滚、悬挂等问题。
</details>
<details>
<summary>TCC 要注意哪些问题？</summary>
	- 幂等：Confirm 或 Cancel 可能被重复调用，必须保证重复执行结果一致。
	- 空回滚：Try 没执行成功却收到了 Cancel，Cancel 要能识别并安全返回。
	- 悬挂：Cancel 先于 Try 到达，后续 Try 不能再真正预留资源。
	- 资源预留：Try 阶段要冻结库存、余额或额度，Confirm 才真正扣减，Cancel 释放冻结。
	- TCC 的优点是控制精细，缺点是业务侵入性强，开发和测试成本高。
</details>
<details>
<summary>最终一致性怎么落地？</summary>
	- 用本地事务保证核心数据和消息记录同时落库。
	- 后台任务或 MQ 可靠投递事件，下游消费失败可以重试。
	- 所有消费者必须幂等，避免重复消息导致重复扣减或重复发放。
	- 做对账任务定期发现不一致数据，再走补偿流程修正。
	- 对用户体验可以先展示“处理中”，最终成功或失败再通知。
</details>
# 分布式组件
<details>
<summary>RPC 的概念是什么？</summary>
	- RPC 是 Remote Procedure Call，远程过程调用，让调用远程服务像调用本地方法一样。
	- 一个 RPC 框架通常包含代理、序列化、网络传输、服务注册发现、负载均衡、超时、重试、熔断等能力。
	- 调用流程：客户端代理封装请求，序列化后通过网络发给服务端；服务端反序列化并执行目标方法，再把结果返回。
	- 常见问题：网络不可靠、超时、重复请求、服务实例变更、版本兼容、链路追踪。
	- 面试提醒：RPC 屏蔽了远程调用细节，但不能把远程调用真的当成本地调用，因为延迟和失败模型完全不同。
</details>
<details>
<summary>ZooKeeper 拿来做什么？核心原理是什么？</summary>
	- ZooKeeper 是分布式协调服务，常用于注册中心、配置管理、分布式锁、选主、集群成员管理。
	- 它的数据模型类似文件系统树，每个 znode 可以存数据，并支持临时节点、顺序节点和 watcher 通知。
	- 临时节点会在客户端会话断开后自动删除，适合服务注册和锁释放。
	- 顺序节点可以实现公平锁和选主；watcher 可以监听节点变化并触发客户端更新。
	- 底层通过 Zab 协议保证集群内数据一致，写请求需要 leader 协调，读请求可以由 follower 提供。
</details>
<details>
<summary>注册中心为什么重要？</summary>
	- 微服务实例会动态扩缩容、重启、迁移，调用方不能写死服务地址。
	- 注册中心保存服务名到实例列表的映射，服务启动时注册，停止或失联时下线。
	- 客户端或网关通过注册中心发现可用实例，再进行负载均衡和调用。
	- 注册中心还要处理健康检查、变更通知、故障剔除和多机房容灾。
</details>
