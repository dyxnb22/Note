# 分布式系统基础

分布式系统由多个独立进程通过不可靠网络协作。它最核心的困难不是“机器多”，而是无法同时获得全局时钟、即时通信和绝对可靠的节点状态。

锁、选主、注册和 ID 生成集中到 [分布式协调](分布式协调.md)；事务、幂等和补偿见 [可靠性与一致性](可靠性与一致性.md)；限流、熔断和容灾见 [高可用与服务治理](高可用与服务治理.md)。

## 先建立故障模型

设计前先回答：

- 节点会崩溃、暂停还是返回错误结果？
- 网络会延迟、丢包、重复、乱序或分区吗？
- 存储写成功但响应丢失时，调用方如何判断结果？
- 时钟是否同步，能否用本地时间判断先后？
- 系统要容忍多少节点或机房同时故障？

超时只说明“在期限内没有得到响应”，不能证明对方没有执行。

## 一致性、可用性与分区

CAP 讨论网络分区发生时，一致性和可用性之间的取舍：

- 一致性：所有客户端看到单一、最新的逻辑值。
- 可用性：每个到达非故障节点的请求都能得到非错误响应。
- 分区容忍：网络分区时系统仍按定义工作。

真实系统不能通过“不选 P”让网络永不分区，而是要说明分区期间哪些操作拒绝、降级或接受旧数据。CAP 也不能替代对正常状态下延迟、吞吐和数据新鲜度的设计。

BASE 是工程风格概括，不是严格协议：允许部分功能基本可用、中间状态暂时不一致，最终通过重试、补偿和对账收敛。

## 一致性模型

| 模型 | 保证 | 典型用途 |
| --- | --- | --- |
| 线性一致性 | 操作像在单一实时顺序中发生 | 锁、选主、唯一元数据 |
| 顺序一致性 | 所有节点同意一个顺序，但不要求符合真实时间 | 部分协调和复制系统 |
| 因果一致性 | 有因果关系的操作保持顺序 | 协作、社交数据 |
| 最终一致性 | 无新写入时副本最终收敛 | 缓存、异步复制 |

“最终一致”必须同时定义收敛条件、最长可接受时间、冲突解决和失败兜底，不能只是一句承诺。

## 复制

复制提高读能力和容错，但引入副本延迟、冲突和故障切换问题。

- 单主复制：写入路径简单，但主节点是写入协调点。
- 多主复制：多地写入能力强，但冲突解决复杂。
- 无主复制：通过法定数量读写获得可调一致性，但要处理读修复和版本冲突。

`N`、`W`、`R` 只是副本数和读写法定数量；`W + R > N` 能增加读到最新版本的交集机会，但不自动提供线性一致性，还要考虑并发写、故障节点和版本判定。

## 共识解决什么

共识让多个节点在部分故障下对一串决议形成相同顺序，常用于复制日志、选主和元数据管理。它不能让业务系统自动获得端到端正确性。

### Raft

Raft 把问题拆成领导者选举、日志复制和安全性：

1. 节点以 Follower 启动，选举超时后成为 Candidate。
2. 获得多数票的节点成为 Leader。
3. 写入先追加 Leader 日志，再复制给多数节点。
4. 多数确认后条目提交，并按顺序应用到状态机。
5. 任期号防止旧 Leader 继续被信任。

多数派系统通常需要 `2f + 1` 个节点容忍 `f` 个节点故障。失去多数派时宁可停止写，也不能让两个分区都确认冲突日志。

### Paxos 与 Raft

Paxos 是共识协议家族和理论基础；Raft 更强调可理解的工程分解。比较时不要简单说“Raft 更快”，应看实现、日志复制、成员变更、存储和网络条件。

### Gossip

Gossip 通过节点间周期性交换信息传播成员状态和数据摘要，扩展性和容错性好，但收敛需要时间，不提供强一致顺序。它适合成员发现、故障怀疑和最终传播，不替代共识。

## 顺序、时间与版本

物理时钟可能漂移和回拨，不能单独作为分布式事件顺序。常用方法包括：

- 单调序列或日志偏移量。
- 逻辑时钟表达 happens-before。
- 版本号和 Compare-And-Set 防止覆盖。
- 向量时钟识别并发版本。
- 混合逻辑时钟在时间可读性和因果关系间折中。

业务上更常见的做法是为聚合或记录维护版本，而不是追求全系统唯一时间线。

## 分片与一致性 Hash

分片把数据或流量按键分配到不同节点。设计时要同时考虑：

- 分片键是否均匀且符合查询路径。
- 热点键如何拆分或隔离。
- 扩缩容需要移动多少数据。
- 跨分片查询和事务成本。
- 故障恢复期间副本和路由如何更新。

一致性 Hash 通过环和虚拟节点减少节点变化时的数据迁移量，但不能自动解决热点、容量差异和多副本放置。

## RPC 的语义边界

RPC 让远程调用看起来像本地方法，但失败语义完全不同：网络超时、请求已执行但响应丢失、重试重复执行都必须显式处理。

RPC 合同至少包括：

- 请求和响应 Schema 的兼容策略。
- Deadline 传播和取消。
- 错误分类，不用异常文本作为协议。
- 幂等性和重试条件。
- 身份、租户和 Trace 上下文。
- 限流、熔断和降级语义。

## 验收清单

- 能区分网络超时、节点故障和业务失败。
- 能解释 CAP 在网络分区时实际要求取舍什么。
- 能区分复制、共识、协调和业务事务。
- 能说明 Raft 为什么需要多数派，以及失去多数派会怎样。
- 能为最终一致性写出收敛、冲突和兜底方案。
- 能解释 RPC 为什么不能拥有本地调用的失败语义。
