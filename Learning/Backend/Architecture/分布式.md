- # 分布式理论

    -   介绍一下 CAP 理论
        -   CAP 指一致性 Consistency、可用性 Availability、分区容错性 Partition tolerance。
        -   一致性：用户读到的数据必须是最新的，或者所有节点看到的数据一致。
        -   可用性：每个请求都能在合理时间内得到非错误响应。
        -   分区容错性：网络分区、节点之间通信失败时，系统仍能继续工作。
        -   分布式系统里网络故障无法完全避免，所以 P 通常必须接受；真正的取舍是在 CP 和 AP 之间。
        -   CP 代表优先保证一致性，必要时牺牲部分可用性，例如强一致配置中心、分布式锁。
        -   AP 代表优先保证可用性，允许短暂不一致，后续通过同步和补偿达到最终一致，例如很多业务缓存和异步系统。
    -   CAP 和 BASE 是什么关系？
        -   BASE 是 Basically Available、Soft state、Eventually consistent，即基本可用、软状态、最终一致。
        -   它不是推翻 CAP，而是在 AP 方向上的工程实践：服务先保证可用，再通过异步复制、重试、补偿、对账让数据最终一致。
        -   面试可以这样讲：强一致成本高，很多互联网业务更关注可用性和吞吐，因此常用 BASE 思路设计最终一致系统。

    # 分布式锁

    -   用 Redis 怎么实现分布式锁？
        -   加锁用 `SET key value NX PX expire_ms`，必须同时满足互斥和过期时间，避免死锁。
        -   value 要放唯一标识，例如 UUID，释放锁时只允许持有者删除自己的锁。
        -   解锁要用 Lua 脚本保证“判断 value + 删除 key”是原子操作，避免误删别人刚获得的锁。
        -   业务执行时间可能超过锁过期时间，所以要么设置合理过期时间，要么使用看门狗续期机制。
        -   Redis 分布式锁适合短临界区和可容忍少量异常的业务；强一致要求特别高时要谨慎。
    -   Redis 分布式锁有哪些坑？
        -   过期时间太短：业务没执行完锁就过期，其他线程进入临界区，导致并发问题。
        -   过期时间太长：持锁服务宕机后恢复慢，影响可用性。
        -   非原子解锁：先 get 再 delete 可能误删别人的锁，需要 Lua 保证原子性。
        -   主从切换：主节点写入锁后还没同步到从节点就宕机，新的主节点没有这把锁，可能出现多个客户端同时持锁。
        -   可重入、续期、公平性都不是 Redis 原生命令直接提供的，通常要靠 Redisson 等客户端封装。
    -   还有没有其他方法实现分布式锁？
        -   数据库锁：用唯一索引插入锁记录，或者用行锁控制互斥；实现简单但性能一般，适合低频场景。
        -   ZooKeeper 锁：利用临时顺序节点和 watcher，节点断开后锁自动释放，一致性更强，适合对正确性要求高的场景。
        -   etcd 锁：基于 lease 和 revision，也适合服务协调、选主、配置一致性等场景。
        -   Redis 锁：性能高、实现方便，适合高频短任务，但要处理过期、续期和主从切换风险。
        -   选型话术：性能优先选 Redis；一致性和协调语义优先选 ZooKeeper/etcd；简单低频任务可用数据库。

    # 分布式事务

    -   分布式事务的解决方案你知道哪些？
        -   2PC：协调者先询问各参与者能否提交，再统一提交或回滚。优点是强一致，缺点是同步阻塞、协调者单点、性能较差。
        -   3PC：在 2PC 基础上增加阶段降低阻塞风险，但实现复杂，实际业务中不常直接使用。
        -   TCC：Try 预留资源，Confirm 确认提交，Cancel 释放资源。适合核心业务，但侵入性强，每个服务都要写三套逻辑。
        -   Saga：把长事务拆成一系列本地事务，每步有对应补偿动作。适合长流程，但只能保证最终一致。
        -   事务消息：本地事务成功后通过可靠消息驱动下游，常用于订单、支付、库存等最终一致场景。
        -   本地消息表 / Outbox：业务数据和消息表写入同一个本地事务，再由后台任务投递 MQ，失败可重试和补偿。
    -   阿里的 Seata 框架了解过吗？
        -   Seata 是分布式事务框架，常见模式有 AT、TCC、Saga、XA。
        -   AT 模式对业务侵入较低，会记录 undo log，一阶段提交本地事务并释放本地锁，二阶段根据全局事务结果提交或回滚。
        -   TCC 模式需要业务实现 Try、Confirm、Cancel，适合对一致性要求高且业务可控的场景。
        -   Saga 模式适合长事务，把流程拆成多个本地事务，失败时按补偿动作回滚。
        -   面试重点：Seata 能降低分布式事务开发成本，但并不是银弹；要考虑性能、锁冲突、补偿、幂等、空回滚、悬挂等问题。
    -   TCC 要注意哪些问题？
        -   幂等：Confirm 或 Cancel 可能被重复调用，必须保证重复执行结果一致。
        -   空回滚：Try 没执行成功却收到了 Cancel，Cancel 要能识别并安全返回。
        -   悬挂：Cancel 先于 Try 到达，后续 Try 不能再真正预留资源。
        -   资源预留：Try 阶段要冻结库存、余额或额度，Confirm 才真正扣减，Cancel 释放冻结。
        -   TCC 的优点是控制精细，缺点是业务侵入性强，开发和测试成本高。
    -   最终一致性怎么落地？
        -   用本地事务保证核心数据和消息记录同时落库。
        -   后台任务或 MQ 可靠投递事件，下游消费失败可以重试。
        -   所有消费者必须幂等，避免重复消息导致重复扣减或重复发放。
        -   做对账任务定期发现不一致数据，再走补偿流程修正。
        -   对用户体验可以先展示“处理中”，最终成功或失败再通知。

    # 分布式组件

    -   RPC 的概念是什么？
        -   RPC 是 Remote Procedure Call，远程过程调用，让调用远程服务像调用本地方法一样。
        -   一个 RPC 框架通常包含代理、序列化、网络传输、服务注册发现、负载均衡、超时、重试、熔断等能力。
        -   调用流程：客户端代理封装请求，序列化后通过网络发给服务端；服务端反序列化并执行目标方法，再把结果返回。
        -   常见问题：网络不可靠、超时、重复请求、服务实例变更、版本兼容、链路追踪。
        -   面试提醒：RPC 屏蔽了远程调用细节，但不能把远程调用真的当成本地调用，因为延迟和失败模型完全不同。
    -   ZooKeeper 拿来做什么？核心原理是什么？
        -   ZooKeeper 是分布式协调服务，常用于注册中心、配置管理、分布式锁、选主、集群成员管理。
        -   它的数据模型类似文件系统树，每个 znode 可以存数据，并支持临时节点、顺序节点和 watcher 通知。
        -   临时节点会在客户端会话断开后自动删除，适合服务注册和锁释放。
        -   顺序节点可以实现公平锁和选主；watcher 可以监听节点变化并触发客户端更新。
        -   底层通过 Zab 协议保证集群内数据一致，写请求需要 leader 协调，读请求可以由 follower 提供。
    -   注册中心为什么重要？
        -   微服务实例会动态扩缩容、重启、迁移，调用方不能写死服务地址。
        -   注册中心保存服务名到实例列表的映射，服务启动时注册，停止或失联时下线。
        -   客户端或网关通过注册中心发现可用实例，再进行负载均衡和调用。
        -   注册中心还要处理健康检查、变更通知、故障剔除和多机房容灾。

    # 分布式场景

    -   常见的限流算法你知道哪些？
        -   固定窗口：按固定时间窗口计数，实现简单，但窗口边界可能出现瞬时流量翻倍。
        -   滑动窗口：把大窗口拆成多个小窗口，统计更平滑，精度比固定窗口高。
        -   漏桶算法：请求像水一样进入桶，按固定速率流出，能平滑流量，但突发能力弱。
        -   令牌桶算法：系统按固定速率生成令牌，请求拿到令牌才能通过，允许一定突发流量。
        -   面试取舍：接口保护常用令牌桶或滑动窗口；要严格匀速处理可用漏桶；简单场景固定窗口也能用。
    -   分布式限流如何实现？
        -   单机限流只能保护单个实例，多实例部署时总流量可能超过全局阈值。
        -   分布式限流常用 Redis 计数器、Redis Lua、令牌桶服务、网关限流或 Sentinel 等组件。
        -   Redis Lua 可以保证计数和过期设置原子执行，适合全局 QPS 控制。
        -   网关限流适合入口流量控制；服务内限流适合保护具体下游资源。
        -   限流后要配合降级返回、排队、重试建议和监控告警，不能只是简单丢请求。
    -   高并发下如何保护下游服务？
        -   限流：控制进入系统或下游的请求速率。
        -   熔断：下游持续失败时快速失败，避免线程和连接被拖垮。
        -   降级：关闭非核心能力，保留主流程。
        -   隔离：线程池、连接池、舱壁隔离，避免一个依赖拖垮整个服务。
        -   缓存和异步化：热点读走缓存，非核心写入异步处理。

    # 分布式一致性算法

    -   Raft 协议和 Paxos 协议的原理？
        -   Paxos 是经典一致性算法，目标是在可能失败的分布式节点之间对某个值达成一致，但概念抽象、实现复杂。
        -   Raft 和 Paxos 解决的问题类似，但更强调可理解性，把一致性拆成 leader 选举、日志复制、安全性三个部分。
        -   Raft 中节点有 follower、candidate、leader 三种角色；超时未收到 leader 心跳时 follower 会转为 candidate 发起选举。
        -   Leader 负责接收写请求并复制日志到 follower，多数节点确认后日志才算提交。
        -   如果 leader 故障，集群重新选主；新 leader 必须包含已提交日志，保证不会丢失已确认的数据。
    -   Raft 的核心流程是什么？
        -   Leader 选举：节点超时后发起投票，获得多数票成为 leader。
        -   日志复制：客户端写请求先到 leader，leader 把日志复制给 follower。
        -   提交规则：日志被多数节点持久化后提交，再应用到状态机。
        -   安全性：任期 term、日志索引和日志任期共同保证旧 leader 或落后节点不会覆盖新数据。
        -   心跳机制：leader 定期发送心跳维持权威，避免 follower 频繁发起选举。
    -   有什么框架或技术用了 Raft 协议？
        -   etcd 使用 Raft 来保证集群元数据一致，Kubernetes 依赖 etcd 存储集群状态。
        -   Consul 使用 Raft 做服务发现和配置数据的一致性复制。
        -   TiKV、CockroachDB、Nacos 的部分一致性场景也能看到 Raft 或类 Raft 思路。
        -   面试表达：Raft 常用于需要强一致元数据、配置、选主、分布式 KV 的系统。

    # 分布式ID

    -   为什么需要分布式ID？

        单库自增ID在分库分表后无法保证全局唯一，多节点各自自增会产生冲突。需要满足：全局唯一、趋势递增（利于B+树索引）、高性能、高可用。

    -   雪花算法（Snowflake）的原理是什么？

        64位 Long 型 = 1位符号位 + 41位毫秒时间戳 + 10位机器ID + 12位序列号。
         • 41位时间戳支持约 69 年
         • 10位机器ID支持 1024 个节点
         • 12位序列号每毫秒最多生成 4096 个ID
         整体趋势递增，天然适合数据库主键索引。

    -   雪花算法有什么缺点？如何解决时钟回拨？

        缺点：强依赖机器时钟，NTP 时钟回拨会导致生成重复ID。

        解决方案：
         ① 检测到回拨则等待至时钟追回
         ② 美团Leaf：用 ZooKeeper 记录上次时间，启动时校验
         ③ 百度UidGenerator：用 RingBuffer 预生成ID，规避实时依赖

    -   还有哪些分布式ID方案，如何选型？

        • UUID：无序、字符串，索引性能差，不推荐做数据库主键
         • 数据库号段模式：批量取号段到内存，美团Leaf号段模式，依赖DB但性能好
         • Redis INCR：高性能，需持久化保证重启后不重复
         • 美团Leaf / 百度UidGenerator：生产可用方案，推荐优先考虑

        选型建议：中大型系统用美团Leaf（同时支持号段模式和Snowflake模式），小项目直接用Snowflake自实现即可。

    # 服务熔断/降级

    -   限流、熔断、降级有什么区别？

        • 限流：控制请求速率，超过阈值直接拒绝/排队，保护服务不被打垮（令牌桶/漏桶/滑动窗口）
         • 熔断：监控下游错误率，超过阈值自动断开调用，防止级联雪崩，类比保险丝
         • 降级：服务不可用时返回兜底数据（缓存值、默认值、提示语），保证用户体验

        三者可以配合使用：先限流控入口，再熔断断下游，最后降级保体验。

    -   熔断器的三种状态是什么？

        • Closed（关闭）：正常通行，持续统计请求的错误率和响应时间
         • Open（打开）：错误率超过阈值，直接拒绝所有请求，不再调用下游，立即返回降级结果
         • Half-Open（半开）：Open 状态持续一段时间后，自动放入少量探测请求；成功则切回 Closed，失败则继续 Open

        状态转换本质是在「保护系统」和「尝试恢复」之间做动态切换。

    -   Sentinel 和 Hystrix 有什么区别？

        Hystrix（Netflix）：
         • 已停止维护（2018年进入维护模式）
         • 基于线程池隔离，每个服务独立线程池，资源开销大
         • 配置静态，修改需重启

        Sentinel（阿里开源，Apache 孵化）：
         • 基于信号量，轻量级，性能更好
         • 支持：流量控制、熔断降级、热点参数限流、系统自适应保护
         • 控制台实时推送规则，无需重启
         • 与 Spring Cloud / Dubbo / Gateway 生态集成完善

        生产推荐 Sentinel，国内大厂主流选择。

    # 分布式链路追踪

    -   什么是分布式链路追踪，解决什么问题？

        微服务架构下，一次用户请求会跨越多个服务（A→B→C→D），一旦超时或报错，很难定位是哪个服务的问题。

        链路追踪通过唯一 TraceId 串联整个调用链，每个节点记录为一个 Span，包含：
         • 服务名、接口名
         • 开始时间、耗时
         • 状态码、异常信息
         • 父子 Span 关系

        最终可视化展示为调用瀑布图，一眼看出哪个服务耗时多、哪里报错。

    -   TraceId 是怎么在服务间传递的？

        • HTTP 调用：放在请求头（如 X-Trace-Id、W3C 标准的 traceparent 字段）
         • RPC 调用：通过框架隐式传参（Dubbo 的 Attachment、gRPC 的 Metadata）
         • 线程内：存在 ThreadLocal / MDC（Mapped Diagnostic Context），日志框架自动打印 TraceId
         • 跨线程：需显式传递，线程池场景用 TransmittableThreadLocal（TTL）

        OpenTelemetry 是目前主流的跨框架统一标准，负责定义传播协议。

    -   常见的链路追踪框架有哪些，如何选型？

        • SkyWalking（推荐）：Apache 顶级项目，Java Agent 字节码增强无侵入，支持多语言，UI 强大，与国内 Spring Cloud/Dubbo 生态最兼容
         • Zipkin：Twitter 出品，轻量简单，Brave 客户端，适合小规模项目
         • Jaeger：CNCF 项目，云原生友好，适合 K8s 环境
         • OpenTelemetry：厂商中立标准，负责采集和传播，通常配合后端（Jaeger/Zipkin）使用

        选型建议：国内项目首选 SkyWalking；云原生 K8s 环境选 Jaeger + OpenTelemetry；追求极简选 Zipkin。
