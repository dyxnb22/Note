# 分布式理论

## CAP 和 BASE 是什么？

CAP 指一致性（Consistency）、可用性（Availability）和分区容错性（Partition tolerance）。在网络分区发生时，分布式系统通常需要在一致性和可用性之间取舍。

| 概念         | 含义                                                         |
| ------------ | ------------------------------------------------------------ |
| 一致性 C     | 在一致性模型要求下，读操作能看到符合顺序约束的最新结果；强一致场景通常指线性一致性 |
| 可用性 A     | 每个请求都能在合理时间内获得非错误响应                     |
| 分区容错性 P | 网络分区、节点之间通信失败时，系统仍能继续工作             |

网络故障无法完全避免，所以 P 通常必须接受；工程上的主要取舍是 CP 和 AP：

- CP：优先保证一致性，必要时牺牲部分可用性，例如基于 ZooKeeper/etcd 的协调服务。具体系统仍要结合实现判断，不能把所有分布式锁都简单归为 CP。
- AP：优先保证可用性，允许短暂不一致，后续通过同步和补偿达到最终一致，例如很多业务缓存和异步系统。

BASE 是 Basically Available、Soft state、Eventually consistent，即基本可用、软状态、最终一致。它不是推翻 CAP，而是在 AP 方向上的工程实践：服务先保证可用，再通过异步复制、重试、补偿和对账让数据最终一致。

# 分布式锁

## Redis 分布式锁如何实现？有哪些风险？

Redis 加锁通常使用：

```text
SET lock_key unique_value NX PX expire_ms
```

关键点：

- `NX` 保证互斥，`PX` 设置过期时间，避免持锁服务宕机后永久死锁。
- `value` 要放唯一持锁者标识，例如实例 ID + 线程 ID + 请求 ID 或 UUID。
- 解锁必须用 Lua 脚本保证“判断 value + 删除 key”原子执行，不能先 `GET` 再 `DEL`，避免误删其他客户端刚获得的锁。
- 业务执行时间可能超过锁过期时间，要么设置合理过期时间，要么使用看门狗续期机制。

常见风险：

- 过期时间太短：业务未执行完锁就过期，其他线程进入临界区。
- 过期时间太长：持锁服务宕机后恢复慢，影响可用性。
- 主从切换：主节点写入锁后还没同步到从节点就宕机，新的主节点可能没有这把锁，导致多个客户端同时持锁。
- 可重入、续期、公平性不是 Redis 原生命令直接提供的，通常需要 Redisson 等客户端封装。

Redis 分布式锁适合短临界区、性能优先且可以容忍少量异常的业务；对正确性要求特别高时，应谨慎评估故障模型。

## 可重入分布式锁如何实现？

可重入锁用“持锁者标识 + 重入次数”区分同一持锁者的重复加锁：Redis 可用 Hash 保存 `owner -> count`，首次加锁设置过期时间，重入时递增次数。

加锁、重入、解锁和续期都要用 Lua 保证原子性；解锁只允许持锁者执行，计数减到 0 才删除 Key。持锁者标识通常由实例 ID、线程 ID 和请求 ID 组成，业务时间不确定时需要看门狗续期。

其他实现方案：

| 方案         | 实现方式与特点                                             | 适用场景                           |
| ------------ | ---------------------------------------------------------- | ---------------------------------- |
| 数据库锁     | 唯一索引插入锁记录，或使用行锁控制互斥；实现简单、性能一般 | 低频、已有数据库事务的场景         |
| ZooKeeper 锁 | 临时顺序节点 + watcher，客户端会话断开后锁自动释放         | 对正确性和协调语义要求较高的场景   |
| etcd 锁      | 基于 lease 和 revision，也适合选主、配置一致性             | 云原生协调、强一致元数据场景       |
| Redis 锁     | 性能高、实现方便，但要处理过期、续期和主从切换风险         | 高频、短任务、性能优先的场景       |

选型时可概括为：性能优先考虑 Redis；一致性和协调语义优先考虑 ZooKeeper/etcd；简单低频任务可以使用数据库。

# 分布式事务

## 分布式事务有哪些解决方案？

| 方案             | 核心思路                                                   | 特点与适用场景                                 |
| ---------------- | ---------------------------------------------------------- | ---------------------------------------------- |
| 2PC              | 协调者先询问各参与者能否提交，再统一提交或回滚             | 实现相对直接，但同步阻塞、协调者依赖强、性能较差 |
| 3PC              | 在 2PC 基础上增加阶段，降低部分阻塞风险                    | 实现复杂，实际业务中不常直接使用               |
| TCC              | Try 预留资源，Confirm 确认提交，Cancel 释放资源            | 控制精细，但业务侵入性强                       |
| Saga             | 长事务拆成本地事务，每步提供补偿动作                       | 适合长流程，通常保证最终一致                   |
| 事务消息         | 本地事务成功后，通过可靠消息驱动下游                       | 适合订单、支付、库存等最终一致场景             |
| 本地消息表/Outbox | 业务数据和消息记录写入同一个本地事务，再由任务投递 MQ      | 失败可重试、可补偿，工程上较常用               |

## Seata 有哪些模式？TCC 有什么注意事项？

Seata 是分布式事务框架，常见模式有 AT、TCC、Saga、XA：

- AT：对业务侵入较低，记录 undo log；一阶段提交本地事务，二阶段根据全局事务结果提交或回滚。需要考虑全局锁、锁冲突和 SQL 兼容性。
- TCC：业务实现 Try、Confirm、Cancel，适合一致性要求高且业务边界可控的场景。
- Saga：把流程拆成多个本地事务，失败时执行补偿动作，适合长事务。
- XA：依赖数据库等资源管理器的两阶段提交，强一致性语义更明确，但性能和资源占用较高。

Seata 能降低开发成本，但不是银弹，仍要考虑性能、锁冲突、补偿、幂等、空回滚和悬挂。

TCC 的关键问题：

- 幂等：Confirm 或 Cancel 可能被重复调用，必须保证重复执行结果一致。
- 空回滚：Try 没执行成功却收到 Cancel，Cancel 要能识别并安全返回。
- 悬挂：Cancel 先于 Try 到达，后续 Try 不能再真正预留资源。
- 资源预留：Try 阶段冻结库存、余额或额度，Confirm 才真正扣减，Cancel 释放冻结。

## 最终一致性如何落地？

- 用本地事务保证核心数据和消息记录同时落库。
- 通过后台任务或 MQ 可靠投递事件，下游消费失败可以重试。
- 所有消费者必须幂等，避免重复消息导致重复扣减或重复发放。
- 定期执行对账任务，发现不一致数据后再走补偿流程修正。
- 用户侧可以先展示“处理中”，最终成功或失败后再通知。

# 分布式组件

## RPC 是什么？

RPC（Remote Procedure Call，远程过程调用）让调用远程服务看起来像调用本地方法，但不能把远程调用真的当成本地调用，因为网络延迟、超时和失败模型完全不同。

一个 RPC 框架通常包含：代理、序列化、网络传输、服务注册发现、负载均衡、超时、重试、熔断和链路追踪。

基本流程是：客户端代理封装请求并序列化，通过网络发送给服务端；服务端反序列化并执行目标方法，再把结果返回。设计时要重点处理网络不可靠、重复请求、服务实例变更和版本兼容。

## Dubbo 一次调用的关键链路是什么？

服务提供者启动后向注册中心注册地址和元数据；消费者订阅服务列表，生成代理对象并根据路由、负载均衡和集群容错策略选择提供者。请求经过序列化和网络传输到服务端，执行完成后再反序列化返回结果。

面试重点不在背组件名称，而在说明调用失败怎么办：设置超时，谨慎重试，按业务决定是否降级；重试可能造成重复写入，因此接口要幂等。服务发布还要考虑版本兼容、优雅停机和连接复用。

## Dubbo 的集群容错和负载均衡怎么选？

- Failover：失败后重试其他节点，适合幂等读请求；重试会放大流量，写请求要谨慎。
- Failfast：失败立即返回，适合对失败敏感、由上层处理的场景。
- Failsafe：失败只记录日志，适合非核心通知类调用。
- Failback：失败后后台重试，适合最终一致的非实时操作。

负载均衡常见有随机、加权随机、最少活跃调用和一致性哈希。选择时要结合节点性能、请求耗时、是否需要会话黏性和数据局部性。

## ZooKeeper 用来做什么？核心原理是什么？

ZooKeeper 是分布式协调服务，常用于注册中心、配置管理、分布式锁、选主和集群成员管理。

- 数据模型类似文件系统树，每个 znode 可以存数据。
- 临时节点会在客户端会话断开后自动删除，适合服务注册和锁释放。
- 顺序节点可以实现公平锁和选主；临时顺序节点常用于分布式锁。
- watcher 可以监听节点变化并触发客户端更新，但通知通常需要配合重新读取数据。
- 底层通过 Zab 协议保证集群内数据一致，写请求需要 leader 协调；读请求可以由 follower 提供，但要注意可能读到旧数据。

Watcher 是一次性通知机制，不是永久订阅；收到事件后客户端通常要重新读取数据并重新注册 Watcher。临时节点依赖会话存活，连接断开或会话过期后会被删除，因此适合服务注册和锁释放，但不能把单次网络断开简单等同于业务进程已经永久宕机。

## 注册中心为什么重要？

微服务实例会动态扩缩容、重启和迁移，调用方不能写死服务地址。注册中心保存服务名到实例列表的映射：

1. 服务启动时注册实例信息。
2. 服务停止或失联时，通过心跳、租约或健康检查将实例下线。
3. 客户端或网关发现可用实例，再进行负载均衡和调用。
4. 注册中心还要处理变更通知、故障剔除和多机房容灾。

# 流量治理与服务容错

## 限流算法和分布式限流怎么实现？

常见限流算法：

- 固定窗口：按固定时间窗口计数，实现简单，但窗口边界可能出现瞬时流量翻倍。
- 滑动窗口：把大窗口拆成多个小窗口，统计更平滑，精度比固定窗口高。
- 漏桶：请求进入桶后按固定速率流出，能平滑流量，但突发能力弱。
- 令牌桶：按固定速率生成令牌，请求拿到令牌才能通过，允许一定突发流量。

单机限流只能保护单个实例，多实例部署时总流量可能超过全局阈值。分布式限流常用 Redis 计数器、Redis Lua、令牌桶服务、网关限流或 Sentinel：

- Redis Lua 可以保证计数和过期设置原子执行，适合全局 QPS 控制。
- 网关限流适合控制入口流量；服务内限流适合保护具体下游资源。
- 限流后要配合降级返回、排队、重试建议和监控告警，不能只是简单丢请求。

面试取舍：接口保护常用令牌桶或滑动窗口；需要严格匀速处理可用漏桶；简单场景固定窗口也能使用。

## 限流、熔断、降级和隔离有什么区别？

| 机制 | 作用                                                         | 常见手段                         |
| ---- | ------------------------------------------------------------ | -------------------------------- |
| 限流 | 控制请求速率，超过阈值后拒绝或排队，保护服务不被打垮       | 令牌桶、漏桶、滑动窗口           |
| 熔断 | 下游持续失败时快速失败，防止线程和连接被拖垮               | 熔断器、失败率/慢调用比例阈值     |
| 降级 | 服务不可用或压力过高时返回兜底数据，保留核心流程           | 缓存值、默认值、提示语、关闭非核心功能 |
| 隔离 | 将资源分开，避免一个依赖拖垮整个服务                       | 线程池、连接池、舱壁隔离         |

高并发保护下游时，通常先限流控制进入系统或下游的请求速率，再通过熔断快速切断异常依赖，最后通过降级保证用户体验，同时用线程池、连接池隔离资源。

## 熔断器有哪些状态？Sentinel 和 Hystrix 有什么区别？

熔断器通常有三种状态：

- Closed：正常通行，持续统计错误率和响应时间。
- Open：错误率超过阈值，直接拒绝请求，不再调用下游，立即返回降级结果。
- Half-Open：Open 持续一段时间后放入少量探测请求；成功则回到 Closed，失败则继续 Open。

Hystrix（Netflix）已停止维护，典型特点是线程池隔离，资源开销较大，规则变更通常需要重启。Sentinel 提供流量控制、熔断降级、热点参数限流和系统自适应保护，并支持控制台推送规则；具体选型应结合项目版本和生态，不宜写成绝对的“生产唯一推荐”。

# 分布式一致性算法

## Paxos 和 Raft 有什么区别？

Paxos 是经典一致性算法，目标是在可能失败的分布式节点之间对某个值达成一致，但概念抽象、实现复杂。

Raft 解决的问题类似，但更强调可理解性，将一致性拆成 leader 选举、日志复制和安全性三个部分。Raft 节点有 follower、candidate、leader 三种角色；follower 超时未收到 leader 心跳时会转为 candidate 发起选举。

## Raft 的核心流程和应用场景是什么？

- Leader 选举：节点超时后发起投票，获得多数票成为 leader。
- 日志复制：客户端写请求先到 leader，leader 把日志复制给 follower。
- 提交规则：日志被多数节点持久化后提交，再应用到状态机。
- 安全性：任期 term、日志索引和日志任期共同保证旧 leader 或落后节点不会覆盖新数据。
- 心跳机制：leader 定期发送心跳维持权威，避免 follower 频繁发起选举。
- 故障转移：leader 故障后重新选主，新 leader 必须包含已提交日志，保证不会丢失已确认的数据。

使用 Raft 或类 Raft 思路的系统包括：etcd、Consul、TiKV、CockroachDB，以及 Nacos 的部分一致性场景。Raft 常用于强一致元数据、配置、选主和分布式 KV 系统。

# 分布式 ID

## 为什么需要分布式 ID？

单库自增 ID 在分库分表后无法保证全局唯一，多节点各自自增会产生冲突。分布式 ID 通常需要满足：全局唯一、趋势递增、高性能和高可用。趋势递增有利于 B+ 树索引写入。

## 雪花算法的原理是什么？有什么缺点？

经典 Snowflake 使用 64 位 Long 型：

```text
1 位符号位 + 41 位毫秒时间戳 + 10 位机器 ID + 12 位序列号
```

- 41 位时间戳通常相对于自定义起始时间计算，约支持 69 年。
- 10 位机器 ID 支持 1024 个节点。
- 12 位序列号每毫秒最多生成 4096 个 ID。
- 整体趋势递增，适合数据库主键索引。

主要缺点是强依赖机器时钟。NTP 时钟回拨可能导致 ID 重复，常见处理方式：

1. 检测到小幅回拨时等待时钟追回。
2. 对较大回拨直接报错或切换备用生成策略。
3. 结合协调服务管理机器 ID，并在启动时校验时间状态。
4. 使用号段模式或预生成缓冲区降低对实时系统时钟的依赖。

## 还有哪些分布式 ID 方案？如何选型？

| 方案                    | 特点                                             |
| ----------------------- | ------------------------------------------------ |
| UUID                    | 全局唯一但无序、字符串较长，索引性能一般         |
| 数据库号段模式          | 批量获取号段到内存，依赖数据库但性能较好         |
| Redis `INCR`            | 性能高，需要持久化和故障恢复设计                 |
| Snowflake               | 趋势递增、性能高，但需要处理机器 ID 和时钟回拨    |
| Leaf / UidGenerator     | 提供号段或 Snowflake 类方案，可减少自研成本       |

选型时，中大型系统可以考虑成熟的号段或 Snowflake 方案；小项目使用经过时钟回拨处理的 Snowflake 实现即可，不必为了分布式 ID 引入过多组件。

# 分布式链路追踪

## 什么是分布式链路追踪？

微服务架构下，一次用户请求会跨越多个服务（A → B → C → D），一旦超时或报错，很难定位具体服务。链路追踪通过唯一 TraceId 串联整个调用链，每个节点记录为一个 Span，包含：

- 服务名、接口名。
- 开始时间、耗时。
- 状态码、异常信息。
- 父子 Span 关系。

最终可以用调用瀑布图查看哪个服务耗时多、哪里发生错误。

## TraceId 如何在服务间传递？链路追踪框架如何选型？

常见传递方式：

- HTTP：放在请求头中，例如 `traceparent` 或自定义 TraceId 字段。
- RPC：通过框架隐式传参，例如 Dubbo Attachment、gRPC Metadata。
- 线程内：放入 ThreadLocal / MDC（Mapped Diagnostic Context），由日志框架打印 TraceId。
- 跨线程：需要显式传递上下文；线程池场景可以使用上下文传递工具，但要注意线程复用导致的上下文污染。

OpenTelemetry 负责定义跨框架的采集和传播标准，通常还需要配合具体后端：

| 方案            | 特点与适用场景                                     |
| --------------- | -------------------------------------------------- |
| SkyWalking      | Java Agent 字节码增强，支持多语言和可视化分析     |
| Zipkin          | 轻量简单，适合小规模项目                           |
| Jaeger          | CNCF 生态，适合云原生和 Kubernetes 环境           |
| OpenTelemetry   | 厂商中立的采集与传播标准，通常搭配上述后端使用     |

选型应结合项目规模、语言栈、部署环境和现有监控体系，不宜只按“国内项目”或“云原生项目”做绝对推荐。
