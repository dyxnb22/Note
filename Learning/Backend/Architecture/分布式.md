# 分布式

这篇不追求把所有名词背全，而是抓住一个核心问题：系统一旦从单机走向多节点，原来默认成立的很多前提就都不成立了。

## 1. 先建立总框架

单机系统里很多事情很简单：

- 数据就在一处
- 调用几乎没有网络成本
- 出问题通常只在本进程、本机器

分布式之后，复杂度主要来自 4 件事：

1. 网络会抖动、超时、丢包
2. 数据会分散在多个节点
3. 节点会宕机、扩缩容、迁移
4. 同一件业务可能跨多个服务和存储

所以分布式系统的核心，不是“多台机器一起跑”，而是：

- 如何在不可靠网络下协作
- 如何在多副本之间处理一致性
- 如何在节点故障时继续服务
- 如何把复杂链路拆成可恢复、可补偿的步骤

## 2. 面试里怎么回答分布式问题

遇到分布式题，优先按这个顺序组织：

1. 先说问题本质
2. 再说核心矛盾
3. 再给常见方案
4. 最后讲适用场景和代价

例如被问“为什么分布式锁难”，不要直接背命令，最好先说：

> 分布式锁难的地方不在于加一个锁，而在于节点宕机、锁过期、主从切换、重复释放这些异常情况下，怎样仍然保证不会误删别人的锁，也不会让系统长时间卡死。

## 3. CAP、BASE 与一致性取舍

### 介绍一下 CAP 理论

- CAP 指一致性 Consistency、可用性 Availability、分区容错性 Partition tolerance。
- 一致性：同一时刻，不同节点对同一份数据的观察结果一致。
- 可用性：每个请求都能在合理时间内得到响应，不要求一定是最新数据。
- 分区容错性：网络分区或节点通信失败时，系统仍要继续运行。
- 分布式系统里网络问题无法完全避免，所以 `P` 基本是必须接受的。
- 真正的取舍通常是：网络异常时，更偏向保一致性还是保可用性。

### CAP 和 BASE 是什么关系？

- BASE 是 Basically Available、Soft state、Eventually consistent，即基本可用、软状态、最终一致。
- 它不是推翻 CAP，而是在很多互联网业务里接受短暂不一致，换取更高可用和吞吐。
- 常见做法是：先把请求处理掉，再通过异步同步、重试、补偿、对账把数据收敛到一致状态。

### 强一致、弱一致、最终一致怎么理解？

- 强一致：写成功后，后续读一定能读到最新值。
- 弱一致：写成功后，读不保证立刻看到最新值。
- 最终一致：短时间内可能不一致，但经过同步后最终会一致。
- 核心写链路通常更看重一致性；查询、推荐、统计等链路更常接受最终一致。

## 4. 分布式锁

### 用 Redis 怎么实现分布式锁？

- 常见加锁方式是 `SET key value NX PX expire_ms`。
- `NX` 保证只有锁不存在时才能设置成功。
- `PX` 给锁设置过期时间，避免服务挂掉后永久不释放。
- `value` 不能只写固定值，必须是唯一持锁者标识，例如 UUID 或 `实例ID + 线程ID + 请求ID`。
- 解锁时要先校验 value 是否还是自己，再删除 key，这一步通常用 Lua 脚本保证原子性。

### Redis 分布式锁有哪些坑？

- 过期时间过短：业务未执行完锁先过期，其他线程进入临界区。
- 过期时间过长：持锁服务宕机后，别人要等很久才能继续。
- 非原子解锁：先 `get` 再 `delete` 可能误删别人的锁。
- 主从切换：主节点刚写入锁还没同步就宕机，新主节点可能根本没有这把锁。
- 长任务续期：如果业务时间不稳定，需要看门狗自动续期。

### 什么时候适合用 Redis 锁，什么时候不适合？

- Redis 锁适合高频、短临界区、可接受少量异常窗口的业务。
- 对强一致和协调语义要求特别高的场景，更适合 ZooKeeper 或 etcd。
- 如果业务频率不高、实现想简单，数据库唯一约束或行锁也可以作为方案。

### 还有哪些分布式锁实现方式？

- 数据库锁：基于唯一索引或行锁，实现简单但性能一般。
- ZooKeeper 锁：基于临时顺序节点，锁语义强，一致性好。
- etcd 锁：基于 lease 和 revision，适合服务协调、选主等场景。
- 选型话术：
  性能优先用 Redis；
  一致性和协调优先用 ZooKeeper/etcd；
  简单低频场景可以用数据库。

## 5. 分布式事务与最终一致

### 分布式事务为什么难？

- 单机数据库事务靠本地日志和锁就能解决。
- 分布式事务涉及多个服务、多个数据库、多个节点，还要面对网络异常和部分成功。
- 真正困难的点不是“能不能提交”，而是“某一步失败后如何恢复，以及恢复期间业务是否还能继续”。

### 常见分布式事务方案有哪些？

- 2PC：强一致，但同步阻塞明显，性能和可用性都一般。
- 3PC：尝试降低阻塞，但复杂度更高，业务里较少直接用。
- TCC：Try 预留、Confirm 提交、Cancel 释放，控制力强，但业务侵入大。
- Saga：一串本地事务加补偿动作，适合长流程，偏最终一致。
- 事务消息：本地事务成功后可靠投递消息，下游异步处理。
- Outbox / 本地消息表：业务数据和消息记录写在一个本地事务里，再异步投递。

### TCC 要注意哪些问题？

- 幂等：Confirm / Cancel 可能重复调用。
- 空回滚：Try 没成功，但 Cancel 先到了。
- 悬挂：Cancel 先执行后，迟到的 Try 不能再真正预留资源。
- 资源冻结：Try 阶段通常是预留资源，不是立即最终扣减。

### 最终一致性怎么落地？

- 先用本地事务保证核心业务数据落库。
- 再可靠记录要发出的消息或事件。
- 由 MQ 或后台任务驱动下游异步处理。
- 消费端必须幂等。
- 再用补偿和对账去修正极端失败情况。

## 6. RPC、注册中心与服务发现

### RPC 的概念是什么？

- RPC 是 Remote Procedure Call，远程过程调用。
- 目标是让调用远程服务像调用本地方法一样方便，但本质仍然是网络调用。
- 一个 RPC 框架通常会包含：代理、序列化、网络传输、注册发现、负载均衡、超时、重试、熔断、链路追踪。
- 面试里要强调：不能把远程调用真的当成本地调用，因为延迟、超时和失败模型完全不同。

### 注册中心为什么重要？

- 微服务实例会动态扩缩容、重启、迁移，调用方不能把 IP 和端口写死。
- 注册中心维护“服务名 -> 实例列表”的映射。
- 服务启动时注册，失联时摘除，调用方通过注册中心拿到可用实例再进行负载均衡。

### ZooKeeper 常见用途是什么？

- 注册中心
- 配置管理
- 分布式锁
- 选主
- 集群成员管理
- 它的关键语义在于临时节点、顺序节点和 watcher。

## 7. ID 生成与顺序问题

### 分布式系统里为什么不能只靠数据库自增 ID？

- 单库自增简单，但一旦多库分片、跨机房部署或写流量过大，就容易变成瓶颈。
- 还可能带来单点问题和扩容困难。

### 常见分布式 ID 方案有哪些？

- UUID：简单、全局唯一，但太长且无序，对索引不友好。
- 数据库号段模式：按批次申请 ID 段，性能比单条自增好。
- 雪花算法：按时间戳、机器号、序列号生成，趋势递增，适合高并发场景。
- 选型时要考虑：唯一性、趋势有序、生成性能、跨机房部署、时钟回拨风险。

## 8. 消息队列在分布式里扮演什么角色

### 为什么很多分布式系统都要引入 MQ？

- 解耦：上游不必同步等待下游完成。
- 削峰：把瞬时高并发摊平。
- 异步：非核心流程不用阻塞主链路。
- 广播事件：订单创建后，库存、积分、通知等系统可以异步订阅。

### 用了 MQ 就一定更好吗？

- 不一定。MQ 引入后也会带来消息丢失、重复消费、顺序性、堆积、重试风暴等问题。
- 所以设计时要同时考虑：
  可靠投递、
  消费幂等、
  死信处理、
  堆积监控、
  补偿和对账。

## 9. 一页答题模板

如果被问“你怎么理解分布式系统”，可以这样回答：

> 分布式系统的难点不在于把服务拆开，而在于拆开后原本单机默认成立的很多前提都失效了，比如调用不再可靠、数据不再天然一致、节点会宕机、同一笔业务会跨多个服务。所以真正要解决的是一致性、可用性、故障恢复和链路治理问题。工程上常见的手段包括缓存和 MQ 做削峰解耦、分布式锁控制临界区、TCC 或事务消息处理跨服务一致性、注册中心和 RPC 做服务发现与调用，再配合幂等、重试、补偿和对账把系统收回来。

## 10. 一页总结

- CAP 讲的是网络分区下的一致性与可用性取舍
- BASE 是很多互联网业务的现实工程方案
- 分布式锁重点不在加锁，而在异常场景下不误删、不死锁
- 分布式事务核心是恢复、补偿和最终收敛
- RPC 只是让调用方便，不会让网络问题消失
- 注册中心解决实例动态变化
- MQ 解决解耦和削峰，也会带来重复消费和堆积治理问题
- 幂等、重试、补偿、对账是分布式系统的基本功
