# 可靠性与一致性

可靠性设计的核心是：重复、乱序、超时、部分成功和依赖不可用时，系统仍能得到可解释、可恢复的结果。

## 先定义事实和状态

- 订单创建不等于支付成功。
- 优惠试算不等于优惠锁定。
- 库存预占不等于最终扣减。
- 消息发送不等于业务副作用完成。
- 抽奖结果不等于奖品已经发放。

把这些事实分开，才能为每一步设计状态、唯一号、回执和补偿。

## 幂等

幂等不是“只执行一次”，而是同一业务操作重复到达时结果可预测。常见手段：

- 业务幂等键和数据库唯一约束。
- 状态条件更新：`where status = expected_status`。
- 消费记录、回调记录和任务记录。
- Redis/Lua 原子判断与更新。
- 重试前读取最新状态，避免盲目重复副作用。

支付回调、消息消费、定时任务、库存释放、发奖和补偿都默认可能重复。

## 远程调用

每个远程调用至少明确：

1. 超时：连接、读取和整体预算分别控制。
2. 重试：只重试可恢复错误，并限制次数和退避。
3. 降级：明确返回什么可接受结果，不能静默吞错。
4. 熔断/隔离：防止下游故障拖垮调用方。
5. 审计：记录业务号、目标、耗时、结果和重试次数。

重试会放大副作用；没有幂等和预算的重试不是可靠性设计。

## 消息与补偿

把事务内事实和外部通知拆开：先记录可恢复的任务/事件，再通过 HTTP、MQ 或定时扫描完成通知、重试、死信和人工处理。任务状态、租约和重放规则见[任务、消息与补偿](./05_任务、消息与补偿.md)。

## 一致性选择

| 场景 | 优先方案 |
|---|---|
| 同库强约束 | 本地事务和条件更新 |
| 跨服务最终一致 | 事件/任务 + 幂等消费 + 补偿 |
| 支付/库存/权益协作 | 状态机 + 回调幂等 + 对账 |
| 高并发资源扣减 | Redis 原子操作 + 异步落库 + 差异修复 |
| 长流程人工可介入 | 工作流状态 + checkpoint + 人工处理 |

## 观测与恢复

日志、指标和 Trace 要能用业务号串起一次链路；至少监控失败率、重试量、队列堆积、死信、库存差异、对账差异和补偿耗时。所有自动补偿都应有停止条件，避免“重试风暴”。
