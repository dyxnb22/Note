# 部署与上线
部署的目标不是“能跑一次”，而是让服务在服务器上稳定、可配置、可重启、可观察、可更新。

## 1. 部署前先明确

上线前先回答：

- 这个服务监听哪个端口？
- 配置从哪里来？
- 数据存在哪里？
- 日志怎么看？
- 崩了怎么重启？
- 如何健康检查？
- 如何更新版本？
- API Key 是否安全？

## 2. Linux 基础

常用命令：

```bash
pwd                         # 当前目录
ls -la                      # 查看文件，包括隐藏文件
cd /opt/app                 # 进入项目目录
cat .env                    # 查看环境变量文件，注意不要随便发给别人
ps aux | grep uvicorn       # 查看 uvicorn 进程
ss -lntp                    # 查看端口监听情况
tail -f app.log             # 实时查看日志
```

权限：

```bash
chmod +x scripts/start.sh       # 给脚本执行权限
chown -R appuser:appuser /opt/app # 修改目录所属用户
```

不要长期用 root 直接跑业务服务。

## 3. 环境变量

`.env` 放真实配置，`.env.example` 放模板。

```plain text
OPENAI_API_KEY=...
MODEL_NAME=gpt-5
DATABASE_URL=postgresql://user:pass@db:5432/app
REDIS_URL=redis://redis:6379/0
```

原则：

- `.env` 不提交 Git。
- 线上和本地用不同配置。
- secrets 不写进 Dockerfile。
- 日志不打印 secrets。

## 4. Uvicorn / Gunicorn

开发：

```bash
uvicorn app.main:app --reload
```

生产基础版：

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

如果是更正式的 Python Web 服务，通常会用 Gunicorn 管理 worker，再由 Uvicorn 作为 worker class：

```bash
gunicorn app.main:app -k uvicorn.workers.UvicornWorker -w 4 -b 0.0.0.0:8000
```

理解重点：

- `uvicorn` 适合开发和轻量部署
- `gunicorn + uvicorn worker` 更适合传统 Linux 服务器部署
- worker 数量不是越多越好，要结合 CPU、内存和接口类型来定

## 5. systemd 托管服务

生产环境不要靠手敲命令长期跑服务，最好交给 `systemd` 托管。

示例：

```ini
[Unit]
Description=myapp
After=network.target

[Service]
User=appuser
WorkingDirectory=/opt/myapp
EnvironmentFile=/opt/myapp/.env
ExecStart=/opt/myapp/.venv/bin/gunicorn app.main:app -k uvicorn.workers.UvicornWorker -w 4 -b 0.0.0.0:8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

常用命令：

```bash
sudo systemctl daemon-reload
sudo systemctl enable myapp
sudo systemctl start myapp
sudo systemctl restart myapp
sudo systemctl status myapp
journalctl -u myapp -f
```

为什么重要：

- 崩了可以自动拉起
- 开机可以自启
- 日志可以统一走 `journalctl`
- 启停流程比手工后台进程更可控

## 6. 反向代理与端口暴露

线上通常不会把应用直接暴露给公网，而是放在 Nginx 或其他反向代理后面。

常见职责：

- 对外暴露 80 / 443
- 做 HTTPS 终止
- 转发请求到后端应用端口
- 处理静态资源
- 统一访问日志

典型链路：

```text
user -> nginx:443 -> app:8000
```

如果应用只监听 `127.0.0.1:8000` 或内网地址，安全性通常会更好。

## 7. 健康检查

至少要有一个轻量健康检查接口，例如：

- `/health`
- `/ping`
- `/ready`

通常会区分两类：

- 存活检查：进程是否还活着
- 就绪检查：服务是否真的能处理请求，例如数据库连接是否正常

不要把健康检查写得太重，否则高频探测本身会增加负担。

## 8. 日志与可观察性

上线后至少回答三个问题：

- 请求有没有进来
- 服务有没有报错
- 出问题时能不能快速定位

最低配建议：

- 应用日志输出到 stdout 或文件
- 反向代理有访问日志
- 错误日志和访问日志分开
- 日志里带时间、级别、请求 ID

常见查看命令：

```bash
tail -f /var/log/nginx/access.log
tail -f /var/log/nginx/error.log
journalctl -u myapp -n 100
journalctl -u myapp -f
```

## 9. Docker 部署思路

如果用 Docker，上线流程通常变成：

1. 写好 Dockerfile
2. 本地构建镜像
3. 推送到镜像仓库
4. 服务器拉取新镜像
5. 重启容器或编排服务

一个最小例子：

```bash
docker build -t myapp:1.0 .
docker run -d --name myapp \
  --restart always \
  --env-file .env \
  -p 8000:8000 \
  myapp:1.0
```

关键点：

- 配置通过 `--env-file` 或平台 secret 注入
- 数据目录需要 volume 持久化
- 不要把数据库数据写在容器可写层里

## 10. 发布流程建议

一个比较稳的人工上线流程可以是：

1. 在测试环境先验证版本
2. 备份关键配置
3. 记录当前运行版本
4. 发布新版本
5. 跑 smoke test
6. 观察日志、错误率、CPU、内存
7. 确认后再扩大流量或通知完成

如果涉及数据库迁移，再多问一句：

- 这次迁移是否向后兼容
- 回滚时数据结构是否还能读

## 11. 回滚意识

上线方案如果没有回滚路径，通常就不算完整。

常见回滚方式：

- 切回上一个版本的二进制 / jar / 镜像 tag
- 切回上一套蓝绿环境
- 回退配置

最危险的不是代码回滚，而是数据库 schema 已经不可逆变化。

## 12. 常见故障排查顺序

如果服务访问失败，建议按这个顺序看：

1. 进程在不在：`ps -ef | grep myapp`
2. 端口有没有监听：`ss -lntp | grep 8000`
3. 本机能否访问：`curl http://127.0.0.1:8000/health`
4. 反向代理是否正常：`systemctl status nginx`
5. 应用日志和代理日志有没有报错
6. 防火墙、安全组、域名解析是否正确

这样排查比一上来就怀疑代码更高效。

## 13. 最小上线清单

- 服务启动命令明确
- 端口明确
- 配置来源明确
- secrets 与代码分离
- 日志可查看
- 健康检查可访问
- 重启方式明确
- 回滚方式明确
- 数据持久化路径明确
- 负责人和变更记录明确

## 14. 面试表达

如果被问“你理解的部署与上线是什么”，可以回答：

> 我理解部署不是把服务启动起来就结束，而是把运行方式标准化。一个可上线的服务至少要解决配置管理、进程托管、日志查看、健康检查、重启和回滚这些问题。实际落地时，我会优先保证服务由 systemd 或容器平台托管，配置和 secrets 与代码分离，并且发布后有可执行的验证和回滚路径。
