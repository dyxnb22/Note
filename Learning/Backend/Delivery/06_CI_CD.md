# CI_CD

CI/CD 解决的不是“炫自动化”，而是让代码从提交到上线这条链路更稳定、更可重复、更少依赖人工记忆。

## 1. CI 和 CD 分别是什么

- `CI`：Continuous Integration，持续集成
  目标是让代码一提交就自动跑构建、测试、静态检查，尽早发现问题
- `CD`：Continuous Delivery / Deployment，持续交付 / 持续部署
  目标是让通过校验的版本能够稳定地交付到测试或生产环境

一句话理解：

- `CI` 关注“代码合进来后是否仍然健康”
- `CD` 关注“健康的版本能否稳定发布出去”

## 2. 一条典型流水线长什么样

```text
push / pull request
-> install dependencies
-> lint
-> unit test
-> build
-> package image / artifact
-> deploy to staging
-> smoke test
-> manual approval or auto deploy
-> production
```

## 3. 为什么需要 CI/CD

- 减少“我本地没问题”的不确定性
- 降低手工上线漏步骤的概率
- 让构建和发布过程可审计、可复现
- 把测试和发布前移，问题更早暴露
- 团队协作时减少口头约定

## 4. CI 常见内容

### 4.1 代码质量检查

- `lint`：代码风格、明显错误
- `format check`：格式统一
- `type check`：静态类型检查
- `security scan`：依赖漏洞、密钥泄露检查

### 4.2 测试

- 单元测试：验证局部逻辑
- 集成测试：验证模块协作
- API / E2E 测试：验证完整链路

### 4.3 构建产物

- Java：`jar` / `war`
- Node：构建后的前端静态资源
- Python：wheel、Docker image

重点不是“是否构建成功”，而是“生成的产物是不是之后部署时真正会用的那份”。

## 5. CD 常见内容

### 5.1 部署策略

- 直接覆盖部署：简单，但风险高
- 滚动发布：逐台替换，影响面较小
- 蓝绿发布：新旧两套环境切换，回滚快
- 金丝雀发布：先给少量流量，观察后再放大

### 5.2 发布前后检查

- 发布前确认数据库变更是否兼容
- 发布前确认环境变量和 secrets 已配置
- 发布后跑健康检查和 smoke test
- 观察错误率、延迟、CPU、内存、日志

## 6. 常见平台

- GitHub Actions
- GitLab CI/CD
- Jenkins
- Argo CD

选型不是重点，重点是把流程拆清楚：

- 谁触发
- 跑哪些检查
- 产物存哪里
- 谁批准上线
- 失败怎么回滚

## 7. 一个最小可用流程

对中小项目，最小可用版本通常是：

1. PR 自动跑 `lint + unit test`
2. 合并到主干后自动构建 Docker 镜像
3. 推送镜像到仓库
4. 测试环境自动部署
5. 生产环境人工审批后部署

这个流程已经能覆盖大部分“别手工漏步骤”的问题。

## 8. 常见坑

- 流水线和本地环境不一致
- 测试通过，但部署脚本用的是另一份配置
- 数据库变更不可回滚
- 只做自动部署，不做自动验证
- 流水线里打印了 secrets
- 构建太慢，团队开始绕过 CI

## 9. 面试表达

如果被问“你理解的 CI/CD 是什么”，可以回答：

> CI/CD 的核心不是工具，而是把代码检查、测试、构建、发布标准化。CI 负责尽早发现问题，CD 负责把通过验证的版本稳定地交付出去。实际落地时我会重点关注三件事：流水线和真实部署流程是否一致、发布后是否有自动验证、失败后是否能快速回滚。

## 10. 自查清单

- PR 是否自动跑测试
- 主干是否只接收通过校验的代码
- 是否有统一构建产物
- 发布流程是否文档化
- 是否有回滚方案
- 发布后是否有健康检查
- secrets 是否和代码分离
