# AI DevOps 与交付

AI 工程的交付闭环不是生成代码，而是从环境、构建、部署、运行、观测到故障修复都可验证。

## 交付链路

```text
需求/代码 → 依赖与配置 → 构建/测试 → 镜像/制品
        → 部署 → 健康检查 → 指标/日志/Trace
        → 压测/故障定位 → 修复/回滚
```

Docker 负责环境和服务编排，Linux/SSH 负责运行边界，CI/CD 负责重复交付，观测系统负责确认结果。Portainer 和 AI Shell 只是操作入口，不能替代权限、审计和回滚设计。

## AI 操作服务器的边界

- 高风险命令要有权限、预览/确认、超时、取消、审计和最小授权。
- 模型输出不是执行权限；命令必须经过 schema/策略校验和服务端执行器。
- 数据库、文件系统、SSH 凭证和生产环境默认视为受保护资源。
- 自动化失败要返回明确错误，不要静默吞掉或无限重试。

## 压测闭环

定义目标和基线 → 设计真实请求比例 → 产生负载 → 同时看吞吐、P95/P99、错误率、CPU、内存、连接池、数据库和队列 → 定位瓶颈 → 修改 → 回归。

压测结果只是证据，不是架构承诺；数据规模、依赖状态、缓存命中率和机器规格都要记录。

## 本地模型

Ollama 适合本地验证和低成本开发。模型适配应通过统一接口隔离，评估上下文长度、推理速度、显存/内存、输出稳定性和安全边界，不把本地验证结果直接等同于生产能力。
