# Java 数据访问

本页连接 [SQL](../Data/SQL.md)、[MySQL](../Data/MySQL.md) 与 Java 应用，重点是 JDBC、连接池、MyBatis、JPA 和事务边界。数据库本身的索引、锁和日志不在这里重复维护。

## 分层边界

```text
Controller
→ Application Service：用例与事务边界
→ Repository / Mapper：持久化合同
→ JDBC / ORM
→ Connection Pool
→ Database
```

Repository 表达领域需要的读写能力，不应只是数据库表的机械镜像；复杂报表和查询可以使用专门 Query Service，避免把聚合查询硬塞进领域实体。

## JDBC

JDBC 的基本流程是获取连接、创建参数化语句、执行、映射结果并释放资源。实际项目应使用连接池和 `try-with-resources`，不要为每个请求建立物理连接。

必须使用参数绑定：

```java
String sql = "SELECT id, amount FROM orders WHERE user_id = ? AND status = ?";
try (PreparedStatement statement = connection.prepareStatement(sql)) {
    statement.setLong(1, userId);
    statement.setString(2, status);
    try (ResultSet rs = statement.executeQuery()) {
        // 显式映射需要的列
    }
}
```

参数绑定解决值注入问题，但动态表名、列名和排序方向不能作为普通参数绑定，必须使用白名单映射。

## 连接池

连接池容量不是越大越好。它受到数据库最大连接数、服务实例数、查询耗时和峰值并发共同约束：

```text
数据库连接预算
÷ 服务实例数
≥ 单实例连接池上限 + 运维余量
```

需要同时配置获取连接超时、连接存活检测、最大生命周期和泄漏检测。连接池耗尽通常是慢 SQL、长事务、连接未释放或下游拥塞的结果，而不只是池太小。

## MyBatis

MyBatis 让开发者保留 SQL 控制权，并负责参数映射和结果映射，适合查询复杂、需要精细优化的系统。

- `#{}` 使用预编译参数，普通值必须优先使用它。
- `${}` 是文本替换，只能用于经过白名单映射的结构元素。
- 动态 SQL 要覆盖空集合、可选条件和批量大小边界。
- 一级缓存属于会话范围；二级缓存会增加一致性和失效复杂度，不应默认开启。
- 插件会影响执行器、参数、结果或 SQL，必须限制职责并测试顺序。

Mapper 方法应体现业务查询意图，例如 `findPayableOrders`，而不是暴露无限组合的通用条件对象。

## JPA 与 ORM

JPA/ORM 适合实体关系清晰、以聚合读写为主的场景，优势是对象映射、脏检查和统一持久化模型。需要注意：

- N+1 查询。
- 懒加载越过事务边界。
- 级联操作扩大写入范围。
- 批量更新绕过当前持久化上下文。
- 对象模型与高性能报表查询之间的张力。

复杂查询可以使用投影、原生 SQL 或独立查询模型，不必强迫所有数据访问都经过实体图。

## 事务边界

事务通常放在 Application Service 的完整业务用例上。事务内应避免长时间远程调用、文件处理和用户等待。

要明确：

- 事务隔离级别及并发异常。
- 哪些异常触发回滚。
- 多数据源和读写分离时连接来自哪里。
- 消息发布与数据库提交如何保持一致。
- 重试发生在事务内还是事务外。

唯一约束、条件更新和锁用于保护数据库事实；应用层的“先查再写”不能代替并发控制。

## 数据库迁移

Schema 变更应通过版本化迁移工具管理，并满足向前、向后兼容的发布窗口。常见顺序是：

1. 先增加兼容的新列或新表。
2. 发布同时兼容新旧结构的代码。
3. 回填和校验历史数据。
4. 切换读写路径。
5. 观察稳定后再删除旧结构。

不要让 ORM 在生产环境中自动推断并修改 Schema。

## 测试清单

- Repository 是否在真实数据库或容器化数据库上做集成测试？
- 唯一约束、外键、事务回滚和并发更新是否验证？
- SQL 是否覆盖空集、NULL、重复值和时间边界？
- 慢查询、连接池耗尽和死锁是否有可观测数据？
- 数据库迁移能否从旧版本实际升级并回滚应用？
