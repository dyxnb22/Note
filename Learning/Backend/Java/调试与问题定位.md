# 调试与问题定位

排障不是猜代码，而是从现象、链路、数据和证据逐步缩小范围。

## 五步流程

1. **复现**：固定输入、环境、版本、时间窗口和最小数据。
2. **定位边界**：判断是入口协议、应用编排、领域规则、数据层、外部依赖还是部署环境。
3. **建立链路**：用请求 ID/业务号串起日志、Trace、SQL、MQ、任务和回调。
4. **验证假设**：用断点、日志、指标、SQL、线程/堆分析或最小测试证明原因。
5. **修复与回归**：修复根因，补测试和观测，验证正常、重复、超时和失败路径。

## 常见定位顺序

```text
现象/错误率 → 时间与版本 → 日志/Trace → 依赖耗时
→ 数据/状态 → 并发与资源 → 最小修复 → 回归验证
```

不要一开始就改代码；先确认是否是配置、数据、依赖不可用、测试未执行或环境差异。

## 业务异常检查

- 状态迁移是否合法，是否被重复回调或超时任务竞争？
- 幂等键、唯一约束、缓存和消息消费记录是否存在？
- 远程调用是否超时、重试放大或返回了部分成功？
- 任务是否卡在处理中，是否有租约、重试和死信？
- 修复后是否补了最小复现、失败注入和监控告警？

## 证据边界

日志、测试和构建结果要区分：构建成功只证明编译/打包，测试命令还要核对实际 `Tests run`，外部服务联调要给出真实请求和响应证据。
