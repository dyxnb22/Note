# Spring Cloud

Spring Cloud 是在 Spring Boot 应用之上组织分布式组件的一组集成方案。它不自动解决微服务复杂度；服务拆分、数据边界和可靠性仍应先按 [Architecture](../Architecture/README.md) 的原则设计。

## 什么时候需要

只有在服务确实需要独立发布、扩缩容、故障隔离或团队自治时，再引入注册发现、配置中心、网关和服务间客户端。单体内部的问题不应通过“先拆微服务”解决。

## 组件地图

| 能力 | 解决的问题 | 关键风险 |
| --- | --- | --- |
| 注册发现 | 实例如何注册和定位 | 陈旧实例、健康判断、网络分区 |
| 配置中心 | 配置如何集中发布 | 权限、版本、动态刷新一致性 |
| OpenFeign/HTTP Client | 服务间合同化调用 | 超时、重试放大、连接池 |
| LoadBalancer | 多实例选择 | 不均衡、局部热点、粘性状态 |
| Gateway | 统一入口和横切治理 | 单点、规则复杂、流式请求 |
| Circuit Breaker | 故障时快速失败 | 阈值错误、恢复抖动、级联降级 |
| Trace/Metrics | 跨服务定位问题 | 上下文丢失、采样和成本 |

## 注册与配置

注册中心保存服务实例及健康状态；配置中心保存运行配置，两者职责不同。Nacos 可以同时提供两种能力，但权限、命名空间、变更审计和故障影响仍应分别设计。

客户端不能无限信任注册表：应配合连接超时、请求超时、熔断和实例摘除。配置动态刷新要说明哪些值能在线修改，重要配置要版本化并支持回滚。

## 服务间调用

OpenFeign 提供声明式 HTTP 客户端，减少样板代码；它不会替你决定正确的超时、重试、幂等和错误合同。

调用链至少要配置：

- 连接超时和请求超时。
- 有明确上限的重试，并限制在幂等操作。
- 连接池和并发上限。
- TraceId、用户/租户身份的安全传递。
- 结构化错误码，不依赖错误文本判断。
- 降级时可接受的业务语义。

## Gateway

网关适合做路由、认证入口、限流、协议适配和通用观测，不适合承载核心业务编排。规则应可测试、可灰度、可回滚；上传下载、SSE 和 WebSocket 等流式场景要单独验证超时和背压。

## 负载均衡与容错

轮询、随机、最少连接和一致性 Hash 各有适用前提。用户黏性会形成局部状态和热点，优先把会话状态外置；确需黏性时要设计实例故障后的恢复。

限流保护容量，熔断隔离持续故障，降级提供替代结果，重试应对短暂错误。四者不能互相替代，错误组合还会放大流量。

## 验收清单

- 能说明为什么拆服务，以及不拆的替代方案。
- 能画出一次调用经过网关、注册发现、客户端、下游和观测系统的链路。
- 能说明配置中心或注册中心不可用时现有请求是否继续工作。
- 能计算超时和重试如何影响整条调用链的延迟与流量。
- 能演练实例摘除、配置回滚、下游熔断和网关降级。
