# Java 设计模式

设计模式不是固定代码模板，而是针对反复出现的变化点，提前安排职责、依赖和扩展边界。基本设计原则见 [Java](01_Java.md)；Spring 中的模式落点见 [Spring](07_Spring.md)。

## 什么时候值得引入设计模式？

| 代码信号 | 优先考虑 | 解决的问题 |
| --- | --- | --- |
| 按类型堆叠 `if/else`，算法经常新增 | Strategy + Factory/Registry | 把选择逻辑和算法实现分开 |
| 构造参数很多、创建步骤固定 | Builder | 避免构造器爆炸，集中组装过程 |
| 外部系统接口或消息字段不一致 | Adapter | 在边界处统一内部模型 |
| 两个维度都可能独立扩展 | Bridge | 避免子类数量出现笛卡尔积 |
| 需要在原对象上叠加多个能力 | Decorator | 组合增强，避免无休止地派生子类 |
| 请求要经过多个处理阶段 | Chain of Responsibility | 拆分流程，明确处理顺序和终止条件 |
| 对象之间出现大量订阅关系 | Observer / Event | 解耦发布者和订阅者 |
| 状态变化导致行为整体变化 | State | 用显式状态机代替状态分支堆积 |
| 创建或使用一个复杂子系统很麻烦 | Facade | 提供稳定的高层入口 |

判断标准是“未来的变化是否能局部完成”。如果需求稳定、变化点只有一处，直接实现通常更清晰；模式本身也会增加类数量和理解成本。

## 22 个模式速记

### 创建型

| 模式 | 核心意图 | 典型场景 |
| --- | --- | --- |
| 工厂方法 | 把一个产品的创建延迟给子类或具体工厂 | 多种商品发放、不同实现的创建 |
| 抽象工厂 | 创建一组相互匹配的产品 | 多个 Redis 集群或多套基础设施 |
| 建造者 | 分离复杂对象的组装过程与最终表示 | 套餐、配置对象、不可变对象 |
| 原型 | 复制已有对象，避免重复初始化 | 复杂题目、模板对象批量复制 |
| 单例 | 控制实例数量并提供统一访问点 | 进程级共享资源；Spring 默认 singleton |

### 结构型

| 模式 | 核心意图 | 典型场景 |
| --- | --- | --- |
| 适配器 | 转换已有接口或数据结构 | MQ 消息、第三方 SDK、旧接口兼容 |
| 桥接 | 把抽象和实现拆成两个可独立变化的维度 | 支付渠道 × 支付方式 |
| 组合 | 用树结构统一处理单个对象和对象集合 | 菜单、组织架构、决策树 |
| 装饰器 | 在不改原类的前提下动态叠加职责 | SSO、日志、缓存、权限增强 |
| 外观 | 为复杂子系统提供简单统一入口 | 门面服务、启动或编排多个组件 |
| 享元 | 共享可复用的细粒度对象，外部传入变化状态 | 大量相似对象、库存/活动查询 |
| 代理 | 控制对目标对象的访问或延迟真实调用 | AOP、事务、远程代理、懒加载 |

### 行为型

| 模式 | 核心意图 | 典型场景 |
| --- | --- | --- |
| 责任链 | 多个处理者按链路依次处理请求 | 审批流、过滤器、校验管道 |
| 命令 | 把请求封装成对象 | 排队执行、重试、审计、撤销 |
| 迭代器 | 隐藏集合内部结构，统一遍历方式 | `Iterable`/`Iterator`、树遍历 |
| 中介者 | 用中介对象集中管理对象间交互 | ORM 协调、复杂组件协作 |
| 备忘录 | 在不破坏封装的前提下保存状态快照 | 配置回滚、撤销、版本恢复 |
| 观察者 | 一个对象变化时通知多个依赖者 | 领域事件、消息订阅、Spring Event |
| 状态 | 对象根据内部状态切换行为 | 订单、审核、营销状态机 |
| 策略 | 封装一组可互换算法 | 折扣、路由、计费、风控规则 |
| 模板方法 | 固定算法骨架，把步骤交给子类或钩子 | `JdbcTemplate`、爬虫流程 |
| 访问者 | 把对稳定对象结构的操作抽离出来 | 对固定元素结构增加多种分析操作 |

## 最容易混淆的边界

### 代理模式和适配器模式

| 对比项 | 代理模式 | 适配器模式 |
| --- | --- | --- |
| 目的 | 控制访问或增加额外行为 | 转换接口，使不兼容的类协同工作 |
| 结构 | 抽象主题、真实主题、代理 | 目标接口、适配器、被适配者 |
| 常见场景 | 日志、权限、事务、缓存 | 兼容第三方 SDK、旧接口和新接口 |

反射提供运行时获取类信息和调用方法的能力，动态代理利用这些能力在不修改原对象的情况下增强方法行为。JDK 动态代理基于接口和 `InvocationHandler`；CGLIB 通过生成目标类子类实现增强。Spring AOP 的具体限制见 [Spring 核心](Spring核心.md)。

| 对比 | 关键区别 |
| --- | --- |
| 工厂方法 vs 抽象工厂 | 工厂方法通常解决一种产品的创建；抽象工厂保证一组相关产品彼此匹配 |
| 适配器 vs 外观 | 适配器解决接口不兼容；外观简化多个子系统的调用流程 |
| 装饰器 vs 代理 | 装饰器重点是叠加同一接口的业务能力；代理重点是控制访问、生命周期或调用位置 |
| 策略 vs 状态 | 策略由调用方选择算法；状态由对象内部状态决定下一步行为并发生迁移 |
| 策略 vs 模板方法 | 策略偏组合、运行时替换整套算法；模板方法偏继承、固定骨架只替换部分步骤 |
| 桥接 vs 适配器 | 桥接在设计之初拆分两个变化维度；适配器通常用于兼容已经存在的类或接口 |
| 组合 vs 责任链 | 组合表达树形“部分—整体”；责任链表达请求沿处理节点传递 |

## 几个高频模式的落地要点

### 工厂方法与策略注册表

当业务按商品类型、支付方式或优惠类型选择实现时，不要让 Controller 直接承担分支：

```java
interface DiscountStrategy {
    Money calculate(Order order);
}

Map<DiscountType, DiscountStrategy> strategies;

DiscountStrategy strategy = strategies.get(order.getDiscountType());
```

工厂负责“拿到哪一个实现”，策略负责“如何计算”。实现数量较少且不会继续增长时，简单 `switch` 也可以，不必为了模式而模式。

### 建造者

适用于“必填参数少、可选参数多，且构造过程需要约束”的对象。建造者应在 `build()` 阶段校验不变量，产出的对象尽量不可变。

如果对象只有一两个可选字段，静态工厂或普通构造器更直接；不要为简单 DTO 建立复杂 Builder 层次。

### 桥接

当两个维度都在增长时，继承容易产生大量组合类。例如支付渠道和支付方式都可能新增：

```text
Pay(抽象) ──持有──> PayMode(实现)
  │                    │
微信支付 / 支付宝       刷脸 / 指纹 / 密码
```

核心不是“多写一个接口”，而是确认这两个维度确实需要独立变化；没有第二个变化轴时，桥接反而会增加复杂度。

### 适配器、装饰器和代理

三者都可能通过包装对象实现，但变化目标不同：

- 适配器：把外部接口转换成内部接口，最好放在系统边界，避免污染核心领域模型。
- 装饰器：保持原接口不变，按组合顺序叠加日志、缓存、权限等能力。
- 代理：对调用进行控制，例如延迟加载、权限检查、远程调用、事务拦截。

Spring AOP 的 JDK/CGLIB 动态代理、声明式事务属于代理思想；具体代理限制和事务失效场景见 [Spring](07_Spring.md)。

### 责任链

责任链的关键不是“把很多类串起来”，而是明确三件事：处理顺序、是否继续传递、异常如何处理。适合校验、审批、过滤和风控管道；如果节点之间强依赖、顺序经常动态变化，应考虑显式流程编排。

### 观察者与事件

观察者适合“一次变化通知多个独立动作”，但生产环境要补充：

- 同步事件中的一个慢或异常订阅者会拖累发布者，需要隔离异常和耗时任务。
- 异步事件要考虑事务提交时机、重复消费、顺序、重试和积压。
- 订阅关系要可取消，避免监听器、线程池或闭包持有对象导致泄漏。

### 状态机

当对象状态超过几个，并且不同状态下允许的操作不同，使用状态对象或显式状态转换表，比在一个方法里堆叠 `if/else` 更容易验证。

```text
待支付 ──支付成功──> 已支付 ──发货──> 已发货
   │                   │
   └──取消────────────> 已取消
```

状态机必须校验非法迁移，并明确重复事件、并发更新和持久化策略；仅仅把 `if/else` 拆成多个类，不等于真正解决了状态混乱。

## 模式组合

- `Strategy + Factory/Registry`：先按类型选策略，再执行算法，适合营销、计费和风控。
- `Adapter + Facade`：先把外部差异归一，再通过门面暴露稳定入口。
- `Decorator + Proxy`：分别叠加业务能力和控制访问，常见于缓存、权限、事务组合。
- `Chain + Template Method`：流程骨架固定，但每个处理节点可以独立扩展。
- `Command + Observer`：命令负责执行和重试，事件负责通知其他模块。

模式组合要围绕变化点展开；如果只是为了展示模式而叠加工厂、代理、策略和模板，最终会形成难以追踪的调用链。

## 复习抓手

- 能根据代码信号选择模式，而不是只背定义。
- 能说明模式引入后减少了哪一种变化成本，以及新增了什么复杂度。
- 能区分适配器、装饰器、代理、外观，以及策略、状态、模板方法。
- 能结合 Java/Spring 说出 `FactoryBean`、AOP 代理、`JdbcTemplate`、Spring Event、拦截器链等落点。
- 看到模式名时，先问“变化轴在哪里、对象生命周期谁负责、异常和并发怎么处理”。
