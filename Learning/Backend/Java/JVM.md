# JVM

## JVM 运行时数据区

## JVM 运行时数据区有哪些？

| 区域 | 线程关系 | 主要作用 | 常见异常 |
| --- | --- | --- | --- |
| 程序计数器 | 线程私有 | 记录当前线程执行的字节码地址；执行 Native 方法时值未定义 | 规范中没有规定该区域的 OOM |
| Java 虚拟机栈 | 线程私有 | 保存方法调用产生的栈帧 | `StackOverflowError`，栈扩展失败时可能 OOM |
| 本地方法栈 | 线程私有 | 为 Native 方法服务，HotSpot 中通常与虚拟机栈合并实现 | `StackOverflowError`、OOM |
| Java 堆 | 线程共享 | 存放对象实例和数组，是 GC 管理的主要区域 | `Java heap space` |
| 方法区/元空间 | 线程共享 | 类元数据、运行时常量池、方法信息等 | `Metaspace` |
| 直接内存 | 线程共享 | NIO、Netty 使用的堆外内存 | `Direct buffer memory` |

堆和方法区是线程共享的；程序计数器、虚拟机栈和本地方法栈是线程私有的。直接内存不属于 JVM 规范定义的运行时数据区，但排查 OOM 时经常需要关注。

## 堆和栈有什么区别？

| 维度 | 栈 | 堆 |
| --- | --- | --- |
| 内容 | 栈帧、局部变量、操作数栈、返回地址 | 对象实例和数组 |
| 线程关系 | 每个线程一份 | 所有线程共享 |
| 生命周期 | 随方法调用入栈、返回出栈 | 由 GC 根据可达性决定 |
| 分配特点 | 栈帧进出，通常较快 | 涉及 TLAB、GC 和内存管理 |
| 常见异常 | 递归过深导致 `StackOverflowError` | 对象过多或泄漏导致 OOM |

一句话：栈管方法调用，堆管对象实例；栈是线程私有且生命周期明确，堆是线程共享且主要由 GC 管理。

## 方法区和元空间是什么关系？

方法区是 JVM 规范中的逻辑区域，元空间是 HotSpot 对它的一种实现：

- JDK 7 及以前主要使用永久代实现。
- JDK 8 移除永久代，改用本地内存中的元空间。

方法区/元空间主要保存：

- 类元数据：类名、父类、接口、字段和方法信息。
- 方法字节码及相关属性。
- 运行时常量池。
- 类、字段和方法的符号引用及解析结果。

JDK 7+ 中，静态变量引用的对象通常在堆中，类元数据仍在方法区/元空间中。不要简单理解为“方法区存放所有静态变量”。

## 运行时常量池和字符串常量池有什么区别？

- 运行时常量池：由 class 文件常量池加载而来，保存字面量、类名、方法名、字段名、描述符和符号引用，属于方法区体系。
- 字符串常量池：保存可复用的字符串对象引用。HotSpot 从 JDK 7 开始将其放入堆中。
- `String.intern()`：尝试将字符串引用放入字符串常量池，后续相同字符串可以复用池中的引用。

## 直接内存是什么？为什么会 OOM？

直接内存是堆外内存，常见于 `ByteBuffer.allocateDirect()` 和 Netty 的直接内存池。它可以减少 Java 堆与 Native 内存之间的数据复制，适合 I/O 场景。

它不受 `-Xmx` 直接限制，受本机内存和 `-XX:MaxDirectMemorySize` 等限制。排查时关注 DirectByteBuffer、Netty buffer 池、释放情况和容器内存上限。

## 对象与字节码执行

## Java 堆如何分代？对象什么时候进入老年代？

传统分代理解中，堆主要包括：

| 区域 | 作用 |
| --- | --- |
| Eden | 新对象通常优先分配在这里，空间不足时触发 Young GC |
| Survivor 0/1 | Young GC 后存活对象在两个区之间复制，年龄增加 |
| Old | 长期存活对象、大对象或晋升对象 |

对象进入老年代的常见情况：

- 对象年龄达到晋升阈值，常见默认最大年龄为 15。
- 动态年龄判断认为 Survivor 空间不足以容纳某年龄段对象。
- 大对象需要连续内存，可能直接进入老年代。
- Young GC 后 Survivor 放不下，触发空间分配担保并晋升。

G1 将堆划分为多个 Region，逻辑上仍保留新生代和老年代概念；超过 Region 一半大小的大对象会使用 Humongous Region。

## 对象创建过程是什么？

1. 检查类是否已加载、解析和初始化，必要时先触发类加载。
2. 在堆中分配对象内存，可能使用指针碰撞或空闲列表。
3. 使用 TLAB 或 CAS 处理多线程分配。
4. 将字段初始化为类型默认值。
5. 设置对象头，包括 Mark Word、类型指针和 GC 年龄等信息。
6. 执行构造方法 `<init>`，完成 Java 层面的初始化。

从 JVM 角度看，内存分配和对象头设置完成后对象已经存在；从业务代码角度看，构造方法执行完成后对象才通常可用。

## 对象的内存布局是什么？

HotSpot 普通对象通常包括：

| 部分 | 内容 |
| --- | --- |
| 对象头 | Mark Word、类型指针；数组对象还包含数组长度 |
| 实例数据 | 对象自身及父类的字段 |
| 对齐填充 | 按对象对齐要求补齐的空间 |

Mark Word 会根据对象状态保存哈希码、GC 年龄和锁状态等信息。压缩类指针和压缩普通对象指针也会影响对象头大小。

## 方法调用和栈帧是如何执行的？

每次调用方法，当前线程的虚拟机栈都会压入一个栈帧，主要包含：

- 局部变量表：方法参数、局部变量和对象引用。
- 操作数栈：字节码执行时的临时计算区。
- 动态链接：指向运行时常量池，用于方法调用解析。
- 方法返回地址：方法结束后恢复调用者的执行位置。

方法返回后当前栈帧出栈，返回值压入调用者的操作数栈或恢复调用者上下文。

## `String s = new String("abc")` 涉及哪些内存区域？

```java
String s = new String("abc");
```

- 栈：局部变量 `s` 位于当前栈帧的局部变量表中，保存引用。
- 字符串常量池：保存字面量 `"abc"` 的可复用引用。
- 堆：`new String("abc")` 创建新的 String 对象。

如果常量池中没有 `"abc"`，通常会创建字面量对象和 `new` 出来的对象；如果已有该字面量，通常只创建 `new` 出来的对象。`s` 指向的是新创建的堆对象。

## 垃圾回收

## 如何判断对象是否可以被回收？

引用计数法简单高效，但无法解决循环引用；主流 JVM 使用可达性分析，从 GC Roots 沿引用链搜索，不可达对象才具备被回收条件。

常见 GC Roots：

- 虚拟机栈中的局部变量和方法参数引用。
- 本地方法栈中的 JNI 引用。
- 类静态字段和运行时常量引用。
- 被同步锁持有的对象。
- JVM 内部引用，例如系统类加载器和基本类型 Class 对象。

对象不可达后也不一定立即回收。`finalize()` 可能产生一次历史上的自救机会，但它已不推荐使用，资源应通过显式关闭或 `try-with-resources` 管理。

## Java 有哪些引用类型？

| 类型 | 对应类 | 回收时机 | 典型场景 |
| --- | --- | --- | --- |
| 强引用 | 普通对象引用 | 强引用存在时不会回收 | 常规业务对象 |
| 软引用 | `SoftReference` | 内存不足时可能回收 | 内存敏感缓存 |
| 弱引用 | `WeakReference` | 下一次 GC 时通常回收 | `WeakHashMap`、ThreadLocalMap 的 key |
| 虚引用 | `PhantomReference` | 不影响生命周期，配合 `ReferenceQueue` 使用 | 跟踪回收、释放堆外资源 |

弱引用只弱化 referent，不代表关联的 value 一定会自动释放。

## 垃圾回收算法有哪些？

| 算法 | 思路 | 优点 | 缺点 | 常见区域 |
| --- | --- | --- | --- | --- |
| 标记-清除 | 标记可回收对象后清除 | 实现简单、不移动对象 | 产生碎片 | CMS 老年代 |
| 复制 | 将存活对象复制到另一块区域 | 无碎片，适合存活率低 | 需要额外空间，存活率高时复制成本大 | 新生代 |
| 标记-整理 | 标记后将存活对象向一端移动 | 减少碎片 | 移动成本高，通常需要 STW | 老年代 |
| 分代收集 | 不同生命周期使用不同算法 | 符合对象朝生夕死规律 | 需要处理跨代引用 | 传统分代 GC |

## Minor GC、Major GC 和 Full GC 有什么区别？

| 类型 | 常见范围 | 常见触发原因 | 说明 |
| --- | --- | --- | --- |
| Minor/Young GC | 新生代 | Eden 空间不足 | 频率较高，通常停顿较短 |
| Major/Old GC | 老年代 | 老年代空间不足或收集器策略触发 | 各资料定义不完全一致，回答时说明上下文 |
| Full GC | 整个堆，通常涉及元空间相关清理 | 老年代不足、元空间不足、分配担保失败、显式 GC 等 | 成本通常最高 |

Full GC 常见原因还包括 CMS Concurrent Mode Failure、大对象连续空间不足和晋升失败。

## 常见垃圾收集器有哪些？

| 收集器 | 范围 | 特点 | 定位 |
| --- | --- | --- | --- |
| Serial / Serial Old | 新生代/老年代 | 单线程、STW、实现简单 | 小堆或兜底 |
| ParNew | 新生代 | Serial 的多线程版本，可配合 CMS | 历史组合 |
| Parallel Scavenge/Old | 新生代/老年代 | 吞吐量优先、自适应调节 | 批处理、后台计算 |
| CMS | 主要老年代 | 并发标记清除、低停顿但有碎片 | 老版本低延迟服务 |
| G1 | 整堆 | Region 化、可预测停顿、局部复制 | 现代服务端常用 |
| ZGC / Shenandoah | 整堆 | 大量并发阶段、超低停顿 | 大堆和低延迟场景 |

## CMS 和 G1 有什么区别？

| 对比 | CMS | G1 |
| --- | --- | --- |
| 范围 | 主要老年代，通常搭配新生代收集器 | 面向整堆，按 Region 管理 |
| 算法 | 标记-清除 | Region 间复制，整体减少碎片 |
| 目标 | 尽量缩短老年代停顿 | 在吞吐和可预测停顿之间平衡 |
| 问题 | 碎片、浮动垃圾、Concurrent Mode Failure | 转移失败或空间不足时可能退化 |
| 阶段 | 初始标记、并发标记、重新标记、并发清除 | 初始标记、并发标记、最终标记、筛选回收/转移 |

CMS、ParNew 等属于传统收集器；现代 JDK 选择收集器时应结合 JDK 版本、堆大小、吞吐和延迟目标。

## G1 有哪些主要特性？

- Region 化：堆被划分为多个大小相等的 Region。
- 分代逻辑：Region 可以动态扮演 Eden、Survivor、Old 或 Humongous。
- 可预测停顿：通过 `-XX:MaxGCPauseMillis` 设置期望目标，优先回收收益高的 Region。
- 局部复制、整体整理：降低碎片。
- 并行与并发结合：标记可以并发，转移阶段通常需要 STW。

Humongous 对象通常是超过 Region 一半大小的对象，会占用一个或多个连续 Humongous Region。

## 哪些 GC 阶段会 Stop The World？

STW 表示暂停用户线程，让 GC 在一致性视图下完成工作。

- 新生代复制存活对象通常需要 STW。
- Serial、Parallel 的主要回收阶段基本是 STW。
- CMS 的初始标记和重新标记需要 STW，并发标记和清除不暂停用户线程。
- G1 的初始标记、最终标记和筛选回收/对象转移需要 STW，并发标记不需要全程 STW。
- 标记-整理中的对象移动和引用更新通常需要 STW。

G1 的停顿瓶颈常在对象转移阶段，因为需要复制存活对象并修正引用。

## GC 只回收堆吗？类卸载需要什么条件？

不是。GC 也可能处理方法区/元空间中的废弃常量和无用类。

类卸载通常要求：

1. 堆中不存在该类及其子类的实例。
2. 加载该类的 ClassLoader 可以被回收。
3. 对应的 `Class` 对象没有被其他地方引用。

类卸载比普通对象回收更严格，动态加载、插件化和容器场景更常见。

## 类加载

## 类加载过程有哪些阶段？

类加载过程通常概括为：加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载。其中验证、准备、解析合称连接。

| 阶段 | 主要工作 |
| --- | --- |
| 加载 | 获取 class 字节流，转为方法区运行时结构并生成 `Class` 对象 |
| 验证 | 校验文件格式、元数据、字节码和符号引用 |
| 准备 | 为类变量分配内存并设置零值 |
| 解析 | 将符号引用替换为直接引用 |
| 初始化 | 执行 `<clinit>`，即静态赋值和静态代码块合并后的逻辑 |
| 卸载 | 满足条件后回收类元数据 |

易错点：`static int a = 1` 在准备阶段先是零值，显式赋值要到初始化阶段才执行。

## 类加载器有哪些？

| 类加载器 | 作用 |
| --- | --- |
| Bootstrap | JVM 实现的启动类加载器，负责核心类库，Java 代码通常拿不到其对象引用 |
| Extension/Platform | JDK 8 是扩展类加载器，JDK 9+ 是平台类加载器 |
| Application | 加载 classpath/module path 上的应用类 |
| Custom | 从网络、数据库、加密文件等特殊来源加载类 |

类的唯一性由“类加载器 + 全限定类名”共同决定。同一个 class 文件被不同类加载器加载，会被 JVM 视为不同的类。

## 双亲委派模型是什么？为什么可能被打破？

一个类加载器收到加载请求后，先委托父加载器加载；父加载器无法加载时，子加载器才尝试自己加载。

作用：

- 保证核心类库安全，避免自定义 `java.lang.String` 覆盖 JDK 类。
- 保证核心类的唯一性，避免重复定义。
- 让不同类加载器各司其职。

典型打破场景：

- SPI：父加载器加载接口，具体实现由子加载器提供。
- Web 容器：不同应用之间需要类隔离。
- 插件化和热部署：需要动态加载、替换和卸载类。

## 对象生命周期和类生命周期有什么区别？

对象生命周期是：创建 → 使用 → 不可达 → GC 回收。判断重点是对象是否仍可从 GC Roots 到达。

类生命周期是：加载 → 连接 → 初始化 → 使用 → 卸载。类卸载还依赖 ClassLoader、Class 对象和该类实例都不再可达，因此普通业务类不会频繁卸载，动态加载场景更常见。

## 常见 JVM 参数如何区分？

| 参数 | 作用 |
| --- | --- |
| `-Xms` / `-Xmx` | 堆初始大小 / 最大大小 |
| `-Xss` | 单个 Java 线程的栈大小 |
| `-XX:MaxMetaspaceSize` | 元空间上限，限制类元数据占用的本地内存 |
| `-XX:MaxDirectMemorySize` | 直接内存上限，影响 NIO/Netty 等堆外缓冲区 |

`-Xmn`、`NewSize` 等新生代参数依赖具体垃圾收集器，现代 G1 等收集器通常不建议手工固定；参数应结合 GC 日志、堆外内存和压测结果调整，而不是套用固定比例。

## JVM 内存问题与诊断

## JVM 常见 OOM 有哪些？

| 异常 | 常见原因 | 排查方向 |
| --- | --- | --- |
| `Java heap space` | 堆对象过多、大对象、内存泄漏 | Heap Dump、MAT/VisualVM、GC 日志 |
| `StackOverflowError` | 递归过深、循环调用、栈帧过大 | 调用栈、递归出口、`-Xss` |
| `Metaspace` | 动态生成类过多、ClassLoader 泄漏、元空间过小 | 类加载数量、ClassLoader 引用链 |
| `Direct buffer memory` | 直接内存过多、Buffer 未释放、Netty 配置不当 | 直接内存上限、NMT、Netty 监控 |
| `GC overhead limit exceeded` | GC 花费大量时间但回收很少 | 内存泄漏、堆大小、对象存活率 |

## 内存泄漏和内存溢出有什么区别？

内存泄漏是对象已经不再使用，但仍被引用链持有，GC 无法回收；内存溢出是 JVM 申请内存失败后的结果。

常见泄漏场景：

- 静态集合只增不删。
- 监听器或回调注册后没有取消。
- 线程池、ThreadLocal 或长生命周期线程持有业务对象。
- 连接、流、Session 等资源没有关闭。
- ClassLoader 泄漏导致类元数据无法卸载。

## JVM 内存问题如何排查？

1. 查看 GC 日志，观察 Full GC 后内存是否下降。
2. 获取 Heap Dump，使用 MAT/VisualVM 查看大对象和支配树。
3. 沿 GC Roots 引用链确认是谁持有本应释放的对象。
4. 遇到堆外内存问题时，结合 Native Memory Tracking、Netty leak detector 和容器内存指标分析。

常用工具包括 `jps`、`jstat`、`jmap`、`jstack`、`jcmd`、GC 日志和 Heap Dump。

## Java 进程 CPU 使用率接近 100% 怎么排查？

先确认是整个进程还是某个线程占用 CPU，再把操作系统线程 ID 转成十六进制，用 `jstack` 定位对应 Java 线程的调用栈。重点排查死循环、异常重试、频繁 GC、锁自旋、正则回溯和热点 SQL/请求。

基本流程：

1. 用 `top`、`pidstat` 或监控确认进程和线程的 CPU 使用率。
2. 用 `top -H -p <pid>` 找到高 CPU 线程。
3. 多次采集 `jstack`，判断线程是否持续停留在同一段代码。
4. 同时查看 GC、请求量、线程池队列和外部依赖，避免只盯着 Java 调用栈。

如果是 GC 线程占用，应结合 GC 日志和堆中对象增长判断；如果是业务线程，则重点看循环、批处理、锁竞争和异常重试。
