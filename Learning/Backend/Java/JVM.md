# JVM

## JVM 内存区域
### JVM 运行时数据区有哪些？

JVM 运行时数据区可以按线程是否共享来记：

| 区域            | 是否线程共享 | 主要作用                                                     | 常见异常                                 |
| --------------- | ------------ | ------------------------------------------------------------ | ---------------------------------------- |
| 程序计数器      | 线程私有     | 记录当前线程正在执行的字节码指令地址。执行 Native 方法时值为 undefined（未定义）。 | 规范中唯一没有规定 OOM 的区域            |
| Java 虚拟机栈   | 线程私有     | 每次方法调用创建一个栈帧，保存局部变量表、操作数栈、动态链接、方法返回地址等。 | StackOverflowError；栈扩展失败时可能 OOM |
| 本地方法栈      | 线程私有     | 为 Native 方法服务。HotSpot 中通常与虚拟机栈合并实现。       | StackOverflowError / OOM                 |
| Java 堆         | 线程共享     | 存放对象实例和数组，是 GC 管理的主要区域。                   | Java heap space                          |
| 方法区 / 元空间 | 线程共享     | 存放类元数据、运行时常量池、方法字节码等。JDK 8+ 使用本地内存实现元空间。 | Metaspace                                |
| 直接内存        | 线程共享     | NIO、DirectByteBuffer、Netty 常用的堆外内存，不属于运行时数据区规范的一部分。 | Direct buffer memory                     |

面试回答要点：堆和方法区是线程共享的；程序计数器、虚拟机栈、本地方法栈是线程私有的；直接内存不是 JVM 规范定义的运行时数据区，但排查 OOM 时经常要看。

### 堆和栈有什么区别？

| 维度     | 栈                                                           | 堆                                                 |
| -------- | ------------------------------------------------------------ | -------------------------------------------------- |
| 存储内容 | 方法调用产生的栈帧，包括局部变量、操作数栈、返回地址等。局部变量表中可以存基本类型值和对象引用。 | 对象实例、数组、字符串常量池中的字符串对象等。     |
| 线程关系 | 线程私有，每个线程有自己的栈。                               | 线程共享，所有线程都可以访问堆中对象。             |
| 生命周期 | 随方法调用入栈，随方法结束出栈，生命周期确定。               | 对象生命周期不确定，由 GC 根据可达性决定是否回收。 |
| 分配效率 | 通常更快，栈帧入栈出栈即可。                                 | 相对复杂，涉及 TLAB、指针碰撞、空闲列表、GC 等。   |
| 常见异常 | 递归过深或栈帧过大导致 StackOverflowError。                  | 对象过多、内存泄漏、大对象分配失败导致 OOM。       |

一句话：栈管方法调用，堆管对象实例；栈是线程私有且生命周期明确，堆是线程共享且主要由 GC 管理。

### 方法区和元空间是什么关系？方法区里有什么？

方法区是 JVM 规范中的逻辑区域，HotSpot 在不同版本有不同实现：

-   JDK 7 及以前：主要用永久代（PermGen）实现。
-   JDK 8 以后：永久代移除，改为元空间（Metaspace），使用本地内存。

方法区 / 元空间主要存：

| 内容               | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| 类元数据           | 类名、父类、接口、访问修饰符、字段信息、方法信息等。         |
| 方法字节码         | 方法的 Code 属性、操作数栈大小、局部变量表大小等。           |
| 运行时常量池       | class 文件常量池加载后的运行时表示，包括字面量和符号引用。   |
| 符号引用和解析结果 | 类、字段、方法的符号引用会在解析阶段转为直接引用。直接引用表示可以直接定位目标，但不等于目标类一定已经完成初始化。 |

静态变量注意版本差异：JDK 7+ 中，静态变量对应的对象数据通常在堆中，类元数据仍在方法区 / 元空间中。面试中可以说“静态变量逻辑上跟类相关，物理存储实现要看 JDK 版本，现代 HotSpot 下主要在堆中保存数据”。

### 运行时常量池和字符串常量池有什么区别？

运行时常量池是方法区 / 元空间的一部分，来源于 class 文件常量池，存放字面量、类名、方法名、字段名、描述符等符号引用，类加载后进入运行时常量池。

字符串常量池保存字符串字面量对应的 String 对象引用。HotSpot 从 JDK 7 开始将字符串常量池移到堆中，所以字符串对象本身在堆里，常量池保存的是可复用的引用。

区别可以这样答：

-   运行时常量池偏 class 元数据，属于方法区体系。
-   字符串常量池偏 String 对象复用，现代 HotSpot 中位于堆。
-   `String.intern()` 会尝试把字符串引用放入字符串常量池，后续相同字面量可以复用。

### 直接内存是什么？为什么会 OOM？

直接内存是堆外内存，不属于 JVM 运行时数据区规范，但常被 Java 程序通过 NIO 使用，例如 `ByteBuffer.allocateDirect()`、Netty 的直接内存池。

直接内存的优势是减少 Java 堆和 Native 堆之间的数据复制，适合 I/O 密集场景。风险是它不受 `-Xmx` 直接限制，受本机内存和 `-XX:MaxDirectMemorySize` 限制。

常见异常是：

```
java.lang.OutOfMemoryError: Direct buffer memory
```

排查时要看：是否大量创建 DirectByteBuffer、Netty 直接内存池配置、是否及时释放、容器内存上限是否小于 JVM 预期。



## 堆与对象

### Java 堆分为哪几部分？

传统分代理解中，Java 堆主要分为新生代和老年代：

| 区域                    | 作用                                                         |
| ----------------------- | ------------------------------------------------------------ |
| Eden                    | 大多数新对象优先分配在 Eden。Eden 满时通常触发 Minor GC。    |
| Survivor 0 / Survivor 1 | Minor GC 后仍存活的对象在两个 Survivor 区之间复制，年龄增加。 |
| Old / Tenured           | 存放长期存活对象、大对象或晋升对象。老年代回收通常成本更高。 |

对象晋升老年代的常见情况：

-   对象年龄达到阈值，默认最大年龄常见为 15，可通过 `-XX:MaxTenuringThreshold` 调整。
-   动态年龄判断：Survivor 中相同年龄对象大小超过 Survivor 空间一半时，大于等于该年龄的对象可直接晋升。
-   大对象需要连续内存，可能直接进入老年代。
-   Minor GC 后 Survivor 放不下，存活对象通过空间分配担保进入老年代。

G1 中要补充一句：G1 把堆切成多个 Region，逻辑上仍有新生代、老年代概念；Humongous Region 是 G1 对大对象的特殊处理，通常用于超过 Region 一半大小的对象，不是所有 JVM 堆的通用分区。

### 对象创建过程是什么？

1.  类加载检查：检查常量池中类的符号引用，确认类是否已加载、解析、初始化；没有则先触发类加载。
2.  分配内存：在堆中为对象分配确定大小的内存。常见方式是指针碰撞或空闲列表。
3.  处理并发分配：通过 CAS 重试或 TLAB（Thread Local Allocation Buffer）保证多线程分配安全。
4.  初始化零值：对象字段先被置为类型默认值，不包括对象头。
5.  设置对象头：写入 Mark Word、类型指针、GC 年龄、锁状态等信息。
6.  执行 `<init>`：调用构造方法，按 Java 代码完成字段赋值和构造逻辑。

面试区分：从 JVM 视角看，内存分配和对象头设置后对象已经有了；从 Java 程序视角看，构造方法执行完才是可用对象。

### 对象的内存布局是什么？

HotSpot 中普通对象通常由三部分组成：

| 部分     | 内容                                          |
| -------- | --------------------------------------------- |
| 对象头   | Mark Word、类型指针；数组对象还包含数组长度。 |
| 实例数据 | 对象真正存储的字段内容，包括父类字段。        |
| 对齐填充 | JVM 要求对象大小按一定字节对齐，不足时填充。  |

对象头中的 Mark Word 会随状态变化保存不同内容，例如哈希码、GC 分代年龄、锁标志位等。开启压缩类指针、压缩普通对象指针时，对象头大小会受影响。

### 方法调用和栈帧执行过程是什么？

每调用一个方法，当前线程的虚拟机栈都会创建一个栈帧。栈帧主要包含：

-   局部变量表：保存方法参数、局部变量、对象引用等。
-   操作数栈：字节码指令执行时的临时计算区。
-   动态链接：指向运行时常量池中该方法所属类的引用，用于方法调用解析。
-   方法返回地址：方法正常返回或异常退出后，恢复调用方执行位置。

执行流程：

1.  方法调用时创建并压入新栈帧。
2.  字节码执行过程中读写局部变量表和操作数栈。
3.  需要调用其他方法时，通过符号引用解析或动态分派找到目标方法。
4.  方法返回后当前栈帧出栈，返回值压入调用者的操作数栈或直接恢复调用者上下文。

### String s = new String("abc") 涉及哪些内存区域？

代码：

```java
String s = new String("abc");
```

涉及的区域：

-   栈：局部变量 `s` 存放在当前方法栈帧的局部变量表中，保存一个引用。
-   字符串常量池：保存 `"abc"` 这个字面量对应 String 对象的引用。JDK 7+ 字符串常量池在堆中。
-   堆：`new String("abc")` 会在堆上创建一个新的 String 对象。

对象数量要分场景：

| 场景                 | 创建对象数量 | 说明                                                         |
| -------------------- | ------------ | ------------------------------------------------------------ |
| 常量池中没有 `"abc"` | 通常 2 个    | 先创建字面量 `"abc"` 对应的 String 对象并放入字符串常量池，再创建 `new String` 的新对象。 |
| 常量池中已有 `"abc"` | 通常 1 个    | 复用常量池中的 `"abc"`，只创建 `new String` 产生的新对象。   |

面试补充：`s` 指向的是 `new` 出来的堆对象，不是字符串常量池中的对象。



## 垃圾回收

### 如何判断对象是否可以被回收？

主要有两类方法：

| 方法       | 原理                                                         | 问题 / 特点                    |
| ---------- | ------------------------------------------------------------ | ------------------------------ |
| 引用计数法 | 每个对象维护引用计数，引用增加则 +1，引用失效则 -1，计数为 0 可回收。 | 简单高效，但无法解决循环引用。 |
| 可达性分析 | 从 GC Roots 出发沿引用链搜索，不可达对象可被回收。           | JVM 主流实现使用该方法。       |

常见 GC Roots：

-   虚拟机栈中引用的对象，例如局部变量、方法参数。
-   本地方法栈中 JNI 引用的对象。
-   方法区 / 元空间中类静态属性引用的对象。
-   方法区 / 元空间中常量引用的对象。
-   被同步锁持有的对象。
-   JVM 内部引用，例如系统类加载器、基本类型 Class 对象等。

对象不可达后也不一定立即死亡。若对象重写了 `finalize()` 且未执行过，可能经历一次自救机会；但 `finalize()` 不可靠，实际开发不建议依赖。

### Java 有哪些引用类型？

| 引用类型 | 对应类                                     | 回收时机                                           | 典型场景                                 |
| -------- | ------------------------------------------ | -------------------------------------------------- | ---------------------------------------- |
| 强引用   | 普通对象引用，如 `Object o = new Object()` | 只要强引用存在，GC 不会回收。                      | 常规业务对象。                           |
| 软引用   | `SoftReference`                            | 内存不足前会尝试回收。                             | 内存敏感缓存，但现在更推荐成熟缓存组件。 |
| 弱引用   | `WeakReference`                            | 下一次 GC 发生时，只要没有强引用就会被回收。       | `WeakHashMap`、ThreadLocalMap 的 key。   |
| 虚引用   | `PhantomReference`                         | 不影响对象生命周期，必须配合 ReferenceQueue 使用。 | 跟踪对象回收、释放堆外资源。             |

弱引用常见用途是避免因为辅助结构持有对象而导致泄漏，但要注意：弱引用只弱化 key 或 referent，不代表 value 也会自动安全释放。

### 垃圾回收算法有哪些？

| 算法      | 思路                                                         | 优点                             | 缺点                                              | 常见适用区域  |
| --------- | ------------------------------------------------------------ | -------------------------------- | ------------------------------------------------- | ------------- |
| 标记-清除 | 先标记可回收对象，再统一清除。                               | 实现简单，不移动对象。           | 容易产生内存碎片，分配大对象可能失败。            | CMS 老年代。  |
| 复制算法  | 将内存分成两块，只使用一块，GC 时把存活对象复制到另一块，再清空原区域。 | 没有碎片，适合存活对象少的场景。 | 可用空间减少，存活对象多时复制成本高。            | 新生代。      |
| 标记-整理 | 先标记存活对象，再把存活对象向一端移动，清理边界外空间。     | 解决碎片问题。                   | 移动对象成本高，通常需要 STW。                    | 老年代。      |
| 分代收集  | 按对象生命周期把堆划分为新生代、老年代，不同区域用不同算法。 | 符合大多数对象朝生夕死的规律。   | 跨代引用需要 Remembered Set / Card Table 等机制。 | 传统分代 GC。 |

### Minor GC、Major GC、Full GC 有什么区别？

| 类型                | 回收范围                                            | 触发场景                                                     | 特点                                          |
| ------------------- | --------------------------------------------------- | ------------------------------------------------------------ | --------------------------------------------- |
| Minor GC / Young GC | 新生代                                              | Eden 空间不足。                                              | 频率高，通常停顿较短。                        |
| Major GC / Old GC   | 老年代                                              | 老年代空间不足或收集器策略触发。                             | 术语容易和 Full GC 混用，回答时要说明上下文。 |
| Full GC             | 整个 Java 堆，通常还会包含方法区 / 元空间相关清理。 | 老年代不足、元空间不足、分配担保失败、显式 `System.gc()` 等。 | 成本最高，通常需要重点排查。                  |

Full GC 常见触发原因：

-   老年代空间不足，大对象或晋升对象放不下。
-   Minor GC 前空间分配担保失败，老年代无法保证容纳新生代可能晋升的对象。
-   元空间不足，JDK 8+ 表现为 `OutOfMemoryError: Metaspace` 或触发 Full GC 尝试卸载类。
-   调用 `System.gc()`，如果没有禁用显式 GC，可能触发 Full GC。
-   CMS 发生 Concurrent Mode Failure，退化为更重的回收。
-   大对象分配需要连续空间，碎片过多导致分配失败。



### 常见垃圾收集器有哪些？

| 收集器            | 作用范围 | 算法               | 目标 / 特点                            | 典型场景                    |
| ----------------- | -------- | ------------------ | -------------------------------------- | --------------------------- |
| Serial            | 新生代   | 复制               | 单线程，简单，STW。                    | 单核、小堆、Client 场景。   |
| Serial Old        | 老年代   | 标记-整理          | 单线程，常作兜底。                     | Client 或 CMS 失败兜底。    |
| ParNew            | 新生代   | 复制               | Serial 的多线程版本，可与 CMS 搭配。   | 传统低停顿组合。            |
| Parallel Scavenge | 新生代   | 复制               | 吞吐量优先，可自适应调节。             | 后台计算、批处理。          |
| Parallel Old      | 老年代   | 标记-整理          | 吞吐量优先的老年代收集器。             | 与 Parallel Scavenge 搭配。 |
| CMS               | 老年代   | 标记-清除          | 并发收集、低停顿，但有碎片和浮动垃圾。 | 老版本低延迟服务。          |
| G1                | 整堆     | 局部复制，整体整理 | Region 化、可预测停顿、适合大堆。      | 现代服务端常用。            |
| ZGC / Shenandoah  | 整堆     | 并发整理           | 超低停顿，适合大堆低延迟。             | 对延迟敏感的大内存服务。    |

### CMS 和 G1 有什么区别？

| 对比     | CMS                                                         | G1                                                           |
| -------- | ----------------------------------------------------------- | ------------------------------------------------------------ |
| 回收范围 | 主要回收老年代，需要搭配新生代收集器，如 ParNew。           | 面向整个堆，内部按 Region 管理。                             |
| 目标     | 尽量缩短老年代回收停顿。                                    | 在吞吐和低停顿之间平衡，支持可预测停顿。                     |
| 算法     | 标记-清除。                                                 | 整体看是标记-整理，Region 之间是复制。                       |
| 碎片问题 | 会产生内存碎片，可能导致 Full GC。                          | 通过 Region 复制整理降低碎片问题。                           |
| 浮动垃圾 | 并发清除时用户线程仍运行，会产生浮动垃圾。                  | 筛选回收阶段 STW 并行转移，浮动垃圾问题相对小。              |
| 失败退化 | Concurrent Mode Failure 后可能退化为 Serial Old，停顿很长。 | Evacuation Failure 或空间不足时也可能退化，但整体设计更适合大堆。 |
| 典型阶段 | 初始标记（STW）-> 并发标记 -> 重新标记（STW）-> 并发清除。  | 初始标记（STW）-> 并发标记 -> 最终标记（STW）-> 筛选回收 / 转移（STW）。 |

### G1 收集器的主要特性是什么？

G1 的核心特性：

-   Region 化：把堆划分为多个大小相等的 Region，不再要求新生代、老年代物理连续。
-   保留分代思想：Region 可以动态扮演 Eden、Survivor、Old、Humongous 等角色。
-   可预测停顿：通过 `-XX:MaxGCPauseMillis` 设置期望停顿时间，G1 根据回收收益选择优先回收的 Region。
-   Garbage First：优先回收垃圾最多、收益最高的 Region。
-   局部复制、整体整理：对象从一个 Region 转移到另一个 Region，减少碎片。
-   并行与并发结合：标记可并发执行，转移阶段多线程并行但需要 STW。

Humongous 对象：超过 Region 一半大小的对象会被放入一个或多个连续 Humongous Region，逻辑上归老年代管理。这是 G1 的实现细节，不是普通堆分区。

### 哪些 GC 阶段会 Stop The World？

STW 是指暂停所有用户线程，让 GC 线程执行某些必须在一致性视图下完成的工作。

常见情况：

-   新生代 Minor GC：复制存活对象、更新引用通常需要 STW。
-   Serial、Parallel 系列：主要回收阶段基本都是 STW。
-   CMS：初始标记和重新标记需要 STW；并发标记、并发清除不暂停用户线程。
-   G1：初始标记、最终标记、筛选回收 / 对象转移需要 STW；并发标记不需要全程 STW。
-   标记-整理：移动对象并更新引用，通常需要 STW。

G1 停顿瓶颈常在对象转移阶段，因为要复制存活对象并修正引用；初始标记只标记 GC Roots，通常较短。

### GC 只回收堆吗？

不是。堆是 GC 的主要区域，但 GC 也会处理方法区 / 元空间中的无用类和废弃常量。

方法区 / 元空间回收主要包括：

-   废弃常量：常量池中的常量没有任何地方引用。
-   无用类：类满足卸载条件后，其类元数据可以被回收。

类卸载比对象回收苛刻得多，通常要同时满足：

1.  堆中不存在该类及其子类的任何实例。
2.  加载该类的 ClassLoader 已经可以被回收。
3.  该类对应的 `java.lang.Class` 对象没有被任何地方引用。

注意：现代 HotSpot 中静态变量数据多在堆中，不要再简单说“方法区存静态变量，所以 GC 回收方法区里的静态变量”。



## 类加载

### 类加载过程有哪些阶段？

类加载过程：加载 -> 验证 -> 准备 -> 解析 -> 初始化 -> 使用 -> 卸载。其中验证、准备、解析合称连接。

| 阶段   | 做什么                                                       |
| ------ | ------------------------------------------------------------ |
| 加载   | 通过类全限定名获取 class 二进制字节流，将静态存储结构转为方法区运行时数据结构，并生成 `Class` 对象作为访问入口。 |
| 验证   | 校验 class 文件格式、元数据、字节码、符号引用，保证不会危害 JVM。 |
| 准备   | 为类变量分配内存并设置零值。例如 `static int a = 1` 在准备阶段先为 0。 |
| 解析   | 将常量池中的符号引用替换为直接引用。                         |
| 初始化 | 执行类构造器 `<clinit>`，即静态变量显式赋值和静态代码块合并后的逻辑。 |
| 卸载   | 当类、ClassLoader、Class 对象都不可达时，类元数据可被回收。  |

易错点：准备阶段不是执行静态赋值语句；初始化阶段才执行  \<clinit \>

### 类加载器有哪些？

| 类加载器                                     | 作用                                                         |
| -------------------------------------------- | ------------------------------------------------------------ |
| Bootstrap ClassLoader                        | 启动类加载器，加载 Java 核心类库，由 JVM 实现，Java 代码中通常拿不到它的对象引用。 |
| Extension ClassLoader / Platform ClassLoader | JDK 8 中是扩展类加载器，JDK 9+ 调整为平台类加载器，负责加载平台相关类库。 |
| Application ClassLoader                      | 应用程序类加载器，加载 classpath / module path 上的业务类，是默认类加载器。 |
| Custom ClassLoader                           | 自定义类加载器，可从网络、数据库、加密文件等来源加载类，常见于容器、插件化、热部署。 |

类的唯一性由“类加载器 + 类全限定名”共同决定。同一个 class 文件被不同类加载器加载，会被 JVM 视为不同的类。

### 双亲委派模型是什么？

双亲委派流程：一个类加载器收到加载请求后，先委托父加载器加载；父加载器无法加载时，子加载器才尝试自己加载。

作用：

-   保证核心类库安全：用户自定义的 `java.lang.String` 不会覆盖 JDK 自带类。
-   保证类的唯一性：核心类由顶层加载器统一加载，避免重复定义。
-   职责清晰：Bootstrap 加载核心类，平台 / 扩展加载公共类库，应用加载业务类，自定义加载器处理特殊来源。

打破双亲委派的典型场景：

-   SPI：父加载器加载接口，但需要子加载器加载具体实现，例如 JDBC。
-   Web 容器：不同 Web 应用需要类隔离。
-   热部署、插件化：需要动态加载和卸载类。

### 对象生命周期和类生命周期有什么区别？

对象生命周期：

1.  创建：`new` 后在堆中分配内存，执行构造方法。
2.  使用：通过引用访问字段和方法。
3.  不可达：没有任何 GC Roots 引用链能到达该对象。
4.  回收：GC 释放对象占用的堆内存。

类生命周期：

1.  加载、连接、初始化后才能被正常使用。
2.  类由对应 ClassLoader 管理，类元数据位于方法区 / 元空间。
3.  类卸载要求更严格：该类及子类实例不存在、ClassLoader 可回收、Class 对象无引用。

区别：对象回收看对象是否可达；类卸载还强依赖 ClassLoader 生命周期，所以普通业务类通常不会频繁卸载，动态加载场景才更常见。



## 内存问题与专题

### JVM 常见 OOM 有哪些？

| 异常                                               | 常见原因                                                | 排查方向                                                     |
| -------------------------------------------------- | ------------------------------------------------------- | ------------------------------------------------------------ |
| `java.lang.OutOfMemoryError: Java heap space`      | 堆对象过多、大对象分配、内存泄漏。                      | heap dump、MAT / VisualVM、对象占用排行、GC 日志。           |
| `java.lang.StackOverflowError`                     | 递归过深、循环调用、栈帧过大。                          | 调用栈、递归出口、`-Xss`。                                   |
| `java.lang.OutOfMemoryError: Metaspace`            | 动态生成类过多、ClassLoader 泄漏、元空间过小。          | 类加载数量、ClassLoader 引用链、`-XX:MaxMetaspaceSize`。     |
| `java.lang.OutOfMemoryError: Direct buffer memory` | 直接内存过多、DirectByteBuffer 未释放、Netty 配置不当。 | 直接内存上限、NIO / Netty buffer 使用、Native Memory Tracking。 |
| `GC overhead limit exceeded`                       | GC 花费大量时间但回收很少内存。                         | 是否内存泄漏、堆是否过小、对象存活率是否异常。               |

### 内存泄漏和内存溢出有什么区别？

内存泄漏：对象已经不会再被业务使用，但仍然被引用链持有，GC 无法回收。它是一种问题原因。

内存溢出：JVM 申请内存失败，抛出 OOM。它是一种结果。

常见泄漏场景：

-   静态集合长期持有对象，数据只增不删。
-   监听器、回调注册后没有取消。
-   线程池中的任务、ThreadLocal、长生命周期线程持有业务对象。
-   连接、流、Session 等资源没有关闭。
-   ClassLoader 泄漏导致类元数据无法卸载。

排查思路：

1.  打开 GC 日志，观察 Full GC 后内存是否下降。
2.  获取 heap dump，查看大对象和支配树。
3.  找 GC Roots 引用链，确认是谁持有了本该释放的对象。
4.  对堆外内存问题，结合 NMT、Netty leak detector、容器内存指标分析。
