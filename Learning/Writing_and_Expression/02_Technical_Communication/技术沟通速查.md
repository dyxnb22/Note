# 技术沟通速查

按读者、决策和维护目标选择文字、图、接口说明、设计文档与事故复盘。

## 什么时候该画图，什么时候该写字

- 图适合表达关系、结构和流程，字适合表达边界、判断和例外。
- 图和字不是二选一，而是分工协作。
- 是否画图，取决于读者最难理解的是哪一层。
- 复杂结构关系用图更快建模。
- 边界、前提和例外用字更准确。
- 图字结合能同时提高可见性和严谨性。
- 系统架构图。
- 流程图配说明。
- 汇报图表配判断。

## 事故复盘怎么写得客观有效

- 复盘的目标是帮助系统学习，不是寻找替罪羊。
- 事实线、影响面、触发条件和改进动作必须分开。
- 机制修复比态度表态更重要。
- 事实先行能降低情绪偏差。
- 根因拆解帮助识别系统问题。
- 改进项明确才能形成长期收益。
- 线上故障复盘。
- 发布事故。
- 数据污染事件。

## 技术方案怎么讲清 trade-off

- 讲 trade-off 是为了提高可信度，不是为了显得专业。
- 任何方案都在成本、复杂度、风险和扩展性之间权衡。
- 不讲代价的方案，往往最不稳。
- 约束不同，最优方案也不同。
- 比较替代方案能暴露真实选择。
- 诚实交代代价能提升评审质量。
- 性能换复杂度。
- 速度换稳定性。
- 自研换外部依赖。

## 技术文档为什么总是难读

- 技术文档常见问题是对象不清、前提缺失和结构失衡。
- 作者理解太深时，更容易高估读者背景。
- 技术文档首先是一种服务，而不是炫耀。
- 缺地图会让读者迟迟进不去。
- 术语堆叠会遮住主线。
- 没有边界和例子时，可用性下降。
- 架构说明。
- 接口文档。
- 内部工具使用说明。

## 图表、流程图与架构图如何服务表达

图不是装饰，而是降低读者理解关系的成本。需要比较时用表；展示时间顺序用流程/时序图；展示组件和依赖用架构图；展示状态转换用状态机。先写一句图要回答的问题，再决定图形。
一张图只保留一个阅读层级：标出入口、关键路径、边界、失败点和数据/控制流；避免把全部实现细节塞进图里。颜色、箭头和图例应有稳定语义，读者不该猜“箭头代表调用还是数据”。
交付前让不了解背景的人用图复述流程；若无法复述，先删减或拆图，而不是继续加标签。

## 写给工程师和写给老板有什么不同

- 工程师更关心机制和边界，老板更关心目标、风险和推进条件。
- 同一件事要换组织方式，不是换事实。
- 对象不同，信息层级必须随之变化。
- 决策者关心能否拍板。
- 执行者关心能否落地。
- 表达层级错位会让信息价值下降。
- 项目周报。
- 系统方案。
- 异常情况同步。

## 决策记录与结论留痕

决策记录（ADR）保存“当时为什么这样选”，避免未来只看到结果却无法理解约束。它不替代设计文档：设计文档说明方案全貌，ADR 记录一个重要且可争议的取舍。
最小模板：背景｜决策｜备选方案｜取舍与后果｜生效日期/owner｜何时复审。记录事实和假设，明确哪些结论依赖成本、流量、团队能力或外部规则；被新事实推翻时写 superseded，而不是悄悄改历史。
好的 ADR 简短、可检索、链接相关代码/监控/事故复盘，并让后来者知道什么不能轻易改。

## 如何写接口文档

- 接口文档首先回答怎么接、会出什么错、边界在哪里。
- 字段、示例、错误码和兼容性说明经常比概述更重要。
- 好接口文档能显著降低协作摩擦。
- 接入方需要先有可执行信息。
- 边界不清会引发大量额外沟通。
- 示例可以快速降低误解概率。
- HTTP API。
- SDK 接入说明。
- 内部服务接口。

## 复杂流程如何用文字解释清楚

- 流程难讲，不在步骤多，而在依赖和异常分支多。
- 先总后分，再补例外，是更稳定的讲法。
- 文字解释流程时，顺序感比辞藻更重要。
- 总流程先给地图。
- 关键节点再细化。
- 异常分支最后补充，避免主线被淹没。
- 审批流。
- 发布流程。
- Agent 调用链路。

## 把系统设计写成别人能复用的知识

- 项目经验只有被抽象成问题、约束和模式后，才会变成知识。
- 只写结果，复用性很低；写出选择过程，复用性才高。
- 别人能不能拿去参考，是很好的检验标准。
- 抽象问题会提升迁移性。
- 保留取舍过程会提高可解释性。
- 统一术语和框架更利于沉淀。
- 项目架构总结。
- 故障经验沉淀。
- 中台能力设计复盘。

## 技术表达里的边界、假设与风险提示

- 成熟表达不只讲能做什么，也讲不能做什么。
- 假设说清楚，后续争议会少很多。
- 风险提示不是自保，而是高质量沟通的一部分。
- 边界限制过度承诺。
- 假设帮助他人判断适用范围。
- 风险提示能改善协作预期。
- 设计文档。
- 项目计划。
- 技术方案评审。
## 使用方法

1. 先写清对象、目标和希望对方采取的行动。
2. 用一句结论和不超过三个一级要点建立骨架。
3. 补足证据、边界、责任和下一步，不用形容词代替事实。
4. 删除不服务目标的背景，并让不了解上下文的人试读。
5. 重要沟通留下可追溯记录，事后根据真实反馈复盘。

这是一份 `reference` 型速查，不代替模块中的完整主文档。需要形成独立文章时，应补充真实材料、修改前后对照和失败案例，而不是复制本页模板。
