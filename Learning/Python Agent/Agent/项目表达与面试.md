# AI 项目表达与面试

本页只补 AI 应用的专项追问；通用的业务目标、个人边界、难点、取舍和证据结构统一看 [Career/项目表达](../../Career/项目表达.md)。

## AI 项目的表达骨架

```text
业务问题 → 个人负责边界 → 架构与关键链路
→ 一个最难的工程问题 → 方案取舍 → 指标/测试/失败证据
```

不要只说“用了 RAG/Agent/MCP”，要说数据如何进入系统、模型如何决策、工具如何执行、权限和失败如何控制，以及结果如何验证。

## RAG 项目追问

至少能说明：解析与切分边界、embedding/关键词的检索组合、重排或 query rewrite、权限过滤、grounding 约束和评测集。

- chunk size 通过 Recall@k 或任务成功率比较，不凭经验拍定；代码按函数/类边界切，长文按语义段落切。
- 混合检索解决专有名词和语义召回的互补问题；用固定测试集比较纯向量、BM25 和融合结果。
- Faithfulness 低时先检查召回证据，再加引用/grounding 约束、拒答策略和人工抽样，不用更强 prompt 掩盖数据问题。

## Agent 项目追问

至少能说明：循环或图编排、工具 schema、状态持久化、预算、人工确认、失败恢复和评测指标。

- 简单任务可用 while loop；需要条件分支、checkpoint、人工中断和多节点协作时再引入图编排。
- 工具失败应结构化返回错误；幂等操作可限次退避重试，扣款、发信、写入等副作用默认不自动重试。
- 评测同时看 task success、tool accuracy、步骤数、成本和延迟；每次改 prompt、schema 或模型都回归。

## AI 系统设计题

1. 先澄清用户、数据、延迟、成本、权限和可用性目标。
2. 画输入、上下文装配、模型、工具、状态、观测和人工兜底链路。
3. 说明模型输出不是执行权限；服务端负责 schema、策略、租约、审计、取消和回滚。
4. 说明数据漂移、模型升级、prompt 变更和依赖故障如何发现与恢复。

高频题型：企业知识库 RAG、多 Agent 协作、工具安全、Agent 成本控制、生产质量漂移、MCP 网关和 Computer Use 沙箱。

## 面试前准备的数字

准备真实或明确标注为估算的：测试集规模、Recall/成功率、P95/P99、token/请求、月成本、并发量、重试率、最重要的优化和当前瓶颈。没有记录就直接说未观测，不要编造线上结果。
