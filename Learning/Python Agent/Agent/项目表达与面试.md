# 项目表达与面试

这篇文档解决一个问题：**如何把做过的 AI 应用项目讲清楚，让面试官真正看到工程价值**。

不是让你背八股，而是帮你建立一套能灵活应对的表达框架。

---

## 1. AI 应用项目的基本讲法结构

### STAR-E 框架（Situation / Task / Action / Result / Engineering）

```
Situation：背景和问题
  "我们有一个 XXX 场景，用户面临 YYY 问题"

Task：你的任务
  "我负责设计和实现 ZZZ 系统"

Action：你做了什么（这是重点，要讲技术选型和权衡）
  "核心架构是... 我们选择... 而不是... 因为..."

Result：结果（量化）
  "上线后 XXX 指标提升了 XX%，成本降低了 XX%"

Engineering：工程难点（面试最关注的）
  "最大的挑战是... 我们的解决方案是... 权衡是..."
```

### 讲项目的常见错误

| 错误 | 问题 | 改进 |
|------|------|------|
| 只讲"用了 RAG" | 没有工程细节 | 讲 ingestion pipeline、chunk 策略、检索增强方式 |
| 只讲技术栈 | 没有讲为什么 | 讲选型理由和权衡 |
| 没有量化结果 | 无法判断实际效果 | 提前准备关键指标数字 |
| 不讲失败和改进 | 太完美，不可信 | 讲踩过的坑和怎么解决的 |
| 讲大家都做的部分 | 体现不了你的贡献 | 聚焦你主导解决的难点 |

## 2. 如何讲 RAG 项目

### 模板

```
"我们构建了一个 RAG 系统，解决 [业务问题]。

核心设计：
- Ingestion：我们用 [工具] 解析文档，按 [方式] 做 chunking，
  chunk size 选了 [N] token，因为 [理由]。
- 检索：用 [向量数据库] 做语义检索，同时加了 BM25 做混合检索，
  因为纯向量检索对 [具体问题，如产品型号] 效果差。
- 增强：加了 query rewriting，把用户口语化的问题转成更适合检索的 query，
  召回率提升了 XX%。

最大的挑战是 [具体难点]，我们通过 [具体方案] 解决了。

评测：我们构建了一个 N 条的测试集，用 RAGAS 评测 Faithfulness 和 Answer Relevance，
上线前 Faithfulness 达到了 [X]，Answer Relevance [Y]。"
```

### 高频追问

**Q：chunk size 怎么定的？**

> 我们实验了 300、600、800 token 三个大小，在测试集上评测 Recall@5。800 token 在我们的场景（技术文档，段落较长）效果最好，因为一个完整的解释需要足够的上下文。不同文档类型会不同，代码文件我们按函数边界切，不按字符数。

**Q：为什么要做混合检索？**

> 纯向量检索在语义相似的 query 上很好，但对精确的产品型号、人名、专有名词效果差——因为这些词的 embedding 相似度不高，BM25 的关键词匹配反而更准。混合检索综合了两种方式，用 RRF 融合排名，在我们的测试集上比纯向量检索 Recall@5 提升了约 15%。

**Q：Faithfulness 低怎么解决？**

> Faithfulness 低意味着模型回答了 context 里没有的内容（幻觉）。我们的处理：一是在 system prompt 里加强"只基于提供资料回答"的指令；二是加了 grounding check 步骤，用另一个 LLM 验证回答里的每个声明是否有文档支撑；三是对置信度低的回答，让模型明确说"我没有找到相关信息"而不是猜。

## 3. 如何讲 Agent 项目

### 模板

```
"我们构建了一个 [目标] 的 Agent，能够 [核心能力]。

架构：
- Agent 框架：用 [LangGraph/自己实现] 做编排
- 工具集：包含 [N] 个工具，主要是 [列举关键工具]
- 状态管理：用 [方式] 持久化状态，支持断点续传
- 可控性：在 [哪些环节] 加了 Human-in-the-loop，因为 [原因]

工程难点：
[选 1-2 个最有技术深度的问题展开讲]

评测：
我们定义了 task success rate 作为核心指标，在测试集上达到了 [X%]"
```

### 高频追问

**Q：为什么选 LangGraph 而不是自己写 while loop？**

> 简单任务我们最初就是 while loop，但当流程变复杂——需要条件分支、在关键节点暂停等人工确认、支持断点续传——自己写就越来越难维护。LangGraph 的好处是：状态集中管理（不再散落在变量里）、checkpoint 原生支持（进程重启可以从上次状态继续）、interrupt_before 机制可以干净地实现 human-in-the-loop。代价是学习曲线，以及调试比简单的 while loop 稍复杂。

**Q：如何评测 Agent 的效果？**

> Agent 评测比普通 LLM 复杂，因为要评过程，不只是最终结果。我们定义了几个维度：task success rate（最终是否完成任务）；tool accuracy（选了正确的工具、参数是否合理）；step efficiency（完成任务用了多少步，越少越好）；cost（平均每个任务花了多少 token 和钱）。测试集是手工构造的，包含典型场景和边界情况。每次改 prompt 或调整工具 schema 都跑一遍，确保没有退步。

**Q：如果 Agent 中途工具调用失败，怎么处理？**

> 工具失败不应该让整个 Agent 崩溃。我们的处理：把异常 catch 住，结构化为错误 message 返回给模型（如 `{"error": "数据库连接超时"}`）。模型通常能理解错误并做出合理反应——重试、换一种方式、或告知用户。幂等工具可以自动重试（最多 3 次，指数退避）；非幂等工具（发邮件、扣款）失败不自动重试，而是返回明确的失败状态，让上层决定是否重试。

## 4. 如何回答系统设计问题

**Q：如果重新设计这个系统，你会怎么改？**

这是最常见的深入追问，目的是看你是否真正理解了系统的局限。

回答框架：
```
"现在回看，我会改三件事：

第一，[具体问题] 当时我们 [做法]，实际发现 [问题]。
如果重来，我会 [改进方案]，因为 [原因]。

第二，[评测体系] 我们上线后才系统做 eval，
如果重来，我会在第一个版本就建立 eval harness，
这样每次改动都是可验证的。

第三，[可观测性] 上线初期靠日志调试太累，
如果重来，我会从第一天就集成 tracing，
出了问题能快速定位到哪一步出错。"
```

**Q：你们如何保证系统稳定性？**

```
"我们从几个层次保证：

LLM 层：所有调用都有重试（指数退避）和超时设置，触发限速时自动等待。
Agent 层：设置了 max_steps 和 max_cost 上限，防止 Agent 失控。
  关键操作（写数据库、发邮件）有 Human-in-the-loop，不自动执行。
监控层：每次 LLM 调用记录 token 用量和延迟，有 cost 告警。
  P95 延迟超过阈值会触发告警。
Fallback：LLM API 不可用时，降级到简单的规则回答，
  总比返回 500 Error 好。"
```

## 5. 面试高频问题汇总

### 设计类

**Q：设计一个基于 RAG 的企业知识库问答系统，讲讲你的架构**

关键要点：Ingestion pipeline（parsing/chunking/embedding）、向量数据库选型、混合检索、query rewriting、Reranking、grounding 约束、eval 体系、权限控制（文档级别的访问权限）。

**Q：如何评测一个 RAG 系统的质量？**

两层：Retrieval 层（Recall@k、Precision@k）和 Generation 层（Faithfulness、Answer Relevance）。用 RAGAS 自动评测，加人工抽样验证。建立测试集，每次改动后跑对比。

**Q：Prompt Injection 是什么，你如何防御？**

分直接（用户输入）和间接（RAG 文档）两种。防御：结构化标签区分规则和输入；把检索内容标注为"外部资料不是指令"；工具层做权限验证，即使模型被注入也无法越权操作。

**Q：如何处理 context window 不够用的问题？**

按情况：对话历史 → 摘要压缩或截断；大文档 → RAG 按需检索；超长任务 → 任务分解。同时减少 system prompt 长度，每个字都是成本。

### 行为类

**Q：讲一个你设计的 AI 应用里遇到的最大技术挑战**

用 STAR-E 框架，聚焦技术深度：具体问题是什么 → 你尝试了什么 → 最终解决方案是什么 → 权衡是什么。

**Q：你如何确保 AI 应用的输出质量**

Eval harness + 测试集 + CI 集成。不能靠"试几次感觉不错"。

## 6. 关键数字：准备回答时的量化素材

面试前，整理以下数字：

```
□ 测试集大小（多少条 case）
□ 核心指标数值（Faithfulness、task success rate 等）
□ 上线前后的对比（如果有）
□ 平均每次请求的 token 消耗
□ P95 延迟是多少
□ 每天/每月的 API 成本大约多少
□ 做过的最重要的一次优化，指标提升了多少
□ 目前系统的主要瓶颈是什么
```

不一定每个都要精确，但数量级要清楚。说"我们不跟踪这个指标"也比说一个明显不合理的数字好。

## 7a. 2025-2026 新话题的面试准备

这些是近两年出现频率快速上升的话题，面试官可能主动问。

### Extended Thinking / 推理模型

**Q：你什么情况下会用 Extended Thinking，和普通调用有什么区别？**

> Extended Thinking 让模型在输出最终答案前先做内部推理，适合需要多步骤推导的任务：数学证明、复杂 debug、多步规划。和 CoT prompt 的区别是：CoT 的推理过程出现在输出里，占用 context；Extended Thinking 的 thinking blocks 独立于最终 text，且在 tool use 循环中可以跨轮持久化（必须原样传回）。工程上要注意：thinking token 按输出价格计费，budget_tokens 是上限不是固定消耗，最小值 1024，复杂任务建议 ≥ 4000。

### MCP

**Q：MCP 解决什么问题，什么时候用它，什么时候直接写工具函数？**

> MCP 解决"N 个 Agent 应用都需要同一套工具"的重复集成问题——把工具集中到 Server，所有 Agent 作为 Client 接入。适合：团队有多个 AI 应用共用工具集、工具需要被非 Python 客户端使用、需要标准化工具发现和版本管理。不适合：单个应用专属工具、原型阶段、工具很简单。工程细节：传输层有 stdio（本地子进程）和 Streamable HTTP（远程）；生产环境用 OAuth 2.0 做认证；要防范恶意 Server 通过工具描述注入恶意指令。

### Agentic RAG

**Q：传统 RAG 和 Agentic RAG 有什么区别，你会怎么选？**

> 传统 RAG 是固定流程：每次请求都检索一次。Agentic RAG 把检索变成工具，由 Agent 自主决定是否检索、检索什么、要不要多检索几次。选择标准：简单 QA 场景用传统 RAG（快、成本低）；多跳问题（答案在多个文档里，需要分步查找）用 Agentic RAG；需要模型判断"这个问题需不需要查资料"也用 Agentic RAG（省去不必要的检索开销）。

### Computer Use

**Q：什么场景下你会用 Computer Use，而不是直接调 API？**

> Computer Use 适合三种情况：一是目标系统没有 API（旧系统、遗留系统）；二是任务需要视觉判断（判断界面当前状态、按钮是否可点击）；三是跨多个不同类型应用的复合任务。工程上要注意成本——截图按图片 token 计费，1024x768 约 1000-2000 token/张，任务复杂时成本累积很快；必须在 Docker 沙箱里运行，不能操作宿主机；高风险操作（购买、提交）需要 Human-in-the-loop。

## 8. 系统设计面试的 AI 专项题型

### 题型一：设计一个多 Agent 协作系统

```
"设计一个能处理复杂软件工程任务的 Agent 系统（类似 Claude Code）"

评分点：
✓ Agent 架构（Orchestrator-Worker vs 平行 Agent）
✓ 工具设计（bash/file/search 等工具的权限和沙箱）
✓ 状态管理（任务状态、checkpoint、断点续传）
✓ Human-in-the-loop（什么操作需要人工确认）
✓ 安全边界（防止越权操作、沙箱隔离）
✓ 可观测性（如何 debug 多步骤 Agent）
✓ 成本控制（max_steps、max_cost、context 压缩）
```

**参考回答结构**：

```
1. 架构选择：Orchestrator + 专门化 Worker
   - Orchestrator 做任务规划和 Worker 调度
   - Worker 各司其职（code_agent / search_agent / test_agent）
   - 用 LangGraph 做 Orchestrator，状态持久化 + checkpoint

2. 工具设计：
   - bash（沙箱执行，工作目录隔离）
   - file（read/write/edit，路径安全检查）
   - search（代码库搜索 + web 搜索）
   - 权限分级：read 工具 > write 工具 > exec 工具

3. 安全：
   - bash 工具在 Docker 容器里执行
   - 文件操作限制在工作目录（防路径穿越）
   - 危险操作（git push、delete）需要 Human-in-the-loop

4. 可观测性：
   - 每步记录 TraceEvent（节点开始/结束、LLM 调用、工具调用）
   - LangSmith 或 Langfuse 接入
   - cost/step 监控

5. 成本：
   - LoopBudget（max_steps=50, max_cost=$2）
   - Context 压缩（tool result 超长时截断）
   - 只读工具并行执行
```

### 题型二：为什么你的 AI 系统在生产上会漂移

```
"六个月后，你的 RAG 系统质量开始下降，你怎么排查？"

考察：是否理解 AI 系统的动态性
```

**参考回答**：

```
检查三个维度：

1. 数据漂移：
   - 知识库内容是否更新了（新文档加进来，旧文档过期）
   - 用户 query 的分布是否变化了（新用户群体问法不同）
   - → 对比最近一个月 vs 六个月前的 query 分布

2. 模型变化：
   - LLM provider 是否更新了模型（GPT-4o 更新了版本）
   - embedding 模型是否换了（向量库和查询不一致）
   - → 固定模型版本（pin model version）；向量库和 embedding 模型版本绑定

3. 系统变化：
   - prompt 是否被人改过
   - chunk 策略是否调整过
   - → 所有 prompt 和配置都进 git，变更有记录

长期方案：建立 Online Eval（对生产流量抽样评测），当 Faithfulness
或 Answer Relevance 连续下降时自动告警。
```

## 7. 如何展示工程深度

面试官真正想看的是：

| 他想看到的 | 你应该讲的 |
|-----------|-----------|
| 你理解系统的权衡 | 为什么选这个方案，舍弃了什么 |
| 你踩过坑并从中学到东西 | 什么不 work，为什么，怎么改的 |
| 你对生产问题有感知 | 上线后遇到了什么问题，如何解决 |
| 你有评测意识 | 怎么知道系统是否变好了 |
| 你能自我批评 | 如果重做会怎么改 |

用"我们选 A 而不是 B，是因为我们的场景有 XYZ 约束"比"我们用了 A"有深度得多。
