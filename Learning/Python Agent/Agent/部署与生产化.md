# 部署与生产化

这篇文档解决一个问题：**AI 应用从本地跑通到稳定上线，需要处理哪些工程问题**。

---

## 1. 部署模式选择

| 模式 | 适合场景 | 优点 | 缺点 |
|------|---------|------|------|
| **Serverless（Cloud Functions）** | 低频调用、短任务 | 按用量付费，无需管服务器 | 冷启动延迟、执行时间有上限 |
| **容器（Docker + K8s/ECS）** | 稳定流量、需要更多控制 | 灵活，可扩展，无执行时间限制 | 需要运维知识 |
| **PaaS（Railway, Render, Fly.io）** | 中小规模应用 | 部署简单，有一定弹性 | 成本比纯云稍高 |
| **GPU 实例** | 需要本地推理（非 API 调用）| 成本低于 API（高吞吐场景）| 需要管理 GPU 实例 |

**大多数 AI 应用（调用 LLM API 的）**：容器化部署到 ECS/Cloud Run，配合 Auto Scaling。

## 2. Docker 容器化

### Dockerfile 最佳实践

```dockerfile
# 使用具体版本，不用 latest
FROM python:3.11-slim

# 设置工作目录
WORKDIR /app

# 先 COPY 依赖文件，利用 layer cache
COPY pyproject.toml uv.lock ./
RUN pip install uv && uv sync --frozen --no-dev

# 再 COPY 代码
COPY src/ ./src/

# 非 root 用户运行（安全）
RUN useradd -m -u 1000 appuser
USER appuser

# 健康检查
HEALTHCHECK --interval=30s --timeout=10s --start-period=10s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

EXPOSE 8000
CMD ["python", "-m", "uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 健康检查端点

Liveness 和 Readiness 要分开，**不要把外部 LLM API 加入 readiness probe**——provider 短暂波动会让 K8s 把你自己的容器误杀，还会产生额外的 API 请求成本。

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/health/live")
async def liveness():
    """Liveness probe：进程存活即返回 200。
    K8s 靠这个判断是否重启容器——只要进程没崩就应该返回成功。"""
    return {"status": "alive"}

@app.get("/health/ready")
async def readiness():
    """Readiness probe：验证本地依赖是否就绪。
    K8s 靠这个决定是否向容器分配流量——失败时摘流量，不重启。
    只检查本服务控制范围内的依赖（数据库、缓存），不检查外部 LLM API。"""
    checks = {}
    
    try:
        await db.execute("SELECT 1")
        checks["database"] = "ok"
    except Exception as e:
        checks["database"] = f"error: {e}"
    
    try:
        await redis_client.ping()
        checks["redis"] = "ok"
    except Exception as e:
        checks["redis"] = f"error: {e}"
    
    all_ok = all(v == "ok" for v in checks.values())
    status_code = 200 if all_ok else 503
    return JSONResponse(
        status_code=status_code,
        content={"status": "ready" if all_ok else "not_ready", "checks": checks},
    )

@app.get("/health/diagnose")
async def diagnose():
    """诊断端点：检查包括 LLM API 在内的所有外部依赖。
    不挂到 K8s probe，仅供运维人工触发排查。"""
    from httpx import AsyncClient, TimeoutException
    checks = {}
    async with AsyncClient(timeout=5.0) as http:
        try:
            resp = await http.get(
                "https://api.openai.com/v1/models",
                headers={"Authorization": f"Bearer {settings.openai_api_key}"},
            )
            checks["llm_api"] = "ok" if resp.status_code == 200 else f"http_{resp.status_code}"
        except TimeoutException:
            checks["llm_api"] = "timeout"
        except Exception as e:
            checks["llm_api"] = f"error: {type(e).__name__}"
    return checks
```

**K8s 配置对应方式**：

```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 8000
  initialDelaySeconds: 10
  periodSeconds: 30

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8000
  initialDelaySeconds: 5
  periodSeconds: 10
```

## 3. 环境与配置管理

### 配置分层

```
.env.local          ← 本地开发，不提交 git
.env.development    ← 开发环境（可提交，无真实密钥）
.env.production     ← 生产环境（通过 secret manager 注入，不提交 git）
```

### 用 Pydantic Settings 管理配置

```python
from pydantic_settings import BaseSettings
from typing import Optional

class Settings(BaseSettings):
    # LLM
    openai_api_key: str
    anthropic_api_key: Optional[str] = None
    default_model: str = "gpt-4o"
    
    # 数据库
    database_url: str
    redis_url: str = "redis://localhost:6379"
    
    # 功能开关
    enable_caching: bool = True
    enable_tracing: bool = True
    
    # 限制
    max_tokens_per_request: int = 4000
    max_concurrent_requests: int = 50
    
    class Config:
        env_file = ".env"
        env_file_encoding = "utf-8"

settings = Settings()  # 启动时验证所有必需的环境变量
```

**生产密钥管理**：使用 AWS Secrets Manager、GCP Secret Manager 或 HashiCorp Vault，不要把密钥放在代码仓库。

## 4. Prompt 版本管理

Prompt 是 AI 应用的"代码"，需要像代码一样版本管理。

### 文件化 Prompt

```
prompts/
  v1/
    system_prompt.txt
    tool_descriptions/
      search_tool.txt
      order_tool.txt
  v2/
    system_prompt.txt
    ...
  current -> v2/  # 符号链接指向当前版本
```

```python
from pathlib import Path

class PromptLoader:
    def __init__(self, prompts_dir: str = "prompts", version: str = "current"):
        self.base_path = Path(prompts_dir) / version
    
    def load(self, name: str, **variables) -> str:
        template = (self.base_path / f"{name}.txt").read_text()
        return template.format(**variables)

loader = PromptLoader()
system_prompt = loader.load("system_prompt", product_name="MyApp")
```

### Prompt 变更流程

```
1. 新建 prompts/v3/ 目录，复制当前版本
2. 修改 prompt 文件
3. 在 staging 环境用新版本 prompt 跑 eval
4. 对比 v2 和 v3 的 eval 结果
5. 通过 → 切换 current 符号链接
6. 保留旧版本（可以快速回滚）
```

## 5. 版本管理与灰度发布

### 灰度发布

```python
import hashlib
import random

class FeatureFlag:
    def __init__(self):
        self._flags = {}
    
    def is_enabled(self, flag_name: str, user_id: str, rollout_pct: float = 0.0) -> bool:
        """
        rollout_pct: 0.0 = 关闭，1.0 = 全量开放
        对同一个 user_id 结果稳定（不会本次是A下次是B）
        """
        if rollout_pct == 0.0:
            return False
        if rollout_pct == 1.0:
            return True
        
        # 用 user_id hash 确保同一用户始终在同一组
        hash_val = int(hashlib.md5(f"{flag_name}:{user_id}".encode()).hexdigest(), 16)
        return (hash_val % 100) < (rollout_pct * 100)

feature_flags = FeatureFlag()

async def handle_request(user_input: str, user_id: str):
    if feature_flags.is_enabled("new_agent_v2", user_id, rollout_pct=0.1):  # 10% 用户
        return await new_agent_v2.run(user_input)
    else:
        return await current_agent.run(user_input)
```

## 6. 生产 Checklist

### 上线前必查

```
基础安全
□ 所有密钥通过环境变量注入，不在代码里硬编码
□ API 端点有鉴权（Bearer token / API Key / OAuth）
□ 有请求频率限制（rate limiting）
□ 用户输入有基本验证和长度限制
□ 工具调用有权限边界

可靠性
□ LLM 调用有重试和超时设置
□ 工具调用有错误处理，失败不会让整个请求崩溃
□ Agent 有 max_steps 和 max_cost 上限
□ 有 fallback 策略（LLM API 挂了有降级）
□ 数据库连接有连接池

可观测性
□ 结构化日志（JSON 格式）
□ 每次 LLM 调用记录 token 用量
□ 有错误告警（出了 5xx 会通知）
□ 有 cost 监控（每日/每月花了多少）
□ 有 latency 监控（P95 延迟是多少）

运维
□ 健康检查端点
□ 优雅关闭（SIGTERM 时完成当前请求再停）
□ 容器化，有 resource limits（CPU/内存）
□ 可以快速回滚（prompt 版本、代码版本）
□ 环境变量齐全（在新环境能启动）

测试
□ 核心功能有 eval 测试集
□ 有回归测试，新 PR 会跑
□ staging 环境与 production 配置一致
□ 做过负载测试（知道系统能承受多少并发）
```

## 7. 优雅关闭

```python
import signal
import asyncio

class GracefulShutdown:
    def __init__(self):
        self.shutdown_event = asyncio.Event()
        signal.signal(signal.SIGTERM, self._handle_sigterm)
        signal.signal(signal.SIGINT, self._handle_sigterm)
    
    def _handle_sigterm(self, signum, frame):
        self.shutdown_event.set()
    
    async def wait_for_shutdown(self):
        await self.shutdown_event.wait()

# 在 FastAPI 里
@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动时
    await startup_tasks()
    yield
    # 关闭时：等待当前请求完成
    await asyncio.gather(*[t for t in asyncio.all_tasks() if t is not asyncio.current_task()])
    await cleanup_tasks()
```

## 8. 面试高频

**Q：AI 应用部署时有哪些特别需要注意的地方？**

> 相比普通服务，几个额外关注点：一是 LLM API 调用超时比普通 HTTP 请求长得多（可能 30-60 秒），连接池和超时配置要调整；二是成本监控，LLM API 按 token 计费，没有监控容易超预算；三是 prompt 版本管理，prompt 变了行为就变了，需要像代码一样管理版本和回滚；四是优雅关闭，LLM 调用可能正在执行，SIGTERM 时要等当前请求完成；五是 Agent 的 max_steps/max_cost 限制，防止 Agent 失控消耗大量资源。

**Q：如何做 prompt 的版本管理和灰度发布？**

> Prompt 文件化（放 prompts/ 目录下，按版本号组织），和代码一起进 git。变更流程：新建版本目录 → staging 环境跑 eval 对比 → 通过后切换 current 指向 → 旧版本保留用于快速回滚。灰度发布用 feature flag，根据用户 ID 的 hash 决定走新版本还是旧版本，同一用户结果稳定。上线后观察 eval 指标、错误率、用户反馈，确认没有退步再全量。
