# 模型行为与工具调用训练

这篇笔记是研究分支，回答：**模型为什么会调用工具、为什么会拒绝工具、什么时候应该改 Prompt，什么时候应该换模型或训练模型**。

它不是 Coding Agent 应用开发的前置条件。做 Harness 时，先用现成模型和可靠工具把系统跑通；只有当模型能力成为瓶颈时，再进入这一章。

## 1. 能力来源的分层

```text
预训练（Pretraining）
  → 语言、代码、世界知识和模式识别

SFT / Instruction Tuning
  → 按指令回答、结构化格式、工具调用示例

Preference / RL / RFT
  → 更符合偏好、更会规划、更愿意修正或遵守边界

Harness
  → 提供工具、环境、上下文、权限、反馈和停止条件
```

不要把“Agent 行为”全部归因于 Prompt，也不要把所有问题都归因于模型训练。最终行为是模型、Harness 和环境共同作用的结果。

## 2. Tool Use 训练轨迹

一条工具调用训练样本通常包含：

```text
用户任务
→ 模型选择工具
→ 生成参数
→ 工具结果
→ 模型根据结果继续调用或结束
```

高质量轨迹应包含：

- 为什么此时需要工具；
- 工具参数和约束；
- 成功、失败、超时和权限拒绝；
- 何时请求澄清或人工确认；
- 何时停止，而不是继续调用。

只收集成功轨迹会让模型学会“总是调用工具”，却不会学会处理失败和不确定性。

## 3. Prompt、Tool Schema 还是训练？

按成本从低到高排查：

```text
协议错误 / schema 错误
  → 修复消息和参数格式

工具选择错误
  → 改 name、description、可用工具集合

参数经常错误
  → 补 enum、required、示例和服务端校验

上下文缺失
  → 改 context 选择、检索和状态注入

模型知道规则但经常违反
  → 增加对抗样本、换模型或考虑训练

任务本身不适合当前模型
  → 改流程设计或加入确定性验证器
```

不要一看到失败就微调模型。先用 trace 判断失败发生在模型决策、context、工具协议、权限还是环境执行层。

## 4. 结构化输出的边界

JSON Schema、函数调用和 Pydantic 可以约束格式，但不能保证：

- 工具选择正确；
- 参数符合业务语义；
- 结果真实；
- 没有越权；
- 任务已经完成。

因此工具层仍需要业务校验，结果层仍需要程序验证，权限层不能由模型自行决定。

## 5. 推理与工具调用的取舍

更长的 reasoning budget 不等于更高的任务成功率。要观察：

| 变量 | 可能收益 | 可能代价 |
|---|---|---|
| 更长推理 | 复杂规划和错误修正 | 成本、延迟、循环风险 |
| 更多工具 | 能力范围扩大 | 选择困难、攻击面扩大 |
| 更多历史 | 保留背景 | context 噪声和成本增加 |
| 更强模型 | 复杂任务能力更好 | 价格、吞吐和可用性 |

用 Eval 做路由，而不是凭印象选择。简单任务可以使用轻量模型；复杂任务需要更多预算；高风险动作必须经过确定性策略和人工控制。

## 6. 何时值得训练

比较适合训练或微调的场景：

- 工具集合固定且调用格式稳定；
- 领域术语和输出格式高度重复；
- 有大量高质量、获授权的任务轨迹；
- 已经有 Eval，可以证明训练前后的行为差异；
- Prompt、schema、context 和工作流优化仍然不足。

不适合贸然训练的场景：

- 任务定义还在快速变化；
- 没有可靠标注或只保存最终答案；
- 失败原因尚未定位；
- 需要的是新工具或新数据，而不是新行为；
- 没有安全回归集。

## 7. 训练数据的安全要求

训练轨迹可能含有密钥、个人信息、专有代码和危险操作。进入数据集前需要：

1. 明确数据授权和用途；
2. 脱敏并删除不需要的秘密；
3. 标记工具权限和审批状态；
4. 保留失败、拒绝和澄清样本；
5. 将训练集、验证集和安全攻击集分离；
6. 训练后做行为、泄露和越权回归。

## 8. 练习与验收

先不训练模型，完成一个行为诊断实验：

1. 为同一个工具设计三种 schema；
2. 为每种 schema 准备正常、歧义、错误和越权任务；
3. 记录工具选择、参数错误、澄清率、成本和延迟；
4. 用 trace 判断问题属于 schema、context、模型还是工具层；
5. 只有在问题稳定属于模型行为后，才设计一组最小训练数据。
