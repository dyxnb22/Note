# 可观测性与调试

这篇文档解决一个问题：**AI 应用上了生产，出了问题怎么看，怎么找，怎么复现**。

AI 应用的可观测性比传统软件更难——因为 LLM 的推理过程不透明，输出是概率性的，调用链可能很长（multi-step agent），中间的任何环节都可能出问题。

---

## 1. AI 应用 Observability 的特殊挑战

传统软件可观测性主要关注：延迟、错误率、吞吐量。

AI 应用还需要关注：

| 额外挑战 | 说明 |
|---------|------|
| **LLM 输出质量** | 回答对不对？格式是否符合预期？ |
| **Tool 选择准确性** | 模型选了正确的工具吗？ |
| **Context 质量** | 发给模型的 context 是否包含正确信息？ |
| **多步骤追踪** | Agent 执行了 10 步，哪一步出的问题？ |
| **Cost 追踪** | 每次调用花了多少钱？哪个功能最贵？ |
| **幻觉检测** | 回答中是否包含不在 context 里的信息？ |

---

## 2. Tracing：让 AI 调用链可见

Tracing 是把一个完整请求（可能包含多个 LLM 调用、工具调用、检索步骤）串起来记录，形成完整的调用树。

### Span 结构

```
Trace（一次完整请求）
  └─ Span: user_request
       └─ Span: intent_classification (LLM call)
       └─ Span: retrieve_context (vector search)
       └─ Span: agent_loop
            └─ Span: llm_call_1 (with tool calls)
            └─ Span: tool_execution: search_orders
            └─ Span: llm_call_2 (final answer)
```

### 自己实现基础 Tracing

```python
import time
from dataclasses import dataclass, field
from typing import Optional
from datetime import datetime
import uuid

@dataclass
class Span:
    span_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    name: str = ""
    parent_id: Optional[str] = None
    start_time: float = field(default_factory=time.time)
    end_time: Optional[float] = None
    metadata: dict = field(default_factory=dict)
    error: Optional[str] = None
    
    def finish(self, error: str = None):
        self.end_time = time.time()
        self.error = error
    
    @property
    def duration_ms(self) -> Optional[float]:
        if self.end_time:
            return (self.end_time - self.start_time) * 1000
        return None

class Tracer:
    def __init__(self, trace_id: str = None):
        self.trace_id = trace_id or str(uuid.uuid4())
        self.spans: list[Span] = []
        self._current_span_id: Optional[str] = None
    
    def start_span(self, name: str, metadata: dict = None) -> Span:
        span = Span(
            name=name,
            parent_id=self._current_span_id,
            metadata=metadata or {},
        )
        self.spans.append(span)
        self._current_span_id = span.span_id
        return span
    
    def finish_span(self, span: Span, error: str = None):
        span.finish(error=error)
        self._current_span_id = span.parent_id
    
    def to_dict(self) -> dict:
        return {
            "trace_id": self.trace_id,
            "spans": [
                {
                    "id": s.span_id,
                    "name": s.name,
                    "parent_id": s.parent_id,
                    "duration_ms": s.duration_ms,
                    "metadata": s.metadata,
                    "error": s.error,
                }
                for s in self.spans
            ]
        }

# 使用示例
tracer = Tracer()

span = tracer.start_span("llm_call", metadata={"model": "gpt-4o"})
try:
    response = client.chat.completions.create(...)
    span.metadata["tokens"] = response.usage.total_tokens
    tracer.finish_span(span)
except Exception as e:
    tracer.finish_span(span, error=str(e))
    raise
```

### 主流 Tracing 工具

| 工具 | 特点 | 适合 |
|------|------|------|
| **LangSmith** | LangChain/LangGraph 原生集成 | 使用 LangGraph 的项目 |

## 3. Logging 规范

AI 应用的日志要记什么：

```python
import structlog
import json

log = structlog.get_logger()

def log_llm_call(
    messages: list[dict],
    response,
    model: str,
    latency_ms: float,
):
    log.info(
        "llm_call",
        model=model,
        input_tokens=response.usage.prompt_tokens,
        output_tokens=response.usage.completion_tokens,
        latency_ms=latency_ms,
        # 不要直接 log 完整 messages（可能含敏感数据）
        # log 精简版本用于调试
        message_count=len(messages),
        has_tool_calls=bool(
            getattr(response.choices[0].message, "tool_calls", None)
        ),
        finish_reason=response.choices[0].finish_reason,
    )

def log_tool_call(tool_name: str, args: dict, result: dict, duration_ms: float):
    log.info(
        "tool_call",
        tool=tool_name,
        args_keys=list(args.keys()),  # 只 log key，不 log value（可能含敏感数据）
        success=result.get("success", False),
        error=result.get("error"),
        duration_ms=duration_ms,
    )

def log_retrieval(query: str, result_count: int, top_score: float, duration_ms: float):
    log.info(
        "retrieval",
        query_length=len(query),
        result_count=result_count,
        top_score=top_score,
        duration_ms=duration_ms,
    )
```

**敏感数据处理原则**：
- 不 log 用户的原始输入（可能含 PII）
- 不 log 工具参数的值（可能含密码、token）
- 需要 debug 时，专门的 debug 模式才输出完整内容
- 生产日志用 structured logging（JSON 格式），方便搜索

## 4. 常见失败模式分类

AI 应用的失败有独特的分类方式：

### LLM 调用层

| 失败类型 | 症状 | 排查方向 |
|---------|------|---------|
| 幻觉 | 回答包含不存在的信息 | 检查 context 是否有支撑，是否需要 grounding |
| 格式错误 | 输出不符合预期 schema | 检查 prompt 的格式要求，考虑 Pydantic 结构化输出 |
| 拒绝回答 | 模型说"无法帮助" | 检查是否触发 safety filter，调整 prompt 措辞 |
| 上下文丢失 | 多轮后忘记前面说的 | 检查 context 截断策略，考虑摘要压缩 |
| 工具选择错误 | 调用了错误的工具 | 检查 tool description 是否清晰 |

### 工具执行层

| 失败类型 | 症状 | 排查方向 |
|---------|------|---------|
| 参数格式错误 | JSON parse 失败 | 检查 schema 是否描述清楚，参数 description 是否准确 |
| 权限错误 | PERMISSION_DENIED | 检查用户权限和工具权限配置 |
| 外部 API 超时 | Tool result 是 timeout error | 检查超时设置，是否需要更长的 timeout 或 fallback |
| 幂等性问题 | 重试导致重复操作 | 检查工具是否正确标注幂等性 |

### RAG 层

| 失败类型 | 症状 | 排查方向 |
|---------|------|---------|
| 召回不到相关内容 | 回答说"找不到相关信息" | 检查 embedding 模型、chunk 策略、query rewriting |
| 召回了无关内容 | 回答基于错误的文档 | 检查 metadata 过滤、reranking |
| 知识过期 | 回答了旧政策 | 检查 ingestion pipeline 的更新频率 |

## 5. Replay 与离线调试

Replay 是把生产环境的一次真实请求复现出来，用于调试。

```python
@dataclass
class RequestReplayData:
    request_id: str
    timestamp: str
    user_input: str
    messages_at_start: list[dict]  # 请求开始时的完整 messages
    tools_available: list[dict]
    full_trajectory: list[dict]    # 完整执行轨迹
    final_response: str
    metadata: dict                 # 用户 ID、session ID 等

def capture_for_replay(request: RequestReplayData):
    """生产环境采样记录，供后续 replay"""
    if random.random() < REPLAY_SAMPLE_RATE:
        replay_store.save(request)

def replay_request(request_id: str, modified_config: dict = None):
    """
    复现一次生产请求：
    - 使用相同的 messages 和 tools
    - 可以改变模型版本、prompt、工具实现
    - 对比新旧结果
    """
    original = replay_store.load(request_id)
    
    config = {
        "messages": original.messages_at_start,
        "tools": original.tools_available,
        **(modified_config or {}),
    }
    
    new_response = run_agent(**config)
    
    return {
        "original": original.final_response,
        "new": new_response,
        "diff": compute_diff(original.final_response, new_response),
    }
```

**Replay 的价值**：
- 修改了 prompt 或模型版本，可以在历史 case 上验证效果
- 调试特定 case 时，不需要用户重新操作
- 配合 eval harness，可以批量 replay 进行回归测试

## 6. 调试流程

遇到 AI 应用 bug，推荐的排查顺序：

```
1. 确认问题是否可复现
   - 相同输入，多次运行，是否稳定出现？
   - 如果不稳定，是 temperature 导致的随机性还是外部依赖的不稳定？

2. 定位到哪个步骤出问题
   - 查看 trace，找到失败的 span
   - 是 LLM 调用？工具执行？检索？

3. 检查输入（这步出了什么问题，是因为输入有什么异常）
   - 发给 LLM 的 messages 是什么？
   - 工具收到的参数是什么？
   - 检索的 query 是什么？

4. 最小化复现
   - 把问题缩小到最简单的可复现案例
   - 去掉不相关的部分

5. 假设 + 验证
   - 先猜原因（通常是 prompt 不清楚 / schema 有误 / context 质量差）
   - 改一个变量，验证猜想

6. 修复 + 加入回归测试集
   - 修复后，把这个 case 加入回归测试集
   - 确保未来不会重现
```

## 7. 面试高频

**Q：AI 应用的 Observability 和传统软件有什么不同，你会关注哪些额外指标？**

> 传统软件主要看延迟、错误率、吞吐量。AI 应用还需要关注：一是 LLM 输出质量（回答是否有幻觉、格式是否正确）；二是工具调用准确性（模型选了正确的工具吗）；三是 context 质量（RAG 检索到的内容是否相关）；四是 cost（每次调用的 token 用量和费用）；五是多步骤的执行轨迹（agent 执行了多步，哪一步出了问题）。我会用 tracing 工具（如 LangSmith 或 Phoenix）记录完整的调用链，structured logging 记录每次 LLM 调用的 token 用量和 tool 选择。

**Q：生产环境 AI 应用的 bug 怎么 debug？**

> 先看 trace，定位是哪一步出问题。通常分几层排查：一是 LLM 层（输出格式不对、幻觉、拒绝回答），主要检查发给模型的 messages 是否正确，prompt 格式要求是否清晰；二是工具层（参数错误、超时、权限问题），检查 tool schema 和实际执行日志；三是 RAG 层（召回不准、知识过期），检查 embedding 和 chunk 策略。如果有 replay 系统，可以直接复现生产 case 再修改配置对比结果。
